**Key-shape clarity.** The two variants are key-disjoint by the `source` tag (no overlap between pool and user keyspaces is possible by construction), and the matching rules explicitly dispatch on the lease record's `source` field. The `CredentialLease` schema (§4.9 line 1225) documents `source` as a first-class field with two values `pool` and `user`; the `credential.leased` audit event (§4.9.2 line 1738) confirms the full field set for each source (`pool_id`, `credential_id` for pool; `user_id`, `credential_ref` for user). No ambiguity in matching rules remains.

**Rebuild-query completeness.** The UNION covers both revocation sources correctly:

- Pool revocations write `status = 'revoked'` to `CredentialPoolStore` (§4.9 step 1 of Emergency Credential Revocation).
- User revocations write `status = 'revoked'` to `TokenStore` (§4.9 line 1348 step 1 of the user revoke handler).

Both stores are queried at gateway startup, scoped to active leases. There is no third credential store in the system (only `CredentialPoolStore` and `TokenStore` hold long-lived credential material), so the UNION is complete against the v1 design.

**Conclusion on CRD-015:** Fixed. The normative contract is now internally consistent across §4.9 user revoke handler, §4.9 pool revoke handler, the Credential deny list section, the LLM Reverse Proxy description, and the §15 admin-API surface.

## 3. New findings (iter6)

### CRD-020. `lenny_credential_revoked_with_active_leases` gauge cannot reflect user-scoped revocation propagation failures; `CredentialCompromised` alert is blind to user-credential deny-list lag [High]

**Status: Fixed** — Added parallel gauge `lenny_user_credential_revoked_with_active_leases{tenant_id, provider}` to `spec/16_observability.md` §16.1 and `docs/reference/metrics.md`; extended `CredentialCompromised` alert in §16.5 with a second clause covering user revocations; updated §4.9 emergency revocation runbook and `docs/runbooks/credential-revocation.md` to include the user path.

**Section:** `spec/16_observability.md` §16.1 line 58 (`lenny_credential_revoked_with_active_leases` gauge), §16.5 line 409 (`CredentialCompromised` alert); `spec/04_system-components.md` §4.9 line 1658–1661 (deny-list tagged union), line 1348 (user revoke handler), line 1674 (emergency revocation runbook — "confirm the `CredentialCompromised` alert clears within 60s")

The iter5 CRD-015 fix introduced the user-shaped deny-list entry `{source: "user", tenantId, credentialRef}` and made user-credential proxy-mode revocation a first-class path. The companion metric and alert that the pool path relies on to detect revocation-propagation failures, however, were not updated to cover the user shape:

1. **Metric cardinality.** `lenny_credential_revoked_with_active_leases` is defined at §16.1 line 58 as "gauge labeled by `pool`, `provider` — number of credentials currently in `revoked` state that still have one or more active leases outstanding against them". User-scoped credentials have no `pool` — they are stored in `TokenStore` keyed by `(tenant_id, user_id, provider, credential_ref)`, not in `CredentialPoolStore`. A revoked user credential with outstanding active leases therefore cannot be counted by this gauge. Any attempt to label it with a sentinel `pool="user"` or `pool=""` collides with real pool names for the same provider, and any attempt to synthesize a pool label from `(tenant_id, credential_ref)` would trip the high-cardinality label rule at §16.1.1.

2. **Alert blindness.** `CredentialCompromised` (§16.5 line 409) fires on `max by (pool, provider) (lenny_credential_revoked_with_active_leases) > 0 sustained for more than 30s`. When the failure is a user-credential revocation that failed to propagate (Redis pub/sub outage, gateway replica missing the notification, pub/sub message lost before the replica's startup-rebuild ran), this alert cannot fire because no time series carries a non-zero value on either the `pool` or the `provider` dimension for user-scoped leases. The operator signal for "a revoked user credential is still being accepted on some replica" is therefore absent.

3. **Runbook inconsistency.** The emergency revocation runbook at §4.9 line 1674 step 4 instructs operators to "confirm the `CredentialCompromised` alert clears within 60s (indicating revocation propagation succeeded)." This instruction is only valid for pool revocations. A user-credential revocation has no equivalent confirmation step documented, and the existing alert cannot provide one. The `TestUserCredentialRevocationDenyListProxy` integration test (line 1672) gives unit-level confidence but gives the production on-call no live signal.

4. **Symmetry gap.** CRD-015 established that user-credential proxy-mode revocation has the same severity as pool-credential revocation — the "immediate invalidation" contract is identical. The observability stack should follow the same symmetry; otherwise the iter5 fix leaves a structural blind spot on the very path it was intended to close. The `CredentialCompromised` alert is the tier-1 production signal for revocation-propagation failure; leaving it incapable of firing on the user path reintroduces the exploitation window CRD-015 nominally closed, only behind a lag-detection gap rather than behind a match-rule gap.

Severity rationale — High (matching iter5 CRD-015 High anchor). CRD-015 was High because the deny-list keying gap meant a revoked user credential could silently keep being used in proxy mode on multi-tenant deployments. CRD-020 is the same failure class expressed one layer later: the rule is now correct, but the operator has no production-grade signal if the rule's propagation (pub/sub or startup rebuild) silently fails on the user path. The exploitation window is the same window CRD-015 was meant to bound. The finding is High, not Critical, because the rebuild-query completeness means the window is bounded to the pub/sub-to-rebuild gap on the affected replica rather than unbounded, and because the `TestUserCredentialRevocationDenyListProxy` test reduces the probability the gap is introduced accidentally; but a deployed regression is blind to production monitoring until a user reports the incident.

**Recommendation:** Either (a) generalize `lenny_credential_revoked_with_active_leases` to carry a `source` label (`pool` or `user`) and leave the `pool` label empty for user-source rows (Prometheus cardinality budget unaffected since `source` is a 2-value dimension), or (b) add a parallel gauge `lenny_user_credential_revoked_with_active_leases` labeled by `provider` only (user revocations are bounded per-tenant-per-provider at 1, so the cardinality remains `tenants × providers` — within budget). Update the `CredentialCompromised` alert expression to include the new source dimension: `max by (source, pool, provider) (lenny_credential_revoked_with_active_leases) > 0` or `max by (provider) (lenny_user_credential_revoked_with_active_leases) > 0` in a second clause. Update §4.9 line 1674's emergency-revocation runbook to add a user-credential step, or add a new line 1348 inline runbook fragment that references the alert confirmation step for the user path. Update `docs/runbooks/credential-revocation.md` (currently pool-only) to include a "user-credential revocation" subsection describing the alert-confirmation procedure.

### CRD-021. `docs/runbooks/credential-revocation.md` covers only pool-credential revocation; user-credential revoke path (`POST /v1/credentials/{credential_ref}/revoke`) has no operator runbook [Medium]

**Status: Fixed** — Added "User-scoped credentials" section (steps U1–U5) to `docs/runbooks/credential-revocation.md` covering identification, revocation via `POST /v1/credentials/{credential_ref}/revoke`, provider-side rotation, re-registration, and audit verification. Cross-references the new `lenny_user_credential_revoked_with_active_leases` gauge and `TestUserCredentialRevocationDenyListProxy` test.

**Section:** `docs/runbooks/credential-revocation.md`; `spec/04_system-components.md` §4.9 line 1348 user revoke endpoint, line 1674 emergency revocation runbook; `spec/17_deployment-topology.md` §17.7 line 766 (runbook pointer)

Iter5 introduced the user-credential revoke path as a first-class invalidation channel equivalent in severity to pool revocation. The spec references `docs/runbooks/credential-revocation.md` for the emergency revocation runbook (§17.7 line 766), but that runbook covers only pool-credential revocation: every `access:` annotation in it references `/v1/admin/credential-pools/…` endpoints; there is no mention of `/v1/credentials/{credential_ref}/revoke`, no `lenny-ctl` user-revocation command (there isn't one — §24 `lenny-ctl` is operator-focused and doesn't cover per-user self-service actions), and no procedure for an operator helping a user revoke their credential during an incident response (e.g., upon credential compromise reported by the user via support channel, the operator may need to revoke on the user's behalf if the user cannot authenticate).

Per the `feedback_docs_sync_after_spec_changes.md` discipline, this is the kind of spec-to-docs divergence that should be reconciled before declaring convergence on the fix that introduced the divergence. The iter5 fix for CRD-015 landed in spec/ without a corresponding docs/ update, and the prior iterations' omission persists unchanged.

Severity rationale — Medium (not High). Operators have workarounds: the spec text at §4.9 line 1348 is self-contained and an on-call familiar with the spec can execute the revocation correctly from first principles. The runbook gap is a paper-trail and operator-guidance defect, not a correctness defect. It is not Low because incident-response runbooks are the primary operator interface in production — a missing runbook during an incident is a real response-time cost, especially when CRD-020 also leaves the alert-signal path incomplete. It is not High because there is no correctness break: the platform does the right thing when the revocation is executed correctly; the gap is in operator mechanics.

**Recommendation:** Extend `docs/runbooks/credential-revocation.md` with a "User-credential revocation" section at top level or as a parallel runbook. Include: the trigger (user-reports credential compromise, operator-detects user-credential suspicious activity), the API call (`POST /v1/credentials/{credential_ref}/revoke` — acting as the user via bearer token delegation, or direct DB mutation for the offline case), the verification procedure (once CRD-020 is fixed, reference the alert expression; in the interim, document `SELECT count(*) FROM tokens WHERE status = 'revoked' AND tenant_id = … AND credential_ref = …` paired with a grep over gateway logs for `CREDENTIAL_REVOKED`), and the follow-up steps (user re-registration flow, replacement credential provisioning). Cross-reference the spec §4.9 line 1348 and the `TestUserCredentialRevocationDenyListProxy` assertion as the contract the runbook is verifying.

### CRD-022. `TokenStore` schema for user credentials is only implicitly defined; rebuild query correctness depends on columns (`status`, lease-index linkage) that are not documented as part of the stated schema [Low]

**Section:** `spec/04_system-components.md` §4.9 line 1335 ("User-scoped credential storage" callout), line 1348 (revoke handler: "marks the credential as `revoked` in `TokenStore`"), line 1668 (startup rebuild query: "FROM `TokenStore`: all `(tenantId, credentialRef)` tuples WHERE `status = 'revoked'` AND EXISTS an active lease backed by that `credential_ref`"); `spec/12_storage-architecture.md` §12.2 line 13 (TokenStore role definition)

The iter5 CRD-015 fix extended `TokenStore` with two requirements it doesn't explicitly document as schema changes:

1. **`status` column with `revoked` value.** Line 1348 writes `status = 'revoked'` and line 1668 reads it. Neither the "User-scoped credential storage" callout at line 1335 nor the `TokenStore` role definition at §12.2 line 13 documents a `status` column or its allowed values. A reader who wants to derive the Postgres schema for user credentials from the spec finds only the envelope-encryption description, RLS policy mention, and the DEK-per-tenant / KMS-wrapped-KEK structure.

2. **User-credential lease index.** Line 1668 references "an active lease backed by that `credential_ref`" as a predicate for the rebuild query. The pool-side analogue at line 1643 explicitly says "via `CredentialPoolStore` lease index". For user credentials, the index location is never stated — is it in `TokenStore`, in a new `user_credential_leases` table, or inferred by joining `CredentialPoolStore.leases` on a `credential_ref` column? The pool-side lease index supports the `CredentialPoolStore` startup rebuild's EXISTS subquery; the user-side analogue is required but undocumented.

3. **`active_leases_terminated` counting.** The user revoke handler at line 1348 emits `credential.user_revoked` with `active_leases_terminated` (line 1737). The count must be derived from the same lease index that the rebuild query uses. Without documenting where this index lives, two independent spec readers (implementation vs. audit tooling) may make divergent assumptions.

Severity rationale — Low (matching iter5 CRD-016/017/019 Low anchor). This is a documentation-completeness defect, not a correctness defect. The behavior at §4.9 line 1348 and the rebuild query at line 1668 are internally consistent, and an implementer following the spec end-to-end will land on compatible column definitions because both references name `status` and both name `credential_ref` / `tenantId`. The defect is that a §12-only reader doesn't learn about `status`, and a §4.9-only reader doesn't learn about the storage location. The downstream risk is that a future refactor (or a separate Token Service binary implementer) defines a schema that doesn't match the deny-list rebuild expectation. The risk is bounded by the integration test `TestUserCredentialRevocationDenyListProxy` at line 1672, which will fail if the column or index doesn't match. Keeping at Low per the iter5 severity calibration.

**Recommendation:** Add a "TokenStore schema — user credentials" subsection at §4.9 (next to the "User-scoped credential storage" callout at line 1335) listing the columns: `tenant_id` (uuid), `user_id` (uuid), `provider` (text), `credential_ref` (text, primary surrogate), `encrypted_material` (bytea), `key_version` (int, for KMS rotation), `status` (text enum: `active` \| `revoked`), `revoked_at` (timestamptz NULL), `revoked_reason` (text NULL), `created_at`, `last_used_at`. Document the user-credential lease index location (either an explicit `user_credential_leases` table keyed by `(tenant_id, credential_ref, lease_id)` with the same lifecycle as `CredentialPoolStore.leases`, or a union view joining both). Document the index in §12.2 alongside the existing `CredentialPoolStore` and `TokenStore` role descriptions.

## 4. Convergence assessment

**New findings:** 3 (CRD-020 High, CRD-021 Medium, CRD-022 Low).

**Carry-forwards open:** 4 (CRD-016 Low, CRD-017 Low, CRD-018 Low, CRD-019 Low) — none escalated; all remain at their iter5 severity.

**Not converged.** CRD-020 (High) is a material gap in the iter5 fix for CRD-015: the deny-list is correct on the matching side but has no matching observability signal on the user path, leaving production monitoring blind to the exact failure class CRD-015 was meant to close. This blocks convergence by the same argument that made CRD-015 a High in iter5.

CRD-021 (Medium) is a docs-sync gap that the iter5 fix should have reconciled per the `feedback_docs_sync_after_spec_changes.md` discipline; it is a second-order effect of the iter5 change landing in spec/ without docs/ being updated.

CRD-022 (Low) and the four iter5 Low carry-forwards (CRD-016, -017, -018, -019) are polish items that do not block convergence on their own but should be cleared in a subsequent fix pass for completeness. Per iter5 severity calibration, none of them warrant escalation in iter6.

**Status:** Not converged — resolve CRD-020 (High) and CRD-021 (Medium) before declaring. The four Low carry-forwards + CRD-022 may be accepted at reviewer discretion as polish items, but the iter5 calibration feedback notes they should not be held open indefinitely across iterations; iter6's fix pass is a reasonable place to clear them together.
