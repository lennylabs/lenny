## New findings (iter7)

### STO-022. Pod-local cumulative fail-open timer file is not durable across node-level pod rescheduling [Low]

**Section:** §12.4 Redis HA and failure modes — "Cumulative fail-open timer (security control)".

**Description:** The cumulative-fail-open security control (introduced to close the original CrashLoopBackOff bypass) persists each replica's sliding-window counter to `/run/lenny/failopen-cumulative.json` and reloads it on restart. The spec explicitly documents that "if the file is missing or corrupted (e.g., fresh pod on a new node), the timer resets to zero — this is the cold-start case and is acceptable because the replica has no prior fail-open history on this pod." However, the file path `/run/` is a tmpfs by Kubernetes convention, and gateway pods do not declare a PVC for it; therefore any pod rescheduling event — eviction, node drain, node failure, HPA scale-down/up — produces a "fresh pod" and resets the cumulative counter for that replica, even when the cluster has been in fail-open continuously. During a sustained Redis outage that also triggers gateway rescheduling (e.g., a noisy-neighbor node-drain during the outage, or a pod OOM), an individual replica's clock of accumulated fail-open seconds resets; a deployment of N replicas that each experience one reschedule during the outage can accumulate `N * quotaFailOpenCumulativeMaxSeconds` of fail-open time cluster-wide before any replica transitions to fail-closed. The defense is not as strong as the text claims; "fresh pod on a new node" is the common case during an infrastructure event, not the rare case.

**Recommendation:** Either (a) call out the reschedule-reset attack surface explicitly and note that `quotaFailOpenCumulativeMaxSeconds` should be set conservatively for deployments with frequent rescheduling (HPA-heavy, spot-node fleets), or (b) persist the cumulative timer through a replica-stable side-channel — the most natural fit is the `ops_replica_state` Postgres row keyed by `service_instance_id`, written on every fail-open state transition, since Postgres is by assumption reachable during a Redis-only outage. Option (b) preserves the security intent of the control in the presence of reschedules; option (a) is the documentation-only path if the reschedule-reset behavior is deemed acceptable.

### STO-023. `audit_redaction_receipts` exemption from `DeleteByUser` / `DeleteByTenant` is documented in the schema paragraph but not in the erasure-scope table [Low]

**Section:** §12.8 Compliance interfaces — "Storage backends in erasure scope" (the 19-row table, lines 772-790) vs. the `RedactionReceipt` schema paragraph (line 827).

**Description:** The `audit_redaction_receipts` table, added to back the OCSF dead-letter PII redaction receipt chain, is described in detail at §12.8 line 812-827. Its exemption is asserted in a single clause at line 827 ("the table is exempt from `DeleteByUser` and `DeleteByTenant` under the same rationale as `gdpr.*` receipts"). The canonical erasure-scope table immediately above (lines 772-790), which enumerates all 14 Lenny-managed stores in erasure scope with an explicit "Data Erased" column, does **not** list `audit_redaction_receipts` at all — neither as an in-scope store with "(exempt)" notation nor as a separately called-out out-of-scope store. The `EventStore` (audit) row carries a gdpr.* exemption note, but the receipts table is physically separate from `audit_log`. An operator reading the scope table to build an RoPA (§12.8 "DSARs and exports" explicitly requires RoPAs to enumerate Lenny stores from this table) will miss `audit_redaction_receipts` and will either (a) omit it from the RoPA inventory, failing the §12.8 completeness requirement, or (b) include it as in-scope and attempt to delete it during tenant off-boarding, violating the schema's INSERT-only grant and losing the receipts that back chain-discontinuity verifiability.

**Recommendation:** Add an explicit row to the erasure-scope table in §12.8 (or a small "Exempt from erasure" table immediately following it) listing `audit_redaction_receipts` with rationale "Holds no personal data (hashes and identifiers only); exempt for the same reason as `gdpr.*` receipts; INSERT-only schema; retained with referenced receipt events under `audit.gdprRetentionDays`." The one-line table entry avoids the RoPA-completeness gap and makes the exemption discoverable from the same place where operators look up the rest of the scope.

### STO-024. `erasure_salt` immediate deletion is within a single DB transaction but the verification step crosses a commit boundary [Low]

**Section:** §12.8 Compliance interfaces — "`erasure_salt` key management" → "Immediate deletion after pseudonymization" (sub-bullets 1-5).

**Description:** The deletion sequence specifies: (1) pseudonymize billing rows using the salt, within a single transaction; (2) in the same transaction, set the salt column to `NULL` and delete the KMS-wrapped ciphertext; (3) **after the transaction commits**, use an in-memory copy of the salt for a re-derivation check. If the erasure worker crashes between step 2 commit and step 3 completion — a window bounded only by network-and-process latency but not infinitesimal — the next crash recovery path described in §12.8 "Erasure job phase field (crash recovery)" states that jobs in phase `verifying` "re-run the salt-deletion verification check." But the `verifying`-phase recovery path does not describe how the resumed worker re-obtains the in-memory salt copy (step 3 requires "an in-memory copy of the salt held for this purpose only"): the salt has been set to `NULL` in Postgres and the KMS ciphertext destroyed, so a freshly-started worker cannot reconstruct the copy. Either the verification is silently skipped on resume (a weaker property than specified, because no re-derivation check is performed), or the verification hard-fails because the salt is unrecoverable — in which case a crash between steps 2 and 3 always produces a `verification_failed` state requiring manual intervention, even though the underlying erasure is correct. Neither outcome matches the spec's stated behavior.

**Recommendation:** Fold step 3's re-derivation check into step 1's transaction by performing the re-derivation **before** the salt is nulled (using the in-memory copy that was read into the transaction to perform the pseudonymization itself), and fail the transaction on verification failure so that a crash either produces no commit (re-run from scratch) or a successful commit that has already verified. The zeroing of the in-memory copy in step 5 still runs after commit but is no longer on the correctness path. Alternatively, explicitly document how the `verifying`-phase resume behaves without the in-memory salt — either "resume-verifying always marks the job `verification_failed` for manual review" or "resume-verifying skips the re-derivation check and records `verification_skipped_by_resume` in the receipt."

### STO-025. Tombstone hard-prune (24h default) can race with a legal-hold application window on soft-deleted rows [Low]

**Section:** §12.5 Artifact Store — "Tombstone hard-prune sweep" vs. §12.8 "Legal hold and checkpoint rotation".

**Description:** Soft-deleted artifact rows (`deleted_at IS NOT NULL`) are retained for `gc.tombstoneRetentionSeconds` (default 24 hours) before physical removal by the hard-prune pass. During that 24-hour window the row still exists in Postgres, but the MinIO object has already been deleted at soft-delete time (per §12.5 GC concurrency model rule 3). If a legal-hold request is issued against a session whose checkpoint was soft-deleted within the last 24 hours — e.g., a routine rotation deleted the third-oldest checkpoint for a session minutes before a regulator issues a preservation order — the `legal_hold.set` path (§12.8) updates `sessions.legal_hold = true` and `artifacts.legal_hold = true`, but the latter update is a no-op on the soft-deleted row's `legal_hold` column (the hard-prune predicate is `deleted_at IS NOT NULL AND deleted_at < now() - retention`, not joined against `legal_hold`). The row and its already-deleted MinIO object are not resurrectable: the MinIO object has been removed, `NoncurrentVersionExpiration: Days: 1` means bucket-versioning restoration is bounded to a 24-hour window that overlaps the Postgres tombstone retention window, and the GC path does not have a "recall from noncurrent version" mechanism. A preservation order issued more than 24 hours after the MinIO delete cannot be honored for that specific object; a preservation order issued within 24 hours could in principle be honored by suspending the next hard-prune cycle and recovering the noncurrent version, but the spec does not describe this recovery path.

**Recommendation:** Either (a) document explicitly that legal-hold orders issued after a routine rotation cannot resurrect soft-deleted checkpoint objects and that tenants with anticipated legal-hold activity should set a longer `gc.tombstoneRetentionSeconds` in line with the expected hold-issuance lag (and that MinIO `NoncurrentVersionExpiration` must then be raised in lockstep), or (b) add a "legal-hold recall" path that, on legal-hold set, scans the audit ledger for recent soft-deletes bound to the target session and either restores the MinIO noncurrent version (if within the bucket-versioning window) or emits a `legal_hold.recall_impossible` audit event listing objects that cannot be preserved. Option (a) is the cheaper documentation fix; option (b) is the cheaper rock-solid preservation guarantee.

### STO-026. T4 KMS continuous-probe rate-limit can allow `2 * t4KmsProbeInterval` alert to fire when the fleet is legitimately sweeping [Low]

**Section:** §12.5 Artifact Store — "Continuous probe (STO-021)" bullet 4 vs. §16.5 `T4KmsKeyUnusable` alert.

**Description:** The probe documentation states that the sweep is rate-limited via `storage.t4KmsProbeRateLimit` (default 10/s) and "under rate-limit pressure the sweep completes asynchronously (it may take longer than `t4KmsProbeInterval` under very high tenant counts, but the rate ceiling is preferred over backend overload)." The alert rule fires when `now() - lenny_t4_kms_probe_last_success_timestamp{tenant_id=$t} > 2 * storage.t4KmsProbeInterval`. At the 10/s default rate and a T4 tenant population of 10,000, a full sweep takes ≥ 1000 seconds ≈ 16.7 minutes even if every probe succeeds; at the 5-minute (300s) default interval, a sweep that takes one full interval of sweep time plus one full interval of backend latency would cross the `2 * 300s = 600s` alert threshold while the probe is simply catching up. An operator with a large T4 fleet will receive spurious `T4KmsKeyUnusable` alerts during the initial bring-up sweep or after a rate-limit-back-off recovery, none of which reflect an actual key-unusable condition.

**Recommendation:** Either (a) tie the alert threshold explicitly to the sweep-time budget — e.g., `2 * max(t4KmsProbeInterval, ceil(t4_tenants / t4KmsProbeRateLimit))` — and describe this coupling in the alert definition, or (b) document that operators with large T4 fleets MUST set `t4KmsProbeRateLimit` high enough that `t4_tenants / rate_limit < t4KmsProbeInterval / 2`, and that the alert PromQL should be cloned with a larger multiplier for those deployments. Option (a) is the self-adjusting fix; option (b) is the operator-tuning documentation fix. The current fixed `2 * interval` threshold is correct only when the sweep completes within one interval — a condition the spec does not require operators to maintain.

### STO-027. `LEGAL_HOLD_ESCROW_REGION_UNRESOLVABLE` fail-closed path leaves Phase 3.5 mid-flight with partial escrow writes [Low]

**Section:** §12.8 Compliance interfaces — `--acknowledge-hold-override` admin action (tenant scope), sub-step 2 ("Resolve the target region and re-encrypt under the region-scoped `legal_hold_escrow_kek`").

**Description:** The region-residency gate at sub-step 2 fails closed if the resolved region has no matching `storage.regions.<region>.legalHoldEscrow` entry or the configured escrow KMS is unreachable — the controller aborts with `LEGAL_HOLD_ESCROW_REGION_UNRESOLVABLE` and "leave[s] the tenant in Phase 3.5 `deleting` state pending operator remediation." However, sub-step 2 enumerates held resources and processes them one at a time (decrypt under tenant KEK, re-wrap under escrow KEK, write ciphertext to escrow bucket in sub-step 3, write ledger row in sub-step 4). If the first several resources succeed and a later one hits the residency failure (e.g., KMS network partition mid-sweep), the escrow bucket now contains partial ciphertext with partial ledger rows, and Phase 4's `DeleteByTenant` skip-logic — which depends on the `legal_hold_escrow: true` marker recorded by sub-step 4 — has only tagged the successfully-migrated resources. The idempotency note ("Phase 3.5 is idempotent: a re-entered Phase 3.5 walks the same ledger and skips holds already migrated") addresses the retry case but not the failure-mid-sweep case: what happens to the partially-written escrow records and ledger rows if the operator gives up, issues `POST /v1/admin/legal-hold` with `hold: false` to release the residual holds, and retries the tenant delete? The escrow bucket now contains orphan ciphertext with ledger rows that reference a ledger entry whose held resource has been released — the escrow GC path (§12.8 sub-step 3 "an escrow GC job deletes it and the resulting `legal_hold.escrow_released` audit event is emitted") may or may not pick them up depending on whether the hold release event pairs correctly with the escrow row.

**Recommendation:** Either (a) wrap sub-steps 1-4 in a per-resource transaction: the escrow bucket write (PUT) and the ledger-row INSERT commit as a unit, with a pre-escrow `legal_hold.escrow_pending` ledger row that is either upgraded to `legal_hold.escrowed` on success or swept by a `escrow_pending` timeout path on failure, or (b) add explicit text describing the partial-sweep-then-release failure mode and confirming the escrow GC job reconciles orphan ciphertext against released holds. Option (a) is the stronger invariant; option (b) is a documentation-only path if the escrow GC is in fact expected to reconcile orphans.

### STO-028. Redis Cluster migration pre-plan's "per-tenant feature flag" route creates a counters-divergence recovery burden the spec does not quantify [Info]

**Section:** §12.4 Redis HA and failure modes — "Redis Cluster migration pre-plan (Tier 2→3 transition)" step 5 (Rollback).

**Description:** Step 5 of the migration plan specifies that "The per-tenant feature flag allows individual rollback to Sentinel if correctness issues are observed. Any quota counters that diverged between the two stores during the migration window must be reconciled by re-reading from Postgres (same procedure as fail-open reconciliation above)." The fail-open reconciliation procedure (§12.4, "Quota counter reconciliation after fail-open") is designed for recovery from a **total** Redis outage, where the in-memory counter on each replica is the authoritative fail-open increment source. In the migration-rollback case, both Sentinel and Cluster are live simultaneously for an arbitrary fraction of tenants; a tenant that was rolled back after several hours in Cluster mode has accumulated `INCR` traffic on Cluster while the replica's in-memory counter was also being flushed to Cluster (not Sentinel). The two-source MAX rule (Postgres checkpoint vs. in-memory counter) does not apply cleanly: the in-memory counter reflects post-Cluster state, the Sentinel counter is stale from before the flag flip, and Postgres has been checkpointed from Cluster. The spec's "same procedure as fail-open reconciliation above" does not produce the right outcome in this asymmetric case, and the operator is left to work out a three-way reconciliation manually.

**Recommendation:** Either (a) dedicate a short subsection to migration-rollback reconciliation that explicitly handles the three counter sources (Sentinel stale, Cluster live, Postgres checkpointed from Cluster) and prescribes `MAX(Cluster current, Postgres checkpoint)` as the authoritative source to write back to Sentinel, or (b) remove the implied "same as fail-open" claim and simply state "Rollback reconciliation is operator-driven; see the Redis recovery runbook in §17.7." Option (a) preserves the migration pre-plan's completeness; option (b) removes the false equivalence without adding a new burden.
