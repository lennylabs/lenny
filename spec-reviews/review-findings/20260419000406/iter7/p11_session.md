### SES-019 (carry-forward from iter4/iter5/iter6) — `POST /v1/sessions/{id}/start` precondition/result table still omits `resume_pending`
**Severity:** Low
**Location:** `spec/15_external-api-surface.md` §15.1 line 618
**Finding:** The `POST /v1/sessions/{id}/start` row still lists the resulting transition as `starting → running` only. The session state machine in `spec/07_session-lifecycle.md` §7.2 at lines 191–195 (`starting → resume_pending` during agent-runtime launch with retries remaining) and `spec/06_warm-pod-model.md` §6.2 both document `starting → resume_pending` as a valid edge that can materialize during the `/start` call (post-attached visibility of a mid-launch pod crash). The §15.1 table remains inconsistent with both authoritative state tables.
**Impact:** Clients consulting only the external API reference will not know that `POST /start` can legitimately leave the session in `resume_pending` awaiting replacement-pod allocation. Incorrect client state machines will treat any non-`running` state after `/start` as an error. No runtime/correctness impact — the underlying state machine is unambiguous — but the API reference is out of sync with the authoritative §7.2/§6.2 tables.
**Recommendation:** Update the §15.1 row for `POST /v1/sessions/{id}/start` resulting-transition cell to `starting → running | starting → resume_pending`, with a note (in the same row or an adjacent paragraph) that the latter fires when the runtime launch encounters a retryable error with retries still remaining, and that the client should listen for `status_change(state: "running")` or `session.resumed` on the event stream rather than assuming the synchronous response covers all outcomes.
**Status vs iter6:** Carry-forward — issue verified still open in iter7 at the same location. The iter6 fix commit (`8604ce9`) did not touch `spec/15_external-api-surface.md` line 618; edits in iter6 to §15 were scoped to the `GIT_CLONE_REF_*` error catalog additions around line 1063 and an iter5 audit-reachability row set around line 647 — the precondition table at line 618 is unchanged from iter5.

### SES-020 (carry-forward from iter4/iter5/iter6) — SSE reconnect/replay has no rate limit or per-reconnect replay cap
**Severity:** Low
**Location:** `spec/07_session-lifecycle.md` §7.2 lines 349–363 (Reconnect / back-pressure semantics); `spec/10_gateway-internals.md` §10.4 line 389 (event replay buffer); `spec/11_policy-and-controls.md`
**Finding:** The replay window is defined as `max(periodicCheckpointIntervalSeconds × 2, 1200s)` and the `OutboundChannel` back-pressure policy bounds a single slow client's memory footprint at the per-connection level, but there is still no (a) per-session or per-principal reconnect rate limit (a client may reconnect with `Last-Event-ID` or `resumeFromSeq` arbitrarily often) and (b) no per-reconnect replay volume cap (a client reconnecting near the 1200s horizon on every attempt will replay the full window each time). A grep for `sessionReconnectRateLimit`, `sessionEventReplayMaxBytesPerReconnect`, and `RECONNECT_RATE_LIMITED` across the entire `spec/` tree in iter7 returns zero matches. Neither `spec/11_policy-and-controls.md` nor `spec/10_gateway-internals.md` §10.4 introduces a rate limit keyed by `(tenant_id, session_id)` or `(principal, session_id)` for reconnect attempts, nor a byte cap beyond which the gateway short-circuits to a `gap_detected` frame and forces the client to re-snapshot via a synthesized `status_change`.
**Impact:** Amplification-style resource-exhaustion vector against the gateway's event replay subsystem (CPU + egress). A malicious or buggy client can impose O(N) replay cost per reconnect, and a script that reconnects every second over a 1200s window amplifies gateway egress by ~1200× relative to an in-protocol client. Not a correctness defect — the data emitted is still correct — but a missing reliability/defense-in-depth control. Consistent with iter3 SES-012 / iter4 SES-020 / iter5-iter6 SES-020 framing.
**Recommendation:** Add a per-session reconnect rate limit (token bucket with configurable burst/refill, e.g., `gateway.sessionReconnectRateLimit.perSessionPerMinute` default 12 and `.burstCapacity` default 6) in `spec/11_policy-and-controls.md` alongside the existing `sessionEventReplayBufferDepth` policy, with a `RECONNECT_RATE_LIMITED` error code (HTTP 429 on the streaming attach). Additionally specify a per-reconnect replay byte cap (e.g., `gateway.sessionEventReplayMaxBytesPerReconnect` default 8 MiB) beyond which the gateway emits a single `gap_detected` frame plus a synthesized `status_change` from durable Postgres state and requires the client to re-snapshot rather than continuing to drain the full historical replay.
**Status vs iter6:** Carry-forward — issue verified still open in iter7. The iter6 fix commit (`8604ce9`) did not add any reconnect-rate-limit or per-reconnect byte-cap policy to `spec/11_policy-and-controls.md`; §10.4 line 389's event replay buffer prose is unchanged from iter5.

### SES-021 (carry-forward from iter4/iter5/iter6) — `awaiting_client_action → expired` trigger is ambiguous across three timers
**Severity:** Low
**Location:** `spec/07_session-lifecycle.md` §7.2 line 202 (state-machine edge); §7.3 lines 421–428 (`awaiting_client_action` semantics)
**Finding:** The edge `awaiting_client_action → expired (lease/budget/deadline exhausted while awaiting client action)` still collapses three distinct timers into a single free-text trigger: (a) `maxAwaitingClientActionSeconds` (900s default, state-specific timer from §11), (b) session budget exhaustion (delegation tree budget counters), and (c) absolute session TTL / deadline (`retryPolicy.maxSessionAgeSeconds`, default 7200s). The spec does not disambiguate which timer fires this specific edge, nor record a distinct `terminal_reason` code per trigger. Related edges elsewhere in §7.2 also collapse multiple triggers into a single `expired` terminus without reason disambiguation (`suspended → expired`, `running → expired`, `input_required → expired`). No canonical `terminal_reason` enum is defined in §7 or §16; a grep for `terminal_reason` across the spec tree returns zero matches.
**Impact:** Operators cannot distinguish "client failed to respond in time" from "budget exceeded while waiting" from "absolute deadline hit" in audit/observability output. Incident triage, billing reconciliation, and automated client resume-vs-fork decisioning all suffer — a CI pipeline polling `GET /v1/sessions/{id}` and receiving `state: "expired"` cannot programmatically distinguish the three cases. Billing teams cannot reconcile pod-hours consumed during a long `awaiting_client_action` wait vs. a short budget-exhaustion terminus. Not a correctness bug; a documentation/policy completeness gap.
**Recommendation:** Introduce a canonical `terminal_reason` enum (sibling to `failureClass` at §7.1 line 100–108) covering at minimum `awaiting_client_action_timeout`, `budget_exhausted`, `session_deadline_exceeded`, `perChildMaxAge_expired`, and `delegation_lease_expired`. Split the §7.2 transition row for `awaiting_client_action → expired` into three rows (or enumerate three sub-triggers) each carrying a distinct `terminal_reason`. Reference the relevant policy knob from §11 (`maxAwaitingClientActionSeconds`, session budget counters, `maxSessionAgeSeconds`) for each. Surface the field on `GET /v1/sessions/{id}` and on the terminal event payload (`status_change(state: "expired", reason: "...")`). Apply the same disambiguation to `suspended → expired`, `running → expired`, and `input_required → expired` so the enum is exhaustive.
**Status vs iter6:** Carry-forward — issue verified still open in iter7 at §7.2 line 202; the `awaiting_client_action` semantics paragraph at §7.3 was unchanged in iter6. No `terminal_reason` enum exists in `spec/07`, `spec/11`, `spec/15`, or `spec/16`.

## 3. New findings in iter7

### SES-022 — §7.1 step flow and atomicity paragraph do not enumerate the pre-claim `git ls-remote` resolution step or its distinct rejection codes
**Severity:** Low
**Location:** `spec/07_session-lifecycle.md` §7.1 lines 5–28 (numbered flow steps 1–8 and Atomicity paragraph)
**Finding:** The §7.1 numbered flow lists steps 1–8 covering auth/policy (2), pre-claim credential availability (3), experiment assignment (3a), pool select / pod claim (4), metadata persist (5), per-provider credential lease (6), AssignCredentials RPC (7), and response (8). The `gitClone.ref` resolution step (`git ls-remote` per §14 line 104), which the iter6 fix introduced as a distinct pre-claim admission step with its own retryable/non-retryable error split (`GIT_CLONE_REF_RESOLVE_TRANSIENT` / `GIT_CLONE_REF_UNRESOLVABLE`), is not enumerated in the numbered flow. The "Atomicity of session creation (steps 2–8)" paragraph at line 28 specifies rollback semantics for failures at step 4 (pool exhaustion), step 6 (credential assignment), and step 7 (pod communication), all collapsed to `503 SESSION_CREATION_FAILED`. It does not mention the `GIT_CLONE_REF_*` codes as legitimate non-`SESSION_CREATION_FAILED` rejection paths, nor does it clarify that these codes surface as direct `CreateSessionRequest` validation failures (before any pool/pod/credential allocation occurs) and therefore do not engage the rollback-without-persist path described in the Atomicity paragraph. A reader comparing §7.1 alone to §14/§15 would incorrectly conclude that all creation-path rejections map to `SESSION_CREATION_FAILED`.
**Impact:** Documentation gap. Clients writing `POST /v1/sessions` handlers guided by §7.1 alone may not handle `GIT_CLONE_REF_UNRESOLVABLE` (422 PERMANENT) and `GIT_CLONE_REF_RESOLVE_TRANSIENT` (503 TRANSIENT) distinctly from `SESSION_CREATION_FAILED` — conflating them breaks retry correctness (422 PERMANENT is not retryable without plan-body changes; the generic `SESSION_CREATION_FAILED` contract prescribes retry with `Retry-After`). Not a correctness defect in the runtime — the gateway's admission path is unambiguous — but the client-facing §7.1 contract is incomplete. Category: documentation completeness; consistent with iter5/iter6 SES-019 framing (external-reference-vs-authoritative-inconsistency, Low).
**Recommendation:** (a) Insert a new step (e.g., step 3b between 3a and 4) in the §7.1 numbered flow: `3b. Gateway: For each gitClone source in workspacePlan, resolve ref to an immutable commit SHA via git ls-remote (see §14 gitClone.ref resolution). On transient network failure, reject with 503 GIT_CLONE_REF_RESOLVE_TRANSIENT; on auth failure or unknown ref, reject with 422 GIT_CLONE_REF_UNRESOLVABLE.` (b) Amend the Atomicity paragraph at line 28 to clarify that request-body validation rejections (including `GIT_CLONE_REF_UNRESOLVABLE`, `GIT_CLONE_REF_RESOLVE_TRANSIENT`, `WORKSPACE_PLAN_INVALID`, `ENV_VAR_BLOCKLISTED`, and the other workspace-plan-validation codes in §14) fire before step 4 and therefore never allocate any resources — the rollback-without-persist path applies only to failures at steps 4 onward.
**Why Low:** No runtime or security impact; the `GIT_CLONE_REF_*` codes are already catalogued in §15.1 with correct `PERMANENT` / `TRANSIENT` classifications, and §14 line 104 is authoritative. The issue is purely that §7.1's numbered flow and atomicity contract are incomplete for a new error family added in iter6.

### SES-023 — §4.4 partial-manifest-cleanup paragraph describes cleanup as reason-agnostic but does not name both `manifest_reason` enum values inline
**Severity:** Low
**Location:** `spec/04_system-components.md` §4.4 line 236 (Partial checkpoint manifest cleanup paragraph)
**Finding:** The §4.4 Partial-checkpoint-manifest-cleanup paragraph at line 236 references a chunked-object cleanup flow that is "reason-agnostic" (a property the §10.1 partial-manifest schema at line 141 and the Cleanup paragraph at line 157 both rely on to preserve GC correctness across the `timeout` and `terminated_during_resume` manifest-reason values). The §4.4 paragraph itself, however, never names either value inline — a reader must follow the reference to §10.1 line 141 to discover that the reason column carries `timeout` or `terminated_during_resume` (or a future-added value). §4.2 line 156 (iter5 SES-017 fix follow-through in iter6) does explicitly name `terminated_during_resume` and cross-reference §10.1, and §10.1 itself enumerates both values. The §4.4 paragraph is thus downstream of both authoritative enumerations and is silent on the reason distinction. A reader who lands on §4.4 for checkpoint-store semantics and does not traverse to §10.1 misses the reason dimension entirely.
**Impact:** Documentation completeness gap. Operators or implementers consulting §4.4 to understand partial-manifest cleanup (the natural entry point, given §4.4's role as the "Event / Checkpoint Store" section) will not see the `manifest_reason` enum without following cross-references. The `lenny_partial_manifest_cleanup_total` counter mentioned at the end of §4.4 line 236 is labeled only by `outcome: success|failed_deleted|gc_collected` — it does not mention the `manifest_reason` label that `lenny_checkpoint_partial_total` carries in §10.1, so an operator reading §4.4 alone might incorrectly assume partial-manifest cleanup observability is reason-blind. Not a correctness defect — cleanup itself is reason-agnostic by design — but the §4.4 paragraph is less self-contained than its §4.2 and §10.1 companions.
**Recommendation:** Insert a one-sentence parenthetical in §4.4 line 236's Partial-checkpoint-manifest-cleanup paragraph reading: "The `manifest_reason` column on the manifest row records why the partial exists — `timeout` for checkpoints that exceeded their tier cap or BarrierAck deadline, `terminated_during_resume` for partials written by an aborted snapshot replay per [§7.2](07_session-lifecycle.md#72-interactive-session-model) snapshot-close semantics; both values are handled identically by this cleanup path (the `partial = true AND deleted_at IS NULL` predicate is reason-agnostic)." Optionally, reference the `lenny_checkpoint_partial_total{manifest_reason}` counter from §10.1 line 157 to make the reason-distinguishing observability discoverable from §4.4.
**Why Low:** No correctness or security impact; §10.1 is authoritative and the cleanup logic itself is reason-agnostic by design. The issue is purely that §4.4 — the natural discovery point for checkpoint-store semantics — does not inline the reason enum.

### SES-024 — Derive source-state rule 1 enumerates `resuming` but §15.1 declares `resuming` never externally observable, creating a client-facing ambiguity
**Severity:** Low
**Location:** `spec/07_session-lifecycle.md` §7.1 derive rule 1 at line 88; `spec/15_external-api-surface.md` §15.1 line 621 and line 643
**Finding:** §7.1 derive rule 1 at line 88 enumerates the following non-terminal source states as requiring `allowStale: true`: "`running`, `suspended`, `resume_pending`, `resuming`, `awaiting_client_action`". `resuming` appears explicitly in the enumeration. However, §15.1 line 621 states that `resuming` is "an internal-only transient state between `resume_pending` and `running`; the API reports the transition as `resume_pending` → `running`", and §15.1 line 643 declares that internal-only pod-state values (including `resuming`) are "**never** returned in external API responses". A client reading the authoritative external API reference cannot observe `resuming` via `GET /v1/sessions/{id}`, yet §7.1 derive rule 1 describes the gateway's state-check behavior against a value the client cannot know is current. The ambiguity is: (a) can a `POST /derive` land while the session is transiently in `resuming` and be subject to the same `DERIVE_ON_LIVE_SESSION` / `allowStale: true` gate as `resume_pending`? (b) if so, how does the client know to pass `allowStale: true` for a state they cannot observe via API?
**Impact:** Client-facing contract ambiguity. In practice, since `resuming` is a transient sub-state of `resume_pending` from the external view, the gate is equivalent to "`resume_pending`" for client purposes — but §7.1's enumeration of `resuming` as a distinct state suggests otherwise. A client that polls `GET /v1/sessions/{id}` and sees `resume_pending`, infers it is safe to call `POST /derive` with `allowStale: true`, and receives a surprise rejection path if the session internally transitions between `resume_pending` and `resuming` during the `POST /derive` admission window would be left without a clear recovery action. Not a correctness defect — both `resume_pending` and `resuming` produce the same external behavior under `allowStale: true` — but the enumeration is misleading.
**Recommendation:** Either (a) remove `resuming` from the §7.1 derive rule 1 enumeration and add a parenthetical "`resume_pending` (internally including the transient `resuming` sub-state)" so the enumeration matches the externally observable state set; or (b) add a sentence to derive rule 1 stating "`resuming` is an internal-only transient sub-state of `resume_pending` (see [§15.1](15_external-api-surface.md#151-rest-api)); clients should treat it as equivalent to `resume_pending` for the purpose of `allowStale: true` gating — the gateway normalises the check accordingly." Option (a) is preferred because it avoids surfacing an internal state name in a client-facing rule at all.
**Why Low:** No runtime or security impact. The gateway's internal state check handles both cases identically; the issue is purely that §7.1 exposes an internal state name in a client-facing derive rule. Consistent with iter4/iter5/iter6 SES-019 framing (external-reference-vs-authoritative-inconsistency, Low).
