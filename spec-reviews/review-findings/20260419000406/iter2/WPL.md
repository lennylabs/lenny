## Findings

No real issues found.

**Regressions/gaps checked and cleared:**

1. **SDK-warm demotion path completeness.** `sdkWarmBlockingPaths` contract (§6.1) correctly specifies: case-sensitive matching, no automatic root anchoring, symlinks not resolved, `workspaceDefaults` files included in the check, `[]` disables demotion entirely, and demotion is mandatory for `preConnect: true` runtimes. Escalation rungs (< 20% / 20–60% / > 60% warn, 90% hardcoded circuit-breaker) are consistent between §6.1 prose and the `SandboxWarmPool.spec.sdkWarmDisabled` field-ownership row (§4.6.3). Circuit-breaker drain semantics (do-not-drain existing idle SDK-warm pods; demote at claim time; natural transition within one cert lifetime) are internally consistent and correctly justified by pool-gap-avoidance reasoning.

2. **Eviction during SDK-warm / `sdk_connecting` preemption.** The SIGTERM-during-`sdk_connecting` protocol (call `DemoteSDK` with bounded 5s timeout → force-terminate → exit → pod transitions to `terminated` not `failed`) is well-specified. The stated `terminationGracePeriodSeconds >= LENNY_DEMOTE_TIMEOUT_SECONDS + 5s` floor (10s) is subsumed by the existing `SandboxWarmPool` CRD validation in §10.1 preStop hook (`max_tiered_checkpoint_cap + checkpointBarrierAckTimeoutSeconds + 30 <= terminationGracePeriodSeconds`), which always yields ≥ 60s — so the 10s SDK-demote floor is trivially satisfied. The `sdk_connecting` watchdog (60s default, configurable via `scalingPolicy.sdkConnectTimeoutSeconds`) prevents phantom-warm accumulation; since pod state lives in `Sandbox.status.phase` (persisted in etcd), a new WarmPoolController leader enforces the watchdog from the persisted entry timestamp — the 25s worst-case failover does not create a bypass window.

3. **Sizing under burst with SDK-warm.** The distinction between `pod_startup_seconds` (container pull + runtime startup) and `pod_warmup_seconds` (adds SDK init, 30–90s) is explicitly called out in §4.6.1 and §17.8.2, with an explicit instruction to substitute the observed SDK-warm startup time for `pod_warmup_seconds` when sizing SDK-warm pools. The burst term `burst_p99_claims × pod_warmup_seconds` therefore correctly scales with SDK init cost for SDK-warm pools and is not silently using the pod-warm baseline.

4. **Concurrent-workspace pod eviction during SDK-warm.** The `preConnect` compatibility table (§6.1) unambiguously rejects the combination `executionMode: concurrent` + `capabilities.preConnect: true` (for both `workspace` and `stateless` styles) at pool-config validation time with explicit error messages. No eviction-during-SDK-warm path exists for concurrent pods — the scenario is structurally unreachable.

5. **Variant pool sizing formula completeness.** The variant pool formula (§4.6.2) is mode-aware: §5.2 explicitly states "For A/B experiment variant pools, apply `variant_weight` as defined in the variant pool formula in §4.6.2, combined with the `mode_factor` and `burst_mode_factor` divisors from this formula." The base pool adjustment on `paused → active`, `active → paused`, and `* → concluded` transitions (§10.7) correctly recomputes `(1 - Σ variant_weights)` using the aggregate across all active experiments on the same base pool — not per-experiment — preventing last-write-wins overwrites. The `initialMinWarm` bootstrap override is correctly bounded to bootstrap mode only and does not persist on `paused → active` re-activation (formula-derived value is used). `active → paused` intentionally leaves `maxWarm` unchanged (warm pods drain naturally) vs `* → concluded` which sets both to 0 for immediate full drain — this asymmetry is correct and documented.

6. **Cross-section consistency.** `failover_seconds = 25` (crash-case `leaseDuration + renewDeadline = 15s + 10s`) is uniformly used across §4.6.1, §4.6.2, §5.2 mode-adjusted formula, §17.8.2 baseline tables and worked examples, and delegation fan-out sizing. `safety_factor` defaults (1.5 agent, 2.0 mcp at Tier 1/2; 1.2 agent at Tier 3) are consistent between §4.6.2 text and the §17.8.2 Tier table. The §17.8.2 worked example (Tier 3 orchestrator preset) correctly combines `mode_factor = 1.0`, `safety_factor = 1.2`, and `pod_warmup_seconds = 35` without arithmetic drift.

7. **Concurrent-workspace termination budget.** The `terminationGracePeriodSeconds` floor for concurrent-workspace pods (`maxConcurrent × max_tiered_checkpoint_cap + checkpointBarrierAckTimeoutSeconds + 30`) is CRD-validated (§5.2), with node-drain-timeout warning (> 600s) and optional hard-ceiling rejection via `maxTerminationGracePeriodSeconds`. Eviction checkpoint serialization (slot-ID order, not parallel) correctly prevents MinIO write-amp cascade. The Redis slot-rehydration blocking protocol (per-pod `rehydrated` flag, `SET NX` lock, bounded spin-wait) eliminates the double-claim race after Redis restart.

8. **Adjacent iter1 fixes verified present.** OBS-005 pool-label on `PodClaimQueueSaturated` (present in §16.5), PoolConfigValidatorUnavailable alert (§16.5), STR-003 T4 KMS lifecycle (§12.5), STR-002 `CheckpointStorageHigh` 80% threshold and sizing (§12.5/§12.8) — all confirmed in current spec and coherent with WPL-facing surface (warm pool sizing, eviction paths, cert expiry handling).
