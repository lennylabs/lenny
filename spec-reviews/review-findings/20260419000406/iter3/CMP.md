# Iter3 CMP Review

Iter2 fixes for CMP-044 (post-restore erasure reconciler) and CMP-045 (per-region backup pipeline) both landed in `spec/12_storage-architecture.md` §12.8 and `spec/25_agent-operability.md` §25.11 and are comprehensive on their primary claims: the reconciler fails closed on ready-gating, uses `gdpr.*` receipts that survive restore under `audit.gdprRetentionDays`, rejects restores predating the retention floor (`RESTORE_INCOMPATIBLE`), and the per-region backup path mirrors `REGION_CONSTRAINT_UNRESOLVABLE` with a new `BACKUP_REGION_UNRESOLVABLE` error + `DataResidencyViolationAttempt` audit event. No regressions introduced by those fixes.

Three residual compliance gaps remain. None are regressions of iter2; CMP-046 was latent before iter2 (pre-existing spoliation risk) but is now much more exposed because the new reconciler is explicitly designed to call `DeleteByUser`/`DeleteByTenant` after a backup restore. CMP-047 and CMP-048 are adjacent gaps surfaced by a careful read of the same sections iter2 modified.

---

### CMP-046 Post-restore erasure reconciler can destroy legally-held data [HIGH]
**Files:** `spec/12_storage-architecture.md` §12.8 ("Backups in erasure scope" reconciler phases 1–3; "Legal hold and checkpoint rotation"; "Legal hold interaction during deletion"); `spec/25_agent-operability.md` §25.11 step 6 (GDPR erasure reconciler)

The iter2 reconciler enumerates every `gdpr.*` receipt with `completed_at > backupTakenAt` and replays `DeleteByUser(user_id)` or `DeleteByTenant(tenant_id)` against the restored databases, with no reference to `legal_hold`. Two paths lead to spoliation:

1. **User-level erasure never checks legal hold.** §12.8 "Legal hold interaction during deletion" (line 840) only gates the Phase-4 tenant-deletion controller on active holds; the `DeleteByUser(user_id)` sequence (steps 1–19, §12.8) and the reconciler's per-subject replay have no equivalent guard. If a user's session was placed on `legal_hold = true` AFTER the backup was taken but BEFORE the restore (e.g., operator placed a hold on user U's session S in response to a subpoena between Day N and Day N+3, and the restore replays a Day-N backup), the reconciler will re-enumerate the pre-hold `gdpr.erasure_completed` receipt for U and invoke `DeleteByUser(U)` on the restored state — destroying the very session S that is under hold. This is exactly the spoliation scenario §12.8 "Legal hold and checkpoint rotation" cites ("destruction of potentially discoverable evidence"), reached via a compliance-triggered code path rather than GC.
2. **Tenant-level replay bypasses the Phase-4 hold gate.** The reconciler replays tenant-level receipts by invoking `DeleteByTenant(tenant_id)` directly, not by re-entering the Phase-1→Phase-4 controller state machine. The `admin.tenant.deletion_blocked` pause at Phase 3 therefore does not fire on replay.

The reconciler's three-phase flow does not mention `legal_hold` in either phase 1 (enumerate) or phase 2 (replay). The `gdpr.erasure_reconciled` event it emits would land in the audit trail simultaneously with a `legal_hold.checkpoint_gap_detected` critical event from the legal-hold reconciler 15 minutes later — compliance teams would learn of the destruction after the fact.

**Recommendation:** Amend §12.8 "Backups in erasure scope" phase 2 (Replay erasure) to perform a legal-hold preflight identical to the Phase-4 tenant-deletion gate: for each enumerated receipt, query `SELECT EXISTS(...) WHERE legal_hold = true AND (user_id = $subject OR tenant_id = $subject OR session_id IN subject's sessions)` against the restored databases. If any row is held, SKIP the replay for that subject, emit a critical `gdpr.backup_reconcile_blocked_by_legal_hold` audit event (carrying `restore_id`, `receipt_id`, `held_resource_ids`), increment a new `lenny_backup_reconcile_legal_hold_blocked_total` counter, and proceed to the next receipt. The restore then completes with a non-empty `blockedSubjects` array in `gdpr.backup_reconcile_completed`; operators must explicitly clear the hold and re-run the erasure for the affected subject via the normal `POST /v1/admin/gdpr/erasure` flow (which itself should gain the same legal-hold check per the separate recommendation below). Additionally, add a legal-hold check to the `DeleteByUser(user_id)` flow itself — not only the reconciler — so that a normal user-level erasure request against a held session also fails closed with a new error code (`ERASURE_BLOCKED_BY_LEGAL_HOLD`, HTTP 409) rather than silently proceeding.

### CMP-047 OCSF dead-letter rows retain raw canonical PII past user erasure [MEDIUM]
**Files:** `spec/11_policy-and-controls.md` §11.7 (OCSF translation failure path, Dead-letter handling); `spec/12_storage-architecture.md` §12.8 (`DeleteByUser` step 13 — `EventStore` (audit); `gdpr.*` audit event exemption)

§11.7 "Dead-letter handling" specifies that an untranslatable audit row transitions to `ocsf_translation_state = 'dead_lettered'` and the translator emits a replacement `translation-failure receipt` whose payload carries the original canonical tuple as `unmapped.lenny.raw_canonical_b64`. The original `audit_log` row remains queryable via `GET /v1/admin/audit-events?ocsf_translation_state=dead_lettered`.

§12.8 step 13 ("`EventStore` (audit) — delete session-level audit events for the user") requires that `DeleteByUser` skip only `gdpr.*` event types (§12.8 `gdpr.*` audit event exemption rule at line 800). The translation-failure-receipt events are an ordinary OCSF `Application Security Finding` class (2004), not `gdpr.*`, so in principle `DeleteByUser` should remove them for a user whose audit events are being erased. However, the receipt payload **duplicates** the original event's canonical JCS payload verbatim in `unmapped.lenny.raw_canonical_b64` — the very bytes the translator could not understand — and the spec does not specify:

- whether the receipt's `tenant_id` / `user_id` mirror the original event (so `DeleteByUser` can find it), or whether it is written under the platform tenant (making it unreachable to a tenant-scoped user-erasure query);
- whether the translator, on encountering a row with `event_type LIKE 'gdpr.%'`, suppresses the raw-canonical dump into `unmapped.lenny.raw_canonical_b64` (preventing the erasure receipt itself from leaking subject identifiers in base64 form when translation of the receipt fails);
- whether `POST /v1/admin/audit-events/{id}/retranslate` re-generates a dead-lettered row's replacement receipt after a successful erasure — in which case the payload pulled from the Postgres row is already nulled by `DeleteByUser`, but the original base64-encoded receipt (already streamed to SIEM and pgaudit sinks) still contains the raw canonical tuple.

The net effect: a user who requests erasure may have their raw canonical audit payloads (including PII fields the translator could not map) retained downstream inside translation-failure receipts that the erasure job has no handle on, in a form that is not `gdpr.*`-exempt and not labelled as subject-scoped.

**Recommendation:** In §11.7 "Dead-letter handling," specify that (1) the translation-failure receipt inherits `tenant_id`, `user_id`, and `session_id` from the original row (so tenant-scoped and user-scoped queries — including `DeleteByUser` step 13 — find and delete it); (2) the `unmapped.lenny.raw_canonical_b64` field is ZEROED (not just marked) when the row's `user_id` or `tenant_id` is subject to an active erasure job (detected via the `processing_restricted` flag or an erasure-job table join); (3) re-translation (`POST /v1/admin/audit-events/{id}/retranslate`) is rejected with `RETRANSLATE_DENIED_ERASED_SUBJECT` when the source row's subject has been erased since the original translation failure; (4) `DeleteByUser` step 13 explicitly names translation-failure receipts as in-scope. Also add an SIEM-side note that downstream consumers are responsible for honoring the `erasure.requested` event for subjects whose data already left as a raw-canonical dead-letter payload (the existing external-sink propagation rule, §12.8 "Erasure propagation to external sinks," is the only lever — make the dead-letter case explicit there).

### CMP-048 ArtifactStore (MinIO) not covered by backup pipeline — restore-time GDPR/residency claims incomplete [MEDIUM]
**Files:** `spec/25_agent-operability.md` §25.11 (Full backup flow, Job Pod Specification, Sensitive Content Policy); `spec/12_storage-architecture.md` §12.8 ("Backups in erasure scope")

The iter2 "Backups in erasure scope" block asserts the reconciler replays erasure against "the restored databases" and claims a backup + restore would resurrect erased users' "sessions, memories, OAuth tokens, billing rows bearing the original `user_id`, and audit entries." But the §25.11 Full backup flow covers only Postgres + K8s CRDs + platform config:

- Step 1: `pg_dump` per Postgres shard
- Step 2: platform config JSON
- Step 3: CRD manifests
- Steps 4–7: tar + encrypt + upload

There is **no `ArtifactStore` (workspace MinIO) backup step** — the `lenny-backup` pod's MinIO access key is scoped to the backup bucket only (§25.11 Job Pod Specification: "`s3:PutObject`, `s3:GetObject` (for verification), and `s3:DeleteObject` (for retention) on the backup bucket only"). The §25.11 restore flow likewise restores Postgres and config but says nothing about workspace artifacts, checkpoints, transcripts, or uploads. The `ArtifactStore` row count in Postgres will restore pointing at MinIO objects that — depending on the ArtifactStore bucket's retention posture — may or may not still exist.

Two consequences for compliance:

1. **GDPR gap:** The post-restore reconciler replays `DeleteByUser` against restored Postgres, which includes `ArtifactStore` rows. But the MinIO objects those rows pointed to are unmanaged by the backup pipeline — they may have been deleted by their own TTL between `backupTakenAt` and restore time (silent success), OR may still be present if MinIO retention is longer than Postgres `backups.retention.retainDays`, in which case the restored Postgres rows repoint at live artifacts that were supposed to be erased but that `DeleteByUser` cannot reach without the MinIO objects in its delete set. Either way, the §12.8 claim "[restore] would resurrect erased users' ... workspace data" is partially unsupportable: workspace data is not *in* the backup, so it cannot be restored or reconciled symmetrically.
2. **Residency gap:** The per-region backup pipeline (§12.8 "Backup pipeline residency," iter2 fix CMP-045) is scoped to `pg_dump` per region. `ArtifactStore` MinIO buckets are per-region at runtime (§12.8 `storage.regions.<region>.minioEndpoint`), but a future artifact-backup path that copies objects into the backup bucket would need to mirror the same per-region guard — there is no `ArtifactStore` backup today, and no per-region gate is specified for a future one. An operator adding artifact backup via any side-channel (e.g., `aws s3 sync`) would do so without the `BACKUP_REGION_UNRESOLVABLE` fail-closed guarantee.

**Recommendation:** Either (a) explicitly declare workspace artifacts OUT of scope for backups in v1 and amend §12.8 "Backups in erasure scope" to restrict its claims accordingly ("backups cover Postgres-resident personal data only; workspace artifacts in MinIO are governed by the ArtifactStore TTL, not by backups" + restore documentation note that Postgres rows repointing at missing artifacts are expected and handled by the existing `artifact_store.deleted_at IS NULL` guard); or (b) extend the backup pipeline to include per-region MinIO-to-backup replication with the same `BACKUP_REGION_UNRESOLVABLE` fail-closed path, the same per-region `kmsKeyId` / bucket / credential configuration as `pg_dump`, and a corresponding extension of `DeleteByUser` step 8 (ArtifactStore) so that user-level erasure deletes both the runtime MinIO objects AND any backup-side artifact segments. Option (a) is the lighter v1 path; option (b) is required if the reconciler's compliance claims are to cover artifacts. In either case, update the §12.8 paragraph to be explicit about which stores the reconciler actually reconciles (Postgres-only) versus which stores a restore cannot resurrect (MinIO artifacts — because they were never captured).
