## 21. Planned / Post-V1

The following items are documented now so data models accommodate them. Implementation is deferred.

**21.1 A2A Full Support.** `ExternalProtocolAdapter` is the mechanism. A2A is implementation two. Inbound: gateway serves `POST /a2a/{runtime}/tasks` via `A2AAdapter`. The adapter creates a Lenny session, maps A2A task fields to `TaskSpec`, and translates A2A state queries to Lenny's canonical task state machine (see [Section 8.8](08_recursive-delegation.md#88-taskrecord-and-taskresult-schema) protocol mapping table). Outbound: external A2A agents registered as connectors, callable via `lenny/delegate_task`. A2A `input-required` state maps directly to Lenny's `input_required` task state. A2A `canceled` maps to Lenny's `cancelled`. A2A `unknown` is not a Lenny state — it surfaces as a 404 from the adapter. A2A artifacts map to Lenny artifact refs (`lenny-blob://`). `agentInterface` auto-generates A2A agent cards. Per-agent A2A endpoints (not aggregated) at `/a2a/runtimes/{name}`. `allowedExternalEndpoints` slot on delegation lease schema exists from v1.

**A2A standards-based discovery (`/.well-known/agent.json`).** The A2A specification defines `GET /.well-known/agent.json` as the canonical discovery URL for external callers performing standards-based agent discovery. The A2A spec requires this endpoint to return a single `AgentCard` object. Because Lenny hosts multiple runtimes behind a single gateway, the gateway's `/.well-known/agent.json` returns a **JSON array** of agent card objects — this is an **intentional Lenny extension** to the A2A spec that enables multi-agent discovery from a single gateway. Each element is the verbatim stored card for one runtime (gateway pass-through — no re-parsing). Runtimes with no public `agent-card` entry are excluded. The endpoint requires no authentication (consistent with the `public` visibility contract defined in [Section 5.1](05_runtime-registry-and-pool-model.md#51-runtime)). To prevent unbounded response sizes, the gateway applies a configurable `wellKnownAgentJsonMaxCards` limit (default: 100 runtimes); deployers with large numbers of public runtimes should use per-runtime endpoints for targeted access. The endpoint is served at gateway startup when the `A2AAdapter` is enabled; it is not available in deployments where `A2AAdapter` is not registered.

**Standard A2A client degradation at the aggregated endpoint.** Standard A2A clients that issue `GET /.well-known/agent.json` and expect a single `AgentCard` JSON object will receive a JSON array, which will typically cause a deserialization error on the client side. To provide a meaningful signal, the aggregated endpoint includes a `Link` header pointing to the per-runtime discovery pattern: `Link: </a2a/runtimes/{name}/.well-known/agent.json>; rel="item"` (one `Link` entry per included runtime, up to `wellKnownAgentJsonMaxCards`). Additionally, the `Content-Type` header is set to `application/json; profile="https://lenny.dev/a2a-multi-agent"` to distinguish the response from a standard A2A single-card response. Standard A2A clients MUST NOT use the aggregated endpoint; deployers that need cross-organization A2A discovery from a single hostname SHOULD configure a reverse proxy to route `/.well-known/agent.json` to a single runtime's per-runtime endpoint.

**Per-runtime standard-compliant A2A discovery.** For standard A2A clients that expect a single `AgentCard` object (per the A2A spec §3), the gateway additionally serves `GET /a2a/runtimes/{name}/.well-known/agent.json`. This endpoint returns the single `AgentCard` object for the named runtime — not wrapped in an array — conforming to the A2A specification. If the named runtime has no public `agent-card` entry, the endpoint returns `404`. Standard A2A clients SHOULD use this per-runtime endpoint; the aggregated array endpoint is intended for Lenny-aware discovery clients that understand the multi-agent extension.

**Elicitation chains under A2A (design constraint).** The MCP elicitation chain ([Section 9.2](09_mcp-integration.md#92-elicitation-chain)) is structurally hop-by-hop, requiring each level to relay elicitations upward to its direct client. A2A has no equivalent hop-by-hop elicitation primitive. When a Lenny session is reached via `A2AAdapter`, any elicitation originating inside that session tree cannot be surfaced to the A2A caller using the standard MCP chain. The `A2AAdapter` therefore imposes the following behavior for v1 A2A support:

1. **Elicitation suppression.** `A2AAdapter`-initiated sessions automatically set `elicitationDepthPolicy: block_all` for agent-initiated elicitations. Gateway-registered connector OAuth flows remain exempt (connector-initiated, not agent-initiated) and are handled out-of-band.
2. **`input-required` as the A2A elicitation surface.** When a Lenny session requires human input that cannot be suppressed (e.g., `lenny/request_input` called by the agent), the gateway transitions the session to Lenny `input_required` state, which maps to A2A `input-required`. The A2A caller polls (or receives a push notification via `OutboundChannel`; see [Section 21.1](#21-planned--post-v1) outbound push) and responds by sending a new A2A task message — the `A2AAdapter` routes this as a `POST /v1/sessions/{id}/messages` call with `inReplyTo` set to the pending `request_input` ID. This is the canonical A2A substitute for the MCP elicitation round-trip.
3. **Limitation documented for callers.** The `A2AAdapter`'s agent card (auto-generated from `agentInterface`) includes a `capabilities` field that explicitly lists `elicitation: false` when `block_all` is in effect, allowing A2A callers to know in advance that elicitation-dependent flows are not supported through this interface.

This constraint is a deliberate v1 scoping decision. A richer A2A elicitation model (e.g., A2A `input-required` combined with structured schema delivery matching MCP elicitation schemas) is a post-v1 concern and does not require any change to the internal elicitation chain or gateway internals — only the `A2AAdapter` surface changes.

**A2A outbound push (OutboundChannel contract).** A2A callers that register a webhook URL with their initial task request receive asynchronous push notifications as the session progresses. The `A2AAdapter` implements this via `OutboundChannel` ([Section 15](15_external-api-surface.md)):

- `OutboundCapabilities()` returns `PushNotifications: true`, `SupportedEventKinds: ["state_change", "output", "error", "terminated"]`.
- On `OpenOutboundChannel`, the adapter stores the caller-supplied `CallbackURL` (from the A2A task request's `pushNotification.url` field) in `OutboundSubscription.CallbackURL`. **SSRF requirement:** The `pushNotification.url` MUST pass the same SSRF validation as `callbackUrl` ([Section 14](14_workspace-plan-schema.md)) — HTTPS-only, private IP rejection, DNS pinning, and optional domain allowlist — before being stored. The A2A adapter MUST validate the URL at `OpenOutboundChannel` time, not at delivery time, and reject task registration with `400 INVALID_CALLBACK_URL` if validation fails. This requirement must be implemented before the A2A adapter is enabled in any deployment.
- `OutboundChannel.Send` serializes the `SessionEvent` to the A2A streaming response format and POSTs it to the callback URL with an HMAC-SHA256 signature in the `X-Lenny-Signature` header. Delivery failures are retried with exponential back-off (3 attempts, max 30s total); after exhaustion the channel emits a `push_delivery_failed` metric and continues — missed events are not re-queued.
- For clients that use long-poll (`GET /a2a/{runtime}/tasks/{id}/stream`) rather than webhooks, `OutboundChannel.Send` writes SSE frames to the held `ResponseWriter` in `OutboundSubscription`.
- `OutboundChannel.Close` is called when the session terminates or the subscription TTL expires (default: `maxSessionAge` + 60s). The channel sends a terminal `A2ATaskState` frame before closing the connection or issuing the final webhook POST.

Adapters with no outbound behavior (MCP, OpenAI Completions, Open Responses) embed `BaseAdapter` and inherit its no-op `OutboundCapabilities` / `OpenOutboundChannel` implementations unchanged.

**21.2 A2A Intra-Pod Support.** Adapter additionally serves per-agent A2A endpoints intra-pod. Adapter manifest gains A2A base URL. All A2A traffic proxied through gateway. Runtime authors opt in explicitly.

**21.3 Agent Protocol (AP) Support.** AP defines `POST /ap/v1/agent/tasks` and step execution. Third `ExternalProtocolAdapter` implementation. If AP defines a push-notification mechanism, `AgentProtocolAdapter` sets `OutboundCapabilitySet.PushNotifications: true` and implements `OpenOutboundChannel` following the same contract as `A2AAdapter`. AP adapters that have no outbound push requirement embed `BaseAdapter` and return an empty `OutboundCapabilitySet`.

**21.4 Future Conversational Patterns.** `MessageEnvelope` with `id`, `from`, `inReplyTo`, `threadId` accommodates all of these without schema changes: threaded messages, multiple participants, non-linear context retrieval, broadcast, external agent participation.

**21.5 Environment Management UI.** Full web UI for browsing environments, editing membership, and previewing selector matches is thin-client work over the admin API. The `?dryRun=true` parameter on `PUT /v1/admin/environments` (semantics defined in [Section 15.1](15_external-api-surface.md#151-rest-api)) is the preview mechanism and ships in v1. The dryRun response includes a `preview` object with `matchedRuntimes`, `matchedConnectors`, and `unmatchedSelectorTerms` arrays, giving the UI immediate feedback on which resources a selector change would include and flagging selector terms that match nothing.

**21.6 Environment Resource — Post-V1 Deferred Items.** Cross-environment delegation richer controls. Runtime-level cross-environment exceptions at granularity beyond the structured form. Integration with experiment-scoped delegation boundaries.

**21.7 Multi-Cluster Federation.** Session IDs must be globally unique. Storage interfaces must not assume single-cluster topology. Connectors registered in one cluster must be expressible by reference in multi-cluster scenarios.

**21.8 UI and CLI.** Admin API is the complete operational surface. Official CLI (`lenny-ctl`) and web portal are separate projects consuming the admin API as thin clients with near-zero business logic (the sole exception is `lenny-ctl preflight` standalone mode; see [Section 24](24_lenny-ctl-command-reference.md)). The `lenny-ctl` command surface is documented in the **`lenny-ctl` Command Reference** appendix ([Section 24](24_lenny-ctl-command-reference.md)). Cross-reference: [§17.7](17_deployment-topology.md#177-operational-runbooks) runbooks use specific `lenny-ctl` subcommands and link to that appendix for flag details.
