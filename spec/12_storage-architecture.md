## 12. Storage Architecture

### 12.1 Design Principle

Abstract by **storage role**, not by raw database API. Each store exposes domain operations, not generic CRUD. Every store role interface (including pluggable ones such as `MemoryStore` and `SemanticCache`) MUST expose the erasure primitives `DeleteByUser(ctx, tenantID, userID) error` and `DeleteByTenant(ctx, tenantID) error` as mandatory methods, not optional extensions — this is enforced at compile time by Go interface satisfaction and at runtime by the startup/per-job preflight described in [Section 12.8](#128-compliance-interfaces). Deployers substituting a custom backend for any pluggable role inherit responsibility for these methods; a backend that omits them will not compile into the gateway binary.

### 12.3 Postgres HA Requirements

**Minimum topology:** Lenny requires a PostgreSQL instance (14+) with synchronous replication and automatic failover. This can be provided by a managed service (AWS RDS Multi-AZ, GCP Cloud SQL HA, Azure Database for PostgreSQL) or a self-managed operator (CloudNativePG, Patroni). Managed services are **recommended** as the default for most deployments. Per-tier Postgres instance sizing is in [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults).

| Deployment             | Recommendation                                             |
| ---------------------- | ---------------------------------------------------------- |
| Cloud (production)     | Managed PostgreSQL with multi-AZ HA (e.g., RDS, Cloud SQL) |
| On-prem / self-managed | CloudNativePG operator or Patroni on Kubernetes            |
| Local dev              | Single PostgreSQL container (via docker-compose)           |

**Connection pooling:** A connection pooler is **required** in front of Postgres. Each gateway replica maintains a connection pool; without pooling, HPA-scaled replicas exhaust Postgres connection limits. **Cloud-managed deployments** should use the provider's built-in connection proxy (AWS RDS Proxy, GCP Cloud SQL Auth Proxy, Azure PgBouncer integration) instead of deploying a self-managed PgBouncer — see [Section 17.9](17_deployment-topology.md#179-deployment-answer-files) for backend guidance. **Self-managed deployments** deploy PgBouncer (or pgcat) as described below. Regardless of implementation, the pooler must operate in **transaction-mode** pooling (not session-mode) to ensure compatibility with RLS enforcement via `SET LOCAL app.current_tenant` (see [Section 4.2](04_system-components.md#42-session-manager)) — `SET LOCAL` is transaction-scoped (cleared on `COMMIT`/`ROLLBACK`), so transaction-mode guarantees the setting does not leak across tenants sharing the same pooled connection. The pooler's `connect_query` (or equivalent initialization hook) must set a sentinel value (`SET app.current_tenant = '__unset__'`) on every fresh connection checkout so that any query reaching RLS without a prior `SET LOCAL` is rejected.

- **Deployment topology:** PgBouncer runs as a separate Deployment (minimum 2 replicas) fronted by a ClusterIP Service — not as a sidecar on each gateway pod. This centralizes pool management and avoids per-pod connection sprawl toward Postgres.
- **Pool mode:** Transaction mode (`pool_mode = transaction`). This is required for RLS enforcement because `SET LOCAL app.current_tenant` is transaction-scoped (cleared on `COMMIT`/`ROLLBACK`); session mode would leak tenant context across unrelated requests sharing a backend connection.
- **Sizing guidance:** Set `default_pool_size` to approximately `max_connections / number_of_PgBouncer_replicas`, leaving headroom for superuser and replication connections. Configure `reserve_pool_size` (e.g., 5–10 per pool) for burst headroom, with `reserve_pool_timeout` set to a short duration (e.g., 3s) so reserved connections are only used under genuine load spikes.
- **HA:** PgBouncer is stateless — multiple replicas behind the Kubernetes ClusterIP Service provide transparent failover. If one replica is terminated or fails a health check, the Service routes traffic to surviving replicas with no client-side changes required.
- **PodDisruptionBudget:** A PDB **must** be configured for the PgBouncer Deployment with `minAvailable: 1` (or `maxUnavailable: 1` when running 3+ replicas). This prevents a rolling update or node drain from terminating all PgBouncer replicas simultaneously. Without the PDB, a bad rollout can take every replica through `CrashLoopBackOff` at once, making Postgres unreachable for all gateway traffic.
- **Readiness probe:** Each PgBouncer pod **must** define a readiness probe that verifies backend Postgres connectivity — not just PgBouncer process liveness. The probe should execute a lightweight query (e.g., `SELECT 1`) through PgBouncer's admin or application port to confirm that at least one backend connection can be established. A pod that starts successfully but cannot reach Postgres must be removed from the Service endpoints so traffic is routed only to replicas with verified backend connectivity. Recommended settings: `periodSeconds: 5`, `failureThreshold: 2`, `timeoutSeconds: 3`.
- **Full PgBouncer unavailability:** If all PgBouncer replicas are simultaneously unreachable (despite the PDB), gateway replicas lose access to Postgres. In this state the system behaves identically to a Postgres outage: Redis remains the primary coordination mechanism and all Redis-backed roles continue normally; new sessions and Postgres-dependent writes (including billing events — [Section 11.2.1](11_policy-and-controls.md#1121-billing-event-stream)) are rejected with `503 Service Unavailable`. If Redis is also unavailable during this window, dual-store unavailability rules apply ([Section 10.1](10_gateway-internals.md#101-horizontal-scaling)). Recovery is automatic once at least one PgBouncer replica passes its readiness probe and rejoins the Service.
- **Monitoring:** Deploy `pgbouncer_exporter` as a sidecar on each PgBouncer pod to expose Prometheus metrics. Key metrics to alert on: pool utilization (`cl_active` / `sv_active` vs. pool size), client wait time (`cl_waiting_time`), and average query duration (`avg_query_time`).

**Cloud-managed pooler limitation — `__unset__` sentinel:** Most cloud-managed connection proxies (AWS RDS Proxy, GCP Cloud SQL Auth Proxy, Azure PgBouncer integration) do **not** support a `connect_query` or equivalent initialization hook. This means the `__unset__` sentinel — which is the second layer of RLS defense ensuring that any query reaching Postgres without a prior `SET LOCAL app.current_tenant` is rejected — cannot be set on connection checkout. Without the sentinel, a bug in the application-layer tenant context logic could silently allow cross-tenant reads, since only the application-layer `WHERE` clauses would remain as the sole defense.

**Alternative RLS defense for cloud-managed poolers:** Deployments using a cloud-managed connection proxy that lacks `connect_query` support **must** implement the following alternative defense:

1. **Per-transaction tenant validation trigger:** Create a Postgres `BEFORE` trigger (or RLS policy with a validation function) on every tenant-scoped table that verifies `current_setting('app.current_tenant', true)` is set and is not `NULL`, empty, or `'__unset__'` before allowing any `SELECT`, `INSERT`, `UPDATE`, or `DELETE`. The trigger explicitly allows `'__all__'` (the `platform-admin` cross-tenant sentinel — see [Section 4.2](04_system-components.md#42-session-manager)) in addition to concrete tenant IDs matching the `^[a-zA-Z0-9_-]{1,128}$` format. The full validation logic is: reject if value is `NULL`, empty string, or `'__unset__'`; allow if value is `'__all__'` or matches the tenant ID format. If the setting is missing or invalid, the trigger raises an exception (`ERRCODE 'P0001'`), blocking the query. This provides equivalent protection to the `connect_query` sentinel — any connection that reaches a tenant-scoped table without a prior `SET LOCAL app.current_tenant` in the current transaction will be rejected at the database level.
2. **Migration-enforced:** The trigger/function **must** be created by the Lenny schema migration ([Section 12.3](#123-postgres-ha-requirements)), not left to manual operator action. The migration detects whether `connect_query` sentinel support is available (via the `postgres.connectionPooler` Helm value) and creates the trigger when `connectionPooler = external`.
3. **`connectionPooler` defaults to `external` when `backends = cloud-managed`:** When the `backends` Helm value (see [Section 17.9](17_deployment-topology.md#179-deployment-answer-files)) is `cloud-managed`, `postgres.connectionPooler` defaults to `external` if not explicitly set. This ensures cloud-managed deployments never silently operate without the `lenny_tenant_guard` trigger. Operators who wish to override this (e.g., because they have deployed a self-managed PgBouncer alongside a cloud database) must explicitly set `connectionPooler: pgbouncer`.
4. **Startup refusal without flag under cloud-managed poolers:** On gateway startup, after Postgres connectivity is confirmed, the gateway detects the active pooler mode by inspecting the `LENNY_POOLER_MODE` environment variable (injected by the Helm chart from `postgres.connectionPooler`). If `LENNY_POOLER_MODE = external` and the `lenny_tenant_guard` trigger does not exist on tenant-scoped tables (verified by querying `pg_trigger`), the gateway **refuses to start** and exits with a fatal error: `"FATAL: cloud-managed pooler mode (LENNY_POOLER_MODE=external) detected but lenny_tenant_guard trigger is absent from tenant-scoped tables — RLS tenant isolation defense is not active; run schema migrations before starting the gateway (Section 12.3)"`. This check is independent of the preflight Job so that it also catches trigger removal after initial installation (e.g., manual migration rollback).
5. **Integration test requirement:** An integration test (`TestRLSTenantGuardMissingSetLocal`) **must** exist that: (a) connects to a test Postgres instance with the `lenny_tenant_guard` trigger deployed; (b) opens a transaction without issuing `SET LOCAL app.current_tenant`; (c) executes a `SELECT` against a tenant-scoped table; and (d) asserts that the query raises an exception (verifying the trigger blocks the access). A second case in the same test must open a transaction for tenant A and attempt to read rows belonging to tenant B (by setting `SET LOCAL app.current_tenant = 'tenant-a'` then querying rows seeded under `'tenant-b'`) and assert zero rows are returned. This test must run in the CI pipeline on every pull request.
6. **Performance note:** The per-transaction trigger adds negligible overhead (a single `current_setting()` call per statement), but operators should monitor `pg_stat_user_functions` to verify trigger execution counts align with query volume.

See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier Postgres and PgBouncer sizing recommendations.

**Write IOPS estimation:** Multiple gateway components generate Postgres writes concurrently: session state updates, quota checkpoint flushes, billing event inserts, and audit log inserts. The following table estimates sustained write IOPS at each capacity tier (see [Section 16.5](16_observability.md#165-alerting-rules-and-slos) for tier definitions, [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for instance sizing):

| Write Source                   | Tier 1    | Tier 2     | Tier 3       | Notes                                                  |
| ------------------------------ | --------- | ---------- | ------------ | ------------------------------------------------------ |
| Session state updates          | ~5/s      | ~50/s      | ~300/s       | Per-session lifecycle transitions, heartbeats          |
| Quota checkpoint flushes       | ~2/s      | ~20/s      | ~100/s       | Redis→Postgres periodic sync ([Section 10.1](10_gateway-internals.md#101-horizontal-scaling))            |
| Billing event inserts          | ~10/s     | ~100/s     | ~600/s       | Per-token and per-session events ([Section 11.2.1](11_policy-and-controls.md#1121-billing-event-stream))      |
| Audit log inserts              | ~5/s      | ~50/s      | ~300/s       | Per-request and lifecycle audit entries ([Section 11.7](11_policy-and-controls.md#117-audit-logging)) |
| **Total estimated write IOPS** | **~22/s** | **~220/s** | **~1,300/s** | Sustained; bursts may reach 2–3× during session storms |

At Tier 3, total sustained write IOPS (~1,300/s) with burst headroom (~3,900/s) approaches the practical write ceiling of a single 8-vCPU Postgres primary with synchronous replication. Operators should monitor `pg_stat_bgwriter` (buffers written), `pg_stat_wal` (WAL write rate), and replication lag to detect write pressure before it impacts latency.

**Batching guidance for write-heavy paths:** Billing event inserts and audit log inserts are the two highest-volume write sources and are append-only (no UPDATE or DELETE). Billing events **should** be batched to reduce per-statement overhead and WAL amplification. Audit log write mode is governed by data classification — T3/T4 audit events are always synchronous; T2 batching is opt-in only (see below):

- **Billing events:** The gateway **should** buffer billing events in memory and flush them as multi-row `INSERT` statements at a configurable interval (`billingFlushIntervalMs`, default: 500ms) or when the buffer reaches a size threshold (`billingFlushBatchSize`, default: 50 events). On graceful shutdown, the gateway flushes any remaining buffered events before exiting. If the buffer exceeds `billingFlushMaxPending` (default: 500), the gateway flushes immediately and emits the `billing_flush_pressure` metric.
- **Audit logs:** Audit log write mode is determined by **data classification tier** ([Section 12.9](#129-data-classification)), not by environment or SIEM configuration alone.

  **Default: synchronous writes for T3/T4 audit events (unconditional).** User-scoped audit events (T3 — Confidential; includes user identifiers and user-scoped records per [Section 12.9](#129-data-classification)) and any T4 — Restricted audit events are written **synchronously by default** with no batching buffer. `auditFlushIntervalMs` is ignored for these events regardless of `LENNY_ENV` or SIEM configuration. The synchronous write path does not block request processing — audit writes use a separate goroutine with a dedicated connection pool (`audit.syncWritePoolSize`, default: 4 connections).

  **T2 (non-PII) audit events — batching is an explicit opt-in.** Non-PII audit events (T2 — Internal; e.g., pool metrics, routing events) may use batched inserts via `auditFlushIntervalMs` (default: **250ms**; `auditFlushBatchSize`, default: 100 entries). **Batching must be explicitly enabled** by setting `audit.batchingEnabled: true` in the Helm configuration. This option is disabled by default. When enabled, the deployment documentation must record the accepted data-loss risk.

  > **WARNING: `audit.batchingEnabled: true` accepts data loss.** The audit flush buffer is in-memory and has no write-ahead log. A gateway replica crash during a flush interval will silently lose buffered T2 audit events not yet written to Postgres. Estimated loss at each tier:
  >
  > | Tier   | Audit write rate | Max events lost per crash (250ms window) |
  > | ------ | ---------------- | ---------------------------------------- |
  > | Tier 1 | ~5/s             | ~2 events                                |
  > | Tier 2 | ~50/s            | ~13 events                               |
  > | Tier 3 | ~300/s           | ~75 events                               |
  >
  > T3 and T4 audit events are **never** batched regardless of this setting.

  **When SIEM is configured — outbox pattern (Postgres-first):** Audit events are **always written to Postgres synchronously first**, regardless of SIEM configuration. SIEM forwarding happens asynchronously via an outbox/change-data-capture pattern: a dedicated forwarder process (or logical-replication consumer) tails the audit table and delivers events to the SIEM after they are durably committed to Postgres. This design ensures a gateway crash before SIEM delivery cannot lose events — the records are already committed in Postgres and will be forwarded by the outbox forwarder on recovery. The SIEM is an independent immutable copy, not the primary durable store.

  Consequently, **SIEM configuration does not relax the T2 batching restriction.** Even with a SIEM endpoint configured, T2 audit event batching remains an explicit opt-in (`audit.batchingEnabled: true`) that accepts data loss for the in-memory buffer window. The rationale: the outbox forwarder reads committed Postgres rows — any T2 event not yet flushed from the in-memory buffer to Postgres before a crash is lost from both Postgres and the SIEM. HIPAA AU-9, FedRAMP AU-10, and SOC2 CC7.2 require completeness; batching is only appropriate when the deployer has accepted the documented data-loss risk for non-PII T2 events.

  **Outbox forwarder requirements:** The forwarder must use Postgres logical replication (`wal_level = logical`) or poll the audit table using the `sequence` column as a high-water mark. It must checkpoint its delivery position durably (e.g., a `siem_delivery_state` table) so that a forwarder restart replays from the last confirmed delivery point without duplication or gap. Delivery acknowledgement from the SIEM must be recorded before the high-water mark is advanced. Operators must monitor `lenny_audit_siem_delivery_lag_seconds` (the gap between the latest committed audit event and the latest SIEM-acknowledged event; see [Section 16.1](16_observability.md#161-metrics)); the `AuditSIEMDeliveryLag` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when lag exceeds the configured threshold (`audit.siem.maxDeliveryLagSeconds`, default: 30s).

  **When no SIEM is configured (Postgres-only deployments):** T3/T4 audit events remain synchronous (as above). For T2 events, `audit.batchingEnabled: true` may be used but the data-loss warning above applies unconditionally — there is no external durable copy to recover from on crash. In `LENNY_ENV=production` mode without SIEM, the gateway logs `"Audit batching is enabled for T2 events but no SIEM is configured — buffered T2 audit events will be lost on gateway crash"` at startup and emits the `AuditBatchingNoSIEM` metric counter. In non-production environments, batching for T2 events is permitted without warning when `audit.batchingEnabled: true`.

  **Startup chain-continuity check:** On gateway startup, after Postgres becomes reachable, the gateway verifies that the audit hash chain is unbroken for each active tenant by querying the most recent N entries (configurable via `audit.startupChainCheckEntries`, default: 1000) and re-verifying the `prev_hash` chain. If a gap is detected (broken chain), the gateway emits a `WARN`-level log message identifying the tenant, the sequence numbers of the gap boundaries, and the approximate timestamp range of the missing entries: `"Audit chain gap detected for tenant <id>: gap between sequence <n> and <m> (~<timestamp range>). This indicates T2 audit events were lost from the in-memory batch buffer during a previous gateway crash. T3/T4 events are synchronous and will not appear in chain gaps."` The `AuditChainGap` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires. The gateway does **not** refuse to start — a gap is a compliance signal, not a fatal error. Operators are responsible for documenting the gap in their compliance records. Because T2 events were never committed to Postgres before the crash, neither the SIEM (which is fed by the Postgres outbox) nor Postgres will have a copy of the lost events; the gap cannot be replayed.

- **Separate Postgres for write-heavy paths (Tier 3):** At Tier 3, operators **may** deploy a dedicated Postgres instance for billing and audit writes to isolate write amplification from the primary instance handling session state and quota checkpoints. The gateway supports separate connection strings for the billing/audit write path (`LENNY_PG_BILLING_AUDIT_DSN`). When configured, billing and audit inserts are routed to this instance while all other writes continue to the primary. This is optional — a single well-provisioned primary handles Tier 3 sustained load, but the separate instance provides headroom for burst absorption and reduces replication lag on the primary.

**Write classification and load shedding strategy (Tier 3):** Not all writes are equally critical. At Tier 3 sustained load (~1,300/s) with burst headroom (~3,900/s), the primary Postgres instance may become write-saturated. The following classification governs how write pressure is relieved before and during saturation:

| Write Source                      | Classification       | Load-Shedding Behaviour                                                                                                                                  |
| --------------------------------- | -------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Session state transitions         | **SLO-critical**     | Never deferred or dropped. Session correctness depends on these writes. If Postgres cannot accept the write, the session creation or transition is rejected with a retryable error. |
| Quota checkpoint flushes          | **SLO-critical**     | Never dropped; may be deferred by increasing `quotaSyncIntervalSeconds` up to 60s (maximum) under sustained write pressure. Redis counters remain accurate; Postgres is a durable checkpoint only. |
| Billing event inserts             | **Best-effort durable** | On Postgres unavailability, events are first staged to Redis stream (`t:{tenant_id}:billing:stream`, MAXLEN per `billingRedisStreamMaxLen` -- see [Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) for per-tier defaults, TTL 3600s) and flushed to Postgres by background goroutine on recovery. If Redis is also unavailable, events fall back to bounded in-memory write-ahead buffer (default: 10,000 events). Buffer overflow rejects new session-progressing requests (503). NOT dropped unless Redis + gateway crash simultaneously. |
| Audit log inserts (T3/T4)         | **SLO-critical**     | Always synchronous; never batched or deferred. Write saturation on the primary Postgres instance is the trigger to activate the separate billing/audit instance (`LENNY_PG_BILLING_AUDIT_DSN`). |
| Audit log inserts (T2)            | **Best-effort**      | Batched (opt-in); may be lost on crash. Highest tolerable drop under write saturation — increasing `auditFlushIntervalMs` to 500ms reduces write rate at the cost of increased data-loss exposure. |

**Per-tier write ceiling reference table.** The `PostgresWriteSaturation` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when sustained write IOPS exceed 80% of the estimated instance ceiling for more than 5 minutes. The following table defines the numeric ceilings used to evaluate that threshold. These values assume the recommended instance class for each tier ([Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference)); operators who provision a smaller instance must scale the ceilings proportionally.

| Tier   | Recommended primary instance | Estimated write ceiling (sustained) | 80% alert threshold | Burst ceiling (~3× sustained) |
| ------ | ---------------------------- | ------------------------------------ | ------------------- | ------------------------------ |
| Tier 1 | 2 vCPU / 4 GB (e.g., `db.t3.medium`) | ~200 IOPS | ~160 IOPS | ~600 IOPS |
| Tier 2 | 4 vCPU / 16 GB (e.g., `db.r6g.xlarge`) | ~600 IOPS | ~480 IOPS | ~1,800 IOPS |
| Tier 3 | 8 vCPU / 32 GB (e.g., `db.r6g.2xlarge`) | ~1,600 IOPS | ~1,280 IOPS | ~4,800 IOPS |

These ceilings are empirical estimates derived from Postgres WAL write benchmarks on the reference instance types, not theoretical maximums. Operators MUST validate them in Phase 2 load tests (Tier 1/2) and Phase 13.5 load tests (Tier 3) and update the Helm `postgres.writeCeilingIops` override if their workload or instance type differs. The `PostgresWriteSaturation` alert uses the configured ceiling value, not a hard-coded constant — set `postgres.writeCeilingIops` to the measured ceiling for accurate alerting.

> **Workload-specific write overhead not captured in the estimates above:** The generic WAL benchmark numbers do not account for two Lenny-specific write overhead sources that must be quantified by the Phase 13.5 Lenny-specific write-pattern benchmark (scenario 8): (1) **RLS trigger overhead** — the `lenny_tenant_guard` trigger ([Section 12.3](#123-postgres-ha-requirements)) executes on every tenant-scoped statement; under high write concurrency the per-call overhead (a `current_setting()` call + conditional abort check) can add measurable WAL amplification, particularly for the audit log and billing event paths where every row write is tenant-scoped. Monitor `pg_stat_user_functions` under load to quantify trigger execution time. (2) **Quota flush burst patterns** — the Redis→Postgres periodic quota checkpoint sync ([Section 10.1](10_gateway-internals.md#101-horizontal-scaling)) produces bursty writes: at the flush interval boundary, all accumulated quota deltas for active sessions are written using batched multi-row `INSERT ... ON CONFLICT UPDATE` statements (`quotaFlushBatchSize`, default: 200 rows per statement). To prevent all gateway replicas from flushing simultaneously, each replica adds per-replica jitter: `quotaSyncIntervalSeconds + random(0, quotaSyncIntervalSeconds × 0.3)`, spreading flush bursts across a ~3-second window at Tier 3. **Per-tier expected peak flush IOPS** (worst case, all sessions with quota changes): Tier 1: ~5 IOPS (negligible); Tier 2: ~50 IOPS; Tier 3: `ceil(10,000 / 200) = 50` batched statements per replica, with 30 replicas staggered across the jitter window, yielding ~500 peak IOPS within any 1-second measurement window. At Tier 3 the margin between estimated sustained load (~1,300/s) and the instance ceiling (~1,600/s) is approximately 18% (~300 IOPS headroom), which accommodates the staggered flush spike but leaves limited margin for coincident overhead. The `PostgresWriteBurstIops` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when instantaneous write IOPS exceed the burst ceiling; repeated triggers indicate quota flush storms that require either increasing the flush interval or provisioning a larger instance class.

**Horizontal write scaling route:** A single Postgres primary has no horizontal write scale-out path. The defined scaling route is:

1. **Vertical scaling first:** Increase the primary instance class ([Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)). An 8-vCPU primary handles ~1,300/s sustained; a 16-vCPU instance extends the ceiling to approximately 2,500–3,000/s.
2. **Instance separation (Tier 3 trigger):** When primary write IOPS sustains above 80% of the instance ceiling, activate the separate billing/audit Postgres instance (`LENNY_PG_BILLING_AUDIT_DSN`). This offloads the two largest append-only write sources (~900/s combined at Tier 3), reducing primary write load to ~400/s (session state + quota checkpoints).
3. **Append-only table partitioning:** If the separated billing/audit instance is also approaching saturation, partition the `billing_events` and `audit_log` tables by time range (monthly or weekly partitions). Postgres declarative partitioning (`PARTITION BY RANGE (created_at)`) allows parallel writes to the current partition and fast detach/archive of old partitions without vacuuming the full table. This is the primary mitigation for sustained write growth beyond vertical scaling.
4. **Write-ahead buffer as back-pressure:** The `billingWriteAheadBufferSize` limit acts as back-pressure — the gateway stops accepting new billable work rather than silently dropping billing records. This is intentional: billing completeness is a hard invariant. Operators must monitor `lenny_billing_write_ahead_buffer_utilization` and alert before it reaches 80%.

Operators must monitor `pg_stat_bgwriter` (buffers_clean, maxwritten_clean), `pg_stat_wal` (wal_bytes/s), and replication lag. The `PostgresWriteSaturation` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when sustained write IOPS exceed 80% of the estimated instance ceiling for more than 5 minutes.

**Shard-key index discipline (R-01).** Every tenant-scoped table MUST have `tenant_id` as the leading column of its primary index or as the first element of every composite index used on high-frequency query paths. This ensures that queries targeting a specific tenant land on the correct shard in a future sharded topology without full-table scans. **Session-sharded table exemption:** Tables routed by `StoreRouter.SessionShard(sessionID)` — i.e., `sessions`, `session_messages`, `session_tree_archive`, and related per-session tables — are exempt from the leading-`tenant_id` primary key requirement because their distribution key is the shard routing prefix embedded in the session ID (see [§12.6](#126-interface-design) session ID format), not `tenant_id`. All sessions in a delegation tree share the same routing prefix and co-locate on the same shard, enabling single-shard tree traversal, cascade cancellation, and delegation lineage queries. These tables MUST still include a `(tenant_id, ...)` secondary index for scatter-gather queries (`GET /v1/sessions`, GDPR erasure, tenant deletion) and RLS filtering. The CI linter must accept either `tenant_id` or `id`/`session_id` as the leading primary key column for tables annotated `-- session-sharded`. **Phase 1.5 CI check requirement:** The CI pipeline MUST include a schema linter (`scripts/lint-schema.sh` or equivalent) that verifies no tenant-scoped table has an index whose leading column is not `tenant_id` (unless annotated `-- session-sharded`). The check must run on every pull request that modifies a migration file. Migrations that introduce non-conforming indexes are rejected at CI time, not discovered in production. **Annotation validation:** The `-- session-sharded` annotation MUST appear only on `CREATE TABLE` statements for tables that have a foreign key referencing `sessions(id)` either directly or transitively (e.g., `session_messages.session_id → sessions.id`, `session_tree_archive.root_session_id → sessions.id`). The linter MUST verify this FK relationship exists in the same migration or a prior committed migration — an annotation without a verifiable FK is a linter error. The set of session-sharded tables is intentionally small and closed: `sessions`, `session_messages`, `session_tree_archive`, and any future table whose primary access pattern is single-session lookup by embedded routing prefix. Adding a new `-- session-sharded` annotation MUST be accompanied by a `-- session-sharded-justification: <reason>` comment on the preceding line (mirroring the R-02 justification pattern for `-- platform-admin-cross-tenant-allowed`). The linter MUST report the total count of `-- session-sharded` annotations and fail if the count increases without a justification.

**No cross-tenant JOIN rule (R-02).** Application code MUST NOT issue SQL queries that JOIN across tenant boundaries — i.e., no query may `JOIN` a tenant-scoped table from one tenant against rows belonging to a different tenant. This rule exists independently of RLS enforcement: even though RLS filters rows at the database level, a cross-tenant JOIN in application code indicates a logic error that will surface as data leakage if RLS is misconfigured or bypassed. A static-analysis linter (`scripts/lint-queries.sh` or equivalent) MUST enforce this rule by detecting JOIN operations between tables that carry `tenant_id` columns where the `ON` clause does not include `a.tenant_id = b.tenant_id`. The linter must run in CI on every pull request. The only permitted exception is the `platform-admin` cross-tenant reporting path ([Section 4.2](04_system-components.md#42-session-manager)), which must be explicitly annotated with `-- platform-admin-cross-tenant-allowed` in the query source to suppress the linter warning. Three additional safeguards apply to every annotated query:

1. **Pairing rule:** Every query annotated `-- platform-admin-cross-tenant-allowed` MUST be in a code path that (a) sets `app.current_tenant = '__all__'` via the platform-admin gateway path ([Section 4.2](04_system-components.md#42-session-manager)), and (b) is unreachable from non-`platform-admin` callers. The linter SHOULD verify that annotated queries appear only in files under a `platform-admin`-designated package (e.g., `internal/admin/platform/`). The annotation alone is not sufficient — it must be paired with a code path that enforces `platform-admin` RBAC before reaching the query.
2. **Audit event:** Every code path that sets `app.current_tenant = '__all__'` MUST emit an audit event (`cross_tenant_read`) recording the caller identity, endpoint, and query category (e.g., `tenant_list`, `usage_dashboard`, `erasure_job`). This is one event per API call, not per SQL statement.
3. **Annotation inventory:** The CI linter MUST report the total count of `-- platform-admin-cross-tenant-allowed` annotations and fail if a new annotation is added without an accompanying `-- platform-admin-cross-tenant-justification: <reason>` comment on the preceding line. This ensures new cross-tenant query paths get explicit justification visible in code review.

**StoreRouter requirement for billing/audit writes (R-03).** All billing event inserts and audit log inserts MUST be routed through the `StoreRouter` interface ([Section 12.6](#126-interface-design)), even in v1 where `StoreRouter` resolves to a single `SingleShardRouter`. This rule prevents the billing and audit write paths from hardcoding a direct Postgres connection that bypasses shard routing. In v1, `StoreRouter.BillingShard(tenantID)` and `StoreRouter.AuditShard(tenantID)` return the sole configured Postgres pool, so there is no behavioral change; the discipline ensures that adding a second shard in a future release requires only a new `StoreRouter` implementation, not a scan for direct connection usage throughout the billing and audit paths. An integration test (`TestBillingAuditRoutedThroughStoreRouter`) MUST verify that billing event inserts and audit log inserts call `StoreRouter` methods rather than accessing a Postgres pool directly.

**Read replicas:** The gateway should use separate connection strings for read and write traffic. Read-heavy queries (session status, task tree, audit reads, usage reports) should be routed to read replicas. Most managed services provide a reader endpoint for this purpose. Write traffic goes to the primary only.

**RPO/RTO targets:**

- RPO: 0 (synchronous replication — no committed transaction lost)
- RTO: < 30s (automatic failover). This is the raw Postgres failover window. Capacity planning for downstream buffers that must survive a failover — notably the billing Redis stream — MUST budget for the outage-plus-recovery envelope (failover RTO + `XAUTOCLAIM` reclaim delay + flush catch-up), not the raw RTO alone. See [§17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) `billingRedisStreamMaxLen` derivation.

**Behavior during Postgres failover:** During the Postgres failover window (up to 30s), Redis remains the primary coordination mechanism and all Redis-backed roles continue normally. If Redis is also unavailable during this window, dual-store unavailability rules apply (see [Section 10.1](10_gateway-internals.md#101-horizontal-scaling), "Dual-store unavailability"). Existing sessions with cached coordination state continue; new sessions and writes requiring Postgres (including billing events — see [Section 11.2.1](11_policy-and-controls.md#1121-billing-event-stream)) are rejected or queued as specified below.

**Write durability categories during failover:**

| Write category                               | Durability during failover                                                                                  | Recovery mechanism                                                                                                                                      |
| -------------------------------------------- | ----------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Committed session state transitions          | **Durable** — synchronous replication guarantees no committed transaction is lost (RPO = 0)                 | Automatic; new primary has all committed writes                                                                                                         |
| In-flight (uncommitted) session state writes | **At-risk** — transactions not yet committed at failover time are lost by definition                        | Gateway retries the state transition after reconnecting to the new primary; `coordination_generation` ([Section 10.1](10_gateway-internals.md#101-horizontal-scaling)) prevents stale or duplicate writes |
| Billing events in write-ahead buffer         | **Durable (Redis available) / At-risk (Redis also down)** — events are first staged to Redis stream (`t:{tenant_id}:billing:stream`) which survives individual gateway crashes; in-memory write-ahead buffer is only used when Redis is also unavailable | Redis stream flushed to Postgres by background goroutine on recovery. In-memory buffer events reconstructed from pod-reported token usage ([Section 7.3](07_session-lifecycle.md#73-retry-and-resume)). See maximum billing gap below. |
| Quota counters (Redis-primary)               | **Unaffected** — Redis continues independently of Postgres                                                  | Postgres checkpoint (every `quotaSyncIntervalSeconds`, default 30s) provides durable fallback                                                           |
| Coordination leases (Redis-primary)          | **Unaffected** — Redis continues independently of Postgres                                                  | Postgres advisory lock fallback if Redis also fails                                                                                                     |
| Audit events (T3/T4) in flush buffer         | **Durable** — synchronous writes by default; no in-memory buffer for T3/T4 events ([Section 12.3](#123-postgres-ha-requirements))           | No recovery needed; synchronous writes are committed before the gateway processes the next request on the audit write path                              |
| Audit events (T2) in flush buffer            | **At-risk if batching enabled** — in-memory buffer lost on gateway crash when `audit.batchingEnabled: true` | T2 events not yet flushed to Postgres are permanently lost — the SIEM outbox forwarder reads from Postgres, so events absent from Postgres are absent from SIEM too ([Section 12.3](#123-postgres-ha-requirements)). Batching is opt-in and disabled by default. |

**Maximum billing gap:** With the Redis stream intermediate buffer, the billing gap scenario is significantly narrowed. The worst-case permanent billing gap now requires: a gateway replica crashes **simultaneously** with both a Postgres failover **and** Redis unavailability, while its in-memory write-ahead buffer (Tier 2 fallback) is non-empty and the pods it coordinated also fail (preventing reconstruction from pod-reported usage). This triple-failure scenario (gateway crash + Postgres failover + Redis outage + pod loss) is the only path to permanent billing data loss. Under the common failure case (Postgres failover with Redis available), all billing events are durably staged to the Redis stream and flushed to Postgres on recovery — zero billing loss. Under the Redis stream path, bounded gap per replica is: `billingFlushIntervalMs` (default 500ms) worth of events that were in-flight between `XADD` to Redis and `XDEL` after Postgres confirmation at the moment of the simultaneous crash — at most ~30 events (60 events/s × 500ms). Under the in-memory-only fallback path (Redis also unavailable), the bounded gap is still ~30 events for the same reasons. If coordinated pods survive (the common case), pod-reported cumulative totals allow full reconstruction with zero billing loss regardless of which buffer tier was active.

**Encryption at rest:** Postgres storage must be encrypted at rest. Managed services (RDS, Cloud SQL, Azure Database) provide this by default via volume-level encryption — verify it is enabled. Self-managed deployments must use LUKS-encrypted volumes or dm-crypt for the Postgres data directory. All WAL archives and base backups must also be encrypted: managed services encrypt backups automatically when storage encryption is enabled; self-managed deployments must use server-side encryption on the backup target (e.g., SSE-S3/SSE-KMS for MinIO/S3 backup destinations) or client-side encryption (e.g., `gpg` or age) before upload.

**Backups:** Daily base backups + continuous WAL archival. Restore tested monthly (see [Section 17.3](17_deployment-topology.md#173-disaster-recovery) for the `lenny-restore-test` CronJob procedure). All backup and WAL archive storage must be encrypted at rest as specified above.

### 12.4 Redis HA and Failure Modes

**Minimum topology:** For self-managed deployments: Redis Sentinel (3 sentinels, 1 primary + 1 replica). Redis Cluster if sharding is needed at Tier 3. For cloud-managed deployments: use the provider's managed cache service (AWS ElastiCache, GCP Memorystore, Azure Cache for Redis) which provides equivalent HA, replication, and scaling — see [Section 17.9](17_deployment-topology.md#179-deployment-answer-files) for backend guidance. See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier Redis topology recommendations.

**Tenant key isolation:** All Redis keys **must** use the prefix `t:{tenant_id}:` followed by the storage role and logical key. The following table lists all canonical key prefix patterns in use:

| Key prefix pattern | Role | Notes |

The `{root_session_id}` hash tag distributes different delegation trees across Redis Cluster slots while co-locating all keys for one tree on the same slot (required for multi-key Lua atomicity in `budget_reserve.lua` / `budget_return.lua`). Using `{tenant_id}` would create a hot slot for tenants with many concurrent trees. Tenant isolation is enforced at the gateway application layer: the gateway validates that `root_session_id` belongs to the calling tenant (via `SessionStore` under RLS) before invoking any delegation budget Lua script. Tree-wide keys (`dlg:tokens`, `dlg:tree_size`, `dlg:tree_memory`) are shared across the entire delegation tree. Per-parent keys (`dlg:parallel_children:{parent_session_id}`, `dlg:children_total:{parent_session_id}`) are scoped to each delegating parent within the tree, enabling per-node `maxParallelChildren` and `maxChildrenTotal` enforcement. DLQ keys and billing stream keys follow the same tenant prefix convention. The `lenny:pod:{pod_id}:*` keys (`active_slots`, `rehydrated`, `rehydrating`), `cb:{name}` circuit breaker keys, and `{root_session_id}:dlg:*` delegation budget keys are the three exception classes to the tenant-prefix rule — pod IDs are cluster-globally unique and pod-scoped, circuit breaker keys are platform-wide operational controls, and delegation budget keys use a tree-scoped hash tag for Redis Cluster slot distribution (see [§8.3](08_recursive-delegation.md#83-delegation-policy-and-lease) R-04 for rationale). This convention is enforced in the Redis wrapper layer; no raw Redis command may be issued without the tenant prefix (or pod prefix for slot counters, or `cb:` prefix for circuit breakers, or `{root_session_id}:dlg:` prefix for delegation budget keys — the wrapper validates the calling tenant owns the `root_session_id` before permitting the operation). An integration test (`TestRedisTenantKeyIsolation`) must verify that operations scoped to one tenant cannot read or mutate keys belonging to another tenant. The test suite **must include DLQ, inbox, semantic cache, and delegation budget key coverage**: (a) a DLQ write for tenant A must not be readable by a DLQ processor scoped to tenant B, and (b) a DLQ processor performing a cross-tenant `ZRANGEBYSCORE` scan must return zero results when the key prefix belongs to a different tenant, and (c) a durable inbox enqueue for tenant A's session must not be visible to a coordinator scoped to tenant B, and (d) a semantic cache write for tenant A must not produce a cache hit for a semantically identical query from tenant B, and (e) a delegation budget `budget_reserve.lua` invocation for a `root_session_id` owned by tenant A must be rejected when the gateway context is scoped to tenant B, and a `budget_return.lua` invocation for the same `root_session_id` must likewise be rejected — verifying the application-layer tenant ownership check that compensates for the absence of a `t:{tenant_id}:` key prefix on delegation budget keys (both scripts must be tested because they have distinct attack profiles: `budget_reserve` steals allocation, `budget_return` inflates available budget), and (f) an EventBus publish on tenant A's `t:{tenant_id}:evt:delegation_tree` channel must not be received by a subscriber scoped to tenant B's channel for the same topic.

**Security:** Redis AUTH (ACLs) and TLS are **required**. Cached access tokens are encrypted before storage in Redis (not stored as plaintext). Tokens are encrypted using AES-256-GCM with a key derived from the Token Service's envelope encryption key; each cached token is stored as `{nonce || ciphertext || tag}`, and the encryption key is rotated alongside the envelope key ([Section 4.9.1](04_system-components.md#491-kms-key-rotation-procedure)).

**Data-at-rest posture:** Redis data is treated as **ephemeral** — every Redis-backed role (leases, quota counters, routing cache, token cache) has a durable fallback or reconstruction path (see failure behavior table below). Redis is not a system of record, and total data loss is recoverable. Nonetheless, sensitive cached values (access tokens, credential lease references) must be encrypted at the application layer before writing to Redis, as specified above. Volume-level encryption for Redis persistence files (RDB/AOF) is recommended for defense-in-depth but is not a substitute for app-layer encryption of sensitive fields.

**Failure behavior per use case:**

| Use Case                   | On Redis Unavailability                                                                                                      |
| -------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| Rate limit counters        | **Fail open with bounded window** — allow requests for up to `rateLimitFailOpenMaxSeconds` (default: 60s), then fail closed  |
| Distributed session leases | **Fall back** to Postgres advisory locks (higher latency)                                                                    |
| Routing cache              | **Fall back** to Postgres lookup                                                                                             |
| Cached access tokens       | **Re-fetch** from TokenStore (Postgres)                                                                                      |
| Quota counters             | **Fail open with per-replica budget ceiling** — enforce conservative per-tenant limits locally, then fail closed (see below) |
| Storage quota counter (`storage_bytes_used`) | **Postgres fallback, then fail closed** — on Redis unavailability, the gateway rehydrates each tenant's `storage_bytes_used` counter from the sum of `artifact_size_bytes` across active artifacts in Postgres (same rehydration path as token quota counters; see [§11.2](11_policy-and-controls.md#112-budgets-and-quotas)). Upload pre-checks and post-upload increments use the Postgres-derived value during the outage window. If Postgres is also unavailable (dual-store outage), storage uploads are rejected with `503` (fail closed) — no bytes are written to MinIO, ensuring the storage quota can never be bypassed due to infrastructure degradation. On Redis recovery, the rehydrated counter is written back to Redis and normal fast-path enforcement resumes. |
| Circuit breakers ([§11.6](11_policy-and-controls.md#116-circuit-breakers))   | **Fail closed (last-known state persists)** — each gateway replica retains the last-known circuit breaker state from its 5-second in-process cache. Open breakers remain enforced; no breaker can transition to closed without a confirmed Redis read. When Redis recovers, replicas re-read all circuit states from Redis and replace the cached snapshot. This ensures operator-initiated safety blocks are never silently lifted during infrastructure degradation. |
| Experiment sticky assignments | **Re-derive from deterministic hash (no data loss)** — the sticky cache is a performance optimization, not a correctness requirement (see [§10.7](10_gateway-internals.md#107-experiment-primitives)). The bucketing algorithm (`HMAC-SHA256(experiment_id, assignment_key)`) is fully deterministic: given the same `(user_id, experiment_id)` pair, `assignVariant` always returns the same variant. On Redis unavailability, the gateway re-computes the assignment from the hash on each session creation instead of reading the cache. On Redis recovery, newly computed assignments are written back to the cache. No experiment statistical validity is affected because assignments are hash-derived, not randomly sampled and stored. |
| Delegation budget counters (`maxTreeSize`, `maxParallelChildren`, `maxTreeMemoryBytes`) | **Fail closed for new delegations** — `delegate_task` calls are rejected with a retryable `DELEGATION_BUDGET_UNAVAILABLE` error during the Redis outage window. Delegation trees are structurally complex to enforce in-memory across replicas, so fail-open is not safe for these counters. `maxDelegationDepth` enforcement is unaffected because it uses the Postgres-backed session lineage (not Redis counters) and remains enforced during Redis outages. In-flight child sessions already delegated before the outage continue running normally; only new `delegate_task` calls are blocked. On Redis recovery, delegation budget counters are reconstructed per the reconciliation procedure below before new delegations are accepted. |
| **Both stores down**       | **Dual-store degraded mode** — new sessions rejected (503), in-flight sessions continue on cached coordination state, `PLATFORM_DEGRADED` event emitted to clients, `DualStoreUnavailable` alert fires. See [Section 10.1](10_gateway-internals.md#101-horizontal-scaling) ("Dual-store unavailability") for the full protocol. |

**Quota counter reconciliation after fail-open:** When Redis recovers, the gateway reconciles quota counters using a two-source MAX rule: for each active session and tenant counter, the gateway reads (1) the last Postgres checkpoint value and (2) the in-memory counter accumulated during the fail-open window on the recovering replica, then writes `MAX(postgres_checkpoint, in_memory_counter)` as the authoritative Redis value. Using the Postgres value alone would reset counters to a value up to `quotaSyncIntervalSeconds` (default 30s) stale — potentially un-enforcing a budget violation that occurred in the fail-open window. The MAX rule ensures that no usage accumulated during the fail-open window is lost. On session completion (terminal state), the gateway writes the final cumulative usage to Postgres as an authoritative reconciliation point, providing a durable baseline for future recoveries. Maximum drift during fail-open is bounded by the window duration (default 60s) multiplied by request rate — at 100 req/s per tenant, worst-case drift is ~6,000 requests. Worst-case drift scales with tier — see [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier estimates. Reconciliation runs automatically when Redis becomes available and completes within seconds. During reconciliation, the gateway falls back to Postgres-backed counters (slower but accurate).

**Delegation budget counter reconciliation on Redis recovery:** On Redis recovery (restart or reconnection), the gateway additionally re-validates all active delegation tree budget counters before resuming delegation operations. The reconciliation procedure follows the two-source reconstruction described in [Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas) (Crash Recovery for Delegation Budget Counters): Postgres checkpoint + live pod enumeration, with irrecoverable trees moved to `awaiting_client_action`. This includes restoring the `maxTreeMemoryBytes` counter via `max(postgres_checkpoint_memory, liveMemoryBytes)` — without this step, a post-recovery counter of zero would allow trees already near their memory cap to accept new delegations they should have been blocked on. New `delegate_task` requests for any tree are rejected with a retryable error during the reconciliation window (typically < 5 seconds). The `lenny_delegation_budget_reconstruction_total` metric (labeled by `outcome`) is emitted for each tree processed, enabling operators to monitor reconstruction volume and detect irrecoverable trees. Operators should add this to the Redis recovery runbook ([Section 17.7](17_deployment-topology.md#177-operational-runbooks)).

**Bounded fail-open for rate limiting:** When Redis becomes unavailable, the gateway starts a fail-open timer per replica. During the fail-open window, requests are allowed, a `rate_limit_degraded` metric is incremented, and an alert fires. After the window expires, if Redis is still unavailable, rate limiting fails **closed** — new requests are rejected with 429 until Redis recovers. **Emergency hard limit:** Each gateway replica maintains an in-memory per-user request counter as a coarse emergency backstop. This counter is not shared across replicas (so the effective limit is `N * per_replica_limit`) but prevents a single user from sending unlimited requests through one replica during the fail-open window. The `rateLimitFailOpenMaxSeconds` is configurable per deployment (default 60s). Higher tiers should use shorter fail-open windows to limit burst damage ([Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)).

**Per-user fail-open ceiling:** The per-tenant fail-open budget (see below) bounds aggregate tenant exposure but does not prevent a single user from monopolizing the entire tenant allocation during a Redis outage. Each gateway replica therefore maintains an additional **in-memory per-user token and request counter** active during the fail-open window. The per-user ceiling is a pure fraction of the effective per-replica ceiling: `per_user_failopen_ceiling = effective_ceiling * userFailOpenFraction`, where `effective_ceiling = min(tenant_limit / max(cached_replica_count, 1), per_replica_hard_cap)` (the same expression defined below under "Per-tenant fail-open budget enforcement") and `userFailOpenFraction` is configurable per tenant (default: 0.25, i.e., a single user cannot consume more than 25% of the tenant's per-replica fail-open allocation). The earlier `min(tenant_limit * userFailOpenFraction, per_replica_hard_cap)` composition is retired: `per_replica_hard_cap` is a per-tenant aggregate bound, not a per-user value, and using it as one operand of the per-user `min()` silently neutralized `userFailOpenFraction` whenever `per_replica_hard_cap` was the smaller operand (e.g., any deployment that lowered `per_replica_hard_cap` or raised `userFailOpenFraction` above 0.5 admitted a single user up to the entire tenant's per-replica budget). Composing the per-user cap as a pure fraction of `effective_ceiling` keeps `userFailOpenFraction` operative regardless of how `per_replica_hard_cap` is tuned. When a user's in-memory counter reaches `per_user_failopen_ceiling`, that user's requests are rejected with 429 for the remainder of the fail-open window, even if the per-tenant counter still has headroom. This prevents monopolization without affecting other users in the same tenant. The `userFailOpenFraction` is configurable via `quotaUserFailOpenFraction` in the gateway Helm values. **Config-time invariant check:** The Helm chart render step and `lenny-ops` startup both validate that `0 < quotaUserFailOpenFraction <= 1.0`; values outside that range are rejected at startup with `CONFIG_INVALID: quotaUserFailOpenFraction (<value>) must satisfy 0 < quotaUserFailOpenFraction <= 1.0`. Additionally, `lenny-ops` emits the `QuotaFailOpenUserFractionInoperative` warning at startup whenever `quotaUserFailOpenFraction >= 0.5` — at that setting the per-user cap is at or above half the tenant's per-replica fail-open budget, so the monopolization-prevention intent of the control is substantially weakened even though the formula itself remains correct. Operators who intentionally raise the fraction above 0.5 should acknowledge the weakened posture in their deployment answer file. Per-user fail-open counters are reset when Redis recovers and counters are reconciled using the standard MAX rule (see "Quota counter reconciliation after fail-open" below).

**Per-tenant fail-open budget enforcement:** In addition to per-user rate limiting, each gateway replica maintains in-memory per-tenant token and request counters that are active during Redis unavailability. Each replica also maintains a `cached_replica_count` variable (in-memory, not in Redis) that is updated every time the Kubernetes Endpoints object is successfully polled; this variable persists across individual poll failures so it always reflects the last-known good replica count. The per-replica ceiling for each tenant is `per_replica_limit = tenant_limit / max(cached_replica_count, 1)`, where `cached_replica_count` is the last successfully observed replica count sourced from the Kubernetes Endpoints object for the gateway Service (polled via the API server, not from Redis), with a maximum staleness of 30 seconds. If the Endpoints object has never been successfully polled (cold start), `cached_replica_count` defaults to `1`. A dual-outage (Redis + Endpoints simultaneously unavailable) will use the most-recently cached count rather than defaulting to `1`, preventing the `N × tenant_limit` overshoot that would otherwise occur if all replicas independently fell back to a count of 1. To further bound worst-case exposure, a hard per-replica cap applies regardless of replica count: `effective_ceiling = min(tenant_limit / max(cached_replica_count, 1), per_replica_hard_cap)`, where `per_replica_hard_cap` defaults to `tenant_limit / 2` and is configurable via `quotaPerReplicaHardCap`. Once a tenant's local counter reaches `effective_ceiling`, the replica rejects further requests for that tenant with 429 until Redis recovers. This ensures that even in the worst dual-outage case, total cluster-wide usage cannot exceed `min(N × (tenant_limit / max(cached_replica_count, 1)), N × per_replica_hard_cap)` — bounded well below `N × tenant_limit`. **Cumulative fail-open timer (security control):** Each replica tracks total cumulative time spent in fail-open mode via a true sliding window counter stored in-memory (not in Redis). The counter accumulates time across all Redis outages within the window — five separate 59-second outages each add 59 seconds to the running total, not just the most recent. If cumulative fail-open time exceeds `quotaFailOpenCumulativeMaxSeconds` (default: 300s) within any rolling 1-hour window (not calendar-aligned), the replica transitions to **fail-closed for quota enforcement** — all new sessions and token-consuming requests are blocked until Redis recovers and counters are reconciled. `quotaFailOpenCumulativeMaxSeconds` is a **financial security control**: its default (300s) should be reviewed per deployment's financial risk exposure and set to the maximum acceptable quota overshoot window. On replica restart, the cumulative timer is initialized from a local file (`/run/lenny/failopen-cumulative.json`) that each replica persists on every fail-open state transition. If the file exists and its timestamp is within the rolling 1-hour window, the cumulative counter resumes from the persisted value rather than resetting to zero. If the file is missing or corrupted (e.g., fresh pod on a new node), the timer resets to zero — this is the cold-start case and is acceptable because the replica has no prior fail-open history on this pod. This prevents the bypass scenario where a replica in CrashLoopBackOff during a sustained Redis outage would reset the cumulative timer on every restart and never reach the `quotaFailOpenCumulativeMaxSeconds` threshold. When a replica enters fail-open mode, it emits a `quota_failopen_started` audit event (catalogued in [§16.7](16_observability.md#167-section-25-audit-events)) containing `tenant_id`, `service_instance_id` (the OTel attribute identifying the gateway replica — see [§16.1.1](16_observability.md#1611-attribute-naming)), and `timestamp`, enabling billing consumers to detect and attribute overshoot windows. The gateway also emits `lenny_quota_failopen_cumulative_seconds` (gauge, catalogued in [§16.1](16_observability.md#161-metrics)) and fires alert `QuotaFailOpenCumulativeThreshold` (catalogued in [§16.5](16_observability.md#165-alerting-rules-and-slos)) when the cumulative timer exceeds 80% of the configured maximum — the pre-breach warning issued before the replica transitions to fail-closed.

**Scalability ceiling of single-instance Redis:** At Tiers 1 and 2, a single Redis Sentinel topology handles all Redis-backed concerns: session leases, quota counters, routing cache, token cache, and event pub/sub. This is simple to operate but creates a shared scalability ceiling — all concerns compete for the same CPU, memory, and network bandwidth on one primary node. Sentinel provides HA (automatic failover) but does not provide horizontal throughput scaling.

**When Sentinel becomes insufficient:** Monitor the following signals to determine when the single Sentinel topology is approaching its ceiling:

- **CPU saturation:** Redis primary sustained above 70% CPU (single-threaded; 70% of one core leaves limited headroom for latency-sensitive operations like lease renewals).
- **Memory pressure:** Used memory exceeds 75% of `maxmemory`, causing eviction risk for cached tokens and routing entries.
- **Operation latency:** P99 latency for `LeaseStore` or `QuotaStore` operations exceeds 5ms (baseline should be sub-1ms). Lease renewals are latency-critical — elevated latency risks false lease expirations.
- **Pub/sub fan-out cost:** At high session counts, pub/sub message delivery competes with key-value operations. If `redis_pubsub_channels` exceeds 5,000 and `redis_connected_subscribers` exceeds 10,000, pub/sub overhead becomes measurable.
- **Operations rate:** Approaching 80% of the budget operations estimate for the current tier ([Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)).
- **Delegation Lua aggregate blocking:** `lenny_redis_lua_script_duration_seconds{script="budget_reserve"}` P99 exceeds 2 ms sustained, or the cluster-wide instantaneous `delegate_task` rate exceeds 50/s — either condition indicates that cross-tenant `budget_reserve.lua` serialization is approaching the 5 ms `LeaseStore` SLO ceiling. See [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease) ("Cross-tenant aggregate Lua contention") for the aggregate blocking formula and dedicated instance remediation steps.

**Logical separation of Redis concerns:** When the signals above indicate the single topology is insufficient (typically at upper Tier 2 or Tier 3 scale), separate Redis into independent instances by concern:

| Redis Instance            | Backed Concerns                                     | Scaling Rationale                                                                                      |
| ------------------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **Coordination**          | `LeaseStore` (session leases), routing cache        | Latency-critical, low throughput, small key space. Sentinel topology is sufficient even at Tier 3.     |
| **Quota / Rate Limiting** | `QuotaStore` (rate limit counters, budget tracking) | High write throughput (per-request increments). Benefits most from Redis Cluster sharding at scale.    |
| **Cache / Pub-Sub**       | Token cache, event pub/sub                          | Tolerates higher latency. Pub/sub fan-out is CPU-intensive and should not compete with lease renewals. |

At Tiers 1 and 2, all concerns run on a single Sentinel instance. The separation is a deployment-time configuration change (separate connection strings per store role) — no code changes are required because each store role already has its own interface ([Section 12.6](#126-interface-design)). See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier Redis topology recommendations including when to split.

**Tier 3 Redis write throughput quantification:** At Tier 3 scale (~10,000 concurrent sessions, 200 new sessions/s sustained — see [§16.5](16_observability.md#165-alerting-rules-and-slos) line "Session creation rate (sustained)"), the estimated write load on the Quota/Rate Limiting Redis instance is the dominant concern. Note that only lease renewals scale with session count; the quota and rate-limit `INCR` sources below scale with gateway→pod RPC rate and inbound Gateway RPS respectively (see [§16.5](16_observability.md#165-alerting-rules-and-slos) "Gateway RPS (all endpoints)" — ~50,000/s at Tier 3), not with session creation. The principal write sources and their rates are:

| Write Source                            | Estimated ops/s (Tier 3) | Notes                                                                         |
| --------------------------------------- | ------------------------ | ----------------------------------------------------------------------------- |
| Quota counter increments (`INCR`)       | ~2,000/s aggregate       | Scales with **gateway→pod RPC rate** (active sessions × turn rate), not session creation; one `INCR` per RPC. At Tier 3 with ~10,000 sessions and typical turn cadence this is the dominant write source but is bounded well below Gateway RPS because not every inbound request produces a pod RPC. |
| Rate limit counter increments           | ~2,000/s                 | Scales with **Gateway RPS** (~50,000/s at Tier 3, [§16.5](16_observability.md#165-alerting-rules-and-slos)); one `INCR` per inbound request per tenant. Sampled/sliding-window implementations bring the observed write rate below raw RPS — the ~2,000/s figure reflects that amortization. |
| Delegation budget decrements (Lua)      | ~500–1,000/s             | Multi-tenant aggregate Redis ops/s (not per-script invocation rate — each Lua invocation executes ~10–15 internal Redis commands; see [§16.5](16_observability.md#165-alerting-rules-and-slos) reconciliation). At burst rates ≥ 50 concurrent scripts, aggregate serialization blocking exceeds 5 ms — see cross-tenant contention analysis in [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease). |
| Lease renewals (`SET` with TTL)         | ~1,000/s                 | Per-session lease heartbeat; one per coordination TTL / 2 — this is the only row that scales directly with concurrent session count. |
| Token cache writes (`SET NX`)           | ~500/s                   | Cold-miss re-fetches from TokenStore                                           |
| **Total estimated write ops/s**         | **~6,000–6,500/s sustained** | Burst may exceed ~20,000/s when Gateway RPS spikes (rate-limit `INCR`) or when orchestrator sessions fan out delegations concurrently (Lua serialization); session creation storms alone do not drive this burst because lease renewal cost grows only linearly with steady-state session count. |

A single-threaded Redis primary can sustain ~100,000–200,000 simple ops/s on modern hardware. The ~6,500/s sustained Tier 3 write rate is within a Redis Cluster topology's capacity but **is NOT trivially safe on a single Sentinel primary** at Tier 3 scale — CPU contention from concurrent read operations, pub/sub fan-out, and Lua script serialization ([Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease)) combine with write throughput to approach single-primary limits. The per-row corrections above do not change the sustained total (~6,000–6,500 ops/s) or the "Sentinel appropriate only through Tier 2" threshold — they refine which workload dimensions each row scales against. The Tier 3 Redis Cluster topology recommendation in [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) is required for sustained Tier 3 write rates; Sentinel is appropriate only through Tier 2. The `~100,000 ops/s` budget estimate in [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) accounts for total operation mix (reads + writes + pub/sub); the ceiling signals in the **"When Sentinel becomes insufficient"** block above are the operational decision criteria.

**Redis Cluster migration pre-plan (Tier 2→3 transition):** For deployments targeting Tier 3 scale, Redis Cluster migration for the Quota/Rate Limiting instance must be planned **before** the Tier 2→3 promotion decision — not reactively once ceiling signals have already fired in production. The migration procedure is:

1. **Trigger:** Initiate planning when any two ceiling signals (CPU > 70%, P99 latency > 5ms, ops rate > 80% of budget) are simultaneously active for more than 30 minutes at Tier 2 peak load.
2. **Topology target:** Redis Cluster with minimum 6 nodes (3 primary + 3 replica shards), using hash-slot-based sharding. The `t:{tenant_id}:` key prefix convention ([Section 12.4](#124-redis-ha-and-failure-modes)) ensures all keys for a given tenant hash to the same slot, preserving multi-key operation correctness for per-tenant Lua scripts.
3. **Interface compatibility:** No application code changes are required. Each store role interface ([Section 12.6](#126-interface-design)) already abstracts the Redis client. The `QuotaStore` and rate limiter implementations must use `CLUSTER KEYSLOT`-aware client libraries (e.g., `go-redis/v9` with cluster mode) — verify this before migration.
4. **Migration steps:** (a) Deploy Cluster in parallel with the existing Sentinel instance; (b) migrate tenants in groups using a feature flag (`redis.quotaStoreBackend: sentinel | cluster`) settable per-tenant; (c) validate quota and rate limit correctness for migrated tenants for 24h before proceeding; (d) migrate remaining tenants; (e) decommission Sentinel instance for Quota/Rate Limiting. The Coordination instance remains on Sentinel throughout — Cluster is not required for low-throughput coordination workloads.
5. **Rollback:** The per-tenant feature flag allows individual rollback to Sentinel if correctness issues are observed. Any quota counters that diverged between the two stores during the migration window must be reconciled by re-reading from Postgres (same procedure as fail-open reconciliation above).

**In-memory quota budgets with Postgres reconciliation:** For high-value limits (e.g., per-tenant token budgets where overshoot has direct cost impact), each gateway replica can maintain an in-memory budget allocation drawn from Postgres rather than relying solely on Redis counters. The replica requests a budget slice from Postgres on startup (e.g., 1/N of the tenant's remaining budget, where N is the replica count), decrements locally per request, and reconciles with Postgres periodically (default: every 30s) or when the local slice is 80% consumed. This approach tolerates full Redis unavailability for quota enforcement with bounded overshoot (at most one slice per replica). It is **not the default** — deployers enable it via `quotaEnforcementMode: in_memory_reconciled` when Redis-based quota drift during outages is unacceptable.

### 12.5 Artifact Store

**Backend:** Any S3-compatible object store. Cloud-managed deployments should use the provider's native object storage (AWS S3, GCP Cloud Storage, Azure Blob Storage) — see [Section 17.9](17_deployment-topology.md#179-deployment-answer-files) for the answer-file catalog. Self-managed deployments use MinIO. For local development, use local disk with the same interface.

**HA topology requirements:**

- **Production topology:** MinIO with erasure coding (minimum 4 nodes, recommended 4+ nodes across availability zones) for data durability. See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier MinIO sizing.
- **Versioning:** Enable bucket versioning for checkpoint objects to prevent accidental overwrites
- **Replication:** For multi-zone deployments, configure MinIO site-to-site replication for near-zero RPO on artifact data
- **TLS:** MinIO connections MUST use TLS (`https://` endpoint). The Helm chart sets `minio.endpoint` to `https://minio.lenny-system:9000` by default. TLS may be disabled in local development mode via `minio.tls.enabled: false` in Helm values; this value defaults to `false` only when `backends: embedded` (the `lenny up` Embedded Mode, [§17.4](17_deployment-topology.md#174-local-development-mode-lenny-dev)). All other backends enforce TLS.
- **Delete-marker lifecycle:** Bucket versioning is bucket-wide, and the GC job (below) deletes artifacts across all object prefixes — not just checkpoints. Each GC deletion on a versioned bucket creates a delete marker. Therefore, the delete-marker expiration and noncurrent-version expiration lifecycle rules MUST apply bucket-wide (not scoped to a single prefix). The rules are: expire delete markers after 24 hours, and expire noncurrent versions after 1 day (`NoncurrentVersionExpiration: Days: 1`). This prevents delete-marker accumulation from degrading `ListObjects` performance at scale across all prefixes (checkpoints, uploads, transcripts, eviction context, workspace snapshots). The Helm chart post-install Job configures this rule via `mc ilm add` with no prefix filter (bucket-wide scope).

**MinIO unavailability during checkpoint:** If MinIO is unreachable during a checkpoint upload, the adapter retries with exponential backoff as described in [Section 4.4](04_system-components.md#44-event--checkpoint-store). Deployers should monitor the `CheckpointStorageUnavailable` critical alert and the `lenny_checkpoint_storage_failure_total` metric. Prolonged MinIO unavailability degrades session durability — any pod eviction during an outage that exhausts all MinIO retries falls back to the Postgres minimal state record ([Section 4.4](04_system-components.md#44-event--checkpoint-store)); sessions resume with conversation context but workspace state is lost.

**Risk: simultaneous MinIO outage and node eviction.** A MinIO rolling upgrade or network partition coinciding with a node drain causes simultaneous preStop hooks across all pods on the drained node, all competing for an unavailable MinIO, all falling back to minimal state. This is a known cascading failure scenario. Mitigations: (1) the `CheckpointStale` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires before eviction if the freshness SLO is already missed, giving operators an early signal; (2) deployers should stagger node drains — drain one node at a time (use `kubectl drain --max-unavailable 1` or cluster-autoscaler drain rate limits) to limit the number of simultaneous eviction checkpoints; (3) schedule MinIO maintenance windows outside of cluster upgrade windows to avoid the combination. The Postgres minimal state fallback is the last-resort safety net, not a substitute for operational discipline around MinIO availability.

**Pre-drain MinIO health check (planned drains only).** This webhook protects against planned drains (operator-initiated `kubectl drain`, cluster-autoscaler scale-downs via the eviction API) only. It cannot protect against spontaneous node failures, OOM kills, or preemptions, which bypass the eviction API entirely. For spontaneous failures, the `CheckpointStale` alert and the periodic checkpoint freshness SLO ([Section 4.4](04_system-components.md#44-event--checkpoint-store)) are the primary defenses against data loss. To prevent draining a node into a known-degraded MinIO state, the gateway exposes a **drain readiness endpoint**: `GET /internal/drain-readiness`. Before any operator or automation initiates a `kubectl drain` on a node, a pre-drain hook MUST query this endpoint. The endpoint performs a MinIO liveness probe (S3 `HeadBucket` on the checkpoints bucket with a 2-second timeout) and returns:

- `HTTP 200 {"status": "ready", "minio": "healthy"}` — drain may proceed.
- `HTTP 503 {"status": "not_ready", "minio": "unhealthy", "reason": "<error>"}` — drain MUST be deferred until MinIO recovers.

The pre-drain hook is implemented as a Kubernetes `ValidatingAdmissionWebhook` named `lenny-drain-readiness` that intercepts `CREATE` operations on pod eviction objects (resource: `pods/eviction`) in the agent namespaces. When the webhook fires, it calls the gateway's `/internal/drain-readiness` endpoint. If the endpoint returns `503` or is unreachable, the webhook rejects the eviction with: `"Node drain blocked: MinIO health check failed — defer drain until MinIO is healthy (STR-006)"`. The webhook is deployed with `failurePolicy: Fail` so that webhook unavailability also blocks drain, preventing silent data loss paths. For cluster-autoscaler-triggered drains, the autoscaler's eviction flow passes through the same Kubernetes eviction API and is therefore subject to the same webhook. Operators can bypass the check for emergency drains by annotating the node with `lenny.dev/drain-force: "true"` and providing justification — the webhook permits the drain but emits a `node.drain.forced` critical audit event. Metric: `lenny_drain_readiness_checks_total` (counter, labeled by `outcome`: `allowed`, `blocked`, `forced`) tracks drain check outcomes.

**Do not use Postgres for blob storage.** Workspace checkpoints (up to 500MB) cause TOAST overhead, vacuum pressure, and degrade transactional workload performance.

**Tenant isolation:** All object keys use the path format `/{tenant_id}/{object_type}/{session_id}/{filename}`. The `ArtifactStore` implementation validates the `tenant_id` prefix on every operation — no S3 call is issued unless the caller's authenticated `tenant_id` matches the path prefix. This validation lives at the interface boundary, not in individual callers. `DeleteByTenant(tenant_id)` performs a prefix-scoped bulk delete on `/{tenant_id}/*`. The GC job (below) inherits the same prefix scoping: it only deletes artifacts under the tenant prefix associated with each expired record.

**Encryption at rest:** All stored objects (checkpoints, workspace snapshots, session transcripts) contain sensitive data including conversation history and workspace file contents. MinIO server-side encryption (SSE-S3 or SSE-KMS) must be enabled for production deployments. For cloud deployments using managed object storage (S3, GCS, Azure Blob), encryption at rest is typically enabled by default. For self-hosted MinIO, enable SSE with a KMS backend (HashiCorp Vault, AWS KMS) or at minimum SSE-S3 with MinIO's internal key management. **T4 tenant encryption:** For tenants with `workspaceTier: T4` ([Section 12.9](#129-data-classification)), workspace snapshots and session transcripts in object storage **must** use SSE-KMS with a tenant-specific KMS key — not the default shared encryption key. This enables per-tenant key rotation and cryptographic erasure (revoking the tenant's KMS key renders all their stored artifacts unreadable). The `ArtifactStore` implementation selects the encryption key based on the tenant's `workspaceTier`: T3 uses the deployment-wide SSE key; T4 uses a KMS key scoped to `tenant:{tenant_id}`. Key provisioning is the deployer's responsibility (e.g., AWS KMS key per tenant, HashiCorp Vault transit key per tenant). The preflight check ([Section 17.6](17_deployment-topology.md#176-packaging-and-installation)) validates that MinIO SSE is enabled but does not validate per-tenant KMS key existence — this is validated at runtime when the first T4 artifact is written, returning `CLASSIFICATION_CONTROL_VIOLATION` if the tenant-scoped KMS key is unavailable.

**T4 per-tenant KMS key lifecycle.** The spec mandates pre-provisioning to eliminate the race between T4 promotion and first artifact write:

1. **Pre-provisioning model (required).** Operators MUST provision the tenant-specific KMS key (`tenant:{tenant_id}` scope — AWS KMS alias, GCP Cloud KMS resource name, or HashiCorp Vault transit key) **before** setting `workspaceTier: T4` on the tenant. On-demand KMS key creation is not supported in v1. The admin API endpoint that sets `workspaceTier: T4` (`PUT /v1/admin/tenants/{id}`) performs an online KMS availability probe (a zero-byte encrypt/decrypt round-trip against the tenant-scoped key) before persisting the change — if the probe fails, the update is rejected with `CLASSIFICATION_CONTROL_VIOLATION` and the tenant remains at its prior tier. This turns the just-in-time runtime failure into an admin-time failure with no data-loss exposure, and removes the eventually-consistent KMS timing window: by the time the tenant is marked T4, the key has already been observed to be usable from the gateway's credentials. The deletion side of this lifecycle (Phase 4a of tenant deletion — `ScheduleKeyDeletion` / `DestroyCryptoKeyVersion` / `DELETE /transit/keys/{tenant_id}`) is defined in [Section 12.8](#128-compliance-interfaces) Tenant deletion lifecycle.

2. **Failure behavior when the KMS key is unavailable at write time.** If the tenant-scoped KMS key is unavailable during a checkpoint or artifact write — for example the operator disabled or deleted it out-of-band after the tenant was marked T4, or the KMS backend is experiencing a transient outage — the gateway MUST reject the write with `CLASSIFICATION_CONTROL_VIOLATION`. The `ArtifactStore` does **not** fall back to the deployment-wide SSE key, because that would silently downgrade the tenant's classification controls and violate the cryptographic-erasure property. For checkpoint writes specifically, this propagates through the `CheckpointStorageUnavailable` critical alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) and the `lenny_checkpoint_storage_failure_total` metric (with `reason="kms_unavailable"`). The session's eviction path falls back to the Postgres minimal-state record as documented in [Section 4.4](04_system-components.md#44-event--checkpoint-store), but durable artifact writes stay blocked until the operator restores KMS key availability. The operator restoration path is: (a) re-enable or restore the `tenant:{tenant_id}` KMS key through the provider's native console or API, (b) re-run the online KMS availability probe by issuing `PUT /v1/admin/tenants/{id}` with the same `workspaceTier: T4` value — an idempotent update that forces the admin-time probe to re-run and returns `200 OK` when the key is reachable again, and (c) the next checkpoint attempt succeeds without any further gateway action — no retry queue, no backfill job. Writes rejected during the outage are lost; operators who require stronger durability during KMS outages must operate KMS in a highly available configuration or run with `workspaceTier: T3`.

3. **Key rotation procedure.** KMS key rotation for T4 tenants is **operator-managed** — the platform does not trigger or schedule rotations. The standard provider-managed rotation semantics apply: AWS KMS automatic key rotation produces a new key material version while retaining older versions for decrypt-only use; GCP Cloud KMS produces new `CryptoKeyVersion`s with the same behavior; HashiCorp Vault transit keys rotate in place and retain prior versions. MinIO's SSE-KMS integration records the key version used at encrypt time on each object and continues to read objects encrypted under prior key versions after rotation — no re-encryption pass is required and no application-layer coordination is needed. Operators SHOULD coordinate any **manual** rotation (e.g., deliberate re-keying for compromise response) with artifact write quiescence to avoid a narrow race in which a write is accepted under the old key version immediately before the old version is disabled; the recommended sequence is: (a) pause new session starts for the tenant via `POST /v1/admin/tenants/{id}/suspend` (which returns `TENANT_SUSPENDED` for new writes — see [Section 15.1](15_external-api-surface.md#151-rest-api)), (b) wait for in-flight checkpoints to drain (observable via `lenny_checkpoint_duration_seconds` falling to zero for the tenant's pools), (c) perform the rotation in the KMS provider, (d) verify reachability with an idempotent `PUT /v1/admin/tenants/{id}` re-assert of `workspaceTier: T4` (the admin-time KMS availability probe re-runs), (e) resume the tenant. Automatic rotations provided by the KMS provider require no quiescence because both old and new key versions remain usable throughout the rotation. The rotation event itself is outside the platform's audit trail (it happens in the KMS provider); operators who require a unified audit trail are responsible for forwarding their KMS provider's key-rotation events into their SIEM alongside the Lenny audit stream.

4. **Continuous probe (STO-021).** The admin-time probe in bullet 1 protects against misconfiguration at tier-transition time and the idempotent re-assert path in bullet 2 protects recovery after an operator-observed outage, but neither signal detects silent post-provisioning lifecycle drift on a tenant that has gone idle (operator disables or revokes the key out-of-band; provider-side key deletion scheduled inadvertently; IAM policy drift that strips the gateway's `kms:Encrypt`/`Decrypt` grant). The gateway therefore runs a **leader-elected background probe** that re-executes the same zero-byte encrypt/decrypt round-trip against every T4 tenant's `tenant:{tenant_id}` KMS key on a fixed cadence, independent of any checkpoint or admin-API activity, making the usable-key state continuously observable. The probe is owned by the gateway process — it runs as a goroutine under the same `lenny-gateway-leader` Lease described in the GC concurrency model above; exactly one gateway replica probes at a time, failover bound matches the GC failover bound. Cadence is configured by `storage.t4KmsProbeInterval` (default **300s** / 5 min); the interval is bounded below by a minimum of 60s enforced at Helm-validate to prevent excessive KMS API spend. On each cycle the leader enumerates tenants with `workspaceTier: T4`, issues the probe for each, and updates two metrics: the `lenny_t4_kms_probe_last_success_timestamp{tenant_id}` gauge is set to `now()` on success, and `lenny_t4_kms_probe_result_total{outcome}` (where `outcome ∈ {success, failure}`) is incremented once per probe. The enumeration is rate-limited via a token bucket at `storage.t4KmsProbeRateLimit` (default **10 probes/sec**) so that a tenant fleet of 10 000 T4 tenants does not burst the KMS backend; under rate-limit pressure the sweep completes asynchronously (it may take longer than `t4KmsProbeInterval` under very high tenant counts, but the rate ceiling is preferred over backend overload). Probe failures do **not** auto-downgrade the tenant from T4 to T3 — that would be a silent classification change; the platform only makes the state observable, and the operator decides whether to re-enable the key, rotate to a new key, or accept the fail-closed write path. The `T4KmsKeyUnusable` critical alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when `now() - lenny_t4_kms_probe_last_success_timestamp{tenant_id=$t} > 2 * storage.t4KmsProbeInterval` for any T4 tenant, catching idle tenants before a checkpoint-path `CheckpointStorageUnavailable` cascade occurs (paired with `lenny_checkpoint_storage_failure_total{reason="kms_unavailable"}` in bullet 2). The continuous probe is a **pure observability addition** — it does not change the fail-closed write semantics in bullet 2, the admin-time probe in bullet 1, or the operator restoration procedure; a tenant with `lenny_t4_kms_probe_last_success_timestamp` flatlined at a stale value is operating in the same fail-closed posture that bullet 2 already defines, and the metric is what tells the operator that they are in that posture. Operators also see `t4KmsLastProbeSuccessAt` as a response field on `GET /v1/admin/tenants/{id}` ([Section 15.1](15_external-api-surface.md#151-rest-api)) for ad-hoc inspection without a metric lookup. The probe costs one `kms:Encrypt` + one `kms:Decrypt` per T4 tenant per interval against the provider's standard API-call billing; at the defaults (5-minute interval, 10/s rate ceiling) the upper bound is 12 probes/tenant/hour.

**Storage quota enforcement.** Per-tenant storage quotas are enforced via a pre-upload size check, post-upload Redis counter increment, and GC-triggered decrement, as defined in [Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas) (Storage quota enforcement mechanism). The `artifact_store` Postgres table stores `artifact_size_bytes` for every artifact; the GC job uses this to decrement the Redis counter on deletion and to rehydrate counters on Redis restart. This table covers **all** MinIO-backed artifacts, including workspace snapshots, checkpoints, uploaded files, session transcripts, and eviction context objects (see [Section 4.4](04_system-components.md#44-event--checkpoint-store) for the `artifact_store` insert requirement for the context > 2KB path). The `lenny_storage_quota_bytes_used` gauge tracks per-tenant consumption. Alert: `StorageQuotaHigh` fires at 80% utilization.

**Checkpoint retention policy:**

- Keep only the latest 2 checkpoints per active session. **Exception: sessions with `legal_hold = true` are exempt from this rotation — all checkpoints are retained until the hold is lifted (see [Section 12.8](#128-compliance-interfaces) Legal hold and checkpoint rotation).** In concurrent-workspace mode ([Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes)), checkpoints are per-slot — the "latest 2" limit applies independently to each slot, meaning a pod with `maxConcurrent: 8` may retain up to 16 checkpoint objects (8 slots × 2). The GC job and retention policy operate on `(session_id, slot_id)` pairs, not on sessions alone.
- Delete all checkpoints when session terminates and resume window expires
- **Eviction context objects:** When a session reaches a terminal state, the gateway deletes the MinIO eviction context object (`/{tenant_id}/eviction/{session_id}/context`) as part of terminal-state cleanup ([Section 4.4](04_system-components.md#44-event--checkpoint-store)). Any objects that escape this primary deletion path (e.g., gateway crash before cleanup) are picked up by the GC sweep: the GC job queries `session_eviction_state` rows for sessions that have reached a terminal state and whose `last_message_context` column stores a MinIO object key (identified by the `/{tenant_id}/eviction/` key prefix), deletes the corresponding MinIO objects, and removes the Postgres rows. This GC path runs within the same cycle as other artifact cleanup.
- **Partial checkpoint manifests:** Partial checkpoint manifest records (`partial: true` in the checkpoint metadata table) are normally soft-deleted immediately after resume completes ([Section 4.4](04_system-components.md#44-event--checkpoint-store)). Additionally, the gateway supersedes any prior active partial manifest for the same `(session_id, slot_id)` in the same transaction that writes a new one ([Section 10.1](10_gateway-internals.md#101-horizontal-scaling) Supersede-on-write for repeated drain-timeouts), and the `partial_manifest_active_uniq` partial unique index enforces the at-most-one-active invariant at the database layer. The intent-row-first ordering ([§10.1](10_gateway-internals.md#101-horizontal-scaling) Intent-row-first ordering) guarantees that every chunk object has a corresponding Postgres row from the moment the first chunk becomes uploadable — no `PutObject` call is ever issued before the manifest row has committed, so the `partial = true AND deleted_at IS NULL` sweep predicate below is sufficient to discover every committed chunk object even when the gateway crashes mid-upload; the previous ordering (chunks committed before the manifest row) could produce orphaned chunks under a `checkpoint_id` that no row referenced and is no longer permitted. With these three primary-path defenses in place (supersede-on-write, partial unique index, intent-row-first ordering), the GC sweep below is a **true backstop** — a rare defense-in-depth path for manifests whose primary cleanup failed mid-transaction (gateway crash between the chunk delete and the Postgres soft-delete, or a crash between the supersession `UPDATE` commit and the new-row `INSERT` completion) rather than a routine cap on unbounded accumulation. The GC job provides a backstop sweep for manifests that escape the primary cleanup path: on each cycle the GC job queries all rows where `partial = true AND deleted_at IS NULL` and either (a) the session has reached a terminal state, or (b) the row's `created_at` timestamp is older than `maxResumeWindowSeconds` (i.e., the resume window has expired and no resume can consume this manifest). For each such row, the GC job lists objects under the manifest's `partial_object_key_prefix` via `ListObjectsV2` (the list is scoped by prefix and is encoding-agnostic — it matches both `partial-{n}.tar` and `partial-{n}.tar.gz` naturally, so the backstop requires no glob or suffix-specific filter and correctly sweeps whichever `chunk_encoding` the manifest declares) and deletes each returned chunk object via `DeleteObject`, then soft-deletes the Postgres row (`UPDATE ... SET deleted_at = now() AT TIME ZONE 'UTC' WHERE ... AND deleted_at IS NULL`). The `deleted_at IS NULL` guard is the idempotency mechanism described in GC concurrency model rule 6 below — a primary cleanup that completes between the backstop SELECT and UPDATE (or a second backstop pass from a stale leader) observes 0 rows affected and safely no-ops. The chunked-object storage model ([Section 10.1](10_gateway-internals.md#101-horizontal-scaling) Partial manifest on checkpoint timeout) commits partial chunks as independent objects via single-part `PutObject`, so the GC path does not issue `AbortMultipartUpload` — there is no open S3 multipart upload to abort. Rows whose `deleted_at` is older than the `artifact_store` tombstone retention window are hard-pruned by the same tombstone sweep that hard-prunes `artifact_store` rows (see Tombstone hard-prune below); partial-manifest rows follow the identical post-soft-delete lifecycle as other GC-managed artifact rows. This sweep prevents partial chunk objects from becoming permanently orphaned when the gateway crashes before performing cleanup or when a session is never resumed within its resume window.
- Background GC job runs every 15 minutes to clean expired artifacts. The interval is configurable via the `gc.cycleIntervalSeconds` Helm value (default: 900, minimum: 60). The job is owned by the gateway — it runs as a leader-elected goroutine inside the gateway process (not a separate CronJob). Only one gateway instance runs GC at a time via the existing leader-election lease. Higher tiers may require more frequent cycles ([Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)). Tenants with `gcPriority: high` (configurable per tenant via the admin API, intended for T4 erasure SLA compliance) trigger an immediate incremental GC sweep scoped to that tenant whenever an erasure job completes, independent of the global cycle interval.
- The GC job is idempotent: it queries Postgres for artifacts past their TTL, deletes them from MinIO, marks the rows as `deleted` in Postgres (commit), and only then decrements the Redis storage quota counter. This ordering is mandatory — the Redis decrement must follow the Postgres commit (see [Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas) step 3 for rationale: decrementing before commit creates a double-decrement on crash recovery because rehydration re-adds bytes for rows that are still `active` in Postgres). Re-running on the same set is safe: MinIO delete-on-absent is a no-op, the Postgres update is conditional on current state (`WHERE deleted_at IS NULL`), and a Redis decrement on an already-deleted artifact is prevented because the Postgres query only returns rows where `deleted_at IS NULL`.
- **No reference counting required in v1:** Each MinIO artifact object is owned by exactly one session. Session derivation ([Section 7.1](07_session-lifecycle.md#71-normal-flow)) always copies the parent's workspace snapshot into a new session-scoped MinIO object — no object is shared across sessions. TTL-based deletion is therefore safe: deleting a parent's expired artifact cannot affect any derived session's workspace. Reference counting will be required only if a future content-addressed deduplication optimization is introduced (see [Section 12.8](#128-compliance-interfaces)).
- Individual artifact deletion failures are retried independently — one stuck artifact does not block cleanup of others. On cycle failure the job logs the error, increments the error counter, and retries on the next 15-minute cycle.
- **Monitoring:** `lenny_gc_runs_total` (counter), `lenny_gc_artifacts_deleted` (counter), `lenny_gc_errors_total` (counter), `lenny_gc_duration_seconds` (histogram). Alert rule `ArtifactGCBacklog` ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when expired artifacts pending cleanup exceeds 1 000.
- Session artifact retention: configurable TTL (default 7 days), extendable per session

**Checkpoint-storage sizing guidance.** Per-tenant `storageQuotaBytes` ([Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas)) is a single bucket that covers **all** artifact types — workspace snapshots, checkpoints, uploaded files, session transcripts, and eviction context objects — not a checkpoint-specific quota. The `CheckpointStorageHigh` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) watches the checkpoint slice of that bucket (`lenny_checkpoint_storage_bytes_total`) against 80% of the same `storageQuotaBytes`; it fires ahead of the artifact-wide `StorageQuotaHigh` only when checkpoints dominate the tenant's footprint. Because the quota is shared, operators MUST account for checkpoint growth when sizing `storageQuotaBytes` per tenant. The per-pod checkpoint-object count depends on the pool's execution mode ([Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes)):
  - **Task / stateless pools:** up to 2 retained checkpoints per session × concurrent sessions assigned to the pod.
  - **Concurrent-workspace pools:** up to `maxConcurrent × 2` retained checkpoints per pod (the "latest 2" rule applies independently per slot). A pod with `maxConcurrent: 8` therefore carries up to 16 checkpoint objects steady-state.
  - **Per-tenant aggregate ceiling (rotation-only, no legal hold):** approximately `Σ_pools (active_pods_for_tenant × per_pod_checkpoint_count) × average_checkpoint_size`. Use this as the lower-bound planning figure before adding workspace-snapshot, upload, transcript, and eviction-context bytes that share the same `storageQuotaBytes` bucket.
  - **Legal-hold accumulation (unbounded).** Sessions with `legal_hold = true` are exempt from the "latest 2" rotation (see [Section 12.8](#128-compliance-interfaces)); checkpoints accumulate at the `periodicCheckpointIntervalSeconds` rate (default 600s / 10 min) for the full session duration and persist until the hold is lifted. A session held for 24 hours can therefore retain ~144 checkpoints instead of 2. Operators MUST size `storageQuotaBytes` for tenants with active or expected legal holds to absorb this accumulation, or raise the quota administratively when a hold is applied. There is no v1 per-tenant legal-hold storage cap or opt-out — suspension of rotation is a compliance requirement (spoliation avoidance), and caps that could force deletion during a hold are explicitly out of scope for v1 (see [Section 12.8](#128-compliance-interfaces) Legal hold and checkpoint rotation). **Predictive alert and runbook (STO-020).** Static sizing is supplemented by the `LegalHoldCheckpointAccumulationProjectedBreach` warning alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)), which fires when the tenant's projected 24-hour hold-driven growth plus current usage would cross 90% of `storageQuotaBytes`; the alert fires ahead of `StorageQuotaHigh` (80%) and `CheckpointStorageHigh` (80%) when the dominant future-growth driver is legal-hold accumulation rather than steady-state rotation. The `docs/runbooks/legal-hold-quota-pressure.md` runbook ([Section 17.7](17_deployment-topology.md#177-operational-runbooks)) is the operator response: enumerate active holds via the audit trail, then either raise `storageQuotaBytes` administratively for the tenant via `PUT /v1/admin/tenants/{id}` (existing admin API in [Section 15.1](15_external-api-surface.md#151-rest-api)) or accept that the held session's own checkpoints will freeze at `STORAGE_QUOTA_EXCEEDED` once the quota is crossed. The platform does not auto-resolve: raising the quota is an explicit operator choice, consistent with "caps that could force deletion during a hold are out of scope for v1."
  - **Per-tier starting points.** See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier MinIO sizing; the checkpoint slice is a subset of the total artifact budget accounted for there.

**GC concurrency model (single-writer + soft-delete guard).** The artifact GC path is designed so that concurrent invocations — whether from a leader-election race, a crash-recovery re-run, or the parallel checkpoint-rotation and TTL-expiration paths — are idempotent and cannot double-delete, double-decrement, or produce observable drift. The concurrency guarantees are:
  1. **Single active writer per gateway deployment (leader-elected).** The GC job runs as a leader-elected goroutine inside the gateway process using the same Kubernetes Lease-based leader-election primitive described for gateway-scoped background jobs (the `lenny-warm-pool-controller` / `lenny-pool-scaling-controller` patterns in [Section 4.6](04_system-components.md#46-pod-lifecycle-controllers) are the analogous primitive for controllers; gateway-scoped singleton jobs use an equivalent `lenny-gateway-leader` Lease scoped to the gateway Deployment). Exactly one gateway replica is the GC writer at any time. During the bounded failover window (near-zero on clean shutdown; up to 25s on crash — matching the controller failover bound in [Section 4.6.1](04_system-components.md#461-warm-pool-controller-pod-lifecycle)), the old leader's lease expires before a new leader can acquire it, so the two cannot overlap. **Per-tenant leader election is not used in v1** — global leader election is sufficient because all GC work is tenant-scoped at the query level (`WHERE tenant_id = $1`) and the gateway is not a scaling bottleneck for GC at Tier 1/2 workloads. Per-tenant GC sharding is a Tier 4 extension (see [Section 12.6](#126-interface-design) scaling extension interfaces) and would partition the `artifact_store` query by `tenant_id` across multiple leader-elected workers; the per-writer guards below remain correct regardless of sharding.
  2. **`WHERE deleted_at IS NULL` guard on every Postgres mutation.** Both the TTL expiration path and the checkpoint rotation path MUST issue their `artifact_store` `UPDATE ... SET deleted_at = now()` with a `WHERE id = $1 AND deleted_at IS NULL` predicate. This makes the state transition strictly monotonic: the first writer to commit moves the row from `active` to `deleted`; any concurrent or crash-resumed writer observes 0 rows affected on its UPDATE and skips the corresponding MinIO delete and Redis decrement. A second GC pass over the same row — whether from a stale leader whose write lost the race, from a retry after a transient Postgres error, or from checkpoint rotation and TTL expiration both selecting the same row in overlapping cycles — is a safe no-op.
  3. **MinIO delete-on-absent is idempotent.** The MinIO `DeleteObject` API returns success when the object is already absent (S3-compatible delete semantics). A stale leader that issued the MinIO delete before losing the guard race, a new leader that re-issues the delete after crash recovery, and checkpoint rotation that overlaps with TTL expiration all converge to the same terminal state: the object is absent in MinIO and the Postgres row is `deleted_at != NULL`. No additional coordination, conditional delete, or object tagging is required on the MinIO side.
  4. **Redis decrement is gated by the Postgres guard.** Per the ordering requirement in [Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas) step 3, the Redis storage-quota decrement is issued only after the Postgres `UPDATE ... WHERE deleted_at IS NULL` has committed and `rows_affected == 1`. A second writer that observes 0 rows affected MUST NOT issue the Redis decrement. This makes double-decrement impossible across concurrent leaders, crash-resumed runs, and the rotation-vs-TTL overlap.
  5. **Checkpoint rotation and TTL deletion coexist via the same guard.** The retention rule "keep latest 2 checkpoints per active session" (per-slot in concurrent-workspace mode — see the first bullet above) and the TTL-based deletion both target rows in the `artifact_store` table and both use the `WHERE deleted_at IS NULL` predicate described above. If rotation selects a checkpoint for deletion because it is the third-oldest-and-newer-than-TTL, and the TTL sweep in the same cycle independently selects the same checkpoint because it is also past TTL, exactly one of the two UPDATEs wins; the other observes `rows_affected == 0` and skips both the MinIO delete and the Redis decrement. Legal-hold rows (`legal_hold = true`, see [Section 12.8](#128-compliance-interfaces)) are excluded at the SELECT level by both paths, so rotation cannot race with a legal-hold grant to delete an artifact that has just been placed on hold.
  6. **Partial-manifest backstop sweep uses the same guard.** The partial-checkpoint manifest sweep (bullet above; row schema in [§10.1 Partial manifest on checkpoint timeout](10_gateway-internals.md#101-horizontal-scaling); primary cleanup path in [§4.4 Partial checkpoint manifest cleanup](04_system-components.md#44-event--checkpoint-store)) issues its Postgres soft-delete under the same `WHERE` predicate that selected the row for cleanup (`partial = true AND (terminal_state OR created_at < now() - maxResumeWindowSeconds) AND deleted_at IS NULL`). The primary cleanup path soft-deletes under the identical `deleted_at IS NULL` guard, so a resume path that completes between the SELECT and the UPDATE — or any stale-leader re-run of either path — observes `rows_affected == 0` on the second writer and safely no-ops. The backstop neither blocks the primary cleanup path nor double-deletes its artifacts. Post soft-delete, the partial-manifest row is hard-pruned by the same `artifact_store` tombstone sweep described below, on the same retention window.

Together these rules make the entire artifact GC path a **convergent, idempotent, single-writer** system: the leader-election guarantee gives single-writer semantics during steady state, the `deleted_at IS NULL` guards give correct single-writer semantics during the bounded failover window, and MinIO's delete-on-absent idempotency absorbs any residual retry. No per-artifact locks, conditional MinIO operations, or inter-path coordination beyond these guards are required.

**Tombstone hard-prune sweep.** Soft-deleted rows (`deleted_at IS NOT NULL`) are retained in Postgres for a bounded window — `gc.tombstoneRetentionSeconds` (default: **86400 / 24 hours**) — before being physically removed. The same leader-elected GC goroutine that performs the soft-delete passes runs a companion **tombstone hard-prune** pass on each cycle: it issues `DELETE FROM artifact_store WHERE deleted_at IS NOT NULL AND deleted_at < now() - (gc.tombstoneRetentionSeconds * interval '1 second')`. The retention window must be longer than the maximum expected stale-leader write delay plus any outstanding backstop cycle — 24 hours is chosen so that any in-flight idempotency guard check has long since converged before the row disappears; operators with unusually long failover windows (e.g., multi-region DR drills) may increase `gc.tombstoneRetentionSeconds` without affecting GC correctness. **Partial-checkpoint manifest rows (§10.1 schema) are swept by the same hard-prune pass** on the same `deleted_at` predicate and the same retention window — this is the single physical-delete path for every GC-managed row in Postgres, avoiding a separate lifecycle for partial manifests. The hard-prune is `artifact_store`-scoped when the row lives in `artifact_store`; partial-manifest rows live in the checkpoint metadata table and the sweep issues the analogous `DELETE ... WHERE partial = true AND deleted_at IS NOT NULL AND deleted_at < now() - (gc.tombstoneRetentionSeconds * interval '1 second')`. MinIO objects have already been deleted at soft-delete time, so the hard-prune is a pure Postgres row removal with no storage-side side effect. Metric: `lenny_gc_tombstones_pruned_total` (counter, labeled by `table: artifact_store|partial_manifest`) tracks hard-prune disposition.

**Scaling extension interfaces.** The following five interfaces are defined in v1 with trivial single-node implementations. They exist to ensure that the components they abstract can be swapped at Tier 4 scale without restructuring the platform core. Each interface ships with exactly one v1 implementation; adding a Tier 4 implementation requires no application code restructuring — the swap is a configuration-only change at the store-router level. Note: some Tier 4 implementations require accompanying infrastructure changes beyond configuration (e.g., `PostgresPodRegistry` requires the `agent_pod_state` schema migration; `NATSEventBus`/`KafkaEventBus` requires deploying and operating a new message broker). All five interfaces are in the `platform/store` package.

#### PodRegistry

Abstracts pod lifecycle state as a data-access layer. The [§4.6.1](04_system-components.md#461-warm-pool-controller-pod-lifecycle) business-logic interfaces (`PodLifecycleManager`, `PoolManager`) delegate their state reads and writes to `PodRegistry` — for example, `PodLifecycleManager.ClaimPod` runs the optimistic-locking CAS loop and `SandboxClaim` creation, calling `PodRegistry.ClaimPod` for the atomic pod-record state transition. Method names overlap intentionally: PodRegistry provides the storage primitive; the [§4.6.1](04_system-components.md#461-warm-pool-controller-pod-lifecycle) interfaces add domain logic (session binding, checkpoint coordination, pool reconciliation) on top. The v1 implementation (`CRDPodRegistry`) reads and writes `Sandbox` CRD status via the Kubernetes API. At Tier 4, `PostgresPodRegistry` replaces CRD status writes with the `agent_pod_state` Postgres table, eliminating ~95% of etcd write pressure (see [§4.6.1](04_system-components.md#461-warm-pool-controller-pod-lifecycle) etcd write pressure table).

```go
type PodRegistry interface {
    GetPod(ctx context.Context, podID PodID) (*PodRecord, error)
    UpdatePodState(ctx context.Context, podID PodID, transition StateTransition) error
    ClaimPod(ctx context.Context, opts ClaimOpts) (*PodRecord, error)
    ReleasePod(ctx context.Context, podID PodID, reason ReleaseReason) error
    ListPodsByPool(ctx context.Context, poolID PoolID, filter PodFilter) ([]PodRecord, error)
    CountByState(ctx context.Context, poolID PoolID) (StateCounts, error)
    CreatePod(ctx context.Context, poolID PoolID, spec PodSpec) (*PodRecord, error)
    DeletePod(ctx context.Context, podID PodID) error
    WatchPods(ctx context.Context, poolID PoolID) (<-chan PodEvent, error)
}
// v1: CRDPodRegistry (Kubernetes API)
```

`ClaimOpts.ClusterID` is a nullable field (type `*ClusterID`). In v1, `ClaimOpts.ClusterID` is always `nil` (single-cluster). The `LocalClusterRegistry` (see below) propagates a non-nil `ClusterID` when routing claims to a remote cluster in a multi-cluster topology.

**`agent_pod_state` table schema.** The `agent_pod_state` Postgres table mirrors `Sandbox` CRD status fields. In v1 it is a read-optimized mirror maintained by the WarmPoolController on every pod state transition; at Tier 4 it becomes the primary store for `PostgresPodRegistry`. The table is platform-global (no RLS — `tenant_id` is a denormalized convenience column set on claim, not an isolation boundary; see [§4.2](04_system-components.md#42-session-manager) data model table).

```sql
CREATE TABLE agent_pod_state (
    pod_id          TEXT        PRIMARY KEY,
    pool_id         TEXT        NOT NULL,
    state           TEXT        NOT NULL,  -- pod state machine value (idle, claimed, running, draining, terminated, failed)
    tenant_id       TEXT,                  -- NULL when idle; set on claim alongside session_id (denormalized for query convenience)
    session_id      TEXT,                  -- NULL when idle; set on claim
    isolation_profile TEXT      NOT NULL,  -- standard, sandboxed, microvm
    execution_mode  TEXT        NOT NULL,  -- session, task, concurrent_workspace
    resource_version BIGINT    NOT NULL,   -- CRD generation for optimistic-locking; incremented on every state transition
    node_name       TEXT,                  -- Kubernetes node hosting the pod
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_agent_pod_state_pool_state ON agent_pod_state (pool_id, state);
CREATE INDEX idx_agent_pod_state_session    ON agent_pod_state (session_id) WHERE session_id IS NOT NULL;
CREATE INDEX idx_agent_pod_state_tenant     ON agent_pod_state (tenant_id) WHERE tenant_id IS NOT NULL;
```

The `pool_id + state` index supports `ListPodsByPool` with state filter and `CountByState`. The partial `session_id` index supports the orphan session reconciler cross-reference ([§10.1](10_gateway-internals.md#101-horizontal-scaling)). `resource_version` is used by `UpdatePodState` for optimistic-locking CAS: `UPDATE ... SET state = $new, resource_version = resource_version + 1 WHERE pod_id = $id AND resource_version = $expected`. The `updated_at` column drives the `lenny_agent_pod_state_mirror_lag_seconds` staleness gauge ([§10.1](10_gateway-internals.md#101-horizontal-scaling)).

**Observability contract.** Every `PodRegistry` implementation MUST emit: `lenny_pod_registry_operation_duration_seconds{operation, pool}` (histogram) and `lenny_pod_registry_error_total{operation, pool}` (counter). The `operation` label is one of: `get`, `update_state`, `claim`, `release`, `list`, `count`, `create`, `delete`, `watch`. These metrics provide per-operation visibility for implementation diagnosis (e.g., detecting a slow `PostgresPodRegistry` at Tier 4). The existing pod claim metrics (`lenny_pod_claim_queue_*`, `lenny_pod_claim_conflict_total`, `lenny_pod_claim_timeout_total`) remain as higher-level business metrics capturing the end-to-end claim flow including CAS retries and queue wait time; the registry-level metrics capture the underlying storage operation only.

**`WatchPods` delivery contract.** `WatchPods` provides **eventually-consistent change notification**, not a reliable event stream. Implementations MUST deliver events within a bounded latency: 500 ms P99 for `CRDPodRegistry` (Kubernetes informer propagation), 2 s P99 for a future `PostgresPodRegistry` (poll or LISTEN/NOTIFY propagation). Implementations MAY coalesce or drop events under backpressure — the channel is a hint, not a ledger. Consumers MUST NOT rely on `WatchPods` as the sole source of truth; they MUST periodically reconcile via `CountByState` or `ListPodsByPool` on a configurable interval (default: 10 s for `PoolScalingController`). The reconciliation interval governs correctness; the watch channel governs responsiveness.

The channel MUST NOT emit an initial state snapshot on creation. Consumers call `ListPodsByPool` for initial state, then `WatchPods` for subsequent deltas. If the implementation detects that the channel has fallen behind (e.g., a Postgres poll missed intermediate states), it SHOULD send a synthetic `PodEvent` with `EventType = resync` carrying no `PodRecord` — the consumer treats this as a signal to re-read via `ListPodsByPool`.

**`PostgresPodRegistry` implementation guidance.** A future `PostgresPodRegistry` implementation SHOULD use Postgres `LISTEN/NOTIFY` on a per-pool channel (`pod_state_change_{pool_id}`) triggered by an `AFTER UPDATE` trigger on `agent_pod_state`. The trigger sends the `pod_id` as the notification payload; the `WatchPods` goroutine reads the full `PodRecord` on receipt. If `LISTEN/NOTIFY` is unavailable (e.g., connection-pooler limitation with PgBouncer in transaction mode), the implementation should fall back to polling `agent_pod_state WHERE pool_id = $1 AND updated_at > $last_poll` at 1 s intervals. The `lenny_pod_registry_watch_lag_seconds{pool, implementation}` gauge tracks the delay between `updated_at` on the row and the time the event is delivered to the channel — this metric is required for all `PodRegistry` implementations that support `WatchPods`.

#### StoreRouter

Abstracts Postgres shard routing and per-concern Redis instance selection. The v1 implementation (`SingleShardRouter`) returns the sole configured connection pool for every call. All billing and audit writes MUST be routed through `StoreRouter.BillingShard` and `StoreRouter.AuditShard` even in v1 (see [§12.3](#123-postgres-ha-requirements) R-03).

```go
type StoreRouter interface {
    // TenantShard returns the pool for tenant-scoped metadata tables
    // (tenants, environments, runtime_definitions, credential_pools,
    // delegation_policies). Always routes by tenant_id.
    TenantShard(ctx context.Context, tenantID TenantID) (*pgxpool.Pool, error)

    // SessionShard returns the pool for session-scoped tables
    // (sessions, session_messages, session_tree_archive).
    // Extracts the shard routing prefix from the session ID
    // (see session ID format below) and consistent-hashes it.
    // All sessions in a delegation tree share the same routing
    // prefix, so they co-locate on the same shard.
    // In v1, returns the same pool as TenantShard.
    SessionShard(ctx context.Context, sessionID SessionID) (*pgxpool.Pool, error)

    // BillingShard returns the pool for billing event writes.
    // Always routes by tenant_id (R-03).
    BillingShard(ctx context.Context, tenantID TenantID) (*pgxpool.Pool, error)

    // AuditShard returns the pool for audit log writes.
    // Always routes by tenant_id (R-03).
    AuditShard(ctx context.Context, tenantID TenantID) (*pgxpool.Pool, error)

    // RedisShard returns the Redis client for a given concern.
    RedisShard(ctx context.Context, tenantID TenantID, concern RedisConcern) (redis.UniversalClient, error)

    // PlatformRedis returns the Redis client for platform-scoped keys
    // (pod slot counters, circuit breakers). Not tenant-routed.
    // In v1, returns the same instance as all RedisConcern values.
    PlatformRedis(ctx context.Context) (redis.UniversalClient, error)

    // AllSessionShards returns all session-table shards for scatter-gather
    // queries (e.g., list sessions for a tenant, GDPR erasure).
    AllSessionShards(ctx context.Context) ([]ShardHandle, error)

    // PlatformPostgres returns the connection pool for platform-global
    // tables that are not owned by any tenant or session (ops-scoped
    // tables — see §12.X "Ops-scoped tables" — platform_upgrade_state,
    // bootstrap_seed_snapshot, etc.). In v1 SingleShardRouter, returns
    // the same pool as TenantShard. In multi-shard deployments, routes
    // to the dedicated platform database instance.
    PlatformPostgres(ctx context.Context) (*pgxpool.Pool, error)

    // AllAuditShards returns every audit shard for scatter-gather
    // cross-tenant audit queries (§25.9). In v1 returns a single handle.
    AllAuditShards(ctx context.Context) ([]ShardHandle, error)

    // ShardCount returns the number of shards for a given store type.
    ShardCount(storeType StoreType) int
}
// v1: SingleShardRouter (single Postgres pool, single Redis per concern)
```

**Section 25 ops-scoped tables.** The `lenny-ops` service (§25.4) stores its state in platform-global tables routed via `PlatformPostgres()`. The v1 `SingleShardRouter` returns the same pool as all other accessors; the dedicated method is present so multi-shard deployments can route platform tables to an isolated instance without touching call sites. The full table list — `ops_remediation_locks`, `ops_lock_epoch`, `ops_lock_conflicts`, `ops_idempotency_keys`, `ops_escalations`, `ops_event_subscriptions`, `ops_event_deliveries`, `ops_backups`, `ops_backup_schedule`, `ops_retention_policy`, `ops_restore_state`, `platform_upgrade_state`, `platform_upgrade_check_cache`, `bootstrap_seed_snapshot`, `audit_log_deferred_writes`, `ops_postgres_outage_log`, `ops_operation_baselines` — is defined in §25.4, §25.5, §25.8, §25.9, §25.10, and §25.11. Schemas are authoritative in Section 25; only routing is documented here.

**Progress column on long-running operation tables.** Each long-running operation table (`platform_upgrade_state`, `ops_restore_state`, `ops_backups`, and any future tables whose `kind` appears in the Operations Inventory of §25.4) carries a `progress JSONB` column conforming to the canonical Progress Envelope (§25.2). The owning subsystem updates `progress` atomically with each step transition; readers scatter-gather via `AllAuditShards()` or query `PlatformPostgres()` directly.

**UTC timestamps.** Every `TIMESTAMPTZ` column authored by Lenny application code MUST use `now() AT TIME ZONE 'UTC'` (not `now()`). Consumers that reason across shards — split-brain resolution (§25.4 Locks), audit chain integrity (§25.9), lock-TTL comparison — depend on this. `PlatformPostgres()` callers are subject to the same convention.

**Sharding strategy.** The StoreRouter separates two routing dimensions: tenant-scoped metadata (`TenantShard`) and session-scoped data (`SessionShard`). In v1 (`SingleShardRouter`), both return the same pool. A future session-sharded topology distributes session-scoped tables by consistent hash of the **shard routing prefix** embedded in each session ID (see session ID format below). All sessions in a delegation tree share the same routing prefix, so they co-locate on the same shard — enabling single-shard tree traversal, cascade cancellation, and delegation lineage queries. Standalone sessions (no delegation) have random routing prefixes and distribute uniformly. Tree sizes are bounded by `maxChildrenTotal` and `maxDepth` delegation policy limits, so no single tree creates a hot shard. Tenant-scoped metadata tables remain tenant-routed — their volume is low and they don't create hot shards.

**Caller guide:** All single-session operations use `SessionShard(sessionID)` — the shard routing prefix is embedded in every session ID, so any session ID (root or child) routes correctly without external lookups. This covers session creation, single-session reads, coordinator handoff, and admin API lookups. Tenant-wide session queries (`GET /v1/sessions`, GDPR erasure, tenant deletion) use `AllSessionShards()` with a `tenant_id` filter on each shard. Tenant config reads (environments, policies, credential pools) use `TenantShard(tenantID)`. Billing and audit writes use `BillingShard` and `AuditShard` per R-03.

**Scatter-gather execution constraints.** All scatter-gather queries using `AllSessionShards()` MUST observe the following bounds. In v1, `SingleShardRouter.AllSessionShards()` returns a single `ShardHandle`, so these constraints are trivially satisfied; they become load-bearing only when a multi-shard `StoreRouter` implementation is deployed.

- **Concurrency limit.** At most `storeRouter.maxScatterGatherConcurrency` shards are queried in parallel (Helm value, default: 16). Remaining shards are queried in subsequent batches. This prevents a tenant-wide query from saturating all shard connection pools simultaneously.
- **Per-shard timeout.** Each shard query has a deadline of `storeRouter.scatterGatherPerShardTimeoutSeconds` (Helm value, default: 10 s). If a shard times out, the scatter-gather operation logs a `scatter_gather_shard_timeout` structured event (including shard ID and query type) and either (a) returns a partial result with a `partial: true` flag for read paths (list sessions), or (b) retries the timed-out shard up to 2 times for write paths (GDPR erasure, tenant deletion) before failing the operation.
- **Aggregate timeout.** The total scatter-gather operation has a deadline of `storeRouter.scatterGatherAggregateTimeoutSeconds` (Helm value, default: 120 s). This caps worst-case latency when many shards are slow.

Every scatter-gather invocation MUST emit `lenny_store_router_scatter_gather_duration_seconds{query_type}` (histogram) and `lenny_store_router_scatter_gather_shard_count{query_type}` (gauge). The `query_type` label is one of: `list_sessions`, `gdpr_erasure`, `tenant_deletion`, `delegation_budget_purge`. A `ScatterGatherSlowQuery` warning alert fires when P99 of `lenny_store_router_scatter_gather_duration_seconds` exceeds 30 s sustained for 5 minutes.

`SessionShard` implementations MUST resolve within 5 ms P99 — this call is on the coordinator handoff hot path.

**Session ID format.** Session IDs are UUIDv8 (RFC 9562) with an application-defined bit layout that embeds a **shard routing prefix** in the first 32 bits:

```
xxxxxxxx-xxxx-8xxx-yxxx-xxxxxxxxxxxx
^^^^^^^^      ^    ^
routing       ver  variant
prefix        (8)  (10xx)
(32 bits)
```

The routing prefix determines the Postgres shard for session-scoped tables. `SessionShard()` extracts the first 32 bits from any session ID and consistent-hashes them — no external state or lookup is needed. Generation rules:

- **Root session creation:** The gateway generates a UUIDv8 with a fully random routing prefix (32 random bits). The remaining ~90 bits (after the version nibble and variant bits) are also random.
- **Child session creation (delegation):** The gateway copies the first 32 bits from the root session's UUID into the child's UUID. The remaining bits are random. This guarantees every session in a delegation tree shares the same routing prefix and lands on the same shard. The `root_session_id` used for prefix copying is available from the delegation lease ([§8.3](08_recursive-delegation.md#83-delegation-policy-and-lease) R-04).
- **Session derivation (`/v1/sessions/{id}/derive`):** The derived session is a new independent root — its UUID gets a fully random routing prefix. The derived session forms a new tree on a potentially different shard.

With 32 bits for routing and ~90 random bits for uniqueness, collision probability within a single delegation tree reaches 50% at ~2^45 children. Tree sizes are bounded by policy at hundreds — collision risk is effectively zero.

External consumers (clients, admin API callers, MCP tooling) treat session IDs as opaque strings. The routing prefix is an internal implementation concern and is not exposed in any API contract.

#### ClusterRegistry

Abstracts cluster topology for multi-cluster delegation routing. The v1 implementation (`LocalClusterRegistry`) exposes only the local cluster and always returns `LocalClusterID()` from `SelectCluster`.

```go
// RemotePodOperations is the subset of PodRegistry permitted over
// cross-cluster connections. Returned by ClusterClient for remote
// clusters. The restricted type enforces the cross-cluster security
// contract at compile time — callers cannot invoke DeletePod,
// UpdatePodState, CreatePod, CountByState, or WatchPods on a
// remote cluster.
type RemotePodOperations interface {
    GetPod(ctx context.Context, podID PodID) (*PodRecord, error)
    ClaimPod(ctx context.Context, opts ClaimOpts) (*PodRecord, error)
    ReleasePod(ctx context.Context, podID PodID, reason ReleaseReason) error
    ListPodsByPool(ctx context.Context, poolID PoolID, filter PodFilter) ([]PodRecord, error)
}

type ClusterRegistry interface {
    ListClusters(ctx context.Context) ([]ClusterInfo, error)
    GetCluster(ctx context.Context, clusterID ClusterID) (*ClusterInfo, error)
    SelectCluster(ctx context.Context, req ClusterSelectionRequest) (ClusterID, error)
    ClusterClient(ctx context.Context, clusterID ClusterID) (RemotePodOperations, error)
    LocalClusterID() ClusterID
}
// v1: LocalClusterRegistry (single cluster, ClaimOpts.ClusterID always nil)
// LocalClusterRegistry.ClusterClient returns the local CRDPodRegistry,
// which satisfies RemotePodOperations (it is a superset).
```

**Cross-cluster transport security contract.** Implementations of `ClusterClient` that return a remote `RemotePodOperations` (i.e., any implementation other than `LocalClusterRegistry`) MUST establish a mutually authenticated TLS connection to the remote cluster's PodRegistry endpoint. Requirements: (1) the connection MUST use mTLS with certificates issued by the platform's cert-manager CA ([§10.3](10_gateway-internals.md#103-mtls-pki)) — the client presents a gateway certificate and the server presents a certificate whose SAN matches the endpoint declared in `ClusterInfo`; (2) `ClusterInfo` MUST include a `CACertBundle` field containing the remote cluster's CA certificate chain, used by the client to verify the server's identity — this prevents a compromised cluster from impersonating another via a different CA; (3) the remote PodRegistry endpoint MUST validate that the caller's certificate SAN matches the expected gateway identity of the originating cluster before accepting any operation; (4) operations not in `RemotePodOperations` (`DeletePod`, `UpdatePodState`, `CreatePod`, `CountByState`, `WatchPods`) are prohibited over cross-cluster connections — this is enforced at compile time by the `RemotePodOperations` return type and at runtime by the remote endpoint, which MUST reject any RPC for a non-permitted operation from a cross-cluster caller. The `LocalClusterRegistry` (v1) is exempt — it returns an in-process `PodRegistry` reference (which satisfies `RemotePodOperations`) with no network transport.

**Cross-cluster CA cert bundle rotation.** When a remote cluster rotates its CA (following the overlap procedure in [§10.3](10_gateway-internals.md#103-mtls-pki)), the local cluster's `ClusterInfo.CACertBundle` must be updated to include the new CA chain before the old CA expires. During the overlap window, `CACertBundle` MUST contain both the old and new CA certificates so that connections succeed regardless of which CA signed the remote endpoint's current certificate. `ClusterRegistry` implementations that persist `ClusterInfo` (e.g., in Postgres or a ConfigMap) SHOULD support a `UpdateClusterCACertBundle` operation or equivalent; implementations that derive `ClusterInfo` from a shared secret store (e.g., Vault) inherit rotation from the store's update mechanism.

**Cross-cluster `PodRecord` compatibility contract.** Every `PodRegistry` implementation — whether accessed locally or via `ClusterClient` — MUST produce `PodRecord` values that satisfy the following invariants. These are implicit in the current single-implementation design but must be stated explicitly to prevent subtle cross-implementation bugs if clusters run different `PodRegistry` backends (e.g., one cluster on `CRDPodRegistry`, another on a future `PostgresPodRegistry`):

1. **State machine values.** The `State` field MUST be one of the canonical pod state machine values defined in [§4.6.1](04_system-components.md#461-warm-pool-controller-pod-lifecycle) (`idle`, `claimed`, `running`, `draining`, `terminated`, `failed`). No implementation-specific states are permitted.
2. **`resource_version` monotonicity.** `resource_version` MUST be strictly monotonically increasing for a given `PodID` within a single `PodRegistry` implementation. Cross-implementation comparisons of `resource_version` are undefined — callers MUST NOT compare `resource_version` values obtained from different `PodRegistry` implementations (e.g., a `resource_version` from a `CRDPodRegistry` on cluster A vs. a future `PostgresPodRegistry` on cluster B).
3. **`ClaimPod` atomicity.** `ClaimPod` MUST be atomic with respect to concurrent callers across all access paths (local and remote). A pod claimed via `ClusterClient` on cluster A MUST NOT be concurrently claimable via direct `PodRegistry` access on cluster B. This is enforced by the remote endpoint's own `PodRegistry` implementation — the `ClusterClient` delegates to the remote cluster's `PodRegistry`, which holds the authoritative state.

**Version negotiation.** Future `ClusterClient` implementations SHOULD include a lightweight version handshake on connection establishment: the remote endpoint reports its `PodRegistry` implementation type and schema version (e.g., `crd/v1`, `postgres/v1`) in the gRPC metadata of the first response. The local cluster logs this as a `cluster_registry_peer_version` structured event. No behavioral branching is required — the handshake is diagnostic, enabling operators to detect mixed-version deployments before they cause issues.

#### EventBus

Abstracts control-plane event distribution for delegation tree signals and session lifecycle events. The v1 implementation (`RedisEventBus`) publishes to Redis pub/sub channels under the tenant-prefix convention ([§12.4](#124-redis-ha-and-failure-modes)). If the EventBus backend is later replaced with NATS JetStream or Kafka for scalability, the swap is a configuration-only change at the store-router level — no callers change.

Every event carries a **[CloudEvents v1.0.2](https://github.com/cloudevents/spec/blob/v1.0.2/cloudevents/spec.md)** envelope. The envelope makes the event transport-independent (the same envelope flows over Redis pub/sub, SSE [§25.4](25_agent-operability.md#254-the-lenny-ops-service), outbound webhook callbacks [§14](14_workspace-plan-schema.md), and §25 event-subscription webhooks). Event payloads never travel as opaque bytes.

```go
// EventTopic identifies a control-plane event channel.
// Adding a topic requires updating this list, the CloudEvents type registry
// below, and the §12.4 key table.
type EventTopic string

const (
    TopicDelegationTree   EventTopic = "delegation_tree"   // budget signals, child completion, tree cancellation
    TopicSessionLifecycle EventTopic = "session_lifecycle"  // state transitions, coordinator handoff notifications
)

// Event is a CloudEvents v1.0.2 envelope. The `go-sdk` cloudevents.Event type
// is used directly; no Lenny-specific wrapper.
type Event = cloudevents.Event

type EventBus interface {
    Publish(ctx context.Context, tenantID TenantID, topic EventTopic, event Event) error
    Subscribe(ctx context.Context, tenantID TenantID, topic EventTopic,
        handler func(context.Context, Event) error) (Subscription, error)
}
// v1: RedisEventBus (Redis pub/sub, tenant-prefix key convention, CloudEvents JSON serialization)
```

**CloudEvents envelope contract.** Every published `Event` MUST set the following context attributes:

| CloudEvents attribute | Value |
|---|---|
| `specversion` | `"1.0"` |
| `id` | Unique identifier `{tenantId}:{publisherId}:{nanoTimestamp}:{nonce}`. `publisherId` is the gateway replica ID (`gw-<5-hex>`); `nanoTimestamp` is `time.Now().UnixNano()`; `nonce` is a 64-bit value generated via `crypto/rand.Int63()` (cryptographic RNG) and encoded as lowercase hex — never `math/rand` and never a sequence counter, because the `id` is the primary de-duplication key when a future at-least-once backend is enabled and a predictable nonce would allow a malicious or buggy publisher to forge collisions with another publisher's `id`. Collision probability across a Lenny cluster is negligible under this construction (per-replica publisherId + nanosecond timestamp + 64-bit CSPRNG nonce). |
| `source` | URI identifying the emitter: `//lenny.dev/{component}/{replicaId}` (e.g., `//lenny.dev/gateway/gw-7f4c2`) |
| `type` | `dev.lenny.<short_name>` where `<short_name>` is the identifier registered in the [§16.6](16_observability.md#166-operational-events-catalog) / [§25](25_agent-operability.md) catalogs (e.g., `dev.lenny.delegation_tree_completed`, `dev.lenny.session_state_changed`, `dev.lenny.operation_progressed`) |
| `time` | RFC 3339 UTC timestamp |
| `datacontenttype` | `"application/json"` (default); audit-bearing events carrying an OCSF record set `datacontenttype: "application/ocsf+json"` |
| `subject` | Opaque domain subject: `session/{sessionId}`, `tree/{rootSessionId}`, or `pool/{poolName}` depending on topic |
| Lenny extensions | `lennytenantid` (always present, mirrors the `tenantID` parameter for cross-cutting filters); `lennyrootsessionid` (present for delegation-tree events); `lennyoperationid` ([§25.1](25_agent-operability.md#251-design-philosophy-and-agent-model)) where applicable |

`data` is the event-specific payload, JSON-encoded. When an EventBus event carries an audit record, `data` is the **OCSF JSON record** ([§11.7](11_policy-and-controls.md#117-audit-logging) wire format) — the CloudEvents envelope is the transport; the OCSF record is the payload. Single-envelope model: nothing is double-wrapped.

**Delivery contract.** EventBus provides **at-most-once** delivery in v1 (Redis pub/sub drops messages when no subscriber is connected). Callers MUST tolerate missed events — any caller that requires guaranteed delivery must also poll durable state (Postgres) as a fallback. If the EventBus backend is later replaced with NATS JetStream or Kafka for scalability, the delivery guarantee strengthens to at-least-once, introducing the possibility of duplicate delivery. Handlers MUST be idempotent to remain correct under either guarantee. See the handler design note below.

**Publish failure after durable commit.** Some events (notably audit-bearing events, [§11.7](11_policy-and-controls.md#117-audit-logging)) are emitted after the originating transaction has already committed durable state to Postgres. When `EventBus.Publish` returns an error after the durable commit — e.g., Redis unreachable, channel buffer exceeded, serialization failure — the gateway MUST NOT attempt to roll back the durable state (it is already committed and, for audit rows, immutable). Instead, the gateway: (a) increments `lenny_event_bus_publish_dropped_total{topic}` labeled with the topic and an `error_type` label (`backend_unavailable`, `serialization_failed`, `timeout`); (b) writes the serialized CloudEvents envelope to a bounded in-memory replay buffer (default 10k events per replica, oldest-first eviction) for opportunistic re-publish when the backend recovers; and (c) for audit-bearing topics, marks the source Postgres audit row with `eventbus_publish_state = 'failed'` and leaves `retry_count = 0` (the `audit_log.retry_count` column is `INT NOT NULL DEFAULT 0`, co-located with `eventbus_publish_state` — see [§12.2](#122-storage-roles) `EventStore`). Subscribers that require strict delivery reconstruct missed events by querying Postgres via the `/v1/admin/audit-events` API ([§25.9](25_agent-operability.md#259-audit-log-query-api)) filtered on `eventbus_publish_state IN ('failed', 'retry_pending')`. The `EventBusPublishDropped` alert ([§16.5](16_observability.md#165-alerting-rules-and-slos)) fires when dropped-event rate exceeds `eventBus.dropAlertThreshold` (default 10/min) for more than 5 minutes. On Redis reconnect, the replay buffer is drained in FIFO order with `publishedAt` preserved (the `time` context attribute stays at the original event time, not the re-publish time); drained events flip the source audit row's state to `published`. The in-memory replay buffer is a **latency optimization** — it keeps the common-case recovery path (same replica that observed the publish failure drains the buffer seconds later when Redis returns) off Postgres. The **correctness** layer, which guarantees that every `failed` audit row is eventually re-published even when the replay buffer is lost, is the retranscribe worker described next.

**EventBus retranscribe worker.** A leader-elected background worker co-located with the gateway (same leader-election primitive as the §4.6 pool controllers, scoped with lock key `lenny.gateway.eventbus.retranscribe.leader`) sweeps the audit shards every `eventBus.retryInterval` (Helm value, default: 60 s). The sweep reads rows matching `WHERE eventbus_publish_state IN ('failed', 'retry_pending') AND retry_count < eventBus.maxRetryAttempts` ordered by `created_at ASC` (FIFO within each shard; `AllAuditShards()` iteration is the outer loop, so the worker is scatter-gather-bounded by `storeRouter.scatterGatherMaxConcurrency` per cycle). For each eligible row the worker re-serializes the CloudEvents envelope from the canonical Postgres tuple (same `datacontenttype: "application/ocsf+json"` payload and same `id` / `source` / `time` attributes — the envelope is byte-identical to the original publish attempt, so downstream de-duplication by CloudEvents `id` continues to work), re-invokes `EventBus.Publish(ctx, tenantID, topic, event)`, and applies the state transition under the per-tenant `pg_advisory_xact_lock` used for audit writes ([§11.7](11_policy-and-controls.md#117-audit-logging) item 3) so the `prev_hash` chain is never re-hashed — the retranscribe worker updates **only** `eventbus_publish_state` and `retry_count`, both of which are excluded from the `payload_canonical_json` hash input. Outcomes:

- **Success** → `eventbus_publish_state = 'published'`, `retry_count` stays at its last value (retained for forensics), and `lenny_event_bus_retranscribe_attempts_total{outcome="success"}` is incremented.
- **Failure** → `eventbus_publish_state = 'retry_pending'`, `retry_count = retry_count + 1`, and `lenny_event_bus_retranscribe_attempts_total{outcome="failure"}` is incremented labeled by `error_type` (same `backend_unavailable` / `serialization_failed` / `timeout` vocabulary as the first-publish metric).
- **Terminal failure** (only reached when the failing sweep would increment `retry_count` to `eventBus.maxRetryAttempts`) → `eventbus_publish_state = 'failed'`, `retry_count = eventBus.maxRetryAttempts` (the row stops being picked up by future sweeps because the `retry_count < eventBus.maxRetryAttempts` predicate excludes it). The worker emits a `EventBusPublishFinalFailure` warning alert ([§16.5](16_observability.md#165-alerting-rules-and-slos)) carrying `event_id`, `topic`, `retry_count`, and `tenant_id` so operators can intervene via the `POST /v1/admin/audit-events/{id}/republish` admin endpoint ([§25.9](25_agent-operability.md#259-audit-log-query-api)). A manual republish resets `retry_count = 0` and state to `pending`, so the row re-enters the worker's eligibility set on the next sweep.

Retranscribe-sweep duration is observed in `lenny_event_bus_retranscribe_duration_seconds` (histogram, labeled by `topic`). The worker does not re-drive events whose source topic has been retired — a topic removal migration marks all pending rows `failed` with a terminal sentinel so they do not accumulate indefinitely; the operator-facing republish endpoint remains the only way to re-emit after a topic retirement. Because the CloudEvents `id` is byte-identical across retranscribes, a durable EventBus backend (future NATS/Kafka) will de-duplicate replays at the subscriber boundary.

Interaction with SSE subscribers ([§25.5](25_agent-operability.md#255-operational-event-stream)) and the admin query API ([§25.9](25_agent-operability.md#259-audit-log-query-api)): SSE subscribers receive retranscribed events exactly as if they were first-time publishes (the replay carries the original `time` and `id`, so client-side "already seen" logic remains correct); operators polling Postgres for reconciliation query `?eventbus_publish_state=retry_pending` (in flight) or `?eventbus_publish_state=failed` (terminal — needing manual republish) to see the complete picture without racing the worker.

**Handler design.** EventBus events are **notifications, not commands.** The payload tells the handler *what changed*; the handler reads durable state (Postgres, Redis) to determine *what to do*. For example, a `TopicDelegationTree` event signaling child completion causes the gateway to read the child's terminal state from `SessionStore` — receiving the same notification twice simply triggers a redundant state check and is a no-op. This convergent design makes handlers naturally idempotent without requiring event IDs, deduplication infrastructure, or exactly-once delivery. Handler errors are logged by the implementation; the v1 `RedisEventBus` does not retry. Future at-least-once implementations SHOULD support configurable retry with dead-letter after exhaustion.

**Delivery-guarantee transition protocol.** Replacing `RedisEventBus` with an at-least-once implementation (NATS JetStream, Kafka) changes runtime delivery behavior for all subscribers. The "notifications not commands" design (above) makes handlers structurally idempotent, but subtle violations can hide in handlers that perform unguarded side effects (e.g., incrementing a metric per event rather than per state change, or logging at a rate that becomes problematic under replay). Before enabling a durable EventBus implementation in production, operators MUST execute the following validation sequence in a staging environment:

1. **Duplicate-injection test.** Configure the staging EventBus to publish every event twice (configurable via `eventBus.duplicateInjectionFactor: 2` test-only Helm value). Run the full integration test suite. Verify that no handler produces doubled side effects — specifically, check that business-metric counters (e.g., delegation tree completion counts) do not correlate with the doubled publish rate.
2. **Redelivery-after-crash test.** Kill a subscriber mid-handler (SIGKILL, not SIGTERM). Verify that the at-least-once implementation redelivers the in-flight event to another subscriber and that the handler converges to the same outcome as a clean first delivery — i.e., the durable state read in the handler returns the same result regardless of whether the prior handler completed.
3. **Metric audit.** Review all `lenny_event_bus_handler_*` metric emissions for counters that increment per event without deduplication. Any counter that is semantically "per-occurrence" (not "per-state-change") must be guarded by a durable-state check or converted to a gauge.

These checks are one-time validation steps during the backend swap, not ongoing CI obligations.

**Tenant isolation.** The `tenantID` parameter is mandatory. The implementation constructs the underlying channel name by prefixing with the tenant ID (e.g., Redis channel `t:{tenant_id}:evt:{topic}`). Callers never construct raw channel names. This ensures tenant isolation is enforced at the interface boundary, not by caller convention. The `t:{tenant_id}:evt:{topic}` channels follow the [§12.4](#124-redis-ha-and-failure-modes) tenant-prefix key convention and must be added to the canonical key table and to `TestRedisTenantKeyIsolation` coverage.

**Future: partition key for per-tree subscriber filtering.** At scale, a gateway replica coordinating one root session receives all `TopicDelegationTree` events for the entire tenant — wasteful when the tenant has many concurrent trees. A future interface revision can add an optional `partitionKey string` parameter to `Publish` and `Subscribe`. When set (e.g., to `root_session_id`), the implementation appends it to the channel name (`t:{tenant_id}:evt:delegation_tree:{root_session_id}`), so each subscriber receives only events for the trees it coordinates. When empty, the channel is tenant-wide (current behavior). In Redis Cluster, the `{tenant_id}` hash tag is still the first `{...}` match, so all EventBus channels for a tenant remain on the same shard — the partition key provides subscriber filtering, not hash-slot distribution. In NATS, the partition key maps to a hierarchical subject segment with wildcard subscribe available for monitoring; in Kafka, it becomes the partition key for both filtering and distribution. This change is additive — it does not require modifying existing callers (they pass `""`) and is compatible with the [§12.4](#124-redis-ha-and-failure-modes) key prefix convention.

**Observability contract.** Every `EventBus` implementation MUST emit: `lenny_event_bus_publish_total{topic}` (counter), `lenny_event_bus_publish_duration_seconds{topic}` (histogram), `lenny_event_bus_handler_duration_seconds{topic}` (histogram), and `lenny_event_bus_handler_error_total{topic}` (counter). The `topic` label is the `EventTopic` constant value (`delegation_tree`, `session_lifecycle`). Handler duration measures time spent in the caller-supplied handler function, not the pub/sub transport. These metrics enable operators to detect silent publish failures, slow handlers, and handler error accumulation — particularly important when diagnosing delegation tree signal delays or missed session lifecycle events.

**Infrastructure pub/sub exclusion.** Three pub/sub pathways in the system intentionally bypass EventBus: circuit breaker state propagation ([§11.6](11_policy-and-controls.md#116-circuit-breakers), channel `circuit_breaker_events`), credential deny list propagation ([§4.9](04_system-components.md#49-credential-leasing-service)), and certificate deny list propagation ([§10.3](10_gateway-internals.md#103-mtls-pki)). These are **gateway-internal replica synchronization** mechanisms that share three properties: (1) they are platform-wide, not tenant-scoped; (2) they use a poll-with-pub/sub-hint pattern where the pub/sub channel is a latency optimization, not a delivery guarantee (each has a polling fallback and startup-read path); (3) they are low-volume and do not benefit from a future NATS/Kafka migration. Routing these through EventBus would require either a nil `TenantID` (undermining the isolation contract) or a separate method signature (complicating the interface for no operational benefit).

### 12.8 Compliance Interfaces

**T4 ephemeral workspace isolation.** Node-level disk encryption ([Section 6.4](06_warm-pod-model.md#64-pod-filesystem-layout)) does not provide per-tenant key isolation for T4 workspace data on emptyDir — a node compromise exposes all T4 tenants co-located on that node. To enforce per-tenant key isolation at the node boundary, T4 workloads are required to run on dedicated node pools ([Section 6.4](06_warm-pod-model.md#64-pod-filesystem-layout) T4 dedicated-node requirement). The `lenny-t4-node-isolation` validating webhook ([Section 6.4](06_warm-pod-model.md#64-pod-filesystem-layout)) enforces this at admission time and is fail-closed. This control is in addition to the SSE-KMS per-tenant key requirement for MinIO artifacts ([Section 12.5](#125-artifact-store)), which covers durable (post-checkpoint) workspace data.

**Legal hold.** Sessions and artifacts support a `legal_hold` boolean flag. When set, the artifact retention policy is suspended — artifacts are not deleted by the GC job regardless of TTL. Legal holds are set and cleared via the admin API (`POST /v1/admin/legal-hold`), which accepts a session ID or artifact ID and the desired hold state.

**Legal hold and checkpoint rotation.** When `legal_hold` is set on a session or artifact, **all** retention rotation policies for that session are suspended — the GC job skips both TTL-based deletion and the "latest 2 checkpoints" rotation for any session whose `legal_hold` flag is `true`. Checkpoints accumulate until the hold is lifted; only then does the normal rotation policy resume and trim to the "latest 2" limit. This prevents checkpoint spoliation — discarding intermediate forensic state while a hold is in effect constitutes destruction of potentially discoverable evidence. **Storage accounting and operator responsibility.** Storage growth for long-held sessions is accepted as a compliance cost; v1 does **not** define a per-tenant legal-hold storage cap or an opt-out that would allow rotation to override a hold. Legal-hold bytes continue to count against the tenant's `storageQuotaBytes` bucket — at 100% utilization, new checkpoint writes for that tenant will be rejected with `STORAGE_QUOTA_EXCEEDED` exactly as they would be for any other artifact class (see [Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas) storage quota enforcement mechanism). Operators MUST therefore (1) monitor `lenny_checkpoint_storage_bytes_total` (gauge, labeled by `tenant_id` and `pool`) and the `CheckpointStorageHigh` / `StorageQuotaHigh` alerts ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)), (2) allocate additional `storageQuotaBytes` for tenants with active or expected holds using the per-pod accumulation rates documented in [Section 12.5](#125-artifact-store) (Checkpoint-storage sizing guidance), and (3) raise the quota administratively via `PUT /v1/admin/tenants/{id}` ([Section 15.1](15_external-api-surface.md#151-rest-api)) when a hold is applied to a high-volume session rather than relying on a fixed pre-allocation. **Predictive early-warning signal (STO-020).** Beyond the reactive 80% `CheckpointStorageHigh` / `StorageQuotaHigh` thresholds, the `LegalHoldCheckpointAccumulationProjectedBreach` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when the tenant's projected 24-hour hold growth plus current usage would exceed 90% of `storageQuotaBytes`. Projection inputs: active hold count (`lenny_tenant_legal_hold_active_count`), each held session's `periodicCheckpointIntervalSeconds`, and the tenant's average observed checkpoint size over a trailing 1-hour window; the controller emits the computed 24-hour projected growth as `lenny_legal_hold_checkpoint_projected_growth_bytes` (gauge, labeled by `tenant_id` and `root_session_id`) so operators can dashboard projected pressure before any threshold is crossed. The alert routes to the `docs/runbooks/legal-hold-quota-pressure.md` runbook ([Section 17.7](17_deployment-topology.md#177-operational-runbooks)), whose decision tree is: (a) enumerate active holds via audit trail, (b) either raise `storageQuotaBytes` via `PUT /v1/admin/tenants/{id}` or accept the held-session checkpoint freeze. No platform-initiated action — the alert is a notification, not an enforcement handoff. This preserves the "no forced deletion during a hold" invariant while eliminating the silent-cascade failure mode where operators first discover the breach via cascading `STORAGE_QUOTA_EXCEEDED` errors.

**Legal hold reconciler.** A background reconciler (co-located with the GC goroutine, running every 15 minutes) scans for sessions where `legal_hold = true` and one or more checkpoints have already been rotated (i.e., there are gaps in the checkpoint sequence recorded in the `artifact_store` table). When such a session is found, the reconciler emits a `legal_hold.checkpoint_gap_detected` critical audit event and increments the `lenny_legal_hold_checkpoint_gaps_total` counter. This surfaces any pre-existing rotation that occurred before the hold was set so that compliance and legal teams can assess the impact. The reconciler does not attempt to recover deleted checkpoints — it provides detection and audit trail only.

**Data erasure (GDPR).** Each store interface includes `DeleteByUser(user_id)` and `DeleteByTenant(tenant_id)` methods. Erasure is implemented as a background job that runs to completion and produces an erasure receipt stored in the audit trail. These methods are **mandatory** on every store interface, not optional extension points — the Go type system enforces this at compile time so that a backend missing either method cannot be wired into the platform. For the `MemoryStore` role specifically (which is pluggable — see [Section 12.2](#122-storage-roles) and [Section 9.4](09_mcp-integration.md#94-memory-store)), the interface definition in Section 9.4 lists `DeleteByUser(ctx, tenantID, userID) error` and `DeleteByTenant(ctx, tenantID) error` as required methods. Deployers who substitute a custom vector database backend (Mem0, Zep, or similar) MUST implement both methods so that user and tenant erasure actually removes memories from the underlying store.

**MemoryStore erasure preflight.** Because `MemoryStore` is the only erasure-scope store that deployers may replace at runtime with a non-default backend, the erasure job performs a two-layer preflight before step 8 of the `DeleteByUser` sequence (and the corresponding step in tenant deletion Phase 4):

1. **Compile-time interface satisfaction (primary defense).** Every concrete `MemoryStore` implementation wired into the gateway binary MUST satisfy the full interface declared in [Section 9.4](09_mcp-integration.md#94-memory-store), including `DeleteByUser` and `DeleteByTenant`. Go enforces this at compile time via `var _ MemoryStore = (*CustomStore)(nil)` declarations in each backend package — a backend that fails to implement either method will not link. The gateway never performs runtime reflection to "detect" missing methods; a binary that boots has, by construction, a `MemoryStore` with both methods present.
2. **Startup preflight (stub detection).** To catch no-op stub implementations that satisfy the signature without performing deletion, the gateway runs a `ValidateMemoryStoreErasure(ctx, store MemoryStore)` check at startup as part of the existing `ValidateMemoryStoreIsolation` contract helper ([Section 9.4](09_mcp-integration.md#94-memory-store)). The helper writes a known memory row under a synthetic `(tenantID=__preflight__, userID=__preflight_user__)` scope, invokes `DeleteByUser(ctx, "__preflight__", "__preflight_user__")`, then issues a `Query` for the same scope and asserts zero rows are returned. If the query returns non-zero rows — indicating the backend's `DeleteByUser` is a silent no-op — the gateway **refuses to start** with the fatal error `FATAL: MemoryStore preflight failed — configured backend (<backend_name>) does not honor DeleteByUser; GDPR erasure would silently succeed while leaving memories in place (Section 12.8)`. The preflight also covers `DeleteByTenant` using the same seeded row. The preflight is skipped only when the feature flag `memory.enabled=false` is set (no MemoryStore wired at all).
3. **Per-job preflight (defense in depth).** The erasure job re-runs the seeded-row check (scoped to a synthetic `__preflight_user__` under a reserved preflight `tenantID`) at the start of each job before step 8. If the per-job preflight fails, the job aborts with `memory_store_preflight_failed`, sets `processing_restricted: true` on the target user remains in place (it is not cleared), increments `lenny_erasure_job_failed_total{failure_phase="memory_store_preflight"}`, and fires the `ErasureJobFailed` alert ([§12.8](#128-compliance-interfaces)). This ensures that a backend which passes startup preflight but later regresses (e.g., after a rolling upgrade of the external vector DB) does not silently fail erasure.

**Deployment guidance for custom MemoryStore backends.** Deployers substituting a pluggable `MemoryStore` backend are responsible for the following (captured in the deployment guide and enforced by the preflight above):

- Implement `DeleteByUser(ctx, tenantID, userID) error` to remove every record the backend holds for `(tenantID, userID)`. The operation MUST be synchronous from the caller's perspective — it may not return `nil` while deletion is still eventually-consistent on the underlying index/shard.
- Implement `DeleteByTenant(ctx, tenantID) error` with the same synchrony requirement for tenant-wide purge.
- Reject empty `tenantID` (and empty `userID` in `DeleteByUser`) with an error; never treat empty arguments as "delete everything".
- Implement both methods idempotently — repeated calls for the same scope after successful deletion MUST return `nil`.
- Pass the `ValidateMemoryStoreIsolation` / `ValidateMemoryStoreErasure` contract tests published in the Lenny SDK before the backend is wired into production. CI in the deployer's fork is expected to run these tests against the real backend (not a mock).
- Emit the `lenny_memory_store_operation_duration_seconds`, `lenny_memory_store_errors_total`, `lenny_memory_store_record_count`, and `lenny_memory_store_user_count_over_threshold_total` metrics ([Section 9.4](09_mcp-integration.md#94-memory-store)) so that the `MemoryStoreErasureDurationHigh` and `MemoryStoreGrowthHigh` alerts function identically to the default backend. Specifically, the `operation_duration_seconds` histogram MUST emit observations under `operation="delete_by_user"` for every `DeleteByUser` call and under `operation="delete_by_tenant"` for every `DeleteByTenant` call (distinct from the per-record `operation="delete"` label used by `MemoryStore.Delete(ctx, scope, ids)`); `MemoryStoreErasureDurationHigh` ([§16.5](16_observability.md#165-alerting-rules-and-slos)) is keyed on these whole-scope erasure labels at a P99 threshold of 60 s for per-user erasure and 300 s for per-tenant erasure, sustained for 10 minutes. Backends whose erasure semantics are implemented internally as a loop over per-record `Delete` calls MUST still emit a single histogram observation under `delete_by_user` / `delete_by_tenant` reflecting the wall-clock duration of the whole synchronous call (the per-record observations are additional, not a substitute), so the alert predicate fires on the SLO-relevant signal rather than on sub-operation noise.

Failure to meet any of the above is a GDPR Article 17 (and HIPAA, for PHI-bearing deployments) compliance risk: the audit trail would record a successful erasure receipt while memories persist in the backend. The compile-time interface and the two preflight layers are the platform's fail-closed guarantee that this divergence cannot occur silently.

**User-level erasure invocation.** User-level erasure is triggered via the Admin API: `POST /v1/admin/users/{user_id}/erase` initiates the background erasure job and returns a job ID. The job status is queryable via `GET /v1/admin/erasure-jobs/{job_id}`, which returns the current phase, completion percentage, time elapsed, and any errors. These endpoints are listed in [Section 15.1](15_external-api-surface.md#151-rest-api).

**Erasure job phase field (crash recovery).** Each erasure job record persists a `phase` field (enum: `initiated | store_deleting | pseudonymizing | verifying | completed | failed`) in Postgres. The phase is updated atomically with each step transition. If the controller crashes and restarts, it reads the persisted `phase` to determine the safe resume point: jobs in `store_deleting` re-run `DeleteByUser` (all store deletions are idempotent); jobs in `pseudonymizing` check whether the billing pseudonymization transaction already committed before re-applying; jobs in `verifying` re-run the salt-deletion verification check. This prevents double-deletion errors and ensures crash recovery cannot silently skip a deletion step.

**Processing restriction during erasure (GDPR Article 18).** When an erasure job is initiated, the platform sets `processing_restricted: true` on the user record in the `UserStore`. While this flag is set: (1) all session creation endpoints (`POST /v1/sessions` and `POST /v1/sessions/start`) are rejected with `403 ERASURE_IN_PROGRESS` for the affected `user_id`; (2) the error body includes `{"code": "ERASURE_IN_PROGRESS", "userId": "<user_id>", "jobId": "<job_id>", "message": "This user has a pending erasure request. New sessions cannot be created until erasure completes."}`. The restriction is enforced at the session creation gate upstream of endpoint-specific logic — both endpoints share the same gate. In-flight sessions that are already running at erasure initiation time are allowed to complete naturally (they will be erased by the job). The `processing_restricted` flag is cleared atomically when the erasure job writes its completion receipt — after all stores have confirmed deletion or pseudonymization. If the erasure job fails, the flag remains set and the job must be retried or cleared explicitly via `POST /v1/admin/erasure-jobs/{job_id}/clear-processing-restriction` (requires `platform-admin`; this endpoint records the operator identity and justification in the audit trail). The restriction is scoped to the specific `user_id`; other users in the same tenant are unaffected. **Database-level enforcement:** To prevent the `processing_restricted` flag from being cleared by application bugs, migration errors, or direct database manipulation outside the erasure job, the `UserStore` table carries a `CHECK` constraint that enforces: `processing_restricted = false` is only valid when no row exists in the `erasure_jobs` table for that `user_id` with `status IN ("initiated", "store_deleting", "pseudonymizing", "verifying")`. This is implemented as a `BEFORE UPDATE` trigger on the `processing_restricted` column: if the new value is `false` and an active erasure job exists for the user, the trigger raises an exception (`ERASURE_JOB_ACTIVE: cannot clear processing_restricted while erasure job <job_id> is in progress`). The trigger exempts the `lenny_erasure` database role (which clears the flag as part of the erasure completion receipt transaction) and the explicit `clear-processing-restriction` admin endpoint (which sets a session-local variable analogous to `lenny.erasure_mode`). This ensures GDPR Article 18 (right to restriction of processing) is enforced at the database level, not solely at the application layer.

**Erasure job failure alerting (CMP-026).** A failed erasure job leaves `processing_restricted: true` set on potentially many users simultaneously if the failure is batch-wide. To prevent silent service denial, the platform applies two controls: (1) **Counter:** `lenny_erasure_job_failed_total` (counter, labeled by `tenant_id`, `failure_phase: store_delete|verification|pseudonymization`) is incremented on each job failure; (2) **Alert:** `ErasureJobFailed` (Warning severity, [§16.5](16_observability.md#165-alerting-rules-and-slos)) fires immediately when `lenny_erasure_job_failed_total` increases, with the `job_id`, `user_id`, and `failure_phase` in the alert labels. The alert body directs operators to retry via `POST /v1/admin/erasure-jobs/{job_id}/retry` or clear the processing restriction manually with justification via `POST /v1/admin/erasure-jobs/{job_id}/clear-processing-restriction`. This alert is separate from `ErasureJobOverdue` (which fires on SLA breach); `ErasureJobFailed` fires on explicit failure so operators are notified without waiting for a deadline to elapse.

**Erasure SLA enforcement.** Erasure jobs must complete within the tier-specific deadlines defined in [Section 12.9](#129-data-classification): T3 data within 72 hours, T4 data within 1 hour. The erasure job records its start time and the applicable deadline. If the job has not completed by the deadline, the `ErasureJobOverdue` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires. Metrics: `lenny_erasure_job_duration_seconds` (histogram) tracks job completion time; `lenny_erasure_jobs_active` (gauge) tracks in-progress jobs. These metrics enable operators to monitor erasure throughput and detect stalled jobs before they breach the SLA.

**`DeleteByUser` dependency-ordered deletion sequence.** The `DeleteByUser(user_id)` erasure job executes store deletions in the following dependency order. Each step is individually idempotent to support crash recovery (see Erasure job phase field above). Foreign-key and referential constraints dictate that child/referencing stores are deleted before parent/referenced stores.

0. **Legal-hold preflight (CMP-049, fail-closed).** Before any destructive operation, the erasure job enumerates active legal holds scoped to the target `user_id`. The check covers: (a) sessions owned by the user where `legal_hold = true` (from `sessions.legal_hold`); (b) artifacts whose owning session belongs to the user and whose `legal_hold = true` (from `artifacts.legal_hold`); (c) audit-range or workspace-snapshot holds recorded in the legal-hold ledger (`legal_hold.set` audit events with no matching `legal_hold.cleared` successor, where the hold's `resourceType` is `session`, `artifact`, `audit_range`, or `workspace_snapshot` and the resolved resource is owned by the target user). If **any** active hold is present, the erasure job **MUST abort before step 1** with error code `ERASURE_BLOCKED_BY_LEGAL_HOLD` (HTTP 409, `POLICY` category), set the job status to `failed` with `failure_phase: "legal_hold_preflight"`, increment `lenny_erasure_job_failed_total{failure_phase="legal_hold_preflight"}`, and emit a `gdpr.erasure_blocked_by_hold` critical audit event carrying `user_id`, `tenant_id`, `job_id`, `hold_count`, and an array of `{resourceType, resourceId, holdSetAt, holdSetBy, note}` tuples describing the blocking holds. The `processing_restricted` flag is **not** set on the target user when the preflight blocks (the erasure never actually initiates, so Article 18 restriction-of-processing is not triggered); the admin receives a synchronous rejection at `POST /v1/admin/users/{user_id}/erase` and may either release the holds via `POST /v1/admin/legal-hold` (with `hold: false`) before re-invoking erasure, or override the preflight via the `--acknowledge-hold-override` admin action (see below). The preflight is symmetric with the post-restore reconciler's legal-hold ledger freshness gate ([§12.8](#128-compliance-interfaces) "Post-restore reconciler" phase 2, §25.11 confirm-legal-hold-ledger endpoint) — the reconciler gate protects replay-after-restore; this preflight protects first-time execution. Destroying evidence subject to preservation orders constitutes spoliation; the preflight is mandatory and fail-closed.

   **`--acknowledge-hold-override` admin action.** An operator with the `platform-admin` role (tenant-admins cannot self-override) may invoke `POST /v1/admin/users/{user_id}/erase` with request body `{"acknowledgeHoldOverride": true, "justification": "<required free-text>"}` to bypass the preflight. The override is accepted only when `justification` is present and non-empty and the caller's role is `platform-admin`; missing `justification` returns `400 INVALID_REQUEST`, a `tenant-admin` caller is rejected with `403 FORBIDDEN`. On a successful override, the erasure job records the override in its receipt (`legal_hold_override: true`, `override_by`, `override_justification`, `override_at`, and the full list of holds that would otherwise have blocked it) and emits a `gdpr.legal_hold_overridden` critical audit event (separate event type from `gdpr.erasure_blocked_by_hold`) carrying the same fields plus `job_id`. The override does **not** clear the underlying legal-hold rows — they remain set on `sessions`/`artifacts` and in the ledger — so that any parallel hold audit or e-discovery export can still observe that the hold was in force when the override was exercised. After the override is recorded, the erasure proceeds to step 1 normally. Operators should treat `--acknowledge-hold-override` as a last resort (e.g., a hold erroneously applied by an automated system and not clearable in time to meet a conflicting regulatory deadline) and document the justification in their legal-hold review process; the `gdpr.legal_hold_overridden` event is surfaced in the SIEM stream and raises the `LegalHoldOverrideUsed` warning alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) so that compliance teams see every override.

14. **OCSF dead-letter PII redaction.** For every `audit_log` row whose `ocsf_translation_state = 'dead_lettered'` and whose `user_id` column equals the target user (equivalently, whose `payload` carries the target user as actor or subject — see [§11.7](11_policy-and-controls.md#117-audit-logging) Lenny → OCSF field mapping), the erasure job redacts the PII-bearing payload fields in place: the `payload` column is rewritten with `{"redacted": true, "redacted_at": "<timestamp>", "user_id_redacted_from_row": true, "error_class": "<preserved from the original failure labels>", "event_type": "<preserved>"}`; the `payload_canonical_json` generated column is recomputed from the redacted payload; the `user_id` column is set to `NULL`; any derived columns that mirror user-identifying fields are set to `NULL`. The `id`, `tenant_id`, `sequence_number`, `prev_hash`, `event_type`, `event_schema_version`, `created_at`, and `ocsf_translation_state` columns are **preserved** so that the hash chain remains verifiable and operators retain pipeline-failure forensics (error class, failure phase, attempt count). Because the redaction is a rewrite of `payload` (not a row delete), the pre-commit hash chain must be re-sealed: the erasure job recomputes the per-row hash using the redacted canonical tuple, writes the new value back to the row's `prev_hash`-equivalent position and to the immediately subsequent row's `prev_hash` input so the chain is re-sealed across the rewrite, sets the row's `chainIntegrity` verifier state to `redacted_gdpr` (see [§11.7](11_policy-and-controls.md#117-audit-logging) `chainIntegrity` enumeration and the `lenny_audit_chain_integrity_total{state="redacted_gdpr"}` counter in [§16.1](16_observability.md#161-metrics) Audit Integrity), persists a signed **`RedactionReceipt`** (schema defined immediately below) pinning the discontinuity to a specific erasure job and legal basis, and records a `gdpr.erasure_deadletter_redacted` audit event per redacted row carrying `audit_event_id`, `tenant_id`, `original_event_type`, `original_error_class`, `redacted_at`, `pre_redaction_prev_hash`, `post_redaction_prev_hash`, and `redaction_receipt_id` (so that external hash-chain verifiers can migrate their snapshot past the rewrite — the chain is explicitly discontinuous at redaction points, analogous to the crypto-shredded boundary described in backup reconciler phase 3). Retranslation (`POST /v1/admin/audit-events/{id}/retranslate`) is rejected with `410 DEADLETTER_REDACTED` for redacted rows, since the original payload is unrecoverable. Chain verifiers that encounter a `redacted_gdpr` row without a signature-verifying `RedactionReceipt` MUST raise `broken` and fire `AuditRedactionReceiptMissing` ([§16.5](16_observability.md#165-alerting-rules-and-slos)) — the receipt is the sole provenance token that distinguishes an authorized GDPR redaction from a tamper that happens to clear `payload`. This ensures that a user's raw canonical PII (subject identifier, session payload snippets) cannot linger in a dead-lettered row past the erasure SLA while still preserving the dead-letter's operational utility for pipeline debugging.

    **`RedactionReceipt` schema.** Stored in a new append-only `audit_redaction_receipts` table (Postgres) co-located with the audit shard of the redacted row. One receipt per redacted `audit_log` row. Columns and fields:

    - `receipt_id` (UUID, primary key) — referenced from the `gdpr.erasure_deadletter_redacted` audit event as `redaction_receipt_id`.
    - `audit_event_id` (UUID, foreign key `audit_log.id`) — the redacted row.
    - `tenant_id` (text) — mirrors the redacted row's tenant scope.
    - `sequence_number` (bigint) — mirrors the redacted row's per-tenant sequence; lets verifiers locate the receipt by position rather than UUID.
    - `original_hash` (bytea, 32 bytes) — SHA-256 of the canonical tuple as hashed **before** redaction (i.e., the pre-redaction `prev_hash` input for the immediately subsequent row).
    - `new_hash` (bytea, 32 bytes) — SHA-256 of the canonical tuple as hashed **after** redaction (the post-redaction value used to re-seal the chain at the rewrite boundary and written into the subsequent row's `prev_hash`).
    - `erasure_job_id` (UUID) — the `DeleteByUser` job that performed the redaction; correlates every receipt in a single erasure to one job row.
    - `legal_basis` (text, enum) — one of `gdpr_art17` (Right to erasure, Article 17), `gdpr_art17_with_art17_3_exception` (Article 17 with a recognized retention exception), or `operator_acknowledged_override` (used only when paired with the legal-hold override path from Step 0, Section [§12.8](#128-compliance-interfaces) CMP-049). Additional values MUST NOT be introduced without a spec revision; this keeps the ground-truth basis auditable.
    - `redactor_identity` (text) — the identity that authorized the redaction, using the same subject-identifier format as `actor.user.uid` in §11.7 OCSF mapping (e.g., `platform-admin:<sub>` or `erasure-job:<job_id>`).
    - `timestamp` (timestamptz, NOT NULL) — the `redacted_at` instant captured when the in-place payload rewrite committed. Equals the `redacted_at` field on the corresponding `gdpr.erasure_deadletter_redacted` event.
    - `signature` (bytea) — detached signature over the canonical JCS serialization of the tuple `(receipt_id, audit_event_id, tenant_id, sequence_number, original_hash, new_hash, erasure_job_id, legal_basis, redactor_identity, timestamp)`. Signed by the platform's KMS-held audit signing key (same key as [§11.7](11_policy-and-controls.md#117-audit-logging) genesis-nonce signing in regulated profiles). External verifiers fetch the public key from the admin API and verify signatures before accepting a `redacted_gdpr` discontinuity as authorized.
    - `signature_kms_key_id` (text) — the KMS key identifier used to produce `signature`; recorded inline so a later key rotation leaves prior receipts independently verifiable.

    The table is grant-restricted: `lenny_erasure` holds `INSERT` only; `lenny_app` holds `SELECT` only (for chain-verification reads); no role holds `UPDATE` or `DELETE`. Receipts are retained for `audit.gdprRetentionDays` in lockstep with the `gdpr.*` audit events they reference — a receipt outlives the redacted row if the partition containing the row is ever force-dropped, but the receipt itself is never purged before its referenced receipt event ([§16.7](16_observability.md#167-section-25-audit-events)). Because `audit_redaction_receipts` holds no personal data (hashes and identifiers only, by construction), the table is exempt from `DeleteByUser` and `DeleteByTenant` under the same rationale as `gdpr.*` receipts.

    **Downstream SIEM erasure notification (CMP-051, GDPR Art. 17(2)).** Because the original dead-lettered row was already forwarded to every configured OCSF sink before redaction (the class 2004 translation-failure receipt embedding the raw canonical payload as `unmapped.lenny.raw_canonical_b64` — see [§11.7](11_policy-and-controls.md#117-audit-logging) Dead-letter handling), the pre-redaction canonical PII may exist in downstream SIEM stores outside Lenny's cryptographic control. For every dead-letter row redacted under Step 14, the erasure job MUST emit a structured **`gdpr.erasure_deadletter_downstream_notified`** audit event that flows through the normal OCSF translator to the same sinks that received the original dead-letter receipt (SIEM forwarder, pgaudit sink consumer, every subscribed webhook). The event is mapped to OCSF class **5001 Entity Management** with `activity_id: 4` (`Delete`) and `category_uid: 5` (Identity & Access Management / Audit Activity) so SIEM rulesets can recognize it as an authoritative erasure signal on a previously-ingested entity. Payload fields: `audit_event_id` (the `id` of the redacted `audit_log` row — identical to the `audit_event_id` on the matching `gdpr.erasure_deadletter_redacted` event so a SIEM operator can join the two), `tenant_id`, `original_event_type`, `original_sequence_number`, `original_hash` (the pre-redaction `original_hash` from the signed `RedactionReceipt` so the SIEM operator can locate the previously-ingested record without Lenny having to know its downstream storage key), `erasure_job_id`, `legal_basis` (mirrors the `RedactionReceipt.legal_basis` enum: `gdpr_art17` \| `gdpr_art17_with_art17_3_exception` \| `operator_acknowledged_override`), `redaction_receipt_id`, `redacted_at`, and `downstream_action_required: true` (a constant flag that SIEM ingestion rules may use to route the event into an operator-review queue). The event is emitted under the same per-tenant advisory-locked write path as all audit events, is retained under `audit.gdprRetentionDays` (it is a `gdpr.*` event), and is itself **not** subject to OCSF translation failure dead-lettering of the same kind — because it carries no raw canonical PII (only hashes and identifiers), a translator failure on this event cannot leak personal data. Downstream erasure is out of Lenny's cryptographic control (SIEM retention and deletion follow the SIEM operator's own policies and backup schedule, not Lenny's); the notification is the **`reasonable steps to inform other processors holding the data`** Lenny takes under GDPR Art. 17(2). Operators MUST document downstream SIEM erasure responsibility in their data-processing agreement — see [§25.9](25_agent-operability.md#259-audit-log-query-api) Downstream SIEM scope boundary for the operator runbook entry.
15. `EventStore` (billing) — pseudonymize or exempt billing events per `billingErasurePolicy` (see tenant-controlled billing erasure below).
16. Delegation budget keys (Redis) — enumerate the user's root session IDs (`SELECT id FROM sessions WHERE user_id = $1 AND parent_session_id IS NULL` via `AllSessionShards()` scatter-gather). For each `root_session_id`, delete tree-wide keys (`{root_session_id}:dlg:tokens`, `{root_session_id}:dlg:tree_size`, `{root_session_id}:dlg:tree_memory`) and scan for per-parent keys (`{root_session_id}:dlg:parallel_children:*`, `{root_session_id}:dlg:children_total:*`) using slot-local `SCAN` (all keys share the same `{root_session_id}` hash tag, so a single-slot scan covers them). Route through `RedisShard(tenantID, RedisConcernDelegation)`. Must precede `SessionStore` deletion — after session rows are removed, root session IDs are irrecoverable.

**`gdpr.*` audit event exemption from user-level deletion.** Erasure receipts and related compliance audit events (event type prefix `gdpr.*`, e.g., `gdpr.erasure_completed`, `gdpr.erasure_verification_failed`) are written to the `EventStore` (audit) as part of the erasure job's completion phase. These records must survive subsequent erasure or deletion operations — they are the authoritative proof that a GDPR erasure was carried out. `DeleteByUser` MUST filter out rows where `event_type LIKE 'gdpr.%'` — these rows are never deleted by user-level erasure jobs, regardless of which `user_id` is being erased. `DeleteByTenant` (Phase 4 of tenant deletion) similarly skips `gdpr.*` rows.

**GDPR erasure receipt retention.** `gdpr.*` rows are subject to a dedicated retention window, **separate from** the general `audit.retentionDays` setting. The audit GC MUST NOT purge `gdpr.*` rows based on the standard partition TTL; instead it applies `audit.gdprRetentionDays` (default: **2555 days / 7 years**) as a minimum floor. This ensures erasure receipts outlive both the erased user's data and any subsequent tenant deletion, and satisfies GDPR enforcement windows of 4–6 years. Deployers that require a longer retention window (e.g., national data-protection authority investigation periods beyond 7 years) may increase `audit.gdprRetentionDays` without affecting the general audit retention window. Helm configuration: `audit.gdprRetentionDays: 2555`. This value may not be set below 2190 (6 years) when `complianceProfile` is any regulated value (`soc2`, `fedramp`, `hipaa`); the gateway rejects the configuration at startup with `CONFIG_INVALID: audit.gdprRetentionDays below compliance floor`.

**Tenant-controlled billing erasure.** Billing events use append-only semantics with monotonic sequence numbers ([Section 11.2.1](11_policy-and-controls.md#1121-billing-event-stream)). Deleting billing rows would break gap detection and audit integrity. By default, the erasure job **pseudonymizes** billing events: `user_id` is replaced with a one-way hash (`SHA-256(user_id || erasure_salt)`) and any free-text fields that could contain PII are cleared. The pseudonymized events retain their `sequence_number`, `tenant_id`, and cost dimensions so that financial reconciliation remains intact.

**`erasure_salt` key management.** The `erasure_salt` is a **per-tenant** cryptographic secret used exclusively for billing pseudonymization. Storage and lifecycle:

- **Storage:** The salt is stored in the tenant configuration record in Postgres, encrypted at rest using envelope encryption via KMS — the same pattern used for OAuth refresh tokens ([Section 4.3](04_system-components.md#43-token-service)). The salt is never stored in plaintext or alongside the pseudonymized billing values.
- **Scope:** Per-tenant. Each tenant has its own independently generated salt (256-bit, generated via `crypto/rand`). A per-tenant salt ensures that pseudonymized `user_id` values from one tenant cannot be correlated with values from another tenant, even by an adversary with access to both tenants' billing data.
- **Access control:** Only the erasure job reads the salt, and only during pseudonymization. The KMS decrypt permission required to unwrap the salt is scoped to the erasure job's service account — gateway replicas and runtime pods cannot decrypt it.
- **Immediate deletion after pseudonymization (GDPR erasure compliance).** Under GDPR Recital 26, retaining the `erasure_salt` in the same database as the pseudonymized billing records means the data remains personal — the salt enables re-identification. To constitute effective erasure, the `erasure_salt` **must be deleted from the tenant configuration record immediately after pseudonymization completes** for a given user-level erasure job. The deletion sequence is:
  1. The erasure job pseudonymizes all billing events for the target `user_id` using the current `erasure_salt` (in a single database transaction).
  2. In the same transaction, the `erasure_salt` column in the tenant configuration is set to `NULL` and the KMS-wrapped ciphertext is deleted.
  3. **Verification step:** After the transaction commits, the erasure job re-derives the hash of a known `(user_id, erasure_salt)` pair using an in-memory copy of the salt held for this purpose only. It then queries the billing event table to confirm no row exists with the original `user_id`. Additionally, it attempts to re-derive the pseudonymized value from Postgres (the salt is now `NULL` in the database) — this attempt must fail with a `NULL`-dereference or decryption error, confirming the salt is gone from persistent storage. The verification outcome is recorded in the erasure receipt.
  4. If verification fails (salt is still resolvable from Postgres), the erasure job marks the job as `verification_failed`, emits a `gdpr.erasure_verification_failed` critical audit event, and halts — the erasure is not marked complete until the failure is investigated and resolved.
  5. The in-memory copy of the salt used for verification is zeroed immediately after the verification step completes (via `runtime.SetFinalizer` or explicit `crypto/subtle`-style zero, depending on language runtime).
  - **Impact on other users:** Because the salt is per-tenant (not per-user), deleting the salt on a per-user erasure request affects the pseudonymization keying for all subsequent erasure requests in that tenant. After deletion, the next erasure request generates a fresh 256-bit salt for that tenant. Previously pseudonymized records from earlier erasures used the prior salt — they remain pseudonymized under the old key, but that key no longer exists anywhere in the system. This is the desired outcome: old pseudonymized records become effectively anonymous because the key needed to re-identify them has been destroyed.
  - **`billingErasurePolicy: exempt` tenants:** For tenants configured as exempt from billing erasure, no pseudonymization occurs and no salt is used or deleted. The erasure receipt records the `exempt` disposition.
- **Rotation:** Salt rotation is supported but optional. On rotation: (1) generate and store the new salt; (2) run a one-time re-hash migration job that re-pseudonymizes all billing events from the old-salt hash to the new-salt hash; (3) delete the old salt only after migration completes and is verified. The old salt is retained in KMS-encrypted form during the migration window. **Erasure during rotation:** If a user-level erasure job is triggered while a salt rotation migration is in progress, the erasure job must wait for the migration to complete before proceeding. The rotation migration holds an advisory lock (`erasure_salt_migration:{tenant_id}`) that the erasure job respects — this prevents the erasure from deleting a salt that the migration is actively using. Once the migration completes and the old salt is deleted, the erasure job proceeds using the new salt, then deletes it per the immediate-deletion rule above.
- **Compromise response:** If a salt is compromised, the tenant admin rotates the salt via the admin API (`POST /v1/admin/tenants/{id}/rotate-erasure-salt`). The compromised salt is deleted immediately (not archived) and a security audit event is emitted.

**GDPR compliance note on pseudonymization.** Under GDPR Recital 26, pseudonymized data remains personal data — it is a risk-reduction technique, not anonymization — as long as the pseudonymization key exists somewhere accessible. The platform addresses this directly: the `erasure_salt` is deleted immediately after pseudonymization completes (see Immediate deletion above), which destroys the only key capable of re-identifying the pseudonymized billing records. Once the salt is deleted and the verification step confirms it is unrecoverable from persistent storage, the pseudonymized billing events become effectively anonymous under GDPR Recital 26. Prior to salt deletion (during the pseudonymization transaction window), deployers must treat the billing events as T3 — Confidential personal data. Tenants with `billingErasurePolicy: exempt` retain original `user_id` values under GDPR Article 17(3)(b) and must document this retention in their Records of Processing Activities.

Tenants that require user-level billing attribution (chargebacks, department allocation, per-user invoicing) may configure billing events as **exempt from user-level erasure** via the tenant configuration (`billingErasurePolicy: exempt`). When exempt, billing events are retained with the original `user_id` intact. This is legally permissible under GDPR Article 17(3)(b) — processing necessary for compliance with a legal obligation or for financial record-keeping. Tenants that enable this exemption accept compliance responsibility for the retained billing data. The erasure receipt records which policy was applied and whether billing events were pseudonymized or exempted.

**`billingErasurePolicy: exempt` with regulated compliance profiles.** When a tenant is created or updated with both `billingErasurePolicy: exempt` and a regulated `complianceProfile` (`hipaa`, `fedramp`, `soc2`), the platform does **not** reject the combination — retaining billing records with original `user_id` can be a legitimate operational need (e.g., HIPAA-covered entities that process billing under 45 C.F.R. §164.502(a)(2)(ii) for payment operations). However, the platform emits a `compliance.billing_erasure_exempt_regulated` audit event at tenant creation or update time. This event carries `tenant_id`, `complianceProfile`, and `billingErasurePolicy: exempt` and appears in the SIEM stream. Its purpose is to make the policy decision visible in the audit trail so that compliance officers and auditors can confirm that the deployer has a documented legal basis for retaining identifiable billing data under their applicable regulatory framework — including the HIPAA minimum-necessary principle (45 C.F.R. §164.502(b)), which requires covered entities to limit PHI use to the minimum necessary to accomplish the intended purpose. Deployers must document their legal basis for this retention in their HIPAA Business Associate Agreement (BAA) or equivalent compliance documentation. The event is also emitted on every gateway startup when any active tenant has this combination, so the posture cannot silently persist across redeployments.

**Tenant deletion lifecycle.** Tenant deletion is a multi-phase process managed by a background controller. Each tenant carries a `TenantState` enum (`active`, `disabling`, `deleting`, `deleted`) persisted in Postgres and exposed via the admin API.

**Legal hold interaction during deletion (CMP-052, fail-closed; Phase 3.5 segregation).** Phase 3.5 is the tenant-scope analog of the `DeleteByUser` Step 0 preflight (CMP-049) and the post-restore reconciler legal-hold ledger freshness gate (CMP-046 / Backups in erasure scope phase 2 above). After Phase 3 completes and before Phase 4 (`DeleteByTenant`) or Phase 4a (tenant KMS key destruction) runs, the deletion controller **MUST** enumerate every active `LegalHold` resource scoped to the tenant. The enumeration covers: (a) sessions under the tenant where `sessions.legal_hold = true`; (b) artifacts whose owning session is under the tenant and whose `artifacts.legal_hold = true`; (c) audit-range or workspace-snapshot holds recorded in the legal-hold ledger (`legal_hold.set` audit events with no matching `legal_hold.cleared` successor, where the hold's `resourceType` is `session`, `artifact`, `audit_range`, or `workspace_snapshot` and the resolved resource belongs to the tenant). If **any** active hold is present, the controller pauses at Phase 3 in state `deleting`, emits an `admin.tenant.deletion_blocked` audit event listing the held resource IDs, and rejects any attempt to advance Phase 4 with `TENANT_DELETE_BLOCKED_BY_LEGAL_HOLD` (HTTP 409, `POLICY`). An operator must either release the holds via `POST /v1/admin/legal-hold` (with `hold: false`) and re-enter the lifecycle so the controller re-evaluates the ledger, or exercise the `--acknowledge-hold-override` admin action below. Destroying tenant-scoped KMS-wrapped ciphertext under Phase 4 / 4a without segregating held evidence would render that evidence cryptographically unrecoverable and constitute spoliation of material subject to preservation orders; the gate is mandatory and fail-closed.

**`--acknowledge-hold-override` admin action (tenant scope).** An operator with the `platform-admin` role (tenant-admins cannot self-override) may invoke `POST /v1/admin/tenants/{id}/force-delete` with request body `{"acknowledgeHoldOverride": true, "justification": "<required free-text>"}` to proceed despite active holds. The override is accepted only when `justification` is present and non-empty and the caller's role is `platform-admin`; missing `justification` returns `400 INVALID_REQUEST` and a `tenant-admin` caller is rejected with `403 FORBIDDEN`. The override does **not** skip Phase 3.5 — it authorizes it. The controller then performs the following four sub-steps, idempotently and in order, before handing control to Phase 4:

1. **Enumerate held resources.** Re-enumerate the active holds using the ledger query described above and pin the resulting `{resourceType, resourceId, holdSetAt, holdSetBy, note}` tuples into the erasure receipt's `overridden_holds` array. The enumeration is performed under the same per-tenant advisory lock used by Phase 4 so that no new sessions or artifacts can acquire a hold mid-migration.
2. **Resolve the target region and re-encrypt under the region-scoped `legal_hold_escrow_kek` (CMP-054, fail-closed).** The controller first resolves the target escrow region from the original tenant's `dataResidencyRegion` (when set) or from the deployment's single-region default (when unset). It then consults the per-region escrow configuration map `storage.regions.<region>.legalHoldEscrow.{endpoint, bucket, kmsKeyId, escrowKekId}` to locate that region's escrow bucket and escrow KEK. If the resolved region has no matching `legalHoldEscrow` entry, or the configured escrow KMS key is unreachable, or the escrow bucket endpoint is unreachable, the controller **MUST** abort Phase 3.5 with error code `LEGAL_HOLD_ESCROW_REGION_UNRESOLVABLE` (HTTP 422, `PERMANENT`; fail-closed mirror of `BACKUP_REGION_UNRESOLVABLE` / `ARTIFACT_REPLICATION_REGION_UNRESOLVABLE`), emit a `DataResidencyViolationAttempt` audit event (critical severity) with `operation: "legal_hold_escrow"`, `requested_region`, `tenant_id`, and `tenant_delete_job_id`, raise the `LegalHoldEscrowResidencyViolation` critical alert ([§16.5](16_observability.md#165-alerting-rules-and-slos)), and leave the tenant in Phase 3.5 `deleting` state pending operator remediation. There is no silent fallback to a default or cross-region escrow bucket — an unresolvable escrow region is a hard error, not a configuration gap to route around. Given a successfully resolved region, for each held resource the controller decrypts the ciphertext under the tenant KEK (`tenant:{tenant_id}`, still live because Phase 4a has not yet run) and re-wraps the data encryption key under the region-scoped **`legal_hold_escrow_kek`** — a platform-managed envelope KEK distinct from every tenant KEK, held in the **region-scoped** platform KMS keyring (`platform:legal_hold_escrow:<region>`), whose decrypt permission is granted only to the platform legal-hold escrow service account in that region and is not delegable to tenant admins. This covers audit-log rows referenced by `audit_range` holds, ArtifactStore objects (workspace snapshots, checkpoints, session transcripts, uploaded files) referenced by `session` / `artifact` / `workspace_snapshot` holds, and any other tenant-scoped ciphertext that the ledger identifies. The escrow KEK is single, long-lived, and **per-region** (never global and never per-tenant) — its semantic role is to outlive any tenant KEK and let held evidence be read after the tenant record is tombstoned, while keeping decrypt capability inside the originating tenant's jurisdiction. Rotation of each regional escrow KEK follows the same KMS rotation policy as the platform audit signing key. A `legal_hold.escrow_region_resolved` INFO audit event is emitted recording `tenant_id`, `tenant_delete_job_id`, `requested_region`, `resolved_region`, and `escrow_kek_id` so the residency-at-write-time decision is observable in the audit trail (parallels `artifact.cross_region_replication_verified`).
3. **Migrate to the region-scoped legal-hold escrow bucket.** Re-encrypted payloads are written to the **region-scoped legal-hold escrow bucket** (`legal-hold-escrow/{original_tenant_id}/{resourceType}/{resourceId}`) in the MinIO (or S3-compatible) endpoint configured via `storage.regions.<region>.legalHoldEscrow.endpoint` / `.bucket` / `.kmsKeyId`, using the resolved region from sub-step 2. The object is written with an independent retention policy tied to hold release rather than tenant lifecycle: the escrow object's retention tag is set to `retain-until-hold-release` and the object is immutable under MinIO object-lock in `COMPLIANCE` mode for the duration of the hold. When the hold is subsequently cleared via `POST /v1/admin/legal-hold` (`hold: false`) — which the legal-hold admin API accepts on tombstoned tenants for this purpose — the escrow object becomes eligible for deletion; an escrow GC job deletes it and the resulting `legal_hold.escrow_released` audit event is emitted. For audit-log rows specifically, the migration copies the row into the escrow bucket as a signed JSON envelope (`audit_event_id`, `original_tenant_id`, canonical payload, the hash-chain neighbors at time of migration, and a detached signature under the audit signing key) so the chain-integrity model used by chain verifiers (§11.7) recognizes the escrow record as an authorized export and the post-Phase-4 deletion of the original audit row as a `redacted_gdpr` discontinuity — but because Phase 4 explicitly excludes escrowed rows from `DeleteByTenant` (see Phase 4 actions above), the originals are not deleted; only the escrow copy exists for post-tenant-tombstone retrieval. A deployment that has not configured any `storage.regions.*` map operates on the single-region default: the scalar values configured via `storage.legalHoldEscrowDefault.{endpoint, bucket, kmsKeyId, escrowKekId}` are treated as the implicit single region, the fail-closed residency gate is a no-op (there is no residency constraint to violate), and the escrow KEK is still resolvable by the single-region identifier `platform:legal_hold_escrow:default`.
4. **Record migration in the legal-hold ledger.** A `legal_hold.escrowed` audit event is written for each migrated resource carrying `tenant_id` (the original tenant — populated into `unmapped.lenny.target_tenant_id`), `resourceType`, `resourceId`, `original_hold_set_at`, `escrow_object_key`, `escrow_region` (the region resolved in sub-step 2), `escrow_kek_id` (the region-scoped identifier `platform:legal_hold_escrow:<region>`), `tenant_delete_job_id`, and `migrated_at`. The ledger update is part of the same transaction that marks the ArtifactStore / EventStore record as `legal_hold_escrow: true` so that Phase 4's `DeleteByTenant` skip logic can observe the marker without re-reading the audit chain. The event is subject to `audit.gdprRetentionDays` (it is a legal-hold ledger event) and is written to the platform tenant under the same cross-tenant write path used by `admin.impersonation_*` events (§11.7 Write-time tenant validation), so that the ledger record survives the tenant tombstone. **Residency (CMP-058, fail-closed):** the ledger row is routed via `StoreRouter.PlatformPostgres(region)` to the target tenant's regional platform-Postgres — the same region resolved in sub-step 2 for the escrow bucket — so that the ledger record describing an EU tenant's held evidence lands in the EU-region platform-Postgres, never in a US-region platform-Postgres. If the target region's `storage.regions.<region>.postgresEndpoint` is missing or unreachable, the ledger write is rejected fail-closed with `PLATFORM_AUDIT_REGION_UNRESOLVABLE` (HTTP 422, `POLICY`; see [§15.4](15_external-api-surface.md#154-errors-and-degradation) and [§11.7](11_policy-and-controls.md#117-audit-logging) "Platform-tenant audit event residency"), the Phase 3.5 migration aborts on the row that triggered the failure, and the controller halts in `deleting` state pending operator remediation — Phase 4 does not proceed because the escrow ciphertext must be accompanied by a durable ledger row or Phase 4's `DeleteByTenant` skip logic cannot distinguish escrowed from non-escrowed records. This residency gate is co-located with sub-step 2's `LEGAL_HOLD_ESCROW_REGION_UNRESOLVABLE` bucket/KEK residency gate: a deployment that correctly configured `storage.regions.<region>.legalHoldEscrow` but omitted the corresponding `storage.regions.<region>.postgresEndpoint` fails fast on the ledger write with a distinct error code rather than writing the escrow ciphertext and losing the pointer.

After the four sub-steps complete, the controller emits a `gdpr.legal_hold_overridden_tenant` critical audit event carrying `tenant_id`, `job_id` (the tenant-delete job), `override_by`, `override_justification`, `override_at`, `overridden_holds` (the full tuple array from sub-step 1), and `escrow_object_keys` (the set of escrow keys written in sub-step 3). The override event is retained under `audit.gdprRetentionDays` (default **2555 days / 7 years**; see GDPR erasure receipt retention above), is written to the platform tenant, and raises the `LegalHoldOverrideUsedTenant` warning alert ([§16.5](16_observability.md#165-alerting-rules-and-slos)) so every tenant-level override is surfaced to compliance teams. The override does **not** clear the underlying legal-hold rows — they remain set in the ledger's escrowed form — so that any parallel hold audit or e-discovery export can still observe that holds were in force at tenant-delete time and that their evidence persists in escrow. Phase 4 then proceeds with `DeleteByTenant` excluding the escrowed records (see Phase 4 actions above), and Phase 4a destroys the tenant KEK as documented — this is safe because held ciphertext has been re-wrapped under `legal_hold_escrow_kek` and no longer depends on the tenant KEK for decryption.

Operators should treat the tenant-level `--acknowledge-hold-override` as a last resort (e.g., a business-wind-down deadline that cannot wait for a hold to clear through e-discovery). The `gdpr.legal_hold_overridden_tenant` event is surfaced in the SIEM stream alongside the user-scope `gdpr.legal_hold_overridden` event from CMP-049; the two are distinct event types so tenant-scope and user-scope overrides can be audited independently. `POST /v1/admin/tenants/{id}/force-delete` called **without** `acknowledgeHoldOverride: true` while active holds exist is rejected with `TENANT_DELETE_BLOCKED_BY_LEGAL_HOLD` — the endpoint does not silently assume the override when holds are present.

**Tenant deletion SLA.** The deletion controller enforces wall-clock SLAs per tenant data classification: T3 tenants must complete all 6 phases within **72 hours** of initiating Phase 1; T4 tenants must complete within **4 hours**. The deletion controller records the Phase 1 start time in the tenant record. The `TenantDeletionOverdue` warning alert fires when elapsed time exceeds 80% of the applicable SLA without reaching the `deleted` state. Metric: `lenny_tenant_deletion_duration_seconds` (histogram, labeled by `tenant_data_tier`) tracks end-to-end deletion time from Phase 1 start to Phase 6 completion. A `TenantDeletionOverdue` warning alert fires when any tenant has been in `disabling` or `deleting` state longer than the tier SLA.

**Idempotency and resumption.** The controller persists the current phase in the tenant record. If the process is interrupted (controller restart, transient failure), it resumes from the last incomplete phase. Each phase is individually idempotent.

**Erasure propagation to external sinks.** Data exported to external systems (SIEM via [Section 11.7](11_policy-and-controls.md#117-audit-logging), billing webhooks/message queues via [Section 11.2.1](11_policy-and-controls.md#1121-billing-event-stream)) is outside direct platform control. The erasure job publishes an `erasure.requested` event to all configured billing and audit sinks, carrying the `user_id` (or `tenant_id`) and a deadline. Deployers are responsible for ensuring their downstream consumers honor this event. The erasure receipt records which sinks were notified and their acknowledgment status. The platform guarantees notification and auditability, not external erasure.

**Backups in erasure scope.** The erasure scope table above enumerates *runtime* stores. Backups ([§25.11](25_agent-operability.md#2511-backup-and-restore-api)) are a second data surface: a `pg_dump` or MinIO snapshot taken before an erasure request captures the pre-erasure state of every runtime store and persists it for up to `backups.retention.retainDays` (90 days at Tier 3). Restoring such a backup without additional controls would resurrect erased users' sessions, memories, OAuth tokens, billing rows bearing the original `user_id`, and audit entries — including clearing `processing_restricted` — which is an unauthorized reinstatement of personal data under GDPR Article 17 and (for PHI-bearing deployments) an unauthorized disclosure under 45 C.F.R. §164.502. The platform therefore treats erasure propagation to backups as a first-class compliance control, implemented through the post-restore reconciler described below. Backup archives themselves are **not** rewritten in place — GDPR erasure on retained backups is satisfied by (a) reconciling at restore time, and optionally (b) crypto-shredding at the tenant level, and (c) bounding `backups.retainDays` to the erasure SLA; the reconciler is the primary control and the only one that satisfies per-user erasure without forcing short backup retention.

- **Post-restore reconciler (primary control).** Between the `restore_completed` event and the gateway rolling restart ([§25.11](25_agent-operability.md#2511-backup-and-restore-api) restore execution step 6), `lenny-ops` runs an **erasure reconciler** against the restored databases before the gateway is marked Ready. The reconciler executes as a dedicated K8s Job with the same security profile as the restore Job, and proceeds in four phases:
  1. **Enumerate receipts.** Scan the restored `audit_log` for rows where `event_type LIKE 'gdpr.%'` AND the receipt's `completed_at` timestamp (recorded in the event payload) is strictly greater than `ops_backups.completed_at` of the restored backup (the `backupTakenAt` boundary). `gdpr.*` receipts survive restore unconditionally because they are retained under `audit.gdprRetentionDays` (default 2555 days / 7 years, see GDPR erasure receipt retention above), which always exceeds the 90-day maximum `backups.retention.retainDays` — so the restored audit log always contains every completed erasure that occurred after the backup was taken.
  2. **Legal-hold ledger freshness gate (CMP-046).** Before any replay is attempted, the reconciler consults the **current** legal-hold ledger — the live `legal_hold` column on `sessions` / `artifacts` plus the `legal_hold.set` / `legal_hold.cleared` audit events in the restored `audit_log` — to ensure no enumerated receipt targets a subject now under an active hold and to detect ledger staleness. Because the `sessions` / `artifacts` rows and their audit events are restored from the same backup, the ledger as seen immediately post-restore reflects the **backup-time** hold state rather than the true current state. The reconciler therefore computes a `ledgerLatestWriteAt` watermark — the maximum of (a) the most recent `legal_hold.set` or `legal_hold.cleared` audit event timestamp and (b) the most recent `sessions.updated_at` / `artifacts.updated_at` where `legal_hold` is referenced — and compares it to `backupTakenAt`. If `ledgerLatestWriteAt <= backupTakenAt`, the reconciler **MUST block replay entirely**: it cannot prove that a hold applied after the backup (which would veto an erasure replay) or a hold released after the backup (which would change which artifacts the replay must delete) does not exist. In that case, the reconciler emits a `gdpr.backup_reconcile_blocked` critical audit event carrying `restore_id`, `backup_id`, `backupTakenAt`, `ledgerLatestWriteAt`, `enumeratedSubjects`, and `blockReason: "legal_hold_ledger_stale"`, increments the `lenny_backup_reconcile_blocked_total{reason="legal_hold_ledger_stale"}` counter, fires the `BackupReconcileBlocked` critical alert, and exits with `RESTORE_ERASURE_RECONCILE_FAILED` — the gateway is not restarted until an operator confirms the ledger is current via `POST /v1/admin/restore/{id}/confirm-legal-hold-ledger` (requires `platform-admin`; the endpoint records operator identity and justification in the audit trail and injects a synthetic `legal_hold.ledger_confirmed_current_at` timestamp that the reconciler accepts as the authoritative watermark on retry). If `ledgerLatestWriteAt > backupTakenAt`, the reconciler proceeds to phase 3 using the current ledger state.
  3. **Replay erasure.** For each enumerated receipt, the reconciler first re-queries the current ledger for an active hold on the target subject (`legal_hold = true` on any session or artifact owned by the `user_id` / `tenant_id`). If an active hold exists whose most recent `legal_hold.set` event post-dates the receipt's `completed_at`, the replay for that subject is **suppressed** (the hold was applied after the prior erasure, and the legal-hold override of retention in §12.8 "Legal hold and checkpoint rotation" takes precedence over the replay); the reconciler writes a `gdpr.erasure_reconciled_suppressed_by_hold` audit event recording `user_id` / `tenant_id`, `receipt_id`, `hold_set_at`, and the specific session/artifact IDs protected by the hold, and the subject is listed in the reconcile receipt's `suppressedSubjects` array. Otherwise the reconciler invokes the corresponding erasure action against the restored stores in dependency order (`DeleteByUser(user_id)` for user-level receipts, `DeleteByTenant(tenant_id)` for tenant-level receipts). The platform's existing `DeleteByUser` / `DeleteByTenant` implementations are idempotent (see erasure job phase field above) and may therefore be replayed verbatim — a subject whose erasure was already applied to the restored state (because the backup post-dated the erasure for that subject) will be a no-op. The reconciler does **not** re-run billing pseudonymization if the restored `erasure_salt` is `NULL` (the salt-deletion verification step above already guaranteed the salt is unrecoverable for the prior erasure); instead, the reconciler verifies that no billing row exists with the original `user_id` and, if any such row is found post-restore, pseudonymizes it using a freshly generated per-tenant salt and then deletes that salt under the same Immediate-deletion-after-pseudonymization rule. A `gdpr.erasure_reconciled` audit event is written for each replay (separate event type so the reconciled replay can be distinguished from the original erasure in the audit trail).
  4. **Complete and emit receipt.** On successful completion of every enumerated receipt (each either replayed or suppressed-by-hold per phase 3), the reconciler emits a single `gdpr.backup_reconcile_completed` audit event (critical severity) carrying `restore_id`, `backup_id`, `backupTakenAt`, `ledgerLatestWriteAt`, `reconciledSubjects` (array of `{user_id | tenant_id, receipt_id, replayed_at}` tuples), `suppressedSubjects` (array of `{user_id | tenant_id, receipt_id, hold_set_at}` tuples from phase 3 suppressions), and `reconcilerDurationSeconds`. The event is written before the gateway restart is triggered so that the reconciled state is durable in the audit chain.
- **Readiness gating (fail-closed).** `lenny-ops` **MUST NOT** patch the gateway Deployment to trigger the rolling restart until the reconciler reports success. If the reconciler fails — whether due to an individual replay failure, Postgres unavailability mid-reconcile, the enumeration step throwing, or the legal-hold ledger freshness gate blocking replay (`gdpr.backup_reconcile_blocked`, phase 2 above) — the restore is aborted with error code `RESTORE_ERASURE_RECONCILE_FAILED`, the `ops_restore_state.status` is set to `"failed"`, the `restore_failed` event is emitted with `failure_phase: "erasure_reconcile"` (and `block_reason: "legal_hold_ledger_stale"` when the ledger gate fired), the `restore:platform` remediation lock remains held, and the gateway is not restarted. The partial-restore platform is not served until the operator investigates and resolves the reconcile failure — for a ledger-stale block, that means confirming ledger currency via `POST /v1/admin/restore/{id}/confirm-legal-hold-ledger` once out-of-band evidence (e.g., an external legal-hold registry, signed operator attestation, or re-application of any holds known to have been set post-backup) establishes that no post-backup hold transitions are being silently overridden; for other failure modes, typically by repairing the affected subject's data manually or by re-running the reconciler via an operator-driven retry. Leaving the gateway Ready after a partial reconcile would serve resurrected personal data or destroy legally-held data, which are the failure modes this control exists to prevent.
- **Crypto-shredding (alternative, tenant-level only — documented but not primary).** A deployer that cannot tolerate the reconciler's additional downtime window may configure **per-tenant backup crypto-shredding** via `backups.encryption.perTenantWrapKeys: true`: each per-tenant dump is encrypted client-side with a KMS key scoped to that tenant, and `DeleteByTenant` destroys the wrap key in KMS (via `ScheduleKeyDeletion` / `DestroyCryptoKeyVersion`), rendering every backup of that tenant's data cryptographically unrecoverable. Crypto-shredding satisfies **tenant-level** erasure only — it cannot shred a single user's rows out of a tenant-scoped encrypted archive. A deployer that enables crypto-shredding MUST also configure the reconciler (which handles user-level `DeleteByUser` receipts); the two are complementary, not mutually exclusive.
- **Short backup retention (alternative, last-resort — documented but not primary).** A deployer that disables the reconciler (e.g., for regulatory environments that forbid modifying restored state) MUST set `backups.retention.retainDays` to a value ≤ the applicable GDPR erasure SLA (72h for T3, 1h for T4, see tier-specific deadlines above). `lenny-ops` rejects `retainDays` values exceeding the SLA when `backups.erasureReconciler.enabled: false` — the configuration is validated at chart render time and at `lenny-ops` startup, with the fatal error `CONFIG_INVALID: backups.retention.retainDays exceeds GDPR erasure SLA and reconciler disabled`. This mode trades backup depth for compliance surface and is not recommended for Tier 3 deployments.
- **Receipt survivability.** `gdpr.*` audit events are written to the `EventStore` (audit) before the completion receipt is returned to the caller, are never deleted by `DeleteByUser` or `DeleteByTenant` (see GDPR erasure receipt retention above), and are retained under `audit.gdprRetentionDays` independently of the audit partition TTL. A restored backup carries these receipts verbatim because the audit log is included in the `pg_dump`. Restoring a backup older than `audit.gdprRetentionDays` is explicitly unsupported — the platform rejects restore at `POST /v1/admin/restore/execute` if `backup.completed_at < now() - audit.gdprRetentionDays` with error `RESTORE_INCOMPATIBLE: backup predates GDPR retention floor` — because without receipts the reconciler cannot establish the `completed_at > backupTakenAt` boundary.

**Workspace deduplication.** In v1, each MinIO artifact object is owned by exactly one session — session derivation ([Section 7.1](07_session-lifecycle.md#71-normal-flow)) always copies the parent's workspace snapshot into a new independent MinIO object, never shares it. There are no cross-session MinIO references in v1. If a future optimization introduces content-addressed deduplication for workspace snapshots in the `ArtifactStore`, the GC job and erasure job must use reference counting: a blob is only deleted when no other session's artifact references it (i.e., `reference_count = 0`). Until deduplication is implemented, each artifact is session-scoped and deleted directly by TTL without reference counting.

**Data residency.** Tenants and environments support an optional `dataResidencyRegion` field (e.g., `eu-west-1`, `us-east-1`) set via the admin API (`PUT /v1/admin/tenants/{id}` and environment configuration). When set, the platform enforces region constraints at three levels:

1. **Pod pool routing.** Each pool carries a `region` label. When a tenant or environment specifies `dataResidencyRegion`, the gateway restricts pod allocation — including delegation targets — to pools whose `region` label matches. A delegation that would route to a pool in a non-matching region is rejected with `REGION_CONSTRAINT_VIOLATED`. This applies transitively: every node in a delegation tree inherits the root session's region constraint.
2. **Storage routing.** The `StorageRouter` interface accepts `dataResidencyRegion` as a parameter and directs writes (Postgres, MinIO, Redis) to the region-local backend. Deployers configure per-region storage endpoints in the Helm values (`storage.regions` map): each region entry must declare `storage.regions.<region>.postgresEndpoint`, `storage.regions.<region>.minioEndpoint`, and `storage.regions.<region>.redisEndpoint`. When `dataResidencyRegion` is set on a tenant but `redisEndpoint` is not configured for that region, the `StorageRouter` fails closed with `REGION_CONSTRAINT_UNRESOLVABLE` — consistent with the Postgres and MinIO behavior. Single-region Redis deployments are incompatible with multi-region data residency; deployers must provision per-region Redis instances. Redis-stored personal data subject to region routing includes: semantic cache entries, session coordination leases, experiment sticky assignments, billing write-ahead buffer entries, and cached access tokens. When `dataResidencyRegion` is unset, the platform uses the default (single-region) storage backend — no behavioral change for existing deployments. When `dataResidencyRegion` is set but the value is not present in `storage.regions`, the `StorageRouter` **fails closed**: the write is rejected with `REGION_CONSTRAINT_UNRESOLVABLE` and a `DataResidencyViolationAttempt` audit event is emitted. There is no silent fallback to the default backend — an unresolvable region constraint is treated as a hard error, not a configuration gap to route around.
3. **Validation at session creation.** The gateway validates that at least one pool and one storage backend are available for the requested region before accepting a session. If not, session creation fails with `REGION_UNAVAILABLE`.

**Data residency admission control.** A `ValidatingAdmissionWebhook` (`lenny-data-residency-validator`) enforces region integrity at the Kubernetes resource layer. The webhook intercepts `CREATE` and `UPDATE` operations on all tenant-scoped CRD resources that carry a `dataResidencyRegion` field (including `SandboxClaim`, and any Tenant or Environment configuration resources). On each admission request, the webhook resolves the effective `dataResidencyRegion` (applying the inheritance rules below) and rejects the resource if the region value is non-empty and not declared in `storage.regions`. The rejection message identifies the offending field and lists the declared regions. The webhook runs with `failurePolicy: Fail` and a 5-second timeout; if the webhook is unavailable, admission is denied (fail-closed) so that a webhook outage cannot silently admit resources with unvalidated region constraints. Alert `DataResidencyWebhookUnavailable` fires when the webhook has been unreachable for more than 30 seconds.

**Data residency audit events.** Every data residency enforcement action emits an audit event to make violations and violation attempts independently observable — not just silently rejected:

- When the `StorageRouter` receives a write request where the resolved storage region does not match the session's `dataResidencyRegion`, it rejects the write and emits a `DataResidencyViolationAttempt` audit event (critical severity). The event records: `tenant_id`, `session_id`, `requested_region`, `resolved_region`, `storage_backend`, `operation` (`write`), and `timestamp`. This turns silent bypasses (e.g., a misconfigured fallback-to-default route) into observable, alertable signals.
- When pod pool routing rejects a delegation due to `REGION_CONSTRAINT_VIOLATED`, a `DataResidencyViolationAttempt` audit event is emitted with the same fields plus the delegation `parent_session_id` and `target_pool`.
- When a platform-tenant audit event referencing a non-platform `target_tenant_id` (e.g., `admin.impersonation_started`, `legal_hold.escrowed`, `gdpr.legal_hold_overridden_tenant`, `compliance.profile_decommissioned`) fails to commit because the target tenant's regional platform-Postgres is misconfigured or unreachable, a `DataResidencyViolationAttempt` audit event (critical severity) is emitted with `operation: "platform_audit_write"`, `target_tenant_id`, `requested_region` (the target's `dataResidencyRegion`), `event_type` (the originating event type that failed to write), `failure_mode` (`missing_entry` | `postgres_unreachable`), and the `PLATFORM_AUDIT_REGION_UNRESOLVABLE` error code is returned to the originating operation (CMP-058). The violation event is itself routed under the CMP-058 rules: if the target region is the source of the failure, the violation falls back to the global `PlatformPostgres()` so the incident does not disappear into the unreachable region. See [§11.7](11_policy-and-controls.md#117-audit-logging) "Platform-tenant audit event residency".
- When session creation fails with `REGION_UNAVAILABLE`, a `DataResidencyRegionUnavailable` audit event (warning severity) is emitted.

These events are forwarded to the SIEM (when configured) alongside all other audit events.

**KMS key residency.** When `dataResidencyRegion` is set, the KMS key used for envelope encryption of tenant data ([Section 4.3](04_system-components.md#43-token-service), [Section 12.5](#125-artifact-store)) MUST reside in the same jurisdiction. Under GDPR and data localization laws, decryption capability located in a different region constitutes a cross-border data transfer even if the encrypted data never leaves the declared region. Deployers configure per-region KMS endpoints in the Helm values (`storage.regions.<region>.kmsEndpoint`). When `dataResidencyRegion` is set on a tenant, the gateway validates at tenant creation and update time that a `kmsEndpoint` is declared for the tenant's region; if absent, the operation is rejected with `422 KMS_REGION_UNRESOLVABLE`. The KMS endpoint used for all T4 SSE-KMS operations for that tenant is the region-local endpoint, not the default KMS endpoint. Deployers who cannot configure a region-local KMS endpoint must leave `dataResidencyRegion` unset and accept that cross-region decryption capability applies.

**Inheritance:** Sessions inherit `dataResidencyRegion` from their environment, which inherits from its tenant unless explicitly overridden. An environment must use the same `dataResidencyRegion` as its tenant or inherit the tenant's value; specifying a different region is rejected with `REGION_CONSTRAINT_VIOLATED`.

**Backup pipeline residency.** Backups ([§25.11](25_agent-operability.md#2511-backup-and-restore-api)) are subject to the same cross-border rules as runtime stores: a `pg_dump` archive or MinIO snapshot is a copy of regulated personal data, and the destination bucket, KMS key, and decrypt capability must all reside in the tenant's `dataResidencyRegion`. A single-bucket, single-KMS-key backup pipeline — the default when no region map is configured — cannot be used in a deployment where any tenant has `dataResidencyRegion` set, because a global aggregated `pg_dump` across `AllSessionShards()` would write EU-resident rows and US-resident rows to the same archive and the resulting object-store PUT (and subsequent GET at restore time) would constitute a prohibited cross-border transfer under §12.8 rules and a GDPR Article 44–46 transfer without a documented legal basis. The platform therefore requires a **per-region backup pipeline** whenever data residency is in use:

1. **Per-region backup endpoints (required when any tenant has `dataResidencyRegion` set).** Deployers configure `backups.regions.<region>.{minioEndpoint, kmsKeyId, accessCredentialSecret}` in the Helm values, mirroring the `storage.regions.<region>.*` structure. Each region entry carries its own MinIO endpoint (typically co-located with that region's `storage.regions.<region>.minioEndpoint`), its own KMS key for client-side and SSE-KMS envelope encryption (which MUST reside in the same jurisdiction per KMS key residency above), and its own access-credential Secret name so that one region's backup Job cannot authenticate to another region's MinIO. `backups.encryption.kmsKeyId` remains valid as the single-region default; when `backups.regions` is non-empty, per-region `kmsKeyId` values override the scalar. `lenny-ops` validates at startup that every region referenced by any tenant's `dataResidencyRegion` has a complete `backups.regions.<region>` entry; a missing entry is a fatal `CONFIG_INVALID` error.
2. **Per-shard dump routing (no global aggregated dump).** The backup Job does **not** run a single `pg_dump` over `AllSessionShards()` in a multi-region deployment. Instead, `lenny-ops` enumerates shards by their resolved region (via the same `StorageRouter`-backed resolution used at runtime) and runs one `pg_dump` per region against that region's Postgres shards only. Each per-region archive is uploaded to that region's `backups.regions.<region>.minioEndpoint` using that region's `accessCredentialSecret` and encrypted with that region's `kmsKeyId`. The `ops_backups` row records one entry per region (with `components` listing the regions covered), and verification / restore operate per-region so no control path ever opens a cross-region stream.
3. **`BackupRegionUnresolvable` (fail-closed).** If a shard's resolved region has no corresponding `backups.regions.<region>` entry, or if the region's MinIO endpoint is unreachable, or if `backups.regions.<region>.kmsKeyId` cannot be resolved against KMS, `lenny-ops` **MUST** abort the backup with error code `BackupRegionUnresolvable` — mirroring the `REGION_CONSTRAINT_UNRESOLVABLE` fail-closed behavior for runtime writes. The aborted backup row is marked `status: "failed"` with `error: "BACKUP_REGION_UNRESOLVABLE: region <region> has no backups.regions entry (or endpoint/KMS is unreachable)"`, the `BackupFailed` alert fires, and a `DataResidencyViolationAttempt` audit event is emitted (critical severity, counter `lenny_data_residency_violation_total`, same counter defined for runtime `StorageRouter` violations above) recording `requested_region`, `shard_id`, `backup_id`, and `operation: "backup"`. There is no silent fallback to a default backup bucket — an unresolvable backup region is a hard error, not a configuration gap to route around. Cross-region configurations that would cause an aggregated dump (e.g., `backups.strategy: "aggregated"` combined with any tenant having `dataResidencyRegion` set) are rejected at chart render time and at `lenny-ops` startup with the same `CONFIG_INVALID` class.
4. **Restore region symmetry.** `POST /v1/admin/restore/execute` against a per-region backup restores only into the same region's shards. A restore that would load EU-origin rows into a US-region shard is rejected with `RESTORE_INCOMPATIBLE: region mismatch`. Cross-region restores are out of scope for v1.
5. **ArtifactStore continuous-replication runtime residency (fail-closed).** The per-shard `pg_dump` pipeline described above is a batch-at-a-time surface gated by `BackupRegionUnresolvable` at job start. The ArtifactStore continuous-replication surface ([§25.11](25_agent-operability.md#2511-backup-and-restore-api) ArtifactStore Backup) is long-lived and stateful — startup-time validation of `minio.regions.<region>.artifactBackup.target.*` is insufficient because a post-startup Helm reconfiguration, DNS rebinding of the destination endpoint, region-tag drift in the target MinIO cluster's metadata, or an operator mis-edit during an incident can silently redirect replication across a jurisdiction boundary without `lenny-ops` restart. The replication path therefore runs a **runtime residency preflight** — before every replication batch and on a periodic tick (`minio.artifactBackup.residencyCheckIntervalSeconds`, default 300s) — that re-resolves the destination, reads the destination bucket's mandatory `lenny.dev/jurisdiction-region` tag, and compares it to the source region's `dataResidencyRegion`. On mismatch, missing tag, DNS rebinding outside `backups.regions.<region>.allowedDestinationCidrs` (when set), or tag-probe failure, replication for the affected region is suspended (`mc replicate disable` or provider equivalent), `lenny-ops` sets `ops_artifact_replication_state.status: "suspended_residency_violation"`, the error code `ARTIFACT_REPLICATION_REGION_UNRESOLVABLE` is surfaced on admin queries, a `DataResidencyViolationAttempt` audit event is emitted (critical severity, same counter `lenny_data_residency_violation_total` used for runtime `StorageRouter` violations and Postgres backup residency, labeled `operation: "artifact_replication"`), a dedicated counter `lenny_minio_replication_residency_violation_total` is incremented (labeled by `region` and `violation_type`), and the `ArtifactReplicationResidencyViolation` critical alert fires. Resume requires operator intervention (`POST /v1/admin/artifact-replication/{region}/resume`, `platform-admin`, audited with justification). A positive-audit `artifact.cross_region_replication_verified` event is emitted on each successful preflight (sampled) carrying `destination_jurisdiction_tag`, `destination_endpoint`, `source_data_residency_region`, `verified_at`, and `batch_object_count` — so chain-of-custody for cross-region replication records residency at write time, not only at startup config-load time. This control is the continuous-replication analog of `BackupRegionUnresolvable` for the batch surface; the two together cover both backup surfaces against post-startup jurisdiction drift.

**Multi-region reference architecture.** A multi-region deployment runs one Lenny control plane per region (gateway, controllers, storage), each serving tenants pinned to that region. Cross-region delegation is not supported — delegation trees are region-local by design. A global load balancer (e.g., DNS-based) routes clients to the correct regional gateway based on tenant configuration. Tenant metadata (including `dataResidencyRegion`) is replicated to a lightweight global catalog so the load balancer can resolve region affinity before the first request reaches a gateway. This architecture avoids cross-region data transfer while allowing a single organization to operate tenants in multiple jurisdictions. The **per-region backup pipeline** described above is the mandatory backup topology for this architecture: each regional control plane operates its own `lenny-ops`, its own `backups.regions.<region>` MinIO bucket, its own per-region KMS key, and its own backup Jobs whose `pg_dump` invocations target only that region's shards. There is no global backup aggregator — cross-region backup aggregation is the deployer's responsibility at a layer above Lenny (e.g., a compliance-approved cross-region archive that is out of Lenny's chain of custody).

**Multi-region consistency model.** Because each region runs its own independent Postgres, Redis, and MinIO instances, the following invariants are **per-region** rather than globally aggregated:

- **Billing sequence monotonicity:** The `billing_seq_{tenant_id}` Postgres sequence is per-region-per-tenant. Consumers that aggregate billing events across regions must not assume global monotonicity — sequence numbers are monotonic within a single region only. Cross-region billing aggregation is the deployer's responsibility (e.g., via a centralized billing aggregation layer that merges per-region streams using `(region, sequence_number)` composite ordering).
- **Per-tenant quotas (storage, token budget):** Redis-backed quota counters (`storage_bytes_used`, token budget counters) are enforced per-region. If a tenant has sessions in multiple regions, each region enforces quotas independently against its own counters. The global per-tenant quota (`storageQuotaBytes`, per-tenant token budget) could be exceeded in aggregate even if each region is individually within limits. Global cross-region quota aggregation is the deployer's responsibility; deployers who require global enforcement should implement a centralized quota coordination service or set per-region sub-quotas that sum to the desired global limit.
- **Cross-region delegation:** Explicitly prohibited. The gateway rejects any delegation that would route to a pool in a non-matching region with `REGION_CONSTRAINT_VIOLATED` (see pod pool routing above). This constraint is enforced transitively through the delegation tree.

**Data Subject Access, Rectification, and Portability (GDPR Articles 15, 16, 20).** Lenny operates as a **data processor** on behalf of the deployer, who is the **data controller**. GDPR obligations under Articles 15 (right of access), 16 (right to rectification), and 20 (right to data portability) are the deployer's responsibility to fulfill — the platform does not expose a built-in DSAR endpoint because the deployer controls the identity relationship with the data subject and the scope of personal data collected.

The platform provides the following primitives that deployers can use to satisfy DSAR obligations:

- **Session and artifact data:** All user-scoped session records, task trees, audit events, and stored artifacts are queryable via the admin API using `user_id` as a filter key. Deployers can build a DSAR data-collection layer that calls these endpoints and assembles a portable export in any required format.
- **Erasure (Article 17):** Fully supported natively via `POST /v1/admin/users/{user_id}/erase` (see Data erasure above).
- **Rectification (Article 16):** Free-text PII fields stored by the platform (e.g., billing event free-text columns) can be updated via the admin API or directly via the erasure pseudonymization flow. Deployers are responsible for any PII stored within agent-generated content (workspace snapshots, session transcripts) — the platform provides artifact access and deletion primitives, but not structured PII search within opaque content.
- **Data portability (Article 20):** Session metadata, audit events, and billing records are available via the admin REST API in JSON format. Deployers are responsible for assembling and delivering portable exports to data subjects. The platform does not include a pre-built DSAR export bundle endpoint in v1 — this is intentionally left to the deployer layer, which has the context needed to scope and format the response correctly.

**Audit event scope for portable exports (normative).** A compliant Article 20 export of a user's audit events MUST include **both** of the following categories, unioned and deduplicated by `audit_log.id`:

  1. **Actor-scoped events** — all rows in `audit_log` where the authenticated actor identity (the caller that caused the event) resolves to the target `user_id`. In the Lenny schema, this is every row whose `user_id` column equals the target user (equivalently, `actor.user.uid == target_user_id` in the OCSF projection, [§11.7](11_policy-and-controls.md#117-audit-logging) Lenny → OCSF field mapping). These rows are written under the target user's tenant.
  2. **Subject-scoped platform-admin impersonation events** — all rows written under the **platform tenant** for cross-tenant flows whose payload identifies the target user as the subject of the operation. In the Lenny schema this is every row where `tenant_id = '<platform_tenant_id>'` AND `payload->>'target_user_id' = '<target_user_id>'` AND `payload->>'target_tenant_id' = '<target_user_tenant_id>'`. This category covers `admin.impersonation_started` / `admin.impersonation_ended` ([§16.7](16_observability.md#167-section-25-audit-events)) and any future cross-tenant admin action that writes under the platform tenant with a `target_user_id` payload field ([§11.7](11_policy-and-controls.md#117-audit-logging) Write-time tenant validation). Excluding these rows would leave impersonation-based reads and modifications invisible to the data subject, contrary to Article 15 (right of access) which Article 20 presupposes.

The export MUST NOT include:

  - Rows where the target user appears only as an incidental string inside opaque `payload` free-text (e.g., a log line that mentions the user but where the operation acted on a different subject) — these are not reliably identifiable and are out of scope for the structured export. Deployers who have surfaced additional PII into payload fields are responsible for any supplementary filtering required by their RoPA.
  - `gdpr.*` rows, which are retained as compliance receipts under `audit.gdprRetentionDays` (see GDPR erasure receipt retention above) and are not personal data subject to erasure — they remain queryable by deployers for compliance recordkeeping but fall outside the data-subject's portable export of their own activity.

**Template DSAR audit query.** The following query returns both categories in a single result set suitable for feeding a deployer DSAR assembler. Parameters: `$1` = target `user_id`; `$2` = target user's `tenant_id`; `$3` = the platform tenant identifier used for cross-tenant admin-originated events ([§11.7](11_policy-and-controls.md#117-audit-logging) Write-time tenant validation):

```sql
SELECT id, tenant_id, sequence_number, event_type, created_at, payload
FROM audit_log
WHERE tenant_id = $2 AND user_id = $1                              -- (a) actor-scoped
UNION ALL
SELECT id, tenant_id, sequence_number, event_type, created_at, payload
FROM audit_log
WHERE tenant_id = $3                                               -- (b) subject-scoped platform-admin
  AND payload->>'target_user_id' = $1
  AND payload->>'target_tenant_id' = $2
ORDER BY created_at ASC, sequence_number ASC;
```

Equivalent invocations via the audit-query API ([§25.9](25_agent-operability.md#259-audit-log-query-api)) cover category (a) directly and category (b) by a platform-tenant query that the DSAR tool then filters client-side on the payload fields (the REST surface does not expose a server-side payload filter — the SQL template is authoritative for category (b) and should be used by deployer tooling that has direct database read access under a scoped `dsar_reader` role):

```
GET /v1/admin/audit-events?tenantId={target_tenant_id}&actorId={target_user_id}   # (a)
GET /v1/admin/audit-events?tenantId={platform_tenant_id}&eventType=admin.impersonation_started,admin.impersonation_ended&since={user_creation_ts}   # (b) — filter payload.target_user_id client-side
```

The deployer DSAR tooling merges the two result sets on `id`, drops any `event_type LIKE 'gdpr.%'` rows per the exclusion above, and emits the combined stream alongside the session, artifact, and billing exports.

Deployers must document their DSAR procedures in their Records of Processing Activities (RoPAs) and ensure their data-collection tooling accounts for all Lenny stores listed in the erasure scope table above. The RoPA MUST enumerate which audit event categories are in scope for the portable export, referencing the two-category rule above; a RoPA that silently omits category (2) is non-compliant with this section.

**Audit trail.** All compliance operations (legal hold set/cleared, erasure requested/completed) are logged in the audit trail ([Section 11.7](11_policy-and-controls.md#117-audit-logging)) with the requesting admin's identity, timestamp, and affected resource IDs.

### 12.9 Data Classification

The platform defines four classification tiers. Every data element maps to exactly one tier, and the tier drives encryption, retention, access, and audit controls.

| Tier | Label            | Description                                                                                                                                                                |
| ---- | ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T1   | **Public**       | Non-sensitive data safe for external exposure (e.g., public runtime catalog metadata, documentation links)                                                                 |
| T2   | **Internal**     | Operational data not intended for external exposure but carrying no regulatory risk (e.g., session metadata, pool metrics, routing tables, non-PII audit events)           |
| T3   | **Confidential** | Business-sensitive data and PII (e.g., workspace files, session transcripts, user identifiers, billing events, memory store contents)                                      |
| T4   | **Restricted**   | Credentials, PHI, and data subject to the strictest regulatory controls (e.g., OAuth tokens, refresh tokens, credential pool secrets, API keys, PHI-tagged workspace data) |

**Default data-type mappings:**

| Data Type                                       | Default Tier      | Store                            |
| ----------------------------------------------- | ----------------- | -------------------------------- |
| Runtime catalog metadata                        | T1 — Public       | Postgres                         |
| Session metadata (IDs, state, timestamps)       | T2 — Internal     | Postgres                         |
| Pool metrics, routing tables                    | T2 — Internal     | Redis                            |
| Audit events (non-PII)                          | T2 — Internal     | Postgres                         |
| Workspace files and snapshots                   | T3 — Confidential | MinIO                            |
| Session transcripts                             | T3 — Confidential | MinIO                            |
| User identifiers, user-scoped audit events      | T3 — Confidential | Postgres                         |
| Billing events                                  | T3 — Confidential | Postgres                         |
| Memory store contents                           | T3 — Confidential | Postgres (or pluggable)          |
| Semantic cache entries                          | T3 — Confidential | Redis (or pluggable)             |
| Eviction state records (`last_message_context`) | T3 — Confidential | Postgres                         |
| OAuth/refresh tokens                            | T4 — Restricted   | Postgres (encrypted)             |
| Credential pool secrets                         | T4 — Restricted   | Postgres (encrypted, KMS-backed) |
| Credential leases                               | T4 — Restricted   | Redis (encrypted)                |

**Per-tenant workspace classification.** By default, workspace data is classified as T3 — Confidential. Tenants that handle PHI or similarly regulated data may elevate workspace classification to T4 — Restricted via tenant configuration:

```yaml
# Tenant level (admin API)
dataClassification:
  workspaceTier: "T4" # default: "T3"
```

When a tenant sets `workspaceTier: T4`, the platform applies Restricted-tier controls (below) to all workspace files, snapshots, and session transcripts for that tenant's sessions. This setting is inherited by all environments under the tenant unless explicitly overridden at the environment level to a **stricter** (never looser) tier.

**Controls driven by classification tier:**

| Control               | T1 — Public             | T2 — Internal            | T3 — Confidential                                  | T4 — Restricted                                                    |
| --------------------- | ----------------------- | ------------------------ | -------------------------------------------------- | ------------------------------------------------------------------ |
| Encryption at rest    | Optional                | Required (storage-layer) | Required (storage-layer, SSE-KMS)                  | Required (envelope encryption via KMS, [Section 4.3](04_system-components.md#43-token-service))                |
| Encryption in transit | TLS                     | mTLS                     | mTLS                                               | mTLS + field-level encryption for token values                     |
| Access control        | Public or authenticated | RBAC role ≥ `viewer`     | RBAC role ≥ `user` + tenant scope                  | RBAC role ≥ `admin` + explicit grant                               |
| Audit logging         | None                    | Write operations         | All read/write operations                          | All operations including access attempts                           |
| Retention default     | Indefinite              | 90 days                  | Deployer-configured (default 7 days, [Section 12.5](#125-artifact-store)) | Deployer-configured (default 24 hours for leases, [Section 4.9](04_system-components.md#49-credential-leasing-service))     |
| Retention override    | Tenant-configurable     | Tenant-configurable      | Tenant-configurable, subject to regulatory floor   | Tenant-configurable, subject to regulatory floor, legal-hold aware |
| Erasure on deletion   | Best-effort             | Required within 30 days  | Required within 72 hours (GDPR-aligned)            | Immediate + cryptographic erasure where supported                  |
| Data residency        | No constraint           | No constraint            | Respects `dataResidencyRegion` ([Section 12.8](#128-compliance-interfaces))      | Respects `dataResidencyRegion`; cross-region transfer prohibited   |

**Enforcement.** Classification is enforced at the storage interface boundary ([Section 12.6](#126-interface-design)). Each store method receives the applicable tier as context and applies the corresponding controls. Tier mismatches (e.g., writing T4 data to a store not configured for envelope encryption) are rejected at write time with a `CLASSIFICATION_CONTROL_VIOLATION` error. The gateway policy engine ([Section 4.8](04_system-components.md#48-gateway-policy-engine)) validates tenant classification configuration at session creation.

**T2 storage-layer encryption definition.** "Required (storage-layer)" for T2 data means **volume-level encryption** enforced at the infrastructure layer — all persistent volumes and node disks used by Postgres and Redis **must** be encrypted using the cluster's CSI driver (e.g., AWS EBS encryption, GCP CMEK, Azure Disk Encryption) or equivalent node-level full-disk encryption. This is distinct from the application-layer encryption required for T3/T4 data. Redis AUTH (ACLs) and TLS are separately **required** for all Redis instances ([Section 12.4](#124-redis-ha-and-failure-modes)). The preflight Job ([Section 17.2](17_deployment-topology.md#172-namespace-layout)) **must** validate that Redis and Postgres volumes are backed by encrypted storage; if the check cannot be performed (e.g., cloud provider API unavailable), a warning is logged and the deployer must attest to compliance via `preflight.attestVolumeEncryption: true` in Helm values. Dev mode (`global.devMode: true`) exempts this check.

**Interaction with existing controls.** Data classification integrates with — and does not replace — existing mechanisms:

- **Legal holds** ([Section 12.8](#128-compliance-interfaces)) override retention at any tier.
- **Data residency** ([Section 12.8](#128-compliance-interfaces)) applies to T3 and T4 data; classification adds the T4 prohibition on cross-region transfer.
- **Credential encryption** ([Section 4.3](04_system-components.md#43-token-service), 4.9) already satisfies T4 encryption requirements; classification formalizes and makes these requirements auditable.
- **Erasure** ([Section 12.8](#128-compliance-interfaces)) honors tier-specific erasure timelines during `DeleteByUser` and `DeleteByTenant` operations.
