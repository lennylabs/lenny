> **Latency comparison note — what each number measures.** The startup figures above are not directly comparable to Lenny's SLO targets in [Section 6.3](06_warm-pod-model.md#63-startup-latency-analysis) and 16.5. Competitors' published numbers (E2B ~150ms, Daytona sub-90ms, Fly.io ~300ms) measure **cold-start time**: the wall-clock duration to boot a fresh VM or container from a stopped state — essentially container scheduling + runtime initialization only. Lenny's SLO target (P95 < 2s runc, < 5s gVisor) measures **full session-ready time**: pod claim + workspace file delivery + setup command execution + agent session start. These are different operations. The pod-claim-and-routing step alone — the operation most directly analogous to a competitor's cold-start number — is in the ~ms range ([Section 6.3](06_warm-pod-model.md#63-startup-latency-analysis)); the remainder of Lenny's hot-path latency is workspace setup work that competitors' numbers do not include at all. A fair latency comparison requires aligning on the same end-point definition. Lenny's numbers are also explicitly **unvalidated targets** to be measured by the Phase 2 startup benchmark harness ([Section 6.3](06_warm-pod-model.md#63-startup-latency-analysis)) — not published benchmarks. Until Phase 2 benchmarks are available, latency parity with competitor cold-start figures is to be demonstrated, not assumed.

### 23.1 Why Lenny?

The differentiators below are architectural commitments reflected in the spec sections cited, not roadmap aspirations.

1. **Runtime-agnostic adapter contract.** Lenny does not bundle or mandate a specific agent framework. Any process that implements the runtime adapter ([Section 15.4](15_external-api-surface.md#154-runtime-adapter-specification)) can run as a Lenny agent pod — Claude Code, a custom LangChain agent, or a bare-metal script. Basic-level integration requires only stdin/stdout JSON Lines (no SDK, ~50 lines of code); Standard and Full levels require gRPC adapter integration with Lenny-specific lifecycle and credential flows. The adapter contract separates platform integration (adapter) from agent logic (runtime), so agent code remains framework-independent even when the adapter layer carries Lenny-specific integration effort.

2. **Self-hosted, Kubernetes-native.** Lenny runs on the operator's own cluster using standard Kubernetes primitives — CRDs, RuntimeClasses, namespaces ([Section 17](17_deployment-topology.md)). There is no dependency on a vendor-hosted control plane. All data stays in the operator's infrastructure. Local development mode ([Section 17.4](17_deployment-topology.md#174-local-development-mode-lenny-dev)) runs with zero cloud dependency.

3. **Security by default.** Agent pods are untrusted by design. Pods run non-root with all Linux capabilities dropped, read-only root filesystem, and default-deny network policies — pods can only reach the gateway ([Section 13](13_security-model.md)). No standing credentials — only short-lived leases and projected ServiceAccount tokens. Gateway-mediated file delivery ensures pods never fetch external data directly. mTLS between gateway and pods with per-replica identity. The Token Service runs as a separate process with its own KMS access. Deployer-selectable isolation profiles (runc, gVisor, Kata microVM) via Kubernetes `RuntimeClass` ([Section 5.3](05_runtime-registry-and-pool-model.md#53-isolation-profiles)). All of these properties are enforced by default — operators do not need to configure them individually.

4. **Flexible runtime types and execution modes.** Two runtime types — `type: agent` (full task lifecycle with sessions, delegation, elicitation, multi-turn) and `type: mcp` (managed MCP server hosting with zero code changes) — serve fundamentally different workloads on the same platform ([Section 5.1](05_runtime-registry-and-pool-model.md#51-runtime)). Three execution modes for agent runtimes — `session` (1:1 pod, strongest isolation), `task` (sequential reuse with workspace scrub, tenant-pinned), and `concurrent` (slot multiplexing with `workspace` and `stateless` sub-variants) — provide deployer control over the pod-usage-to-isolation trade-off ([Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes)). The PoolScalingController applies per-mode adjustment factors (`mode_factor`, `burst_mode_factor`) so operators do not manually account for reuse or multiplexing when sizing pools. This combination of runtime types and execution modes allows a single Lenny deployment to serve diverse workload patterns without forcing operators into a one-size-fits-all model.

5. **Recursive delegation as a platform primitive.** Any agent pod can spawn child sessions through the gateway with enforced scope, token budget, and lineage tracking ([Section 8](08_recursive-delegation.md), Principle 5). Delegation is a first-class gateway operation with policy enforcement at every hop: scope narrowing, isolation monotonicity, content policy inheritance, and cycle detection.

6. **Multi-protocol gateway.** A single gateway edge serves REST, MCP, OpenAI, and Open Responses clients ([Section 15](15_external-api-surface.md), [Section 3](03_high-level-architecture.md)). Operators do not need separate infrastructure per client protocol. Lenny implements MCP Tasks at the gateway's external MCP interface ([Section 9](09_mcp-integration.md)); internal delegation between gateway and agent pods uses a custom gRPC protocol with equivalent semantics (see [§4.7](04_system-components.md#47-runtime-adapter) and [§9](09_mcp-integration.md)).

7. **Enterprise controls at the platform layer.** Multi-tenancy with Postgres row-level security, per-tenant quotas, RBAC, audit logging with hash-chained integrity (SOC2, HIPAA, FedRAMP presets), GDPR erasure, legal holds, and data residency controls are built into the gateway and controller layers (Sections 11, 13). Rate limiting, token budgets, and concurrency controls enforce resource boundaries at every level.

8. **Ecosystem-composable via hooks-and-defaults.** Every cross-cutting AI capability — memory, caching, guardrails, evaluation, routing — is defined as an interface with a sensible default, disabled unless explicitly enabled, and fully replaceable ([Section 22.6](22_explicit-non-decisions.md)). Lenny never implements AI-specific logic (eval scoring, memory extraction, content classification); that belongs to specialized tools the deployer already uses. Lenny provides the platform layer and stays out of the ecosystem layer, so deployers compose their preferred tools without fighting the platform.

#### Additional Platform Capabilities

Beyond the 8 architectural differentiators, Lenny includes additional platform features:

9. **Credential management and LLM Proxy.** Credential pools with round-robin or weighted selection, health scoring, automatic failover, and emergency revocation ([Section 4.9](04_system-components.md#49-credential-leasing-service)). Per-session credential leasing with TTL, rotation, and SPIFFE-bound lease tokens in multi-tenant deployments. The LLM Proxy gateway subsystem injects real API keys into LLM provider requests on behalf of pods — pods never see raw credentials. Pluggable `CredentialRouter` interface for cost-aware, latency-based, or intent-based routing across providers. Optional `SemanticCache` interface on credential pools for LLM response caching.

10. **Gateway request interceptor chain.** 12 hook phases spanning the full request lifecycle: `PreAuth`, `PostAuth`, `PreRoute`, `PreDelegation`, `PreMessageDelivery`, `PostRoute`, `PreToolResult`, `PostAgentOutput`, `PreLLMRequest`, `PostLLMResponse`, `PreConnectorRequest`, `PostConnectorResponse` ([Section 4.8](04_system-components.md#48-gateway-policy-engine)). External interceptors are invoked via gRPC (like Kubernetes admission webhooks) and can `ALLOW`, `DENY`, or `MODIFY` content at any phase. The `GuardrailsInterceptor` (disabled by default) provides the hook point for external content classifiers (AWS Bedrock Guardrails, Azure Content Safety, Lakera Guard, or custom gRPC services). The `ExperimentRouter` operates at `PreRoute` phase. Content policy interceptors can be attached to delegation chains via `contentPolicy.interceptorRef` on `DelegationPolicy`.

11. **Built-in A/B experimentation.** `ExperimentDefinition` as a first-class admin API resource with `ExperimentRouter` as a built-in `RequestInterceptor` ([Section 10.7](10_gateway-internals.md#107-experiment-primitives)). Deterministic HMAC-SHA256 bucketing with cumulative-weight partitioning, sticky assignment (per-user/per-session/none), automatic variant pool sizing via PoolScalingController, delegation propagation modes (`inherit`/`control`/`independent`), and integration with external targeting services (LaunchDarkly, Statsig, Unleash, generic webhook). The experiment system automatically adjusts both variant and base pool `minWarm` to prevent over-provisioning. Isolation monotonicity checks prevent experiments from silently weakening a session's security profile.

12. **Pull-based evaluation hooks.** `POST /v1/sessions/{id}/eval` accepts multi-dimensional scores from any authenticated principal — agent runtime, session owner, or external scorer pipeline ([Section 10.7](10_gateway-internals.md#107-experiment-primitives)). The gateway auto-populates `experiment_id` and `variant_id` from the session's experiment context. Results API (`GET /v1/admin/experiments/{name}/results`) provides per-variant aggregation with mean, p50, p95, and per-dimension breakdowns. `delegation_depth` and `inherited` fields enable filtering for sample contamination in delegated sessions. Lenny explicitly does not build statistical significance testing, automatic winner declaration, multi-armed bandits, or LLM-as-judge integration — eval computation is the deployer's responsibility.

13. **Session replay for regression testing.** `POST /v1/sessions/{id}/replay` re-runs a completed session's prompt history against a different runtime version ([Section 15.1](15_external-api-surface.md#151-rest-api)). Two modes: `prompt_history` (replays full transcript) and `workspace_derive` (clean start with identical filesystem). Replayed sessions can be linked to eval experiments via `evalRef`, enabling systematic A/B comparison across runtime versions.

14. **Pluggable memory store.** `MemoryStore` interface with `Write`, `Query`, `Delete`, and `List` operations, scoped by tenant, user, agent type, and session ([Section 9.4](09_mcp-integration.md#94-memory-store)). Default implementation: Postgres + pgvector with full RLS tenant isolation. Fully replaceable with Mem0, Zep, or any vector database. A `ValidateMemoryStoreIsolation` contract test verifies tenant isolation for custom implementations. Retention controls: configurable `maxMemoriesPerUser` (default: 10,000) with oldest-first eviction. Accessed via `lenny/memory_write` and `lenny/memory_query` platform MCP tools.

15. **Compliance and data governance.** Legal holds on sessions and artifacts ([Section 12.8](12_storage-architecture.md#128-compliance-interfaces)), GDPR data erasure with 19-step dependency-ordered `DeleteByUser` including billing pseudonymization via per-tenant `erasure_salt` with immediate salt deletion for GDPR Recital 26 compliance, processing restriction during erasure (Article 18), and cryptographic erasure receipts. Tenant deletion lifecycle with 6 phases and per-tier SLA enforcement (T3: 72h, T4: 4h). Data residency controls with fail-closed storage routing via `StoreRouter`. Audit logging with hash-chained integrity, configurable retention presets (SOC2, HIPAA, FedRAMP, NIS2/DORA), and SIEM forwarding ([Section 11.7](11_policy-and-controls.md#117-audit-logging)).

16. **Metering and billing integration.** Append-only billing event stream with monotonic sequence numbers and gap detection ([Section 11.2.1](11_policy-and-controls.md#1121-billing-event-stream)). `GET /v1/usage` with filtering by tenant, user, time window, and labels. `GET /v1/metering/events` provides a paginated billing event stream for integration with external billing systems. Per-tenant and per-user quota enforcement with Redis-backed atomic accounting. Billing pseudonymization during GDPR erasure preserves financial reconciliation integrity.

**Trade-offs.** The differentiators above involve deliberate trade-offs that evaluators should weigh:

- **No shared storage mounts** means workspace materialization can add latency to every session start.
- **Least privilege by default** means more complex credential management (credential pools, lease rotation, per-session scoping) compared to simply mounting API keys.
- **Integration with adapter effort** means runtime authors meaning to take full advantage of Lenny's capabilities need to integrate with Lenny's adapter.
- **No built-in eval scoring or memory extraction** means deployers must bring their own tools. Reduces lock-in but increases initial setup for teams without existing tooling.
- **No automatic experiment lifecycle management** means all experiment transitions (pause, conclude, rollback) are operator-initiated. Prevents unsafe auto-rollbacks but requires manual monitoring and Prometheus alerting rules.
- **Pre-warmed pods consume resources when idle**. Warm pools trade resource cost for startup latency. Formula-driven scaling and scale-to-zero mitigate waste, but over-provisioning is possible during traffic ramp-up or experiment variant pool creation.

### 23.2 Community Adoption Strategy

**Target personas:**

| Persona                       | Motivation                                                                                  | Entry point                                                        |
| ----------------------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| **Runtime authors**           | Integrate their agent framework (LangChain, CrewAI, custom) with a managed session platform | Runtime adapter contract ([Section 15.4](15_external-api-surface.md#154-runtime-adapter-specification)); local dev: Embedded Mode (`lenny up`) is the recommended primary path, Source Mode (`make run`) is for platform contributors who need to modify the gateway itself |
| **Platform operators**        | Run multi-tenant agent infrastructure on their own clusters                                 | Helm chart, `lenny-ctl bootstrap`, Admin API ([Section 15.1](15_external-api-surface.md#151-rest-api))          |
| **Enterprise platform teams** | Evaluate Lenny against E2B/Daytona for internal agent hosting with policy controls          | Comparison guides, enterprise controls documentation (Sections 4.8, 8, and 10)  |

**Time to Hello World (TTHW) target: < 5 minutes.** A new contributor must be able to run `lenny up` (Embedded Mode, the recommended primary path — single-binary embedded stack, no source checkout required) and complete a round-trip echo session within 5 minutes on a standard development machine. Platform contributors who need to modify the gateway itself instead clone the repo and run `make run` (Source Mode). The Phase 2 local dev mode ([Section 17.4](17_deployment-topology.md#174-local-development-mode-lenny-dev)) — Embedded Mode with embedded k3s/Postgres/Redis/OIDC and the pre-installed reference runtime catalog, or Source Mode with embedded stores and the echo runtime — is the primary vehicle for this target. CI includes a TTHW smoke test that validates both paths on every merge.

**Governance model.** Lenny adopts a lightweight governance structure established during Phase 2:

- **Benevolent Dictator for Now (BDfN):** Single maintainer makes final decisions until the project reaches 3+ regular contributors, at which point governance transitions to a multi-maintainer steering committee.
- **Decision records:** All architectural decisions tracked via ADRs in `docs/adr/` (see [Section 19](19_resolved-decisions.md)). Community-proposed changes above a defined scope threshold require an ADR with alternatives analysis.
- **Contribution path:** `CONTRIBUTING.md` published in Phase 2 covering: local dev setup, test expectations, PR review process, and the runtime adapter contract as the primary extension point. The Phase 2 `CONTRIBUTING.md` must include a prominent early-development notice stating that the project is in active pre-release development and that unsolicited PRs will not be reviewed or merged until Phase 17a (community launch). Bug reports and discussion via the issue tracker are welcome at any phase. This notice is removed or replaced as part of the Phase 17a documentation and governance review.
- **Governance artifact:** `CONTRIBUTING.md` is published in Phase 2 alongside the `make run` quick-start. `GOVERNANCE.md` is drafted in Phase 2 and finalized in Phase 17a (alongside documentation and governance review). It documents the BDfN → steering committee transition criteria, decision-making process, ADR requirement thresholds, and the license/CLA policy (see open-source license note below). Both files are v1 launch deliverables.
- **Communication:** Public issue tracker for bugs and feature requests. Discussions forum for design proposals and RFC-style conversations.

**Open-source license.** *Resolved — MIT (ADR-008).* License selection was a Phase 0 gating item and is now complete: the `LICENSE` file (MIT) is committed at the repository root. Evaluation criteria applied: competitive landscape alignment (E2B uses Apache 2.0 with a commercial offering, Temporal and LangChain use MIT), enterprise legal review, runtime author copyleft clarity, and compatibility with `kubernetes-sigs/agent-sandbox` upstream. Candidates considered were MIT, Apache 2.0, AGPL + commercial exception, and BSL. MIT was selected for lowest enterprise adoption friction, copyleft clarity for runtime authors, and ecosystem alignment. The decision and rationale are recorded as ADR-008 in `docs/adr/`; any change requires a superseding ADR.

**Community runtime registry.** A community runtime registry — where runtime authors publish versioned adapter packages for operator discovery and installation — is planned as a post-v1 platform service. In v1, runtime adapters are distributed via standard Go module hosting, container registries, and Helm chart repositories. The runtime adapter specification ([Section 15.4](15_external-api-surface.md#154-runtime-adapter-specification)) provides the stable interface contract for v1 adapter distribution.

**Comparison guides.** Phase 17 deliverables include published comparison guides covering Lenny vs E2B, Daytona, Fly.io Sprites, Temporal, Modal, and LangGraph — focusing on self-hosting, recursive delegation, and runtime-agnostic design as differentiators (see [Section 23.1](#231-why-lenny)).
