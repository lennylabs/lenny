### 16.1.1 Attribute Naming

Metric labels and trace/log attributes follow the [OpenTelemetry semantic conventions](https://opentelemetry.io/docs/specs/semconv/) where a convention exists, and Lenny-specific underscore-case names elsewhere. OTel dotted attribute names map to Prometheus labels by replacing dots with underscores (per [OTel's Prometheus interop rules](https://opentelemetry.io/docs/specs/otel/compatibility/prometheus_and_openmetrics/)); both forms refer to the same attribute.

This table is the **single source of truth** for attribute names used across metrics ([Â§16.1](#161-metrics), [Â§16.8](#168-section-25-metrics)), structured logs ([Â§16.4](#164-logging)), tracing spans ([Â§16.3](#163-distributed-tracing)), and alerting PromQL ([Â§16.5](#165-alerting-rules-and-slos)). A label/attribute must not appear anywhere in the spec unless it is in this table or documented as a local-only extension of one of these entries.

| OTel attribute | Prometheus label | Domain | Used on |
|---|---|---|---|
| `k8s.pod.name` | `k8s_pod_name` | Name of the Kubernetes pod the signal originates from or refers to | Task-mode scrub/retirement metrics, slot failure/replacement metrics, task reuse histograms |
| `k8s.namespace.name` | `k8s_namespace_name` | Kubernetes namespace | Where a namespace is a meaningful dimension (rare; `lenny-system`/`lenny-agents` is usually implied) |

**Distinguishing `error.type` from `reason`.** A label named `reason` is reserved for **lifecycle or operational classification** that is not an error â€” for example, pod retirement triggers (`task_count_limit`, `uptime_limit`, `scrub_failure_limit`), pod release causes (`session_complete`, `eviction`, `drain`), or credential lease terminations that are not failures. When the label values describe the cause of an error, failure, rejection, abort, or rotation caused by upstream failure, the label is `error_type` (OTel `error.type`), not `reason`. This mirrors the OTel distinction between `error.type` (failure classification) and `*.reason` domain extensions (lifecycle classification).

**Other domain labels.** Additional Lenny-specific labels (e.g., `outcome`, `operation`, `backend`, `trigger`, `resolution`, `failure_phase`, `event_type`, `statement_type`, `isolation_profile`, `level`, `execution_mode`, `proxy_dialect`, `delivery_mode`, `subsystem`, `deployment_tier`) are documented inline at their point of use in [Â§16.1](#161-metrics) and [Â§16.8](#168-section-25-metrics). They do not appear in this table because they do not correspond to a published OTel convention; they are Lenny-specific enumerations. The `level` label, which appears on multiple metrics in [Â§16.1](#161-metrics) (e.g., `lenny_checkpoint_duration_seconds`, `lenny_checkpoint_size_bytes`, `lenny_checkpoint_size_exceeded_total`, `lenny_checkpoint_storage_failure_total`, `lenny_checkpoint_stale_sessions`), always carries one of `basic`, `standard`, or `full` â€” the runtime integration levels defined in [Â§15.4.3](15_external-api-surface.md#1543-runtime-integration-levels). The `deployment_tier` label, used by the `Tier3GCPressureHigh` alert in [Â§16.5](#165-alerts) and the OpenSLO export in [Â§16.10](#1610-openslo-export), is a static label sourced from Helm (`global.deploymentTier`) and always carries one of `tier1`, `tier2`, or `tier3` â€” the capacity tiers defined in [Â§17.8](17_deployment-topology.md#178-capacity-planning-and-defaults). It is attached at scrape time to every Lenny metric series via `external_labels` on the Prometheus job (or the equivalent OpenTelemetry resource attribute), so it is available on all alerts without requiring per-metric instrumentation. `pool`, `runtime_class`, and `provider` are in the table above because their cardinality is bounded by Helm configuration (number of pools, small enumerated set of runtime classes, small set of provider types) and they are stable across workloads. Introducing a new domain label requires adding it to the metric's inline description; introducing a new global attribute requires adding it here.

**High-cardinality attributes forbidden as metric labels.** `session_id`, `operation_id`, `root_session_id`, `credential_id`, `user_id`, `lease_id`, and `request_id` are valid identifiers on **traces** ([Â§16.3](#163-distributed-tracing)), **structured logs** ([Â§16.4](#164-logging)), and **audit records** ([Â§11.7](11_policy-and-controls.md#117-audit-logging)) but MUST NOT be used as Prometheus metric labels. Their cardinality is unbounded at the per-operation or per-session scale, which would cause Prometheus time-series explosion on ingestion. The only per-tenant unbounded label permitted on metrics is `tenant_id`, and only because its cardinality is bounded at the platform scale by deployer-controlled tenant provisioning. Any new metric introduced by a PR that includes one of these attributes as a label is a CI failure (enforced by the `pkg/metrics/lint` validator that scans metric registrations for label-name membership in a denylist).

### 16.3 Distributed Tracing

Lenny supports two tiers of distributed tracing, serving different purposes:

**Tier 1: Infrastructure tracing (automatic).** OpenTelemetry trace ID propagation through the entire delegation tree. This covers gateway, controller, and adapter spans â€” platform-level observability that operators use for debugging latency, errors, and resource usage. Fully automatic; runtimes do not need to participate.

**Tier 2: Runtime-level tracing (runtime-managed).** Opaque `tracingContext` propagation through delegation for runtime-native eval and observability platforms (LangSmith, Braintrust, Humanloop, etc.). Runtimes register tracing identifiers via `lenny/set_tracing_context` (MCP) or `set_tracing_context` (JSONL), and the gateway attaches them to child delegation leases automatically. Child runtimes receive the parent's `tracingContext` in the adapter manifest and use it to stitch their native traces into the parent's trace tree. This enables end-to-end eval tracing across delegation boundaries without Lenny needing to understand the tracing platform's wire format. See [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease) for `tracingContext` validation rules.

**Tier 1 trace context flows through:**

- Client â†’ Gateway (HTTP headers)
- Gateway â†’ Pod (gRPC metadata)
- Pod â†’ Gateway (delegation tool calls carry parent trace context)
- Gateway â†’ Child Pod (inherited trace context)
- Gateway â†’ External MCP tools (HTTP headers)

Because sampling decisions are made in the Collector after spans are complete (tail-based), error and latency conditions are known at decision time. The trace pipeline uses OpenTelemetry Collector; the platform emits OTLP traces and the collector handles sampling, batching, and export. The backend is deployer-configured â€” Jaeger, Tempo, Zipkin, or cloud-native options (Cloud Trace, X-Ray) are all supported. In dev mode, 100% sampling is enabled with a local Jaeger instance (or stdout exporter for `make run`).

- `PERMANENT` â€” not retryable (invalid workspace, policy denial)
- `POLICY` â€” denied by policy engine (quota exceeded, unauthorized runtime, `CREDENTIAL_POOL_EXHAUSTED`)
- `UPSTREAM` â€” external dependency failure (MCP tool error, auth failure)

### 16.4 Logging

- Structured JSON logs from gateway, token service, pool controller, runtime adapter, and `lenny-ops`
- Correlation fields in every log line: `session_id`, `tenant_id`, `trace_id`, `span_id`
- **Agent-operability correlation fields** (all components, [Â§25.4](25_agent-operability.md#254-the-lenny-ops-service)): every log line MUST include `ts` (RFC 3339 UTC), `level`, `msg`, `component`, and â€” when present on the originating request â€” `operation_id` (from the `X-Lenny-Operation-ID` header) and `agent_name` (from `X-Lenny-Agent-Name`). This shape is shared between the gateway and `lenny-ops` so a single orchestrated agent task can be joined across both binaries via `operation_id`.
- Setup command stdout/stderr captured and stored in EventStore
- Audit events for all policy decisions
- Error events include structured error codes (TRANSIENT/PERMANENT/POLICY/UPSTREAM)
- **Credential-sensitive RPCs** (`AssignCredentials`, `RotateCredentials`) are excluded from payload-level logging, gRPC access logs, and OTel trace span attributes. Only RPC name, lease ID, provider type, and outcome are recorded.

**Log retention and EventStore management.** EventStore tables (audit events, session logs, stream cursors) are partitioned by time using native Postgres range partitioning. A background job drops partitions beyond the retention window: 365 days for audit events (default), 30 days for session logs, 7 days for stream cursors. **SIEM delivery guard:** When `audit.siem.endpoint` is configured, the partition GC MUST NOT drop an audit partition whose most recent event has a `sequence_number` greater than the SIEM forwarder's last acknowledged high-water mark (read from the `siem_delivery_state` table). If the forwarder has not consumed all events in a partition that is past its retention TTL, the GC holds the partition and emits an `AuditPartitionDropBlocked` warning alert identifying the partition, its age, and the SIEM delivery lag. This prevents permanent data loss when the SIEM forwarder is stalled or misconfigured for an extended period. The held partition is dropped once the forwarder's high-water mark advances past its last event, or when an operator explicitly acknowledges the data loss via `lenny-ctl audit drop-partition <partition> --force --acknowledge-data-loss` (see [Â§25.9](25_agent-operability.md#259-audit-log-query-api) for the backing `POST /v1/admin/audit-partitions/{partition}/drop` endpoint). Estimated volume is ~10 KB per session for audit events and ~50 KB for logs; at 10K sessions/day this is ~600 MB/day before retention cleanup. This estimate corresponds to Tier 3 daily throughput; see [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier volume estimates. Deployers should configure an external log aggregation stack (ELK, Loki, CloudWatch, etc.) for long-term retention beyond the Postgres window.

**GDPR erasure receipt rows (`event_type LIKE 'gdpr.%'`) are exempt from the standard partition GC.** The audit GC applies a separate `audit.gdprRetentionDays` window (default: 2555 days / 7 years) to these rows. This is a hard minimum floor: the GC will not delete `gdpr.*` rows until they exceed this window regardless of the general `audit.retentionDays` setting. See [Section 12.8](12_storage-architecture.md#128-compliance-interfaces) for the full GDPR audit event exemption policy.

**Compliance-aware audit retention.** The default 365-day Postgres retention satisfies SOC2 (12-month audit evidence window) and FedRAMP Low/Moderate (1-year AU-11). Deployers subject to stricter frameworks should set `audit.retentionDays` using named presets or a custom value:

| Preset           | Retention           | Frameworks                       | `complianceProfile` pairing        |
| ---------------- | ------------------- | -------------------------------- | ---------------------------------- |
| `soc2` (default) | 365 days            | SOC2, FedRAMP Low/Moderate       | `soc2` or `fedramp` (default)      |
| `fedramp-high`   | 1095 days (3 years) | FedRAMP High                     | `fedramp` + this preset            |
| `hipaa`          | 2190 days (6 years) | HIPAA Â§164.312(b)                | `hipaa`                            |
| `nis2-dora`      | 1825 days (5 years) | EU NIS2, DORA (financial sector) | `none` or `soc2` + this preset     |
| `custom`         | Deployer-specified  | Any                              | Any                                |

The `audit.retentionPreset` is independent of `complianceProfile`: the profile governs which runtime controls are enforced (SIEM requirement, grant-check interval, pgaudit), while the preset governs how long data is retained in Postgres. FedRAMP High deployments must set both `complianceProfile: fedramp` and `audit.retentionPreset: fedramp-high`. See [Section 11.7](11_policy-and-controls.md#117-audit-logging) for the FedRAMP baseline note.

Helm configuration: `audit.retentionPreset: soc2` or `audit.retentionDays: 2190`. The preset drives the Postgres partition retention policy automatically.

**SIEM responsibility split.** When an external SIEM is configured (see [Section 11.7](11_policy-and-controls.md#117-audit-logging)), the SIEM is the system of record for long-term regulatory retention. The Postgres audit partition serves as a queryable hot tier for operational use. In this configuration, the Postgres retention period may be shorter than the regulatory requirement â€” the SIEM satisfies the long-term obligation. If `audit.retentionDays` is set below 365 and `LENNY_ENV=production`, the gateway emits an `AuditRetentionLow` warning at startup. This warning is suppressed if `audit.siem.endpoint` is configured, since the SIEM provides the durable long-term copy.

Total ops/s is higher for three reasons: (1) `budget_return.lua` runs at a comparable rate (~15/s), giving ~30/s total Lua invocations; (2) each Lua script atomically executes ~10â€“15 internal Redis commands across 5â€“6 keys (GET, INCR, DECR, SET, EXPIRE); (3) gateway reads of budget state (e.g., remaining tokens for progress reporting) add ~5/s of non-Lua reads. The [Â§12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes) figure of ~500â€“1,000/s applies to multi-tenant Tier 3 (up to 500 active tenants, though not all delegate simultaneously) and includes burst headroom â€” the upper bound reflects peak instantaneous rates when multiple tenants burst delegations concurrently. Operators must update these Helm values before relying on any formula-derived sizing output.
