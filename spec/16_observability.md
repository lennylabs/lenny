### 16.1.1 Attribute Naming

Metric labels and trace/log attributes follow the [OpenTelemetry semantic conventions](https://opentelemetry.io/docs/specs/semconv/) where a convention exists, and Lenny-specific underscore-case names elsewhere. OTel dotted attribute names map to Prometheus labels by replacing dots with underscores (per [OTel's Prometheus interop rules](https://opentelemetry.io/docs/specs/otel/compatibility/prometheus_and_openmetrics/)); both forms refer to the same attribute.

This table is the **single source of truth** for attribute names used across metrics ([§16.1](#161-metrics), [§16.8](#168-section-25-metrics)), structured logs ([§16.4](#164-logging)), tracing spans ([§16.3](#163-distributed-tracing)), and alerting PromQL ([§16.5](#165-alerting-rules-and-slos)). A label/attribute must not appear anywhere in the spec unless it is in this table or documented as a local-only extension of one of these entries.

| OTel attribute | Prometheus label | Domain | Used on |
|---|---|---|---|
| `k8s.pod.name` | `k8s_pod_name` | Name of the Kubernetes pod the signal originates from or refers to | Task-mode scrub/retirement metrics, slot failure/replacement metrics, task reuse histograms |
| `k8s.namespace.name` | `k8s_namespace_name` | Kubernetes namespace | Where a namespace is a meaningful dimension (rare; `lenny-system`/`lenny-agents` is usually implied) |
| `service.instance.id` | `service_instance_id` | Stable identity of a process-level service instance (gateway replica, controller pod, admission webhook replica) | GC-pause P99, admission webhook rejection counters, replica-scoped gauges |
| `service.name` | `service_name` | Service identifier (`lenny-gateway`, `lenny-ops`, `lenny-token-service`, `lenny-warm-pool-controller`) | Resource attribute on every signal |
| `service.version` | `service_version` | Build tag of the service | Resource attribute |
| `error.type` | `error_type` | Classification of an error, failure, rejection, or rotation cause (e.g., `image_pull_error`, `rate_limit`, `memory_budget_exhausted`, `zip_bomb`) | Warmup failures, slot failures, credential rotation causes, upload extraction aborts, subsystem error counters, delegation tree memory rejections, policy denial causes |
| `tenant_id` | `tenant_id` | Lenny tenant — Lenny-specific; no OTel convention | Every tenant-scoped metric |
| `pool` | `pool` | Warm pool / `SandboxWarmPool` name — Lenny-specific | Pool-scoped metrics |
| `runtime_class` | `runtime_class` | Kubernetes `RuntimeClass` name (`runc`, `gvisor`, `kata`) — Lenny uses the K8s field name, no dot form required | Pool and pod metrics |
| `provider` | `provider` | External provider name — Lenny-specific domain (credential provider, experiment-targeting provider) | Credential pool metrics, experiment metrics |

**Distinguishing `error.type` from `reason`.** A label named `reason` is reserved for **lifecycle or operational classification** that is not an error — for example, pod retirement triggers (`task_count_limit`, `uptime_limit`, `scrub_failure_limit`), pod release causes (`session_complete`, `eviction`, `drain`), or credential lease terminations that are not failures. When the label values describe the cause of an error, failure, rejection, abort, or rotation caused by upstream failure, the label is `error_type` (OTel `error.type`), not `reason`. This mirrors the OTel distinction between `error.type` (failure classification) and `*.reason` domain extensions (lifecycle classification).

**Other domain labels.** Additional Lenny-specific labels (e.g., `outcome`, `operation`, `backend`, `trigger`, `resolution`, `failure_phase`, `event_type`, `statement_type`, `isolation_profile`, `level`, `execution_mode`, `proxy_dialect`, `delivery_mode`, `subsystem`, `deployment_tier`, `experiment_id`, `variant_id`) are documented inline at their point of use in [§16.1](#161-metrics) and [§16.8](#168-section-25-metrics). They do not appear in this table because they do not correspond to a published OTel convention; they are Lenny-specific enumerations. `experiment_id` and `variant_id` are attached by the experiment-routing metrics family ([Section 10.7](10_gateway-internals.md#107-experiment-primitives)) and are bounded by deployer-declared experiment configuration — one `experiment_id` per active experiment and a small enumerated set of `variant_id` values per experiment (typically `control` plus 1–3 treatment arms), so cardinality stays within the small-enumerated-set discipline applied to the other labels in this list. The `level` label, which appears on multiple metrics in [§16.1](#161-metrics) (e.g., `lenny_checkpoint_duration_seconds`, `lenny_checkpoint_size_bytes`, `lenny_checkpoint_size_exceeded_total`, `lenny_checkpoint_storage_failure_total`, `lenny_checkpoint_stale_sessions`), always carries one of `basic`, `standard`, or `full` — the runtime integration levels defined in [§15.4.3](15_external-api-surface.md#1543-runtime-integration-levels). The `deployment_tier` label, used by the `Tier3GCPressureHigh` alert in [§16.5](#165-alerting-rules-and-slos) and the OpenSLO export in [§16.10](#1610-openslo-export), is a static label sourced from Helm (`global.deploymentTier`) and always carries one of `tier1`, `tier2`, or `tier3` — the capacity tiers defined in [§17.8](17_deployment-topology.md#178-capacity-planning-and-defaults). It is attached at scrape time to every Lenny metric series via `external_labels` on the Prometheus job (or the equivalent OpenTelemetry resource attribute), so it is available on all alerts without requiring per-metric instrumentation. `pool`, `runtime_class`, and `provider` are in the table above because their cardinality is bounded by Helm configuration (number of pools, small enumerated set of runtime classes, small set of provider types) and they are stable across workloads. Introducing a new domain label requires adding it to the metric's inline description; introducing a new global attribute requires adding it here.

**High-cardinality attributes forbidden as metric labels.** `session_id`, `operation_id`, `root_session_id`, `credential_id`, `user_id`, `lease_id`, and `request_id` are valid identifiers on **traces** ([§16.3](#163-distributed-tracing)), **structured logs** ([§16.4](#164-logging)), and **audit records** ([§11.7](11_policy-and-controls.md#117-audit-logging)) but MUST NOT be used as Prometheus metric labels. Their cardinality is unbounded at the per-operation or per-session scale, which would cause Prometheus time-series explosion on ingestion. The only per-tenant unbounded label permitted on metrics is `tenant_id`, and only because its cardinality is bounded at the platform scale by deployer-controlled tenant provisioning. Any new metric introduced by a PR that includes one of these attributes as a label is a CI failure (enforced by the `pkg/metrics/lint` validator that scans metric registrations for label-name membership in a denylist).

### 16.2 Key Latency Breakpoints

Instrument four timestamps per session:

1. Pod claimed
2. Workspace prep done
3. Session ready
4. First event/token emitted

This lets operators identify whether bottlenecks are in pod allocation, file upload, session start, or the agent runtime itself.

### 16.3 Distributed Tracing

Lenny supports two tiers of distributed tracing, serving different purposes:

**Tier 1: Infrastructure tracing (automatic).** OpenTelemetry trace ID propagation through the entire delegation tree. This covers gateway, controller, and adapter spans — platform-level observability that operators use for debugging latency, errors, and resource usage. Fully automatic; runtimes do not need to participate.

**Tier 2: Runtime-level tracing (runtime-managed).** Opaque `tracingContext` propagation through delegation for runtime-native eval and observability platforms (LangSmith, Braintrust, Humanloop, etc.). Runtimes register tracing identifiers via `lenny/set_tracing_context` (MCP) or `set_tracing_context` (JSONL), and the gateway attaches them to child delegation leases automatically. Child runtimes receive the parent's `tracingContext` in the adapter manifest and use it to stitch their native traces into the parent's trace tree. This enables end-to-end eval tracing across delegation boundaries without Lenny needing to understand the tracing platform's wire format. See [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease) for `tracingContext` validation rules.

**Tier 1 trace context flows through:**

- Client → Gateway (HTTP headers)
- Gateway → Pod (gRPC metadata)
- Pod → Gateway (delegation tool calls carry parent trace context)
- Gateway → Child Pod (inherited trace context)
- Gateway → External MCP tools (HTTP headers)

**Sampling and backend:** The gateway emits 100% of traces to the OpenTelemetry Collector; the **Collector** applies tail-based sampling to decide which traces to export to the backend. The default probabilistic sampling rate is 10% for normal operations (configurable via `global.traceSamplingRate` Helm value). 100% sampling is applied for errors (any span with error status), slow requests (session creation exceeding 500ms, the P99 SLO target), and delegation trees (all spans in a tree are sampled if the root is sampled, preserving trace completeness). Because sampling decisions are made in the Collector after spans are complete (tail-based), error and latency conditions are known at decision time. The trace pipeline uses OpenTelemetry Collector; the platform emits OTLP traces and the collector handles sampling, batching, and export. The backend is deployer-configured — Jaeger, Tempo, Zipkin, or cloud-native options (Cloud Trace, X-Ray) are all supported. In dev mode, 100% sampling is enabled with a local Jaeger instance (or stdout exporter for `make run`).

**Error codes:** Structured error taxonomy with categories:

- `TRANSIENT` — retryable (pod crash, network timeout)
- `PERMANENT` — not retryable (invalid workspace, policy denial)
- `POLICY` — denied by policy engine (quota exceeded, unauthorized runtime, `CREDENTIAL_POOL_EXHAUSTED`)
- `UPSTREAM` — external dependency failure (MCP tool error, auth failure)

### 16.4 Logging

- Structured JSON logs from gateway, token service, pool controller, runtime adapter, and `lenny-ops`
- Correlation fields in every log line: `session_id`, `tenant_id`, `trace_id`, `span_id`
- **Agent-operability correlation fields** (all components, [§25.4](25_agent-operability.md#254-the-lenny-ops-service)): every log line MUST include `ts` (RFC 3339 UTC), `level`, `msg`, `component`, and — when present on the originating request — `operation_id` (from the `X-Lenny-Operation-ID` header) and `agent_name` (from `X-Lenny-Agent-Name`). This shape is shared between the gateway and `lenny-ops` so a single orchestrated agent task can be joined across both binaries via `operation_id`.
- Setup command stdout/stderr captured and stored in EventStore
- Audit events for all policy decisions
- Error events include structured error codes (TRANSIENT/PERMANENT/POLICY/UPSTREAM)
- **Credential-sensitive RPCs** (`AssignCredentials`, `RotateCredentials`) are excluded from payload-level logging, gRPC access logs, and OTel trace span attributes. Only RPC name, lease ID, provider type, and outcome are recorded.

**Log retention and EventStore management.** EventStore tables (audit events, session logs, stream cursors) are partitioned by time using native Postgres range partitioning. A background job drops partitions beyond the retention window: 365 days for audit events (default), 30 days for session logs, 7 days for stream cursors. **SIEM delivery guard:** When `audit.siem.endpoint` is configured, the partition GC MUST NOT drop an audit partition whose most recent event has a `sequence_number` greater than the SIEM forwarder's last acknowledged high-water mark (read from the `siem_delivery_state` table). If the forwarder has not consumed all events in a partition that is past its retention TTL, the GC holds the partition and emits an `AuditPartitionDropBlocked` warning alert identifying the partition, its age, and the SIEM delivery lag. This prevents permanent data loss when the SIEM forwarder is stalled or misconfigured for an extended period. The held partition is dropped once the forwarder's high-water mark advances past its last event, or when an operator explicitly acknowledges the data loss via `lenny-ctl audit drop-partition <partition> --force --acknowledge-data-loss` (see [§25.9](25_agent-operability.md#259-audit-log-query-api) for the backing `POST /v1/admin/audit-partitions/{partition}/drop` endpoint). Estimated volume is ~10 KB per session for audit events and ~50 KB for logs; at 10K sessions/day this is ~600 MB/day before retention cleanup. This estimate corresponds to Tier 3 daily throughput; see [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier volume estimates. Deployers should configure an external log aggregation stack (ELK, Loki, CloudWatch, etc.) for long-term retention beyond the Postgres window.

**GDPR erasure receipt rows (`event_type LIKE 'gdpr.%'`) are exempt from the standard partition GC.** The audit GC applies a separate `audit.gdprRetentionDays` window (default: 2555 days / 7 years) to these rows. This is a hard minimum floor: the GC will not delete `gdpr.*` rows until they exceed this window regardless of the general `audit.retentionDays` setting. See [Section 12.8](12_storage-architecture.md#128-compliance-interfaces) for the full GDPR audit event exemption policy.

**Compliance-aware audit retention.** The default 365-day Postgres retention satisfies SOC2 (12-month audit evidence window) and FedRAMP Low/Moderate (1-year AU-11). Deployers subject to stricter frameworks should set `audit.retentionDays` using named presets or a custom value:

| Preset           | Retention           | Frameworks                       | `complianceProfile` pairing        |
| ---------------- | ------------------- | -------------------------------- | ---------------------------------- |
| `soc2` (default) | 365 days            | SOC2, FedRAMP Low/Moderate       | `soc2` or `fedramp` (default)      |
| `fedramp-high`   | 1095 days (3 years) | FedRAMP High                     | `fedramp` + this preset            |
| `hipaa`          | 2190 days (6 years) | HIPAA §164.312(b)                | `hipaa`                            |
| `nis2-dora`      | 1825 days (5 years) | EU NIS2, DORA (financial sector) | `none` or `soc2` + this preset     |
| `custom`         | Deployer-specified  | Any                              | Any                                |

The `audit.retentionPreset` is independent of `complianceProfile`: the profile governs which runtime controls are enforced (SIEM requirement, grant-check interval, pgaudit), while the preset governs how long data is retained in Postgres. FedRAMP High deployments must set both `complianceProfile: fedramp` and `audit.retentionPreset: fedramp-high`. See [Section 11.7](11_policy-and-controls.md#117-audit-logging) for the FedRAMP baseline note.

Helm configuration: `audit.retentionPreset: soc2` or `audit.retentionDays: 2190`. The preset drives the Postgres partition retention policy automatically.

**SIEM responsibility split.** When an external SIEM is configured (see [Section 11.7](11_policy-and-controls.md#117-audit-logging)), the SIEM is the system of record for long-term regulatory retention. The Postgres audit partition serves as a queryable hot tier for operational use. In this configuration, the Postgres retention period may be shorter than the regulatory requirement — the SIEM satisfies the long-term obligation. If `audit.retentionDays` is set below 365 and `LENNY_ENV=production`, the gateway emits an `AuditRetentionLow` warning at startup. This warning is suppressed if `audit.siem.endpoint` is configured, since the SIEM provides the durable long-term copy.

**Bundled alerting rules.** The alert catalog above is authored once as a shared Go package `pkg/alerting/rules`. Two consumers derive from this package:

1. **Helm-rendered deployer manifests** — the chart renders the rules into a `PrometheusRule` CRD (when `monitoring.format: prometheusrule`) and/or a ConfigMap (`configmap`), controlled by `monitoring.format` ([§25.13](25_agent-operability.md#2513-bundled-alerting-rules)).
2. **In-process alert state tracker** — the gateway compiles the rules into an in-process evaluator used as a fallback when Prometheus is unreachable ([§25.3](25_agent-operability.md#253-gateway-side-ops-endpoints)). When `gateway.healthTracker.useCompiledRules: true`, the gateway evaluates the bundled rules against its in-process metric registry and against Prometheus through the `MetricReader` interface when available.

Because both consumers share the same source, any rule change is picked up simultaneously by the rendered Helm manifests and the gateway binary on the next release. `docs/alerting/rules.yaml` is regenerated on each release and committed for deployers on non-Prometheus monitoring stacks.

**Capacity recommendation rules** follow the same pattern: authored in `pkg/recommendations/rules`, compiled into both the gateway (evaluated against the in-process `MetricReader`) and `lenny-ops` (evaluated against a Prometheus-backed `MetricReader`). See [§25.3](25_agent-operability.md#253-gateway-side-ops-endpoints) for the API shape.

**Operator-driven alerts (not bundled).** Two additional rules are recommended but not shipped because they depend on operator-owned Prometheus instrumentation: `PrometheusQueryLatencyHigh` on `lenny_prometheus_query_duration_seconds` P95, and a Prometheus-side rule on `prometheus_rule_evaluation_duration_seconds` P95. Document these in the deployer runbook alongside the monitoring stack setup.

**Tier 4 (Platform) definition.** Tier 4 is the first tier that requires infrastructure changes beyond horizontal scaling of v1 components. Tiers 1–3 are achievable with the v1 architecture by scaling replica counts, instance sizes, and Redis/Postgres topology. Tier 4 requires swapping one or more of the [§12.6](12_storage-architecture.md#126-interface-design) scaling extension interfaces to their high-scale implementations: `PostgresPodRegistry` (replacing `CRDPodRegistry` to eliminate etcd write pressure), multi-shard `StoreRouter` (replacing `SingleShardRouter`), durable `EventBus` (replacing `RedisEventBus`), and potentially `ClusterRegistry` for multi-cluster delegation routing. The interface boundaries in [§12.6](12_storage-architecture.md#126-interface-design) are designed so that each swap is a configuration-level change — no application code restructuring is required, though accompanying infrastructure provisioning (Postgres schema migrations, NATS/Kafka deployment, additional cluster federation) is needed. Tier 4 targets are indicative planning figures (10× Tier 3 on most dimensions); exact values must be validated by load testing after the relevant interface swaps are deployed. Tier 4 is **not** a GA deliverable — it is a post-GA scaling milestone. All references to "Tier 4" elsewhere in this document describe the design-time intent of the [§12.6](12_storage-architecture.md#126-interface-design) interfaces, not a committed implementation timeline.

**Workload profile assumptions.** The tier sizing formulas throughout this document (warm pool sizing, Redis ops budgets, Postgres IOPS, gateway replica counts) depend on assumptions about session duration, delegation rates, and workspace size. These assumptions are expressed as explicit Helm values under `capacityPlanning.*` so that operators with atypical workloads can substitute their own measurements without reverse-engineering the formulas. The following table lists the parameters, their defaults, and where each feeds into sizing calculations:

| Helm value | Default | Unit | Used by |
| --- | --- | --- | --- |
| `capacityPlanning.avgSessionDurationSeconds` | 333 | seconds | Warm pool claim rate (Little's Law: `concurrent_sessions / avg_duration`); delegation budget Redis ops/s |
| `capacityPlanning.delegationParticipationRate` | 0.05 | fraction | Fraction of sessions that perform at least one delegation; scales delegation budget operations |
| `capacityPlanning.avgDelegationsPerDelegatingSession` | 10 | integer | Average number of child delegations per delegating session; scales `budget_reserve.lua` invocation rate |
| `capacityPlanning.avgChildSessionSeconds` | 60 | seconds | Average duration of a child delegation session; used in delegation fan-out `minWarm` formula ([§17.8.2](17_deployment-topology.md#1782-capacity-tier-reference)) |
| `capacityPlanning.avgWorkspaceSizeMB` | 100 | MB | Checkpoint bandwidth estimates; MinIO throughput budget ([§17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) object storage table) |
| `capacityPlanning.sessionIdleFraction` | 0.30 | fraction | Fraction of active sessions that are idle (no active LLM call); scales LLM Proxy concurrent connection estimate |

Derived quantities: at Tier 3 defaults, implied claim rate = `10,000 / 333 ≈ 30/s` (matching the warm pool sizing table); delegation budget Lua invocation rate = `10,000 × 0.05 × 10 / 333 ≈ 15/s` sustained `budget_reserve.lua` invocations (time-averaged mean; the denominator is `avgSessionDurationSeconds` because it represents the parent session lifetime over which delegations are spread — not `avgChildSessionSeconds`, which governs concurrent child pod demand). Peak instantaneous Lua invocation rates can be significantly higher than the sustained average because orchestrator sessions typically burst all child delegations at session start; operators should monitor `lenny_redis_lua_script_duration_seconds{script="budget_reserve"}` burst percentiles and the [§8.3](08_recursive-delegation.md#83-delegation-policy-and-lease) cross-tenant aggregate contention formula for burst sizing. **Reconciliation with [§12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes) and [§17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) Redis ops/s figures.** The ~15/s figure is the `budget_reserve.lua` script invocation rate, not the total Redis ops/s for delegation budgets. Total ops/s is higher for three reasons: (1) `budget_return.lua` runs at a comparable rate (~15/s), giving ~30/s total Lua invocations; (2) each Lua script atomically executes ~10–15 internal Redis commands across 5–6 keys (GET, INCR, DECR, SET, EXPIRE); (3) gateway reads of budget state (e.g., remaining tokens for progress reporting) add ~5/s of non-Lua reads. The resulting sustained total is ~30 × 12 + 5 ≈ **365 ops/s** for a single-tenant Tier 3 deployment, consistent with the ~400 ops/s estimate in the [§17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) single-tenant topology table. The [§12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes) figure of ~500–1,000/s applies to multi-tenant Tier 3 (up to 500 active tenants, though not all delegate simultaneously) and includes burst headroom — the upper bound reflects peak instantaneous rates when multiple tenants burst delegations concurrently. Operators must update these Helm values before relying on any formula-derived sizing output. The PoolScalingController reads `capacityPlanning.*` values at startup and logs a warning if defaults are in use for a Tier 2 or Tier 3 deployment: `[WARN] capacityPlanning Helm values are at defaults — review Section 16.5 workload profile table and substitute observed values before production promotion`.

**Design-time scale target:** All architectural decisions (queue depths, connection pool sizes, controller reconciliation intervals, warm pool sizing) must be validated against **Tier 2** as the minimum. Tier 3 must be achievable with horizontal scaling only (no architectural changes). Tier 4 requires swapping one or more [§12.6](12_storage-architecture.md#126-interface-design) scaling extension interfaces to their high-scale implementations — see "Tier 4 (Platform) definition" above. Tier 1 exists for single-node development and CI environments.

**Benchmark requirement:** Before GA, load tests must demonstrate that all SLOs below hold at sustained Tier 2 load, and that Tier 3 is reachable with linear horizontal scaling of gateway, controllers, and data stores. Phase 13.5 establishes pre-hardening baselines; Phase 14.5 re-validates all SLOs with full security hardening active — Phase 14.5 is the final GA gate for SLO compliance.

| Startup latency (pod-warm, gVisor)      | P95 < 5s             | Pod claim through agent session ready, excluding client file upload and workspace materialization (narrower than the indicative ≤ 9s total platform-controlled envelope in [§6.3](06_warm-pod-model.md#63-startup-latency-analysis)) |
| Checkpoint duration (≤ 100MB workspace) | P95 < 2s             | Quiescence request through snapshot upload complete          |
| Warm-pool failover-window serve fraction | ≥ 0.99 per event     | `lenny_warmpool_failover_claims_served_fraction` (per observed failover, or per chaos-drill synthetic failover). Validation gate for `safety_factor` ([§17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) "Rationale and validation gate for the Tier 3 1.2 factor"). Not met → raise `poolScaling.safetyFactor` universally. |

**Startup warning requirement:** The gateway MUST emit the warning above at startup whenever the SLO configuration values remain equal to the defaults listed above and the `slo.validated` Helm value is not set to `true`. The `slo.validated` flag is set by the Phase 14.5 benchmark automation after it records benchmark results to the Phase 14.5 exit gate ADR and annotates the Helm release. This ensures that any deployment that has not completed the validation gate cannot silently operate with unvalidated SLO defaults.

**SLO error-budget burn-rate alerts:**

Multi-window burn-rate alerting is required for all availability and latency SLOs. The fast window (1 h at 14× burn rate) catches acute outages quickly; the slow window (6 h at 3× burn rate) catches slow-burn degradation that would exhaust the monthly budget without triggering threshold-only alerts.

| Alert                                   | SLO                         | Fast window | Slow window | Severity |
| --------------------------------------- | --------------------------- | ----------- | ----------- | -------- |
| `SessionCreationSuccessRateBurnRate`    | Session creation ≥ 99.5%    | 1 h, 14×   | 6 h, 3×    | Critical (fast) / Warning (slow) |
| `SessionCreationLatencyBurnRate`        | Session creation P99 < 500ms | 1 h, 14×  | 6 h, 3×    | Critical (fast) / Warning (slow) |
| `SessionAvailabilityBurnRate`           | Session availability ≥ 99.9% | 1 h, 14×  | 6 h, 3×    | Critical (fast) / Warning (slow) |
| `GatewayAvailabilityBurnRate`           | Gateway availability ≥ 99.95% | 1 h, 14× | 6 h, 3×    | Critical (fast) / Warning (slow) |
| `StartupLatencyBurnRate`               | Startup latency P95 < 2s (runc); metric: `lenny_session_startup_duration_seconds{isolation_profile="runc"}` | 1 h, 14× | 6 h, 3×  | Critical (fast) / Warning (slow) |
| `StartupLatencyGVisorBurnRate`         | Startup latency P95 < 5s (gVisor); metric: `lenny_session_startup_duration_seconds{isolation_profile="gvisor"}` | 1 h, 14× | 6 h, 3× | Critical (fast) / Warning (slow) |
| `TTFTBurnRate`                          | Time to first token P95 < 10s; metric: `lenny_session_time_to_first_token_seconds` | 1 h, 14×  | 6 h, 3×    | Critical (fast) / Warning (slow) |
| `CheckpointDurationBurnRate`            | Checkpoint duration P95 < 2s (≤ 100MB) | 1 h, 14× | 6 h, 3× | Critical (fast) / Warning (slow) |

**Burn-rate calculation:** For availability SLOs, burn rate is `error_rate / (1 - slo_target)` over the window. For latency SLOs, burn rate is `rate(slow_requests[window]) / rate(total_requests[window]) / error_budget_fraction`. A burn rate of 1× means the budget is consumed at exactly the rate that would exhaust it over the 30-day window. At 14× for 1 hour the alert fires if more than `14 * (1h / 720h) = 1.94%` of the monthly budget is consumed in one hour. At 3× for 6 hours the alert fires if more than `3 * (6h / 720h) = 2.5%` is consumed. Both conditions must be present simultaneously for a page (fast window page; slow window warn only) to reduce false positives. Burn-rate thresholds are configurable via Helm values (`slo.burnRate.fastMultiplier`, default 14; `slo.burnRate.slowMultiplier`, default 3).

### 16.6 Operational Events Catalog

Lenny emits **operational events** distinct from audit events. Operational events describe platform state transitions relevant to live operators and autonomous agents; they flow through the Redis stream and the gateway in-memory buffer described in [§25.3](25_agent-operability.md#253-gateway-side-ops-endpoints) and [§25.5](25_agent-operability.md#255-operational-event-stream). Every event is a [CloudEvents v1.0.2](https://github.com/cloudevents/spec/blob/v1.0.2/cloudevents/spec.md) JSON record. The CloudEvents `type` attribute is `dev.lenny.<short_name>` where `<short_name>` is the identifier listed below (e.g., `dev.lenny.alert_fired`, `dev.lenny.pool_state_changed`). Section 25 is the source-of-truth for payload shapes; the catalog below is the canonical enumeration.

**Gateway-emitted:** `alert_fired`, `alert_resolved`, `upgrade_progressed`, `pool_state_changed`, `circuit_breaker_opened`, `circuit_breaker_closed`, `credential_rotated`, `credential_pool_exhausted`, `session_completed`, `session_failed`, `session_terminated`, `session_cancelled`, `session_expired`, `session_awaiting_action`, `delegation_completed`, `backup_completed`, `backup_failed`, `platform_upgrade_available`, `health_status_changed`. Session lifecycle events (`session_*`, `delegation_completed`) are the same events delivered via the `callbackUrl` webhook ([§14](14_workspace-plan-schema.md)), the agent-operability SSE stream, and webhook subscriptions ([§25.5](25_agent-operability.md#255-operational-event-stream)) — one envelope, multiple transports.

**Experiment events (gateway-emitted, operational).** The `ExperimentRouter` ([§10.7](10_gateway-internals.md#107-experiment-primitives)) emits the following operational events during external targeting, multi-experiment routing, and isolation-monotonicity enforcement. All are CloudEvents with `type: dev.lenny.<short_name>` and flow through the same Redis stream / in-memory buffer as the gateway-emitted events above.

- `experiment.unknown_variant_from_provider` (warning) — emitted when an OpenFeature provider returns a `Variant` or `Value` shape the gateway cannot resolve to a registered variant ID; the session falls back to `"control"`. Payload fields: `tenant_id`, `user_id`, `experiment_id`, `provider`, `raw_variant` (the unmapped value).
- `experiment.unknown_external_id` (info) — emitted when the provider returns an experiment/flag ID that is not registered in Lenny's `mode: external` experiment catalog; the experiment is skipped. Payload fields: `tenant_id`, `user_id`, `provider`, `external_experiment_id`.
- `experiment.targeting_failed` (warning) — emitted on OpenFeature client timeout, transport error, or `ErrorResolutionDetails` response; no experiment assignment is made and the session runs the base runtime. Payload fields: `tenant_id`, `user_id`, `provider`, `error`.
- `experiment.multi_eligible_skipped` (info) — emitted under the first-match multi-experiment rule when one or more active experiments are skipped because an earlier-created experiment already assigned a non-control variant. Payload fields: `tenant_id`, `user_id`, `enrolled_experiment_id`, `skipped_experiment_ids[]`.
- `experiment.isolation_mismatch` (warning) — emitted when the `ExperimentRouter` fails closed because a variant pool's `isolationProfile` is weaker than the session's `minIsolationProfile`; the session creation is rejected with `VARIANT_ISOLATION_UNAVAILABLE` ([§15.1](15_external-api-surface.md#151-rest-api) error catalog). Payload fields: `tenant_id`, `user_id`, `experiment_id`, `variant_id`, `sessionMinIsolation`, `variantPoolIsolation`.
- `experiment.variant_weaker_than_tenant_floor` (warning) — emitted at experiment creation or update time by the admission-time tenant-floor advisory check ([§10.7](10_gateway-internals.md#107-experiment-primitives)) when a variant pool's resolved `isolationProfile` is weaker than the tenant-level `minIsolationProfile` floor. Distinct from `experiment.isolation_mismatch`, which is emitted at runtime when the `ExperimentRouter` fails closed on an individual session. This event is advisory — the admission request is still 2xx and the experiment is creatable — but it warns the operator that any session whose resolved `minIsolationProfile` is at the tenant floor will be rejected with `VARIANT_ISOLATION_UNAVAILABLE` at routing time. Emitted in both real and `?dryRun=true` admission paths. One event per offending variant per admission call. Payload fields: `tenant_id`, `experiment_id`, `variant_id`, `variant_pool_isolation`, `tenant_floor`, `actor_sub`, `emitted_at`.

**`lenny-ops`-emitted:** `ops_health_status_changed`, `escalation_created`, `remediation_lock_acquired`, `remediation_lock_released`, `remediation_lock_expired`, `remediation_lock_stolen`, `remediation_lock_split_brain_detected`, `drift_detected`, `platform_upgrade_completed`, `platform_upgrade_verification_failed`, `platform_upgrade_image_pull_failed`, `restore_started`, `restore_shard_completed`, `restore_completed`, `restore_failed`, `event_delivery_failed`, `prometheus_query_timeout`, `lock_split_brain_detected`, `operation_progressed` (emitted per operation state transition or percent threshold; CloudEvents `data` carries: `operationId`, `kind`, `prevStatus`, `newStatus`, `progress`).

### 16.9 Prometheus Scrape Targets and CRDs

The Helm chart renders a NetworkPolicy allowing Prometheus (in `monitoring.namespace`) to scrape the `lenny-system` metrics ports ([§25.4](25_agent-operability.md#254-the-lenny-ops-service)). The canonical scrape target set is: gateway (port 9090), controller, token service, and — mandatory in every tier — `lenny-ops` (port 9090).

**Prometheus Operator CRDs (`ServiceMonitor`, `PodMonitor`, `PrometheusRule`).** Deployers who run the Prometheus Operator (kube-prometheus-stack, Red Hat OpenShift Monitoring, etc.) should set `monitoring.format: prometheusrule` (the default when the operator's CRDs are detected). With this setting:

- The chart renders one `ServiceMonitor` (for the gateway, controller, and token service) and one `PodMonitor` (for `lenny-ops`, which requires Pod-level label selection) into `monitoring.namespace`. Both are rendered with the operator-discoverable labels configured via `monitoring.prometheusRule.additionalLabels`.
- The chart renders one `PrometheusRule` containing the full alert catalog (§16.5 + [§25.13](25_agent-operability.md#2513-bundled-alerting-rules)). The rule manifest is generated from the shared `pkg/alerting/rules` source so that the rules are identical to the in-process evaluator.
- The preflight Job checks that the operator's CRDs are installed before rendering these manifests (`monitoringnamespace-monitors.coreos.com/v1` and peers). If CRDs are absent, the chart falls back to `monitoring.format: configmap` and prints a warning.

Deployers without the Prometheus Operator may either (a) set `monitoring.format: configmap` to render a ConfigMap that their Prometheus scrapes via static file provider, or (b) point a pre-existing Prometheus configuration at the scrape targets above. Both modes consume the same shared alert source.

### 16.10 OpenSLO Export

For deployers who manage SLOs in an [OpenSLO](https://openslo.com)-compatible tool (Sloth, Nobl9, OpenSLO Reliably), the Helm chart renders the latency and availability SLOs from §16.5 as OpenSLO v1 YAML documents under `monitoring.openslo.enabled: true` (default: `false`). The rendered documents reference the canonical metric names (`lenny_session_creation_duration_seconds`, `lenny_session_availability_ratio`, etc.) and the Lenny tier labels (`deployment_tier="tier1" | "tier2" | "tier3"`) so that external SLO tooling produces burn-rate alerts consistent with the bundled multi-window alerts above.

OpenSLO is not a runtime dependency: when enabled, the chart emits a `ConfigMap` containing the OpenSLO `SLO`, `SLI`, and `AlertPolicy` objects for the deployer's tooling to consume. Deployers who use Sloth specifically can generate a `PrometheusRule` from the OpenSLO documents via `sloth generate`. The authoritative source remains the bundled alert catalog in §16.5; the OpenSLO export is a view of the same source.
