## 11. Policy and Controls

### 11.1 Admission and Fairness

| Control                              | Granularity                             |
| ------------------------------------ | --------------------------------------- |
| Rate limits (requests/min)           | Global, per-user, per-runtime, per-pool |
| Concurrency limits (active sessions) | Global, per-user, per-team, per-runtime |
| Active delegated children            | Per-session, per-user                   |
| Concurrent uploads                   | Per-session, global                     |
| Upload size                          | Per-file, per-session                   |

**Access-path default policy (`noEnvironmentPolicy`).** Admission of a session that does not name an environment is governed by the tenant's `noEnvironmentPolicy` setting (platform default `deny-all`; see [Section 10.6](10_gateway-internals.md#106-environment-resource-and-rbac-model) for semantics). The platform-level value is a required configuration key: the gateway validates that `noEnvironmentPolicy` is set to either `deny-all` or `allow-all` during startup and refuses to become ready if the key is missing (see the configuration validation table in [Section 10.3](10_gateway-internals.md#103-mtls-pki)). This closes the misconfigured-Helm gap where an omitted value could otherwise leave admission under undefined behavior.

### 11.2 Budgets and Quotas

| Budget                      | Scope                                                                                      |
| --------------------------- | ------------------------------------------------------------------------------------------ |
| Token limits (LLM tokens)   | Per-request, per-session, per-user/window, per-tenant/window, global/window, per-task-tree |
| Runtime limits (wall clock) | Per-session, per-child                                                                     |
| Retry budget                | Per-session (client-set, deployer-capped)                                                  |
| Delegation budget           | Per-session (depth, fan-out, total children)                                               |
| Concurrent sessions         | Per-tenant, per-user                                                                       |
| Storage quota               | Per-tenant (total artifact storage)                                                        |
| Eval submission rate limits  | Per-session, per-tenant (configurable via `evalRateLimit.perSessionPerMinute`, `evalRateLimit.perTenantPerMinute`) |

**Budget inheritance:** Children inherit strictly narrower budgets. A parent cannot bypass top-level limits by spawning many children.

Delegation budget enforcement uses atomic reservation (see [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease)). Budget slices are reserved at delegation time via Redis atomic counters, preventing concurrent delegations from over-committing the parent's budget. Unused budget is returned to the parent when children complete. The Redis counters for delegation budgets follow the same durability model as token usage counters: Postgres checkpoint every `quotaSyncIntervalSeconds` (default: 30s, minimum: 10s), final reconciliation on session completion. **Delegation budget counters included in Postgres checkpoint:** The periodic Postgres checkpoint explicitly persists all delegation budget counters for each active delegation tree: `maxTreeSize` current count (active pods in the tree), `maxTokenBudget` consumed (tokens reserved and spent across the tree), `maxTreeMemoryBytes` current accumulated value (sum of in-memory footprints for all active, non-archived nodes in the tree), and per-node budget allocations. These are written to the `delegation_tree_budget` table (keyed by `root_session_id`) alongside token usage counters in the same checkpoint transaction, ensuring atomicity between token and delegation budget state. The `maxTreeMemoryBytes` counter must be checkpointed separately from `maxTreeSize` because completed subtrees are offloaded to Postgres and their memory reclaimed (see [Section 8.2](08_recursive-delegation.md#82-delegation-mechanism)); the two counters can therefore diverge, and the memory counter is the binding constraint for gateway memory protection.

**Hierarchical Quota Model:** Quotas are enforced hierarchically: global → tenant → user. A user quota cannot exceed its tenant's quota. Tenant quotas are configured via the admin API or Helm values. Soft warnings are emitted at 80% of quota utilization, surfaced as billing events per [Section 11.2.1](#1121-billing-event-stream). Hard limits are enforced at 100% — new sessions or requests are rejected with a `QUOTA_EXCEEDED` error. Quota reset periods are configurable per quota type: hourly, daily, monthly, or rolling window.

**Storage quota enforcement mechanism.** MinIO OSS has no built-in per-prefix quota. The gateway enforces the per-tenant storage quota using Redis atomic counters and a three-step lifecycle:

1. **Pre-upload size check (atomic reservation).** Before accepting any workspace upload or checkpoint write, the gateway executes an atomic Redis Lua script (`storage_quota_reserve.lua`) that reads the tenant's current `storage_bytes_used` counter, computes `projected = current + incoming_bytes`, and — only if `projected ≤ storageQuotaBytes` — atomically increments the counter by `incoming_bytes` as a reservation, all in a single `EVAL` call. If `projected > storageQuotaBytes`, the script returns a rejection code without modifying the counter and the gateway rejects the request immediately with `STORAGE_QUOTA_EXCEEDED` — no bytes are written to MinIO. This atomic read-check-reserve pattern (matching the delegation budget `budget_reserve.lua` script in [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease)) eliminates the race window where two concurrent uploads could both pass a non-atomic pre-check and collectively exceed the quota. If the upload subsequently fails, the gateway decrements the reservation (releasing the reserved bytes). The incoming size is known at the upload boundary: for workspace uploads the client supplies a `Content-Length` header (enforced by the gateway); for checkpoint writes the adapter supplies the tar size from the workspace-size probe before upload begins. **Hard stream cap:** Regardless of the declared `Content-Length`, the gateway immediately wraps the inbound request body in an `io.LimitedReader` set to `remaining_quota_bytes = storageQuotaBytes − storage_bytes_used`. Any attempt to stream more bytes than this limit causes the read to return `io.EOF` and the upload handler to abort with `STORAGE_QUOTA_EXCEEDED`. This ensures that a client cannot bypass the pre-check by declaring a small `Content-Length` and then streaming a larger body.
2. **Post-upload reconciliation.** After a successful MinIO `PutObject` or multipart complete, the gateway reconciles the reservation: it atomically adjusts the Redis counter by the delta between the confirmed object size (bytes actually written) and the pre-reserved `incoming_bytes`. If the confirmed size is smaller than reserved (e.g., compression), the counter is decremented by the difference; if larger, incremented. Using the confirmed size prevents quota drift from compression or partial upload failures.
3. **GC-triggered decrement.** When the GC job ([Section 12.5](12_storage-architecture.md#125-artifact-store)) deletes an artifact from MinIO, it atomically decrements the Redis counter by the deleted object's size, as recorded in the `artifact_store` Postgres table. The Postgres record is the source of truth for artifact sizes; Redis is a fast-path enforcement cache. **Ordering requirement:** The Redis decrement MUST be issued only after the Postgres `artifact_store` row has been durably committed with `deleted_at` set (i.e., the `UPDATE artifact_store SET deleted_at = now() WHERE id = $1` transaction has committed). Issuing the decrement before the Postgres commit creates a double-decrement hazard: if the gateway crashes after decrementing Redis but before committing the Postgres `deleted_at`, crash recovery rehydrates the counter from Postgres (the row is still `active`), restoring the decremented bytes — and the GC re-run on the next cycle decrements again for the same artifact. By committing Postgres first, any crash after that point leaves the artifact marked `deleted` in Postgres; recovery correctly excludes it from the rehydration sum, and the already-issued Redis decrement is not re-applied. On Redis restart, the gateway rehydrates per-tenant storage counters from the sum of `artifact_size_bytes` across active (non-deleted) artifacts in Postgres (same recovery path as token quota counters). The `lenny_storage_quota_bytes_used` gauge (per tenant) is exported from Redis and alerts when a tenant exceeds 80% of their quota (`StorageQuotaHigh`, [Section 16.5](16_observability.md#165-alerting-rules-and-slos)).

**Quota Update Timing:** Token usage quotas are updated in real-time via Redis counters. The source of token counts depends on delivery mode:

- **Proxy mode (`deliveryMode: proxy`):** The gateway's LLM Proxy ([Section 4.9](04_system-components.md#49-credential-leasing-service)) extracts `input_tokens` and `output_tokens` directly from upstream LLM provider responses. These gateway-observed counts are the **authoritative** usage record — they are written to Redis immediately when the response is received. `ReportUsage` calls from the pod adapter are **ignored** for proxy-mode sessions; a malicious or misconfigured runtime cannot influence quota accounting by underreporting.
- **Direct mode (`deliveryMode: direct`):** The runtime adapter extracts token counts from LLM provider responses and reports them to the gateway via the `ReportUsage` RPC ([Section 4.7](04_system-components.md#47-runtime-adapter)). The gateway increments Redis counters on each usage report. Because the gateway has no independent view of LLM responses in this mode, a malicious runtime could underreport token consumption. This is an accepted residual risk for direct mode, which is already restricted to single-tenant or development deployments (see [Section 4.9](04_system-components.md#49-credential-leasing-service)). Deployers should monitor `lenny_gateway_token_usage_anomaly_total` (counter, labeled by `session_id` and `tenant_id`) — the gateway emits this metric when a session's `ReportUsage` delta is zero or implausibly small relative to LLM call frequency (more than 3 `ReportUsage` calls within a session with zero tokens reported while LLM proxy activity is absent). Persistent anomalies should trigger operator review of the affected runtime image.

Postgres is updated periodically at a configurable sync interval (`quotaSyncIntervalSeconds`, default: 30s, minimum: 10s) as a durable checkpoint, and on session completion as final reconciliation. The gateway enforces budget limits against Redis counters (fast path); if a session exceeds its token budget, the gateway terminates it immediately rather than waiting for session completion. During Redis unavailability (fail-open window), the gateway tracks usage in-memory per session and enforces the per-session budget locally using the per-replica ceiling formula from [Section 12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes) (`tenant_limit / max(cached_replica_count, 1)`). Only cross-session and per-tenant quotas may drift during this window (bounded by the per-replica ceiling formula). **On Redis recovery**, the gateway applies the MAX rule when restoring counters: `restored_counter = MAX(in_memory_counter, postgres_checkpoint)` for each active session and tenant scope. This prevents a stale Postgres checkpoint (up to `quotaSyncIntervalSeconds` old) from resetting a counter to a value lower than actual accumulated usage, which would otherwise silently un-enforce a budget violation that occurred during the fail-open window. On session completion, the final cumulative token usage is always written to Postgres as an authoritative value, ensuring subsequent recovery events have an accurate baseline.

**Crash Recovery for Quota Counters:** If a gateway replica crashes between Postgres sync intervals, in-flight usage data held only in Redis (or in-memory during fail-open) may be lost. On recovery, the gateway reconstructs quota counters from two sources: (1) the last Postgres checkpoint for each active session, and (2) pod-side `ReportUsage` records — each pod's runtime adapter retains a cumulative usage total that is re-reported on reconnection to a new gateway replica ([Section 7.3](07_session-lifecycle.md#73-retry-and-resume)). The gateway takes the **maximum** of the Postgres checkpoint and the pod-reported cumulative total for each session to avoid under-counting. This bounds the maximum unrecovered usage to at most one sync interval's worth of tokens for sessions whose pods were also lost simultaneously.

**Crash Recovery for Delegation Budget Counters:** Delegation tree budget counters (`maxTreeSize` active count, `maxTokenBudget` consumed, `maxTreeMemoryBytes` accumulated) are reconstructed using the same two-source approach: (1) the last Postgres checkpoint from `delegation_tree_budget`, and (2) the gateway's authoritative knowledge of which pods are currently alive (from the SessionStore's `session → pod` mapping). On Redis recovery (restart or reconnection after outage), the gateway **must** re-validate every active delegation tree's budget state before accepting new `delegate_task` requests for that tree. The reconstruction procedure is: (a) load the last Postgres checkpoint for each active tree's `root_session_id`, (b) count currently-alive descendant pods from the SessionStore to derive the actual `treeSize`, (c) sum allocated token budgets across all active descendants for `tokenBudgetConsumed`, (d) compute `liveMemoryBytes` as the number of currently-alive (non-archived) descendant nodes multiplied by the per-node footprint estimate (`nodeMemoryFootprintBytes`, default: 12288 / 12 KB; configurable via gateway Helm value `delegationNodeMemoryFootprintBytes`), (e) set Redis counters to `max(postgres_checkpoint, live_count)` for tree size, `max(postgres_checkpoint, sum_of_allocations)` for token budget, and `max(postgres_checkpoint_memory, liveMemoryBytes)` for tree memory. Using the MAX rule for the memory counter ensures that a stale checkpoint (up to `quotaSyncIntervalSeconds` old) never resets the counter below the reconstructed live estimate, preventing the tree from accepting delegations it would have been blocked on before the outage. **Irrecoverable trees:** If a tree's budget state cannot be reconstructed with confidence — specifically, if the Postgres checkpoint is older than `2 × quotaSyncIntervalSeconds` AND the coordinating gateway replica for the tree's root session was also lost (preventing live pod enumeration) — the tree's root session is moved to `awaiting_client_action` with reason `BUDGET_STATE_UNRECOVERABLE`, and no new delegations are accepted until the client resumes. This prevents potentially over-budget delegations. The gateway emits `lenny_delegation_budget_reconstruction_total` (counter, labeled by `outcome`: `success`, `irrecoverable`) on every reconstruction event.

**Maximum Overshoot Formula:** The worst-case quota overshoot during normal operation is: `max_overshoot = quotaSyncIntervalSeconds × max_tokens_per_second × active_sessions_per_tenant`. During a Redis fail-open window, additional drift is bounded by the per-replica ceiling ([Section 12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes)): `effective_ceiling = min(tenant_limit / max(cached_replica_count, 1), per_replica_hard_cap)`. In the normal case (Endpoints available), `fail_open_overshoot = effective_ceiling × replica_count ≤ tenant_limit`. In the dual-outage case (Redis + Endpoints simultaneously unavailable), each replica uses its `cached_replica_count` (last-known good value) rather than falling back to `1`, so total overshoot is bounded by `min(N × (tenant_limit / max(cached_replica_count, 1)), N × per_replica_hard_cap)` — always less than `N × tenant_limit`. The cumulative fail-open timer ([Section 12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes), `quotaFailOpenCumulativeMaxSeconds`) further limits repeated drift accumulation. Deployers should set `quotaSyncIntervalSeconds` lower (minimum 10s) for tenants with high token throughput to reduce crash-recovery exposure.

**Null/absent field contract:** Fields annotated "(for X events only)" MUST be omitted from the JSON payload for all other event types — they are neither present nor null. Consumers MUST treat absent fields as "not applicable" for the event type (not as zero). `corrects_sequence` uses type `uint64 | null`: `null` (or absent) means the event is not a correction; `0` is never a valid sequence number (sequences start at 1). Analytics consumers MUST handle absent conditional fields as equivalent to `null` in their schema, e.g., Parquet `optional` columns. The `event_type` field is the discriminant for all conditional fields.

**Delivery sinks:**

Events are published to a deployer-configurable sink. Supported sink types:

- **Webhook URL** — Events are POSTed as JSON with HMAC-SHA256 signature headers. Failed deliveries are retried with exponential backoff and dead-lettered after exhaustion.
- **Message queue** — SQS, Google Pub/Sub, or Kafka topic. The gateway publishes asynchronously but only after the synchronous Postgres write confirms.
- **Both** — Webhook and message queue simultaneously for redundancy.

**Immutability guarantees:**

- Billing events are **always written to Postgres synchronously** via the `EventStore`, regardless of Redis availability. The Redis fail-open behavior ([Section 12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes)) applies only to rate-limit counters — billing events are never deferred, batched, or dropped. **During Postgres failover:** If the synchronous write fails due to Postgres unavailability, the billing event is routed through a **Redis stream intermediate buffer** (`t:{tenant_id}:billing:stream`, an XADD-based Redis stream with `MAXLEN ~billingRedisStreamMaxLen` per tenant; per-tier defaults: 50,000 entries at Tier 1/2, 72,000 entries at Tier 3 -- see [Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) operational defaults) before falling back to the in-memory write-ahead buffer. The two-tier failover path is:

  1. **Tier 1 (Redis stream):** On Postgres write failure, the gateway publishes the billing event to the Redis stream via `XADD` using the tenant-scoped key. Redis stream entries carry the full billing event payload plus a `stream_seq` (the event's `sequence_number`). Multiple gateway replicas write to the same tenant stream concurrently; the stream provides durable ordering across replicas. A background flusher goroutine per tenant consumes the Redis stream via a **consumer group** (`billing-flusher`, created with `XGROUP CREATE ... MKSTREAM`). Each gateway replica joins the group as a distinct consumer (consumer name = replica pod ID). The consumer group ensures each stream entry is delivered to exactly one replica — no duplicate billing inserts across replicas. The flusher calls `XREADGROUP` and re-attempts Postgres INSERTs in `stream_seq` order using the existing `billingFlushIntervalMs` schedule. On successful INSERT, the flusher `XACK`s and `XDEL`s the entry from the stream. A periodic pending-entry reclaim (`XAUTOCLAIM`, interval: `billingReclaimIntervalSeconds` (default: 15s), min-idle: `billingReclaimMinIdleSeconds` (default: 30s)) recovers entries assigned to crashed replicas. The worst-case reclaim delay after a replica crash is `billingReclaimMinIdleSeconds + billingReclaimIntervalSeconds` (default: 45s). Additionally, on startup each gateway replica immediately performs an `XAUTOCLAIM` with `min-idle: 0` scoped to its own consumer name (pod ID) to fast-recover any entries assigned to a predecessor replica that held the same pod name. **Deduplication on reclaimed entries:** Because `XAUTOCLAIM` may redeliver an entry that the original consumer already flushed to Postgres but failed to `XACK` before crashing, the flusher uses `INSERT ... ON CONFLICT (tenant_id, stream_entry_id) DO NOTHING` on the billing events table. The `stream_entry_id` column stores the Redis stream entry ID (e.g., `1680000000000-0`) and carries a unique constraint per tenant. This prevents duplicate billing records from reclaimed entries — the second INSERT is a no-op, and the reclaiming consumer proceeds to `XACK` and `XDEL`. The stream TTL is set to `billingStreamTTLSeconds` (default: 3600s) — events not flushed to Postgres within this window are permanently lost (this scenario requires Redis uptime > 1 hour while Postgres is simultaneously down, which is treated as a catastrophic dual-outage; the `BillingStreamBackpressure` alert fires at 80% of `billingRedisStreamMaxLen`; the `BillingStreamEntryAgeHigh` alert fires when the oldest unacknowledged entry age exceeds 80% of `billingStreamTTLSeconds`). **Flusher TTL-proximity logging:** On each `XREADGROUP` cycle, the billing flusher inspects the timestamp embedded in each pending entry's Redis stream ID. When any entry's age exceeds 80% of `billingStreamTTLSeconds` (default: 2880s), the flusher emits a **CRITICAL**-level structured log (`billing_stream_entry_near_ttl_expiry`) containing the tenant ID, stream entry ID, entry age in seconds, and the TTL threshold. This provides per-entry operator visibility into imminent data loss before the TTL expires, complementing the metric-based `BillingStreamEntryAgeHigh` alert.

  2. **Tier 2 (in-memory write-ahead buffer):** If the Redis stream write also fails (Redis unavailable), the billing event falls back to the **bounded in-memory write-ahead buffer** on the gateway replica (max `billingWriteAheadBufferSize`, default: 10,000 events per replica). Queued events are flushed to Postgres in sequence-number order once connectivity is restored, preserving the monotonic ordering guarantee. If the buffer fills before either store recovers, the gateway rejects new session-progressing requests (returning `503`) to maintain the invariant that no billable work occurs without a corresponding billing record. The in-memory buffer is not persisted to disk — if a gateway replica crashes with buffered events and both Redis and Postgres remain unavailable, those events are reconstructed from pod-reported token usage during session recovery ([Section 7.3](07_session-lifecycle.md#73-retry-and-resume)).

  This two-tier design eliminates the billing consistency gap that existed when a gateway replica crashed during the Postgres failover window with only an in-memory buffer: the Redis stream provides a durable intermediate store that survives individual gateway crashes. The common case (Redis available, Postgres transiently unavailable) now routes through the durable Redis stream instead of in-memory state.
- The `EventStore` billing table uses append-only semantics (INSERT only, no UPDATE or DELETE grants), matching the audit log integrity model.
- Each event carries a monotonic `sequence_number` scoped to the tenant, enabling consumers to detect gaps and request replays via the metering API. **Sequencing authority:** The `sequence_number` is assigned by a **Postgres sequence** (`billing_seq_{tenant_id}`) — a dedicated per-tenant sequence object. Gateway replicas call `nextval('billing_seq_{tenant_id}')` inside the `EventStore` INSERT transaction; this guarantees monotonicity across replicas (Postgres sequences may produce gaps on transaction rollback; gaps are handled by the replay mechanism below). Events buffered via the Redis stream or in-memory write-ahead buffer use a provisional `stream_seq` during the outage window and are renumbered by the flusher on the actual Postgres INSERT (the flusher acquires the Postgres sequence value at flush time, not at buffer time). **Gap detection and replay:** Consumers detecting a gap in the `sequence_number` stream may request a replay via `GET /v1/metering/events?since_sequence={N}&limit={L}` — the gateway returns all events with `sequence_number > N` in ascending order, up to `limit`. This endpoint is authenticated (requires `billing-viewer`, `tenant-admin`, or `platform-admin` role) and paginated using the standard cursor envelope ([Section 15.1](15_external-api-surface.md#151-rest-api)).
- Events are retained for a deployer-configurable retention period (`billing.retentionDays`, default: 395 days / ~13 months) to support annual billing cycles and dispute resolution. **Compliance-profile-aware minimum floors:** When a regulated `complianceProfile` is active on any tenant, the platform enforces a minimum `billing.retentionDays` value: `hipaa` — 2190 days (6 years, per 45 C.F.R. §164.530(j)), `soc2` — 365 days, `fedramp` — 365 days. The gateway rejects configurations below the applicable floor at startup with `CONFIG_INVALID: billing.retentionDays below compliance floor for complianceProfile '<value>'`. This is consistent with the `audit.gdprRetentionDays` floor enforcement pattern ([Section 16.4](16_observability.md#164-logging)).

**Correction semantics:**

A `billing_correction` event adjusts a previously emitted event without violating append-only immutability. The correction carries its own `sequence_number` and references the original event via `corrects_sequence`. The `tokens_input`, `tokens_output`, and `pod_minutes` fields on a correction represent the **replacement values** for the original event's corresponding fields. Consumers reconstruct the accurate billing ledger by processing events in `sequence_number` order and applying each correction to the referenced original: when a `billing_correction` is encountered, its values supersede the original event's values for that field set. Multiple corrections to the same original are applied in sequence-number order, with the latest correction taking precedence. The original event remains in the stream unchanged, preserving the full audit trail.

**Billing correction authorization:**

Billing corrections alter the authoritative revenue record. Corrections fall into two categories with different authorization models:

**Category 1 — Gateway-emitted automated reconciliation.** The gateway emits `billing_correction` events through the normal `EventStore` write path during routine operational reconciliation. These corrections are not operator-initiated — they are an automated consequence of crash recovery, retry deduplication, and self-healing metering. The following reason codes belong to this category:

- `RETRY_OVERCOUNTING` — the gateway detects duplicate metering from retried requests and emits a correction to remove the double-count.
- `GATEWAY_CRASH_RECONSTRUCTION` — after crash recovery ([Section 7.3](07_session-lifecycle.md#73-retry-and-resume)), the gateway reconciles billing from pod-reported token totals and emits corrections for any discrepancies.
- `METERING_BUG` — the gateway detects and self-corrects a metering inconsistency (e.g., a token count that disagrees with the pod's reported usage).

These corrections are written to the `EventStore` by the gateway like any other billing event. Each carries a `correction_reason_code` and emits a `billing.correction_issued` audit event recording the gateway replica identity, reason code, original `sequence_number`, and replacement values.

**Category 2 — Operator-initiated manual corrections.** These are issued by human operators through a dedicated admin API endpoint (`POST /v1/admin/billing-corrections`) and require the following controls:

1. **RBAC gating:** The endpoint requires the `platform-admin` role. `tenant-admin` and `user` roles receive `403 FORBIDDEN`.
2. **Structured reason code:** Each correction must include a `correction_reason_code` from this category's codes: `TEST_SESSION_CLEANUP`, `OPERATOR_MANUAL_ADJUSTMENT`, or any deployer-added custom reason code. Free-text `correction_detail` remains available for additional context but is not a substitute for the structured code.
3. **Dual-control approval (four-eyes principle):** Corrections whose absolute adjustment value exceeds the deployer-configurable threshold `billing.dualControlThreshold` (default: `0`, meaning **all** operator-initiated corrections require dual-control) must be approved by a second `platform-admin` before they are committed. The workflow:
   - The initiating admin submits the correction request; the `EventStore` records it in a `billing_correction_pending` state with a unique `approval_request_id`.
   - The gateway notifies eligible approvers via the configured notification channel (`billing.approverNotificationWebhook`, if set).
   - A second `platform-admin` (different identity from the submitter — the gateway rejects self-approval) calls `POST /v1/admin/billing-corrections/{id}/approve` or `.../reject`.
   - Only after approval is the `billing_correction` event written to the immutable billing stream with the full monotonic `sequence_number`. Rejected requests remain in `billing_correction_pending` state with `rejected` outcome for audit purposes and are never promoted to the billing stream.
   - Pending approvals expire after `billing.approvalTimeoutSeconds` (default: 86400 = 24 h). Expired requests are moved to `billing_correction_pending` state `expired` and emitted as a `billing.correction_approval_expired` audit event.
   - When `billing.dualControlThreshold` is set to a positive value, corrections at or below the threshold may be committed by the submitting admin without a second approval (single-control path). Corrections above the threshold always require the dual-control path.
4. **Audit trail:** Every correction — and every approval, rejection, or expiry of a pending correction — emits a `billing.correction_issued` (or `billing.correction_approval_*`) audit event recording the issuing admin's identity, the approver's identity (if applicable), the correction reason code, the original event's `sequence_number`, and the replacement values. These audit events are distinct from the billing events themselves and cannot be suppressed.

**Metrics:** `lenny_billing_correction_pending_total` (gauge, labeled by `state`: `pending`, `approved`, `rejected`, `expired`) tracks the correction approval queue. Alert: `BillingCorrectionApprovalBacklog` fires when `pending` count exceeds a deployer-configurable threshold (default: 10) for more than `billing.approvalBacklogAlertMinutes` (default: 60) minutes.

**Controls that apply to both categories:**

- **Structured reason code (all corrections):** Every `billing_correction` event — whether gateway-emitted or operator-initiated — must carry a `correction_reason_code` from the appropriate category. The reason code unambiguously identifies the correction's origin.
- **Anomaly alerting:** A `BillingCorrectionRateHigh` warning alert fires when correction events (both categories combined) exceed a deployer-configurable percentage (default: 5%) of total billing events in a rolling 24h window (see [Section 16.5](16_observability.md#165-alerting-rules-and-slos)).

The `correction_reason_code` enum is extensible via the admin API (`POST /v1/admin/billing-correction-reasons`) to allow deployers to add domain-specific reason codes without modifying the platform binary. Deployer-added reason codes are classified as operator-initiated and require the admin endpoint. New reason codes require `platform-admin` and are themselves audit-logged.

**`maxRequestInputWait` — inter-agent blocking timeout.** `lenny/request_input` is an inter-agent blocking call (distinct from `lenny/request_elicitation`, which is the human-facing tool governed by `maxElicitationWait`). When a child agent calls `lenny/request_input`, it blocks in `input_required` sub-state until a peer responds via `lenny/send_message` with a matching `inReplyTo`. `maxRequestInputWaitSeconds` (default: 600s, configurable per pool in the `limits:` block of the RuntimeDefinition — same placement as `maxElicitationWait`) caps this wait. When the timeout fires, the gateway resolves the blocked tool call with a `REQUEST_INPUT_TIMEOUT` error delivered as a normal tool-call error to the runtime; the child can then handle it (retry, fall back, or fail). The `maxSessionAge` timer continues running during `input_required` — `maxRequestInputWaitSeconds` provides an earlier, dedicated bound. This timeout is distinct from `maxElicitationWait` and both may be configured independently. When a `request_input` times out, the gateway emits a `request_input_expired` event on the parent's `lenny/await_children` stream: `{ "type": "request_input_expired", "childId": "...", "requestId": "...", "expiredAt": "<ISO8601>" }`, allowing the parent to distinguish "child's input request timed out" from "child expired for other reasons."

**Session expiry warning:** The gateway sends a `session_expiring_soon` event to the client and pod 5 minutes before `maxSessionAge` expires. This gives the agent time to checkpoint and the client time to extend or wrap up. The agent receives a `DEADLINE_APPROACHING` signal via the adapter.

> **Note:** The 2h default is a conservative starting point. Deployers should tune `maxSessionAge` per Runtime based on expected workload duration. There is no platform-imposed upper bound — certificate renewal and SA token refresh are handled transparently at any session duration (see [§10.3](10_gateway-internals.md#103-mtls-pki)).

### 11.4 User Invalidation

Three levels:

| Level        | Effect                                                                           |
| ------------ | -------------------------------------------------------------------------------- |
| Soft disable | Deny new sessions                                                                |
| Hard disable | Also block new delegated tasks; reject pending delegation approvals for the user |
| Full revoke  | Terminate active sessions, invalidate cached auth, deny reconnects               |

**Full revoke propagation mechanism:**

1. Gateway looks up all active sessions for the user (via SessionStore).
2. For each session in the user's task tree: gateway sends a `Terminate` RPC to the pod with reason `USER_REVOKED`.
3. The pod's runtime adapter initiates graceful shutdown (SIGTERM to agent, wait up to 10s, then SIGKILL).
4. Gateway marks all sessions as terminated in SessionStore.
5. Cached auth tokens for the user are invalidated in Redis.
6. Credential leases held by the user's sessions are revoked (returned to pool).
7. Pending elicitations for the user are dismissed.

> **Note:** Invalidation is asynchronous — the API call returns immediately and propagation completes within seconds. The `GET /v1/sessions` endpoint reflects the updated state once propagation completes.

### 11.5 Idempotency

Critical operations support idempotency keys:

- CreateSession
- FinalizeWorkspace
- StartSession
- SpawnChild
- Approve/DenyDelegation
- Resume

**Mechanism.** Clients provide an idempotency key via the `Idempotency-Key` HTTP header (for REST) or the `idempotencyKey` field in the MCP tool input (for MCP). The key is an opaque string of at most 128 characters; UUIDs are recommended. The gateway stores the key, request hash, and response in Postgres (table `idempotency_keys`, keyed by `(tenant_id, idempotency_key)`) with a TTL of 24 hours. Within this window, a duplicate request with the same key returns the cached response (same HTTP status and body) without re-executing the operation. If the same key is reused with a different request body (detected via SHA-256 hash comparison), the gateway returns `422 IDEMPOTENCY_KEY_REUSED` to prevent accidental misuse. After the 24-hour window, the key record is garbage-collected and the key may be reused. Idempotency keys are scoped per tenant — the same key string used by different tenants is treated independently.

Prevents duplicate sessions or children during gateway failover or client retries.

### 11.6 Circuit Breakers

**Two types of circuit breaker exist in the platform.** (1) **Automatic, per-subsystem circuit breakers** — in-memory, managed by each gateway replica, triggered by consecutive failure thresholds (e.g., the gateway subsystem breakers in [Section 4.1](04_system-components.md#41-edge-gateway-replicas), the Token Service breaker in [Section 4.3](04_system-components.md#43-token-service), and the KMS signing breaker in [Section 10.2](10_gateway-internals.md#102-authentication)). These are not operator-controlled and reset automatically via half-open probes. (2) **Operator-managed circuit breakers** — Redis-backed, declared and toggled via the admin API, described in the remainder of this section. The two types are independent: an operator-managed breaker does not affect automatic subsystem breakers, and vice versa.

Operators can declare degraded states:

- Runtime X degraded / offline
- Pool Y full
- External connector Z down
- Uploads temporarily disabled
- Delegation depth > N disabled during incident

**Storage and propagation.** Circuit breaker state is stored in Redis (the Quota/Rate Limiting instance, [§12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes)) as a key per circuit name: `cb:{name}` → `{state: "open"|"closed", reason: string, opened_at: ISO8601, opened_by: user_id}`. State changes are published on the `circuit_breaker_events` pub/sub channel so all gateway replicas pick up the change within one round-trip (~1ms). Gateway replicas also re-read all circuit states from Redis at startup and cache in-process. The cache TTL is 5 seconds; replicas poll Redis every 5 seconds to detect out-of-band changes (e.g., a Redis write that missed the pub/sub channel due to a reconnect).

**In-process fallback for the health service.** The same in-process cache is read by the gateway health service (§25.3) when Redis is unreachable — reporting stale circuit breaker state is strictly better than reporting nothing during a Redis outage. The cache is eventually consistent with Redis; entries up to 5 s stale may be served while Redis is unavailable. The health response marks the dependency as `degraded` via the canonical `degradation` envelope (§25.2) so callers see the staleness rather than silently trusting the cached value.

**Event emission.** On every state transition, the circuit breaker subsystem invokes `EventEmitter.Emit(ctx, event)` (from `pkg/gateway/events`) with `circuit_breaker_opened` or `circuit_breaker_closed` and a payload including `name`, `reason`, and `operator_user_id`. The emitter is injected via constructor (§25.3 Event Emission). Operators reading `/v1/admin/events/*` observe state changes in near real time.

**Admin API.** Circuit breakers are managed via the admin API (requires `platform-admin` role):

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/v1/admin/circuit-breakers` | List all circuit breakers and their current state |
| `GET` | `/v1/admin/circuit-breakers/{name}` | Get state for a single circuit breaker |
| `POST` | `/v1/admin/circuit-breakers/{name}/open` | Open (activate) a circuit breaker. Body: `{ "reason": string }` |
| `POST` | `/v1/admin/circuit-breakers/{name}/close` | Close (deactivate) a circuit breaker |

**AdmissionController evaluation.** The gateway evaluates all active (open) circuit breakers at the start of every session-creation and delegation admission check, before quota and policy evaluation. If any open circuit breaker applies to the requested runtime, pool, connector, or operation type, the request is rejected immediately with `CIRCUIT_BREAKER_OPEN` (HTTP 503, `retryable: false`). The error body includes `circuit_name`, `reason`, and `opened_at`.

> **Pre-chain gate (POL-014).** Circuit-breaker evaluation is a **pre-chain gate** and is NOT an interceptor. It runs **after `AuthEvaluator` completes at `PreAuth`** and **before the `PostAuth` and `PreDelegation` interceptor chains run**, and is not listed in the built-in interceptor priority table at [§4.8](04_system-components.md#48-gateway-policy-engine). The canonical admission chain ordering is therefore `AuthEvaluator` (`PreAuth`, priority 100) → `AdmissionController` (pre-chain gate) → `PostAuth` / `PreDelegation` interceptor chains (priority > 100). Running the gate after authentication is required so that the `admission.circuit_breaker_rejected` audit payload carries the authenticated caller identity (`caller_sub`, `caller_tenant_id`; see [§16.7](16_observability.md#167-section-25-audit-events)) and so tenant-scoped rejection accounting is available even when the breaker's match criteria (currently runtime/pool/connector/operation_type — not tenant) do not themselves reference the tenant. Circuit-breaker evaluation is not subject to interceptor short-circuit semantics; a REJECT produces the distinct audit event type `admission.circuit_breaker_rejected` (not `interceptor.rejected`). See [§4.8](04_system-components.md#48-gateway-policy-engine) "AdmissionController is a pre-chain gate, not an interceptor" for the chain-ordering rationale.

**Audit events.** Every state change emits a `circuit_breaker.state_changed` audit event containing: `circuit_name`, `old_state`, `new_state`, `reason`, `operator_user_id`, `timestamp`. Every request REJECTed by a tripped breaker emits an `admission.circuit_breaker_rejected` audit event containing: `circuit_name`, `reason`, `opened_at`, and the admission-time request snapshot (session_id or delegation parent, requested runtime/pool, caller_user_id). Both events are written to the append-only audit tables ([§11.7](#117-audit-logging)) and appear in the catalogued audit event list in [§16.7](16_observability.md#167-section-25-audit-events) / equivalent.

**Sampling under breaker storms.** When a breaker is open, rejection volume for the affected runtime/pool/connector can be arbitrarily large — misbehaving client retry loops against a tripped breaker produce exactly the saturation pattern that contends the per-tenant advisory-locked audit write path ([§11.7](#117-audit-logging) item 3) and can starve legitimate audit writes (`session.created`, `token.exchanged`, `interceptor.rejected`) for the same tenant, surfacing as `AUDIT_CONCURRENCY_TIMEOUT` on otherwise-healthy traffic. To protect that write path, `admission.circuit_breaker_rejected` is audit-sampled using the same discipline as `token.exchange_rate_limited` ([§13.3](13_security-model.md#133-credential-flow) Audit sampling for rate-limit rejections): the **first** rejection per `(tenant_id, circuit_name, caller_sub)` tuple within any rolling 10-second window is written as a full audit row (preserving per-caller audit trail and SIEM visibility); subsequent rejections in the same window increment `lenny_circuit_breaker_rejections_suppressed_total` (labels: `tenant_id`, `circuit_name`, `limit_tier` — where `limit_tier` carries the breaker scope dimension, one of `runtime` \| `pool` \| `connector` \| `operation_type`) but do **not** write individual rows. The rejection counter `lenny_circuit_breaker_rejections_total` uses the same `limit_tier` vocabulary so operators can correlate a metric spike to its sampled audit rows with an exact label equality. **Sampling window locality.** The sampling window is tracked in **each gateway replica's in-memory admission-controller state**, keyed by `(tenant_id, circuit_name, caller_sub)` — explicitly per-replica, no Redis or EventBus coordination — for the same reasons as §13.3 Sampling window locality: cross-replica coordination would add a synchronous dependency to every admission check and defeat the latency-preserving purpose of sampling, and slight audit amplification across replicas (at most N audit events per window for N replicas serving the same caller) is preferable to under-auditing. Security teams querying audit logs still see at least one authoritative event per `(caller, circuit)` per window per replica; the volumetric magnitude and cross-replica sum are observable via `lenny_circuit_breaker_rejections_total` and `lenny_circuit_breaker_rejections_suppressed_total` and are surfaced in the `CircuitBreakerActive` alert body alongside `lenny_circuit_breaker_open` so on-call can size the rejection storm.

**Observability.** `lenny_circuit_breaker_open` gauge (labeled by `circuit_name`) is 1 when the named breaker is open, 0 when closed. `lenny_circuit_breaker_rejections_total` (counter, labeled by `tenant_id`, `circuit_name`, `limit_tier`) counts every admission rejection caused by a tripped breaker, including those whose audit row was elided by sampling. `lenny_circuit_breaker_rejections_suppressed_total` (counter, same labels) counts only the sampled-away rejections (the delta between total rejections and audit rows written). Alert `CircuitBreakerActive` (severity: warning) fires when any breaker has been open for more than 5 minutes without a close action; the alert body includes both `lenny_circuit_breaker_open{circuit_name=...}` and `sum by (circuit_name) (rate(lenny_circuit_breaker_rejections_suppressed_total[5m]))` so on-call operators can size the rejection storm against the tripped breaker without paging through individual audit rows.

### 11.7 Audit Logging

Every session/task/delegation produces durable records:

- Who requested it (user_id, tenant_id)
- What runtime, pool, isolation profile
- Parent/child lineage
- Which policies were applied and their results
- Token usage
- Retries, cancellations, failures
- External tool access and auth prompts
- Denial reasons
- **(optional)** `operation_id` — from the `X-Lenny-Operation-ID` request header (§25.1). Correlates audit events with operational events and structured logs for post-incident analysis (§25.9).
- **(optional)** `caller_kind` — derived from the OIDC `caller_type` claim (§25.1). Values: `"human"`, `"service"`, `"agent"`. Enables agent-vs-human activity breakdowns in audit queries.

**Integrity:** Audit tables use **append-only semantics** — the gateway's database role (`lenny_app`) has INSERT-only grants on audit tables (no UPDATE, DELETE). These grants are defined in the schema migration files and enforced through the following layered integrity controls:

1. **Startup grant verification (hard-fail in production):** At startup, the gateway queries `information_schema.role_table_grants` and checks that no UPDATE or DELETE grants exist on audit tables for the `lenny_app` role. In production mode (`LENNY_ENV=production`), any unexpected grant causes the gateway to **refuse to start** (fatal error). In non-production environments, a warning is logged instead. This prevents a compromised or misconfigured database from silently undermining audit integrity. Note: startup verification alone is insufficient — a superuser can grant UPDATE/DELETE access after startup and revoke it before the next periodic check. Items 2 and 5 (below) close this window.

   **Trigger-enabled check:** The startup grant verification additionally checks that all billing and audit integrity triggers are enabled: the gateway queries `SELECT tgname, tgenabled FROM pg_trigger WHERE tgname IN ('lenny_tenant_guard', 'lenny_billing_immutability', 'lenny_audit_immutability')` and fails startup (fatal error in production) if any trigger has `tgenabled = 'D'` (disabled). A superuser can `ALTER TABLE ... DISABLE TRIGGER` to bypass these protections without revoking grants. This check is also performed by the `lenny-preflight` Job (see [Section 17.6](17_deployment-topology.md#176-packaging-and-installation)) and by the periodic background integrity check (item 2).

2. **Periodic background integrity check:** A background goroutine re-verifies audit table grants at a configurable interval. The default interval is profile-dependent:
   - **Regulated profiles** (`complianceProfile` in `{soc2, fedramp, hipaa}`): `audit.grantCheckInterval` defaults to **60 seconds** and may not be set higher than 120 seconds via config (the gateway clamps the value at startup with a fatal error if the configured value exceeds the maximum for the active profile).
   - **Unregulated profiles** (`complianceProfile: none`): `audit.grantCheckInterval` defaults to **5 minutes** (300 seconds), configurable up to 15 minutes.
   If drift is detected at runtime (e.g., a superuser added UPDATE grants after startup), the gateway emits a critical alert, increments the `lenny_audit_grant_drift_total` Prometheus counter, and — if `audit.hardFailOnDrift` is enabled — initiates a graceful shutdown. This reduces the detection window to ≤60 seconds for regulated deployments, narrowing the tamper-and-revert window described above.

3. **Hash chaining:** Each audit log entry includes a `prev_hash` column containing the SHA-256 hash of the previous entry's `(id, prev_hash, tenant_id, sequence_number, event_type, event_schema_version, payload_canonical_json, created_at)` tuple. The hash input is computed over the **canonical Postgres-stored tuple** — not over the OCSF-translated wire bytes that leave the gateway. Clarifications on the canonical input and concurrency control:

   - **`payload_canonical_json` scope.** This is the complete Lenny-internal JSON serialization of the event payload column (a `jsonb` column in the `audit_log` table). Every payload field written by Lenny — whether or not it has a mapping in the OCSF Wire Format table below — is included. New payload fields introduced in future Lenny releases become part of `payload_canonical_json` automatically; they do not alter hashes for events written before the field existed because older payloads simply did not contain the field. The OCSF translator reads this payload field-by-field to produce the wire OCSF record; payload fields with no explicit OCSF mapping are routed into `unmapped.lenny.*` (see Wire Format below).
   - **Canonicalization.** RFC 8785 JSON Canonicalization Scheme (JCS): lexicographic key order at every nesting level, UTF-8 NFC normalization of string values, no whitespace, numbers in shortest canonical form. The `pg_jcs_canonicalize()` PL/pgSQL function (installed by the Phase 13 migration) computes the canonical form at INSERT time and stores it in an immutable generated column; the `prev_hash` trigger reads from that column, not from the raw `payload` jsonb.
   - **`event_schema_version`.** Lenny's event-type registry versions each event type independently (e.g., `credential.leased@v1`, `credential.leased@v2`). `event_schema_version` is included in the hash input so a payload shape change for the same `event_type` produces distinct hashes. External verifiers recomputing the chain MUST use the schema version that was current when the hash was computed; Lenny publishes the schema-version registry as `schemas/audit-events/v*.json` on every release. The `unmapped.lenny.event_schema_version` OCSF extension field echoes the version on the wire so verifiers can locate the correct schema.
   - **Authoritative total order (`sequence_number`).** The per-tenant `sequence_number` column — a `bigint` allocated from a per-tenant Postgres sequence inside the same transaction that inserts the row — is the **authoritative total order** for chain verification. Verifiers reconstruct the chain by reading rows ordered by `sequence_number`, not by `created_at`. `created_at` is recorded for human-readable diagnostics and for SIEM time-range queries, but may be non-monotonic under NTP correction or database clock jumps; relying on it for order would produce false `broken` reports after any backward clock step. The hash input includes `sequence_number` so that any attempt to reinsert a row at a different position breaks the chain at that position.
   - **Concurrency control for `prev_hash`.** To prevent two concurrent writers from computing `prev_hash` against the same tail row and creating a forked chain, every audit INSERT within a tenant is serialized by a per-tenant Postgres advisory lock held for the duration of the transaction: `SELECT pg_advisory_xact_lock(hashtext('audit:' || tenant_id::text))` is the first statement of the audit-write transaction, taken before the tail-row SELECT that reads `prev_hash` and `sequence_number`. The lock is released automatically on `COMMIT`/`ROLLBACK`. Advisory locks are keyed by `hashtext(tenant_id)` so cross-tenant writes proceed in parallel; within a tenant, audit writes are strictly serialized. The `lenny_audit_immutability` trigger additionally rejects any INSERT whose `prev_hash` does not match the current tail's hash (under the same transaction-scoped lock), providing defense-in-depth against an application-layer bug that bypasses the lock.
   - **Lock acquisition SLO and retry.** `audit.lock.acquireTimeoutMs` (default `5000`) is set as `statement_timeout` on the `pg_advisory_xact_lock` call; the lock is acquired in blocking mode up to that timeout. Inside the gateway process, audit writes that fail to acquire the lock (caused by `statement_timeout` or Postgres backend termination) return `AUDIT_CONCURRENCY_TIMEOUT` to the caller; the gateway retries on the same replica up to `audit.lock.maxRetries` (default `3`) with exponential backoff (`audit.lock.retryBaseMs` default 20ms, doubling per attempt, jittered ±20%). If all retries fail, the top-level request returns `503 audit_unavailable` and the `AuditLockContention` alert (§16.5) fires. The P99 lock acquisition latency SLO is `lenny_audit_lock_acquire_seconds{p=99} < 0.05` (50ms) across any 5-minute window; breach triggers the same alert. This SLO bounds per-tenant audit write throughput: at 50ms P99 serialization, a single tenant sustains ≥20 audit writes/second worst-case, which exceeds the per-tenant audit event rate for all practical workloads (a single session generates O(10) audit events over its entire lifetime). Bursty workloads with momentary contention are absorbed by the retry loop; sustained breach of the SLO indicates a real capacity problem (e.g., Postgres I/O saturation) and pages the on-call operator.

The OCSF translator (see Wire Format below) reads the canonical tuple to compute the chain and writes the translated OCSF record to external sinks — the chain is verifiable end-to-end against the Postgres source of truth regardless of wire format. The verifier reports chain state via the `chainIntegrity` field with the following enumeration: `verified` (hash chain validated end-to-end), `broken` (mismatch detected and not attributable to a known outage), `unchecked` (verification not yet run or disabled), `rechained_post_outage` (chain rewritten after reconciliation from `audit_log_deferred_writes` — see §25.9; expected value when `lenny-ops` deferred writes are reconciled after a Postgres outage), `gap_suspected` (sequence-number gap detected; cross-reference `ops_postgres_outage_log` for cause). The first entry in each tenant partition uses a **per-tenant genesis hash** derived from a random nonce generated at tenant creation time: `genesis_hash = SHA-256("lenny-audit-genesis" || tenant_id || genesis_nonce)`. The `genesis_nonce` (256-bit, generated via `crypto/rand`) is stored in the tenant configuration record and also written to the SIEM (when configured) alongside the first audit event. This ensures that an adversary who reconstructs a hash chain from the genesis point forward must possess the tenant-specific nonce — a well-known fixed genesis hash would allow undetectable chain reconstruction by a database superuser. This creates a tamper-evident chain — any retroactive modification or deletion of an entry breaks the chain for all subsequent entries. The periodic background check (item 2) also samples random chain segments and verifies hash continuity; a broken chain triggers the same critical alert path.

4. **SIEM connectivity validation (optional):** For deployments that require compliance-grade audit integrity, audit events **should** be streamed to an external immutable log (SIEM, cloud audit service, or append-only object storage) in addition to Postgres storage. SIEM is not required — Lenny operates fully without it, with the integrity trade-offs documented below. When `audit.siem.endpoint` is configured, the gateway validates SIEM connectivity at startup: a test event is sent and the gateway **refuses to start** until acknowledgement is received. At runtime, a health check monitors SIEM delivery success rate; if the failure rate exceeds the configured threshold (`audit.siem.failureThresholdPercent`, default 5%), the gateway emits a critical alert and the `/healthz` endpoint reports degraded status. This ensures audit integrity even if the database is compromised, and prevents silent SIEM delivery failures from creating gaps in the external audit trail.

5. **pgaudit logging to external append-only sink.** To close the residual tamper window that remains between periodic grant checks (items 1–2), regulated deployments **must** enable `pgaudit` on the Postgres cluster and stream DDL/GRANT events to an external append-only sink. Configuration:
   - **`audit.pgaudit.enabled`** (bool, default `false`): when `true`, the gateway's startup preflight check ([Section 17.6](17_deployment-topology.md#176-packaging-and-installation)) validates that the `pgaudit` extension is installed and that `pgaudit.log` includes `DDL` and `ROLE` classes. If validation fails in production mode with a regulated `complianceProfile`, the gateway refuses to start.
   - **`audit.pgaudit.sinkEndpoint`** (string): the external append-only endpoint (e.g., a cloud audit logging service, SIEM, or append-only S3 bucket with object lock) to which `pgaudit` log records are forwarded. The gateway does not ship the pgaudit records itself — this is a Postgres-level configuration (e.g., `pg_log` forwarded via a log shipper such as Fluentd/Fluent Bit). The gateway's startup check verifies that the Postgres cluster's `log_destination` configuration includes the configured shipper endpoint.
   - **Regulated tenant requirement:** When any active tenant has a regulated `complianceProfile`, `audit.pgaudit.enabled` must be `true` and `audit.pgaudit.sinkEndpoint` must be configured. The gateway enforces this at startup (fatal error if not satisfied in production mode) and at regulated tenant creation/update (HTTP 422 `COMPLIANCE_PGAUDIT_REQUIRED` if not configured).
   - **What pgaudit captures:** Any `GRANT`, `REVOKE`, or DDL statement on audit tables executed by any database role — including superusers — is logged with role identity, timestamp, and statement text. Because these records are written to the external append-only sink (not Postgres), a superuser who grants UPDATE access and then revokes it leaves an indelible trace in the sink, even if no periodic check cycle fired in the interim.
   - **Metric:** `lenny_pgaudit_grant_events_total` (counter, labeled by `statement_type`: `GRANT`, `REVOKE`, `DDL`) tracks pgaudit-sourced events forwarded to the sink. Alert: `PgAuditSinkDeliveryFailed` fires if the log shipper reports delivery failures for more than `audit.pgaudit.sinkFailureAlertMinutes` (default: 5) minutes.

6. **Deferred-write path for Postgres outages.** During a Postgres outage, audit events generated by `lenny-ops` (§25.4) are buffered — first into an in-memory queue inside `lenny-ops`, and, when Postgres briefly recovers, written to the `audit_log_deferred_writes` platform-scoped table (§25.9). On full recovery, a reconciliation pass inserts the deferred events into `audit_log` with their **original timestamps** and recomputes the hash chain for the affected range (the resulting range is stamped `chainIntegrity = rechained_post_outage` per item 3). The reconciliation is idempotent and tolerates partial completion. Each outage boundary is also recorded in `ops_postgres_outage_log` so verifiers can cross-reference suspected gaps. Deferred writes apply only to events originating from `lenny-ops`; the gateway's own audit events are written synchronously and block session creation when Postgres is unreachable.

7. **Erasure role and immutability trigger bypass.** The GDPR erasure job ([Section 12.8](12_storage-architecture.md#128-compliance-interfaces)) must UPDATE billing events (pseudonymization) and DELETE audit events, which conflicts with `lenny_app`'s INSERT-only grants and the `lenny_billing_immutability` / `lenny_audit_immutability` triggers. A dedicated `lenny_erasure` database role resolves this:
   - **Grants:** `lenny_erasure` has `UPDATE` on billing tables (scoped to the `user_id` and free-text PII columns only — not `sequence_number`, `tenant_id`, or cost dimensions), `DELETE` on audit tables, and `INSERT` on audit tables restricted to erasure receipt rows (`event_type LIKE 'gdpr.%'`) via a row-level `BEFORE INSERT` trigger that rejects any `lenny_erasure` INSERT where `event_type NOT LIKE 'gdpr.%'`. It has no `SELECT` beyond what erasure verification requires, or any grants on non-erasure tables. These grants are defined in the schema migration alongside the `lenny_app` grants.
   - **Usage:** Only the erasure background job connects as `lenny_erasure`. Gateway replicas and all other services connect as `lenny_app` and never hold `lenny_erasure` credentials.
   - **Immutability trigger bypass:** The `lenny_billing_immutability` and `lenny_audit_immutability` trigger functions check `current_setting('lenny.erasure_mode', true)`. When the setting is `'true'`, the trigger allows the UPDATE or DELETE and returns without raising an exception. The erasure job sets `SET LOCAL lenny.erasure_mode = 'true'` at the start of each erasure transaction — `SET LOCAL` scopes the setting to the current transaction only, so it cannot leak to other connections via the connection pooler. The startup grant verification (item 1) and periodic background check (item 2) verify that `lenny_app` does not hold UPDATE or DELETE grants; the `lenny_erasure` role's grants are expected and excluded from drift detection.
   - **Startup verification extension:** The trigger-enabled check (item 1) additionally verifies that the `lenny.erasure_mode` guard clause is present in both immutability trigger function bodies (by inspecting `pg_proc.prosrc`). If the guard is missing, the gateway refuses to start in production — this prevents a migration rollback from silently removing the bypass and breaking erasure.

**Wire format — OCSF.** Every audit event leaving the Postgres hot tier — to the SIEM forwarder, the pgaudit sink consumer, webhook subscribers ([§25.5](25_agent-operability.md#255-operational-event-stream)), the gateway-emitted EventBus `data` payload when the event is audit-bearing, and all §25 external-facing audit query responses — is serialized as an **[OCSF (Open Cybersecurity Schema Framework) v1.1.0](https://schema.ocsf.io/1.1.0/)** JSON record. OCSF is the single, canonical wire format; there is no alternative, no per-deployer format flag, and no `audit.siem.format` configuration. Lenny-internal storage in Postgres retains the canonical tuple described under item 3 (hash chaining) — OCSF translation happens at the egress boundary.

**Lenny → OCSF field mapping.** The OCSF translator (located in the [§4.4](04_system-components.md#44-event--checkpoint-store) EventStore egress path) maps Lenny's internal audit-row columns and payload fields onto OCSF class attributes. The mapping is deterministic and reversible:

| Lenny column / payload field | OCSF attribute | OCSF class notes |
|---|---|---|

| `sequence_number` | `metadata.sequence` | Monotonic per-tenant sequence; enables external reconciliation with hash chain |
| `tenant_id` | `metadata.tenant_uid` | Lenny-specific tenant identifier |
| `created_at` | `time` | UNIX ms (OCSF requires numeric epoch ms, not ISO 8601) |
| `event_type` | `class_uid` + `activity_id` | See event-type → class/activity table below |

Deployers consuming OCSF records can filter on `type_id` to distinguish agent-initiated from human-initiated activity without parsing the `type` string. |
| `operation_id` ([§25.1](25_agent-operability.md#251-design-philosophy-and-agent-model)) | `metadata.correlation_uid` | Correlates audit events belonging to the same operation |

| Remaining Lenny-specific payload fields | `unmapped.lenny.*` | Preserved verbatim; extends OCSF without breaking schema |

The `class_uid`/`activity_id` mapping for Lenny's event-type catalog ([§16.6](16_observability.md#166-operational-events-catalog), [§16.7](16_observability.md#167-section-25-audit-events), [§4.9.2](04_system-components.md#492-credential-audit-events)) assigns each event type to an OCSF class (primarily classes 3002 Authentication, 6003 API Activity, 6004 File System Activity, 3006 Account Change, 5001 Entity Management, and 2004 Application Security Finding). The full mapping is maintained in `schemas/ocsf-mapping.yaml` committed alongside the spec and regenerated in CI from Lenny's event-type catalog.

**OCSF version pinning.** The wire version is **OCSF v1.1.0** at launch. Upgrading the wire version is a deployer-observable change: every active SIEM/webhook consumer must tolerate it. The OCSF version is advertised in every record via `metadata.version`, and the translator version is surfaced on the `/v1/admin/audit-events` query API response envelope as `ocsfVersion`. Upgrading is planned as a coordinated release — there is no runtime-selectable dual-version emission.

**Hash chain preservation across OCSF translation.** Because the chain is computed over the canonical Postgres tuple and not the OCSF bytes (item 3 above), OCSF translation is a pure rendering step that cannot affect chain integrity. External verifiers can either (a) recompute the chain from the `unmapped.lenny_chain.prev_hash` extension plus the reversible OCSF fields, or (b) request the raw canonical tuple via `GET /v1/admin/audit-events/{id}?format=raw-canonical` (a scope-restricted admin endpoint for chain auditors).

**OCSF translation failure path.** OCSF translation runs at the egress boundary (EventStore forwarder), after the canonical Postgres row has already been committed. If translation fails for any reason — a malformed payload that violates the translator's schema, an unexpected OCSF class mapping gap, or a translator panic — the Postgres row is **not** rolled back; the canonical tuple and the hash chain remain intact.

The `audit_log.ocsf_translation_state` column (enum: `pending` → `retry_pending` → `succeeded` | `dead_lettered`) records the translator's progress per row. On first-attempt failure the translator writes `retry_pending` and increments `lenny_audit_ocsf_translation_failed_total` labeled by `event_type` and `error_class` (`schema_violation`, `class_mapping_missing`, `translator_panic`, `other`). A background retry loop re-attempts translation every `audit.ocsf.retryInterval` (default 30s) up to `audit.ocsf.maxAttempts` (default 10) with exponential backoff. On final-attempt failure the row transitions to `dead_lettered`; on success it transitions to `succeeded`.

**Dead-letter handling (avoids SIEM head-of-line blocking).** The SIEM delivery pointer MUST advance past `dead_lettered` rows so that a single persistently-failing event does not halt the entire per-tenant SIEM stream. The translator achieves this by emitting, in place of the untranslatable event, a **translation-failure receipt** — an OCSF `Application Security Finding` (class 2004) record describing that translation failed, with the problematic payload encoded as an opaque base64 `unmapped.lenny.raw_canonical_b64` field and the failure `error_class` in the receipt's `finding.title`. The receipt itself flows through the normal translator (it is schema-compliant by construction). The original row remains queryable via `GET /v1/admin/audit-events?ocsf_translation_state=dead_lettered` and can be re-translated by a new translator version once the mapping gap is fixed (`POST /v1/admin/audit-events/{id}/retranslate`). **Erasure interaction:** because a `dead_lettered` row holds the untranslated raw canonical payload (including any PII present in the original event), user-level erasure **MUST** redact the PII-bearing payload in place rather than simply deleting the row — see [§12.8](12_storage-architecture.md#128-compliance-interfaces) `DeleteByUser` OCSF dead-letter PII redaction step for the redaction rule and the `gdpr.erasure_deadletter_redacted` receipt. A subsequent `POST /v1/admin/audit-events/{id}/retranslate` against a row whose payload has been redacted is rejected with `410 DEADLETTER_REDACTED`.

The `OCSFTranslationBacklog` alert ([§16.5](16_observability.md#165-alerting-rules-and-slos)) fires when either `retry_pending` count exceeds `audit.ocsf.alertThreshold` (default 10) for more than 5 minutes, or any row transitions to `dead_lettered` (every dead-letter is an operator-visible event regardless of backlog size). This applies equally to the SIEM stream, the pgaudit sink consumer, and webhook-delivered audit events — the translator runs once and its output is multicast to all consumers, so head-of-line blocking avoidance benefits all downstream sinks uniformly.

**Write-time tenant validation.** The `tenant_id` column populated on every audit row (which becomes `metadata.tenant_uid` in the OCSF record) MUST equal the `tenant_id` scope of the request or operation that produced the event. The gateway's audit-write path derives `tenant_id` from the authenticated caller's JWT claim ([§13.3](13_security-model.md#133-credential-flow)) or from the session/operation context that originated the event; callers cannot supply `tenant_id` directly via the payload. Any audit-write attempt whose derived `tenant_id` does not match the available scope is rejected at the Postgres-write boundary with `AUDIT_TENANT_SCOPE_MISMATCH` and emits a `security.audit_write_rejected` event (itself audited, under the platform tenant). Events that legitimately cross tenants — the narrow platform-admin impersonation flow ([§13.3](13_security-model.md#133-credential-flow)) and the `security.audit_write_rejected` event above — are explicitly modeled as platform-tenant events with additional `unmapped.lenny.target_tenant_id` fields rather than being written under a tenant scope that does not own them. This is a complementary control to the hash chain: the chain detects tampering after the fact; write-time validation prevents a compromised caller from injecting forged-tenant events at write time in the first place.

**Compliance profile enforcement gate.** Each tenant carries an optional `complianceProfile` field (set via `POST /v1/admin/tenants` or `PUT /v1/admin/tenants/{id}`) with the following allowed values: `none` (default), `soc2`, `fedramp`, `hipaa`. When `complianceProfile` is `soc2`, `fedramp`, or `hipaa`, the platform enforces SIEM as a hard requirement:

> **FedRAMP baseline note:** `complianceProfile: fedramp` activates the platform controls required for **FedRAMP Moderate** (AU-9 independent audit copy via SIEM, AU-11 1-year retention, AC-3 INSERT-only grants with periodic drift detection, and AU-12 pgaudit DDL/ROLE event capture). FedRAMP Moderate is the assumed baseline because it covers the majority of cloud SaaS use-cases and its 1-year AU-11 requirement is already the platform default. Deployers pursuing **FedRAMP High** must additionally set `audit.retentionPreset: fedramp-high` (3-year retention, see [Section 16.4](16_observability.md#164-logging)) and are responsible for the additional High-baseline controls that fall outside the platform's scope (e.g., FIPS 140-2 cryptographic modules, SC-28 encryption at rest for classified data, and any agency-specific control enhancements). **FedRAMP Low** is fully covered by `complianceProfile: fedramp` — Low's AU-11 requirement (1 year) and AU-9 requirements are a subset of Moderate. The platform does not claim FedRAMP authorization; `complianceProfile: fedramp` enables the relevant technical controls to support a deployer's own ATO process.

- **Tenant creation / update is rejected** with `COMPLIANCE_SIEM_REQUIRED` (HTTP 422) when `complianceProfile` is a regulated value and `audit.siem.endpoint` is not configured at the platform level. The error body includes: `"tenant.complianceProfile '<value>' requires audit.siem.endpoint to be configured. A database superuser can bypass INSERT-only grants; an independent SIEM copy is mandatory for compliance-grade audit integrity."` Deployers cannot bypass this gate — the only path forward is to configure the SIEM endpoint before creating or enabling a regulated tenant.
- **Environment creation within a regulated tenant is rejected** with the same error if `audit.siem.endpoint` becomes absent (e.g., removed from config) after the tenant was created. The gateway checks SIEM availability on every environment creation request when the parent tenant has a regulated `complianceProfile`.
- **Gateway startup with a regulated tenant and no SIEM is a fatal error** in production mode (`LENNY_ENV=production`). If any active tenant has `complianceProfile` in `{soc2, fedramp, hipaa}` and `audit.siem.endpoint` is absent, the gateway refuses to start with: `"FATAL: one or more tenants have a regulated complianceProfile but audit.siem.endpoint is not configured. Configure SIEM or downgrade tenant complianceProfile to 'none'."` This prevents a misconfiguration (e.g., SIEM endpoint accidentally removed from Helm values) from silently invalidating a live compliance posture.

**Audit integrity gap — Postgres-only deployments:** When `audit.siem.endpoint` is not configured and no tenant has a regulated `complianceProfile`, audit logs are stored exclusively in Postgres. In this configuration, the hash-chain mechanism (item 3) provides tamper-detection after the fact, but a database superuser who also has access to the `genesis_nonce` in the tenant configuration record can modify or delete rows and reconstruct a valid hash chain — the external SIEM is the only copy a superuser cannot alter. This means:

- **Tenants with a regulated `complianceProfile`** (soc2, fedramp, hipaa) **require** `audit.siem.endpoint` — enforced at tenant creation and gateway startup (see above). The SIEM provides the independent, immutable audit copy that satisfies regulatory requirements for separation of control from the audited system.
- **Development and single-tenant low-risk deployments** may operate without a SIEM, accepting this integrity limitation. This is appropriate for non-regulated workloads where a database superuser compromise is within the deployer's accepted risk profile.
- **Multi-tenant deployments with `complianceProfile: none`** are explicitly operating below compliance-grade integrity, regardless of tenant count. This must be a deliberate deployer decision (expressed by leaving `complianceProfile` at its default `none`), not an accidental configuration omission.

**Startup behavior when SIEM is unconfigured:**

- In **production mode** (`LENNY_ENV=production`) with any tenant whose `complianceProfile` is regulated (`soc2`, `fedramp`, `hipaa`): the gateway **refuses to start** with a fatal error (see enforcement gate above).
- In **production mode** with **multi-tenant configuration** (`tenancy.mode: multi`) and all tenants having `complianceProfile: none`: the gateway logs a `WARN`-level message at startup and every 60 seconds while running: `"WARNING: audit.siem.endpoint is not configured — audit logs are stored in Postgres only. A database superuser can bypass INSERT-only grants. This deployment does not meet compliance-grade audit integrity requirements (SOC2 CC7.2, FedRAMP AU-9, HIPAA §164.312(b)). Configure audit.siem.endpoint to resolve."` The `AuditSIEMNotConfigured` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires immediately. The gateway **does not refuse to start** — SIEM is not mandatory for unregulated deployments — but the integrity gap is made visible and persistent.
- In **production mode** with **single-tenant configuration** (`tenancy.mode: single`) and `complianceProfile: none`: the same `WARN` message is logged once at startup, without the repeating 60-second reminder. The `AuditSIEMNotConfigured` alert fires at `Warning` severity.
- In **non-production environments**: no warning is emitted. SIEM configuration is not expected outside production.

**Compliance-grade requirement summary:**

| `complianceProfile`       | SIEM configured? | Integrity Level                                                   | Compliance-Grade? | Allowed?                         |
| ------------------------- | ---------------- | ----------------------------------------------------------------- | ----------------- | -------------------------------- |
| `soc2` / `fedramp` / `hipaa` | Yes           | Independent external copy; superuser bypass detectable            | Yes               | Yes                              |
| `soc2` / `fedramp` / `hipaa` | No            | —                                                                 | No                | **No — blocked at creation/startup** |
| `none`                    | Yes              | Independent external copy; superuser bypass detectable            | Yes               | Yes                              |
| `none`                    | No               | Hash-chain tamper-detection only; superuser can reconstruct chain | No                | Yes (with persistent warning)    |

**Superuser bypass mitigation:** Database superusers can bypass INSERT-only grants. The hash chaining mechanism (item 3) provides a detection layer — any superuser modification that alters or deletes rows is detectable by chain verification, provided an independent copy exists to compare against. The external SIEM stream (item 4) is that independent copy — a database superuser cannot modify the SIEM. When no SIEM is configured, hash-chain verification remains useful for detecting accidental corruption but provides weaker guarantees against a determined insider threat. Deployers operating under strict compliance requirements **should** configure `audit.siem.endpoint` and **should** additionally restrict superuser access via connection-level controls (e.g., separate superuser credentials stored in a hardware security module, accessed only through a privileged access management workflow).

### 11.8 Security Incident Response

The platform surfaces security signals and provides first-responder primitives; incident response procedures are the deployer's responsibility.

**Platform-surfaced security signals:**

| Signal | Section | Severity |
| --- | --- | --- |
| `AuditGrantDrift` — unexpected UPDATE/DELETE grants on audit tables | [§11.7](#117-audit-logging), [§16.5](16_observability.md#165-alerting-rules-and-slos) | Critical |
| `CredentialCompromised` — revoked credential still has active leases | [§4.9](04_system-components.md#49-credential-leasing-service), [§16.5](16_observability.md#165-alerting-rules-and-slos) | Critical |
| `DataResidencyViolationAttempt` — cross-region write or delegation bypass | [§12.8](12_storage-architecture.md#128-compliance-interfaces), [§16.5](16_observability.md#165-alerting-rules-and-slos) | Critical |
| `AuditChainGap` — broken hash chain detected at startup | [§11.7](#117-audit-logging), [§16.5](16_observability.md#165-alerting-rules-and-slos) | Warning |
| `AuditSIEMNotConfigured` — no external audit copy in production | [§11.7](#117-audit-logging), [§16.5](16_observability.md#165-alerting-rules-and-slos) | Warning |

**Platform first-responder primitives:**

- **Credential revocation:** `POST /v1/admin/credential-pools/{name}/credentials/{credId}/revoke` immediately marks the credential as `revoked` and triggers active-lease propagation. The `CredentialCompromised` alert clears once all active leases are terminated. See [Section 4.9](04_system-components.md#49-credential-leasing-service).
- **User invalidation:** `POST /v1/admin/users/{user_id}/invalidate` terminates all active sessions for a user and revokes their tokens, stopping an active attacker's sessions. See [Section 11.4](#114-user-invalidation).
- **Legal hold:** `POST /v1/admin/legal-hold` suspends retention rotation for any session or artifact implicated in an incident, preserving forensic state. See [Section 12.8](12_storage-architecture.md#128-compliance-interfaces).
- **Audit trail:** All admin operations (revocations, holds, invalidations) are written to the append-only audit trail ([Section 11.7](#117-audit-logging)) with operator identity and timestamp, providing a tamper-evident record of responder actions.

**Breach notification.** Lenny operates as a data processor. Breach notification obligations (GDPR Article 33/34, HIPAA §164.410, state data breach laws) are the deployer's responsibility as data controller. The platform provides the audit trail, SIEM stream, and erasure primitives deployers need to assess scope and demonstrate response, but it does not implement a notification workflow. Deployers must document their breach notification procedures in their incident response plans and ensure those procedures reference the platform signals listed above.

**Incident response plan guidance.** Deployers should define procedures covering at minimum: (1) triage criteria using the Critical alerts above; (2) first-responder steps using the primitives above; (3) scope assessment using the audit trail and SIEM; (4) breach notification timelines required by applicable law; (5) post-incident review and credential rotation. The platform's runbooks (`docs/runbooks/`) cover operational failures; security incident procedures are maintained separately by the deployer.
