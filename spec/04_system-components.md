### 4.1 Edge Gateway Replicas

**Role:** The only externally-facing component. All client interaction enters through the gateway.

**Responsibilities:**

- Authenticate clients (OIDC/OAuth 2.1)
- Expose multiple external interfaces: REST API and protocol adapters via `ExternalAdapterRegistry` (MCP, OpenAI Completions, Open Responses ŌĆö see [Section 15](15_external-api-surface.md))
- Route sessions to the correct runtime pod
- Proxy long-lived interactive streams (WebSocket / gRPC bidi / Streamable HTTP)
- Run the Policy Engine and `RequestInterceptor` chain on every request
- Handle reconnect/reattach after disconnection
- Mediate all file uploads to pods
- Host virtual MCP child interfaces for delegation
- Serve dedicated MCP endpoints for `type: mcp` runtimes at `/mcp/runtimes/{name}`

**Deployment:**

- Stateless-ish replicas behind ingress/load balancer
- HPA on CPU, request queue depth, and active streams (see [┬¦4.1](#41-edge-gateway-replicas) HPA metric role table for the canonical metric-to-role mapping). `lenny_gateway_active_sessions` is a capacity ceiling alert signal only ŌĆö it is NOT used as an HPA trigger.
- Per-tier HPA configuration (min/max replicas, target utilization, scale-down policy) is in [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults).
- Sticky routing is an optimization, not a correctness requirement
- PodDisruptionBudget to limit simultaneous disruptions

**Per-replica session capacity budget:**

Each gateway replica has a maximum number of concurrent sessions it can handle before session-acceptance latency and stream proxy queue depth degrade. This limit is expressed as `gateway.maxSessionsPerReplica`.

**HPA metric roles ŌĆö canonical definition (SCL-026):** Three gateway metrics are used in HPA and alerting, each with a distinct role. This is the authoritative mapping; [┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) and [┬¦17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) cross-reference it:

| Metric | Role | Where used |
| ------ | ---- | ---------- |
| `lenny_gateway_request_queue_depth` | **Primary HPA scale-out trigger.** Reflects instantaneous back-pressure (requests queued awaiting a handler goroutine). Reacts faster than CPU and does not depend on Prometheus Adapter cache TTL when surfaced via KEDA. Target `averageValue: 10` (Tier 2); see [┬¦17.8.2](17_deployment-topology.md#1782-capacity-tier-reference). | HPA / KEDA ScaledObject |
| `lenny_gateway_active_streams` | **Secondary HPA metric.** Per-replica gauge of in-flight streaming connections, surfaced via Prometheus Adapter or KEDA alongside `request_queue_depth` and CPU. | HPA / KEDA ScaledObject |
| `lenny_gateway_active_sessions / gateway.maxSessionsPerReplica` | **Capacity ceiling alert, not an HPA trigger.** Fires `GatewaySessionBudgetNearExhaustion` ([┬¦16.5](16_observability.md#165-alerting-rules-and-slos)) when > 90% on any replica, giving a last-resort signal if HPA scale-out is lagging. Must NOT be used as the sole HPA trigger ŌĆö it measures Postgres-backed session count (not live goroutine pressure) and lags real load. | Alert only |

`gateway.maxSessionsPerReplica` defines the capacity ceiling used by the alert above; it is calibrated by the Phase 2 benchmark methodology below.

| Tier | Provisional `maxSessionsPerReplica` | HPA target utilization |
| ---- | ----------------------------------- | ---------------------- |
| Tier 1 (Starter) | 50 | 80% |
| Tier 2 (Growth) | 200 | 80% |
| Tier 3 (Scale) | 400 | 80% |
| Tier 4 (Platform) | 400 | 80% |

**These values are provisional first-principles estimates.** The Tier 1 and Tier 2 values are derived from the respective max concurrent sessions targets (100 and 1,000) and assumed starting replica counts. The Tier 3 value (400) assumes that the LLM Proxy subsystem has been extracted to a dedicated service before the Tier 2ŌåÆ3 promotion decision ŌĆö removing the long-lived upstream goroutine workload from the gateway process reduces heap and GC pressure, roughly doubling the per-replica session capacity. **Tier 3 deployment therefore carries the following prerequisite:** the Phase 13.5 load tests must confirm that (a) the LLM Proxy subsystem has been extracted or `lenny_gateway_llm_proxy_active_connections / lenny_gateway_active_sessions` is sustainably below 0.3:1 at Tier 2 peak (see extraction threshold guidance in the "Extraction triggers" subsection below), and (b) `lenny_gateway_gc_pause_p99_ms` remains below 50 ms at Tier 2 peak load. If either condition is not met, the Tier 3 provisional value reverts to 200 and additional replicas must be used to reach the 10,000-session target. All values must be replaced with empirically validated measurements produced by the Phase 2 (Tier 1/2) and Phase 13.5 (Tier 3) benchmark harnesses before any production deployment.

**Capacity budget calibration methodology (Phase 2 deliverable):**

1. **Ramp test.** Drive a single gateway replica from 0 to `maxSessionsPerReplica ├Ś 1.5` concurrent sessions in 10% increments. At each step record: `lenny_stream_proxy_p99_attach_latency_seconds`, `lenny_stream_proxy_queue_depth`, `lenny_gateway_gc_pause_p99_ms`, and RSS.
2. **Saturation point.** Identify the session count at which `lenny_stream_proxy_p99_attach_latency_seconds` first exceeds 0.8 s (800 ms) or `lenny_stream_proxy_queue_depth` first exceeds 500. That count is the empirical saturation point.
3. **Budget setting.** Set `maxSessionsPerReplica` to the saturation point minus 20% headroom. For example, if saturation occurs at 250 sessions, set `maxSessionsPerReplica` to 200.
4. **HPA validation.** Confirm that a horizontal scale-out triggered by `lenny_gateway_request_queue_depth` reaching its HPA target fires at least one full HPA scale cycle (typically 2ŌĆō3 minutes) before the replica reaches its empirical saturation point. Verify that `lenny_gateway_active_sessions / gateway.maxSessionsPerReplica` reaching 90% triggers the `GatewaySessionBudgetNearExhaustion` alert (this metric is an alert-only signal, not an HPA trigger ŌĆö see metric role table above).
5. **Document and replace.** Replace each provisional value in the table above with the calibrated value, annotated with the benchmark run identifier and date. Provisional values must not remain in place for any Tier 2 production deployment.

This calibration must be completed as a Phase 2 exit criterion, aligned with the subsystem extraction threshold calibration described below. The `GatewaySessionBudgetNearExhaustion` alert in [Section 16.5](16_observability.md#165-alerting-rules-and-slos) fires when `lenny_gateway_active_sessions / gateway.maxSessionsPerReplica` exceeds 90% on any replica, giving a last-resort signal if HPA scale-out is lagging.

**Key invariant:** A client can land on any gateway replica. Session state is always in durable stores.

#### Gateway Internal Subsystems

The gateway binary is internally partitioned into four subsystem boundaries. These are **not** separate services ŌĆö they are Go interfaces within a single binary that enforce isolation at the concurrency and failure-domain level, so that a problem in one area (e.g., a slow upload) cannot starve or crash another (e.g., MCP streaming).

**Subsystems:**

1. **Stream Proxy** ŌĆö MCP streaming, session attachment, event relay, and client reconnection handling.
2. **Upload Handler** ŌĆö File upload proxying, payload validation, staging to the Artifact Store, and archive extraction.
3. **MCP Fabric** ŌĆö Delegation orchestration, virtual child MCP interfaces, and elicitation chain management.
4. **LLM Proxy** ŌĆö Credential-injecting reverse proxy for LLM provider traffic ([Section 4.9](#49-credential-leasing-service)). Validates lease tokens, injects real API keys, and forwards streaming requests to upstream LLM providers.

**Per-subsystem isolation guarantees:**

Each subsystem is defined as a Go interface with its own:

- **Goroutine pool / concurrency limits:** A saturated Upload Handler cannot consume goroutines needed by the Stream Proxy. Each subsystem has independently configured `maxConcurrent` and queue-depth settings. See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier recommended values.
- **Per-subsystem metrics:** Latency histograms, error rates, and queue depth are emitted per subsystem, enabling targeted alerting (e.g., upload p99 latency spike does not hide a stream proxy degradation).
- **Circuit breaker:** Each subsystem has its own automatic, in-memory circuit breaker (not the operator-managed circuit breakers in [Section 11.6](11_policy-and-controls.md#116-circuit-breakers)). The Upload Handler can trip to half-open or open state ŌĆö returning 503 for uploads ŌĆö while the Stream Proxy and MCP Fabric continue serving normally. This is the primary mechanism for partial gateway degradation. **These breakers are intentionally per-replica (not shared across replicas).** When a downstream dependency failure is infrastructure-wide (e.g., MinIO down), each replica independently discovers the failure and opens its own breaker after its local failure threshold. This means the first N requests per replica fail before the breaker opens, and clients behind a load balancer may see non-deterministic 503s during the convergence window. This is the intended design: shared-state breakers would add a Redis dependency on the critical upload and streaming paths. Clients should retry on 503; the load balancer distributes retries across replicas, and all replicas converge to open state within seconds. For infrastructure-wide outages that require immediate platform-wide rejection, operators should use the operator-managed circuit breakers ([Section 11.6](11_policy-and-controls.md#116-circuit-breakers)), which are Redis-backed and propagate across all replicas within one round-trip.

**Extraction triggers:**

These internal boundaries are designed so that any subsystem can be extracted to a dedicated service if scaling requires it. The following table defines the observable metrics and indicative threshold ranges for each subsystem. All threshold values in this table are **provisional first-principles estimates** derived from capacity reasoning against the Tier 2 load targets. They must be replaced with empirically validated values produced by the Phase 2 benchmark harness (see calibration methodology below) before those thresholds are used to make any extraction decision.

| Subsystem          | Key Metrics                                                                                                           | Indicative Extraction Threshold (provisional ŌĆö see calibration note)                                                                                                                                                                                                                                                                                    | Failure Mode if Not Extracted                                                                               |
| ------------------ | --------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| **Stream Proxy**   | `lenny_stream_proxy_queue_depth`, `lenny_stream_proxy_goroutines`, `lenny_stream_proxy_p99_attach_latency_seconds`         | Queue depth sustained >500 **or** p99 attach latency >800 ms under normal (non-burst) load for Ōēź5 min. These figures are provisional; final values must be set per the calibration methodology below.                                                                                                                                                   | Head-of-line blocking on session attach; reconnection storms amplify the problem                            |
| **Upload Handler** | `lenny_upload_handler_active_uploads`, `lenny_upload_handler_queue_depth`, `lenny_upload_handler_p99_latency_seconds`      | Active uploads sustained >200 concurrent **or** HPA signals consistently driven by upload load rather than session load. These figures are provisional; final values must be set per the calibration methodology below.                                                                                                                                  | Upload bursts cause gateway over-provisioning that wastes resources on stream proxy replicas                |
| **MCP Fabric**     | `lenny_mcp_fabric_active_delegations`, `lenny_mcp_fabric_goroutines`, `lenny_mcp_fabric_p99_orchestration_latency_seconds` | Active delegations sustained >1,000 concurrent **or** p99 orchestration latency >2,000 ms. | **LLM Proxy**      | `lenny_gateway_llm_proxy_active_connections`, `lenny_llm_proxy_upstream_goroutines`, `lenny_llm_proxy_p99_ttfb_seconds`    | Active upstream connections sustained >2,000 **or** `lenny_gateway_llm_proxy_active_connections` exceeds 60% of `maxConcurrent` across all replicas for Ōēź10 min and is growing faster than gateway HPA can respond **or** HPA signals dominated by LLM proxy connection load. These figures are provisional; final values must be set per the calibration methodology below. | Long-lived upstream streaming goroutines inflate heap and GC pause time for all subsystems                  |

**Extraction threshold Helm values.** All indicative threshold values in the table above are configurable at deploy time via `gateway.extractionThresholds.<subsystem>.<metric>` Helm values (e.g., `gateway.extractionThresholds.streamProxy.queueDepth`, `gateway.extractionThresholds.uploadHandler.activeConcurrent`, `gateway.extractionThresholds.mcpFabric.activeDelegations`, `gateway.extractionThresholds.llmProxy.activeConnections`). The defaults encode the provisional first-principles estimates shown in the table; operators must replace them with empirically calibrated values produced by the Phase 2 benchmark harness before any extraction decision is made. Storing the thresholds as Helm values (rather than hardcoded prose) ensures that validated measurements can be applied without a code change and that the values used for an extraction decision are auditable in the Helm release history.

**LLM Proxy-to-session ratio guidance:** When the ratio of `lenny_gateway_llm_proxy_active_connections` to `lenny_gateway_active_sessions` approaches 1:1 (i.e., nearly every active session has an open LLM proxy stream), the LLM Proxy subsystem is effectively as large as the Stream Proxy subsystem. At ratios above 0.5:1 sustained for more than 15 minutes, operators should begin planning extraction. Below 0.3:1, the combined binary is expected to remain sustainable at Tier 3 scale. Between 0.3:1 and 0.5:1, operators should monitor `lenny_gateway_gc_pause_p99_ms` for early signs of shared-process GC pressure. These ratio boundaries are also provisional and subject to revision after Phase 2 benchmarking.

**Shared-process GC pressure signal:** When any two subsystems simultaneously approach their indicative thresholds, the combined goroutine count and heap allocation rate can cause GC pauses that affect all subsystems. The metric `lenny_gateway_gc_pause_p99_ms` is emitted at the process level; a sustained value >50 ms is an additional signal that the process boundary is becoming a bottleneck regardless of per-subsystem queue depth.

**Threshold calibration methodology (Phase 2 deliverable):**

Before any extraction threshold is treated as actionable, the following procedure must be executed against the Phase 2 benchmark harness running a **monolithic gateway** (all four subsystems in a single binary):

1. **Baseline sweep.** Drive the gateway through four synthetic load levels: 25%, 50%, 75%, and 100% of the Tier 2 session concurrency target. At each level, record p50, p95, and p99 latency for every per-subsystem key metric listed in the table above, plus `lenny_gateway_gc_pause_p99_ms`.
2. **Stress identification.** Identify the load level at which any subsystem's p99 latency or queue depth first shows a statistically significant inflection (non-linear growth relative to the prior level). This is the empirical saturation point for that subsystem.
3. **Threshold setting.** Set the extraction threshold for each subsystem at the metric value observed at the saturation point **minus 20% headroom**. For example, if Stream Proxy p99 attach latency first inflects at 1,000 ms at 75% Tier 2 load, the extraction threshold for that metric is set to 800 ms.
4. **Sensitivity check.** Verify that the 20%-headroom threshold fires at least one full HPA scale cycle (typically 2ŌĆō3 minutes) before the subsystem reaches the measured saturation point, giving operators time to react.
5. **Document and replace.** Replace each provisional threshold in the table above with the calibrated value, annotated with the benchmark run identifier and date. The provisional values must not remain in place for any Tier 2 production deployment.

This calibration must be completed as a Phase 2 exit criterion. Phase 13.5 load testing will perform a second validation at Tier 3 scale; at that point the thresholds may be revised upward if the combined binary remains healthy, or extraction may be triggered if the empirical saturation point has moved.

Until those triggers are hit, a single binary with internal boundaries is preferred for operational simplicity.

### 4.2 Session Manager

**Role:** Source of truth for all session and task metadata.

**Backed by:** Postgres (primary), Redis (hot routing cache, short-lived locks)

**Manages:**

- Session records (id, **tenant_id**, user_id, state, pool, pod assignment, cwd, recovery_generation, coordination_generation, **schema_version**). **`recovery_generation`** is incremented on each pod recovery (visible to clients via the session API and `session.resumed` events); it tracks how many times this logical session has been recovered onto a new pod. **`coordination_generation`** is incremented on coordinator handoff across gateway replicas (internal only, used for split-brain fencing); it tracks which gateway replica is the authoritative coordinator. These are independent counters: a coordinator handoff does not increment `recovery_generation`, and a session recovery does not reset `coordination_generation`. Both counters are **monotonically non-decreasing** across every state transition ŌĆö never rolled back, never reset. In particular, mid-resume terminal collapses (`resuming ŌåÆ cancelled` / `resuming ŌåÆ completed`, see [┬¦7.2](07_session-lifecycle.md#72-interactive-session-model) snapshot-close semantics) bump `coordination_generation` under the terminal write (fencing any stale coordinator still attempting resume) and freeze `recovery_generation` at its current value ŌĆö the interrupted resume is recorded as failed-by-terminal rather than retried, preserving monotonicity without rollback. The partial-checkpoint manifest row written by the aborted replay carries both counter values plus `manifest_reason = 'terminated_during_resume'` per [┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) partial-manifest schema, so audit reconstruction has a consistent generation tuple at every terminal transition and crash-recovery can distinguish an intentional mid-resume cancel/complete from a lost mid-flight.
- Task records and parent/child lineage (task tree)
- Retry counters and policy enforcement
- Resume eligibility and window
- Pod-to-session binding
- Delegation lease tracking

**Multi-tenancy:** `tenant_id` is carried on all session, task, quota, token, memory store, and credential pool store records. **Runtime and pool records are platform-global** ŌĆö they have no `tenant_id` field and are not subject to RLS. Instead, visibility and write access are enforced at the application layer via a `runtime_tenant_access` join table (columns: `runtime_name`, `tenant_id`, `granted_by`, `granted_at`) and an equivalent `pool_tenant_access` table. A `tenant-admin`'s calls to `/v1/admin/runtimes` and `/v1/admin/pools` are filtered to the rows in these access tables for their tenant; `platform-admin` calls are unfiltered. This design allows a single runtime or pool to be made available to multiple tenants without data duplication, and avoids applying RLS policies to resources that are inherently shared infrastructure. The **primary** tenant isolation mechanism is PostgreSQL Row-Level Security (RLS) policies tied to the database session role. Every tenant-scoped table has an RLS policy that filters rows using `current_setting('app.current_tenant', false)` ŌĆö the `false` parameter causes a hard error if the setting is unset, preventing silent fallback to an empty string. Every database call is wrapped in an explicit transaction that begins with `SET LOCAL app.current_tenant = '<tenant_id>'`. `SET LOCAL` is transaction-scoped: the value is automatically cleared on `COMMIT` or `ROLLBACK`, which is essential under PgBouncer transaction-mode pooling where bare `SET` could leak tenant context to subsequent requests sharing the same pooled connection. As defense-in-depth, PgBouncer's `connect_query` is configured to set a sentinel value (`SET app.current_tenant = '__unset__'`) on every fresh connection checkout; any query that reaches RLS evaluation with `__unset__` will be rejected by the policy. Application-layer `tenant_id` filtering in queries remains as additional defense-in-depth, but a missing WHERE clause in application code **cannot** break isolation ŌĆö the database enforces it regardless. An integration test must verify tenant isolation at startup by confirming that a query without `SET LOCAL` is rejected and that cross-tenant reads return zero rows. Namespace-level or cluster-level isolation is a future goal. All quotas, rate limits, and usage reports can be scoped by tenant.

**`platform-admin` cross-tenant database access.** `platform-admin` code paths that require cross-tenant reads (e.g., `GET /v1/admin/tenants`, usage dashboards, erasure jobs) use a dedicated sentinel value: `SET LOCAL app.current_tenant = '__all__'`. Every RLS policy on tenant-scoped tables includes an additional clause: `OR current_setting('app.current_tenant', false) = '__all__'`, which disables row filtering for the transaction. The `__all__` sentinel is restricted to the `platform-admin` code path: the gateway sets it only after RBAC verification confirms the caller holds the `platform-admin` role. Non-`platform-admin` callers can never trigger this sentinel ŌĆö the gateway always sets the caller's actual `tenant_id`. The `lenny_tenant_guard` trigger ([Section 12.3](12_storage-architecture.md#123-postgres-ha-requirements)) treats `__all__` as a valid value (alongside concrete tenant IDs) and does not reject it. The `__unset__` sentinel and empty-string checks remain unchanged ŌĆö they continue to reject queries with no tenant context. An integration test (`TestRLSPlatformAdminAllSentinel`) must verify that: (a) `SET LOCAL app.current_tenant = '__all__'` returns rows from multiple tenants, (b) a non-`platform-admin` caller cannot reach the `__all__` code path, and (c) the `__all__` sentinel is rejected by the `lenny_tenant_guard` trigger when `LENNY_POOLER_MODE = external` and the trigger is configured to allow only concrete tenant IDs (i.e., `__all__` must be explicitly allowlisted in the trigger alongside concrete IDs).

**Single-tenant default:** For single-tenant deployments, `tenant_id` defaults to a built-in value (`default`). The field is always present internally ŌĆö RLS policies and queries always filter by it ŌĆö but single-tenant deployers never need to set or think about it; the gateway automatically applies the default tenant. Multi-tenant mode is enabled by configuring OIDC claims that carry tenant identity, or by setting `tenant_id` explicitly in the API. No additional configuration is required for single-tenant use.

**`tenant_id` claim extraction:** The OIDC claim used to extract the tenant identifier is configurable via the `auth.tenantIdClaim` Helm value (default: `tenant_id`). In single-tenant deployments this value is ignored ŌĆö all requests are assigned the `default` tenant. In multi-tenant deployments the gateway reads the configured claim from the validated OIDC ID token and uses its string value as the `tenant_id`. If the claim is absent from the token or its value is an empty string, the gateway rejects the request with `401 Unauthorized` and error code `TENANT_CLAIM_MISSING`; it does not fall back to the default tenant. If the claim value is present but does not match any registered tenant, the gateway rejects the request with `403 Forbidden` and error code `TENANT_NOT_FOUND`. Both rejection reasons are logged (INFO level, with `user_id` and `jti` for traceability) and emitted as `auth_failure` audit events.

### 4.3 Token Service

**Role:** Manages two categories of credentials plus serves the OAuth token endpoint:

1. **Connector credentials** ŌĆö OAuth tokens for external tools and agents (GitHub, Jira, external A2A agents, etc.) accessed via registered connectors (see [Section 9.3](09_mcp-integration.md#93-connector-definition-and-oauthoidc) for the connector definition and OAuth/OIDC flow)
2. **LLM provider credentials** ŌĆö API keys, cloud IAM roles, and service accounts that runtimes need to access their backing LLM (see [Section 4.9](#49-credential-leasing-service) for the full credential leasing design)
3. **Canonical token endpoint** ŌĆö `POST /v1/oauth/token` ([┬¦15.1](15_external-api-surface.md#151-rest-api)) per [RFC 6749](https://www.rfc-editor.org/rfc/rfc6749) and [RFC 8693](https://www.rfc-editor.org/rfc/rfc8693). All Lenny bearer tokens ŌĆö admin tokens, session tokens, delegation child tokens, credential-lease tokens, operability-scope tokens ŌĆö are minted here (either via direct caller request or via internal Token Service calls from other gateway subsystems). See [Section 13](13_security-model.md#133-credential-flow) for the claim-mapping table and scope-narrowing rules.

**Deployment:** Runs as a **separate process** (Deployment) with its own ServiceAccount and KMS access. This is the only component with KMS decrypt permissions for downstream OAuth tokens. Gateway replicas call the Token Service over mTLS ŌĆö they cannot directly decrypt stored tokens.

**Key rules:**

- Pods never hold downstream OAuth tokens ŌĆö the Token Service does
- Refresh tokens stored encrypted at rest (envelope encryption via KMS)
- Access tokens short-lived, cached in Redis (encrypted, not plaintext)
- Scoped by user + connector + tenant + environment
- Kubernetes Secrets used only for bootstrap/internal credentials, not per-user OAuth tokens
- Gateway replicas request tokens via the Token Service API; they receive short-lived access tokens, never refresh tokens or KMS keys
- Each gateway replica has a distinct mTLS identity so compromise of one is attributable and revocable independently

**High availability and failure handling:**

The Token Service is deployed as a **multi-replica Deployment (2+ replicas)** with a `PodDisruptionBudget` (`minAvailable: 1`) to survive voluntary disruptions. The service is fully stateless ŌĆö all persistent state lives in Postgres (encrypted tokens) and KMS, so any replica can handle any request with no affinity requirements.

The gateway wraps all Token Service calls in an **automatic, in-memory circuit breaker** (a per-subsystem breaker, distinct from the operator-managed circuit breakers in [Section 11.6](11_policy-and-controls.md#116-circuit-breakers)). If the Token Service becomes unavailable:

- Sessions that already hold credential leases **continue operating** ŌĆö leases are self-contained and do not require the Token Service until renewal.
- New sessions that require LLM or OAuth credentials **cannot start** and fail with a retryable error, allowing clients to back off and retry.
- Sessions that do not need LLM credentials (e.g., file-processing-only runtimes) are **unaffected**.

Additionally, the gateway **caches active credential leases in memory**. Token Service unavailability does not affect already-leased credentials until the lease expires, providing a grace period proportional to the lease TTL.

### 4.4 Event / Checkpoint Store

**Role:** Enables session recovery and observability.

**Stores:**

- Event cursors / stream offsets
- Session logs and runtime stderr
- Workspace checkpoint references
- Agent session file snapshots
- Resume metadata
- Audit records (`audit_log`, [Section 11.7](11_policy-and-controls.md#117-audit-logging))

**Audit egress translator.** The EventStore's audit-egress path includes an **OCSF translator** ([Section 11.7](11_policy-and-controls.md#117-audit-logging) Wire Format) that maps the canonical Postgres-stored tuple to OCSF v1.1.0 JSON for every consumer that sits outside the authoritative store: the SIEM forwarder (`audit.siem.endpoint`), pgaudit sink consumers, the `/v1/admin/audit-events` query API ([┬¦25.9](25_agent-operability.md#259-audit-log-query-api)), and the CloudEvents-wrapped audit events published on the EventBus ([┬¦12.6](12_storage-architecture.md#126-interface-design)). The translator is pure (no state outside Postgres) and deterministic ŌĆö identical input produces identical OCSF output ŌĆö so the same chain hash can be recomputed by an external auditor from either the OCSF record plus `unmapped.lenny_chain` extension or the raw canonical tuple. The translator version and OCSF wire version are surfaced on every response envelope.

**Checkpoint Atomicity:** A checkpoint is an atomic unit comprising a workspace snapshot (tar of `/workspace/current`), a session file snapshot (copy of `/sessions/` contents), and a checkpoint metadata record (generation, timestamp, pod state). If either snapshot fails, the entire checkpoint is discarded ŌĆö partial checkpoints are never stored as valid checkpoints. The metadata record in Postgres references both artifacts and is written only after both are successfully uploaded to MinIO. **Exception ŌĆö preStop timeout partial manifest:** When an eviction checkpoint exceeds the preStop tiered cap ([Section 10.1](10_gateway-internals.md#101-horizontal-scaling)) and the upload is incomplete, the gateway writes a *partial checkpoint manifest* (not a full checkpoint record) to Postgres flagged with `partial: true`. This record is not a valid checkpoint; it is a recovery aid that tracks the indexed chunk objects (`partial-{n}.{chunk_encoding}` ŌĆö i.e., `partial-{n}.tar` or `partial-{n}.tar.gz` matching the manifest's `chunk_encoding` field, see [┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) Partial manifest on checkpoint timeout) that were successfully committed under a dedicated prefix, enabling partial workspace reconstruction on resume. The chunked-object storage model ŌĆö separate `PutObject`-committed chunks rather than S3 multipart upload parts ŌĆö is defined in [┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) Partial manifest on checkpoint timeout. This exception does not override the normal atomicity guarantee for non-eviction checkpoints.

**Partial checkpoint manifest cleanup:** Partial checkpoint manifest records and their referenced chunk objects have a defined cleanup path to prevent permanent orphaning. On resume ŌĆö whether the partial reconstruction succeeds or fails ŌĆö the gateway MUST delete every chunk object listed under the manifest's `partial_object_key_prefix` via per-key `DeleteObject` calls, then **soft-delete** the Postgres row by issuing `UPDATE ... SET deleted_at = now() AT TIME ZONE 'UTC' WHERE ... AND deleted_at IS NULL`. Soft-delete (not hard `DELETE`) is required so that idempotent re-runs are safe: a stale-leader retry, a crash-resumed resume path, or the ┬¦12.5 backstop sweep that races the primary cleanup all observe `rows_affected == 0` on the second writer and correctly skip any side effects, matching the same `deleted_at IS NULL` monotonicity guard used for `artifact_store` rows in [┬¦12.5 GC concurrency model rule 6](12_storage-architecture.md#125-artifact-store). MinIO chunk deletion remains a hard-delete batch ŌĆö MinIO's delete-on-absent semantics make it independently idempotent on repeated keys. `AbortMultipartUpload` is not used on this path: partial chunks are committed as independent objects via single-part `PutObject` ([┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) Partial manifest on checkpoint timeout), so there is no open multipart upload to abort. This cleanup is executed as part of the resume path in both the success and failure branches, so the manifest is consumed exactly once in the common case. Rows whose `deleted_at` is older than the `artifact_store` tombstone retention window are hard-pruned by the same sweep that prunes `artifact_store` rows ([Section 12.5](12_storage-architecture.md#125-artifact-store)) ŌĆö the partial-manifest row follows the identical lifecycle as any other GC-managed artifact row. If the MinIO delete call fails (e.g., transient MinIO unavailability), the gateway retries with the same exponential backoff as other artifact deletions; any manifest that escapes this path (e.g., gateway crash before cleanup, or a session that is never resumed before its resume window expires) is picked up by the GC backstop sweep ([Section 12.5](12_storage-architecture.md#125-artifact-store) GC concurrency model rule 6). The `lenny_partial_manifest_cleanup_total` counter (labeled by `outcome: success|failed_deleted|gc_collected`) tracks cleanup disposition for observability.

**Checkpoint quiescence strategy by level:**

- **Full-level runtimes (lifecycle channel):** Cooperative `checkpoint_request`/`checkpoint_ready` handshake via the lifecycle channel (see [Section 4.7](#47-runtime-adapter)) is the only mechanism that produces consistent checkpoints under all isolation profiles (runc, gVisor, Kata). The adapter sends `checkpoint_request`, the runtime quiesces and replies `checkpoint_ready`, snapshots are captured, and the adapter resumes the runtime with `checkpoint_complete`.
- **Basic / Standard-level runtimes (no lifecycle channel):** Best-effort snapshot without pausing. Checkpoint tagged `consistency: best-effort`. Deployers should expect minor workspace inconsistencies on resume. Consistent checkpoints require Full-level.
- **Embedded adapter mode only:** `SIGSTOP`/`SIGCONT` may be used when the adapter is embedded in the same container as the agent (not the default sidecar model). In the default sidecar model the adapter runs in a separate container, so the two processes do not share a PID namespace and the adapter cannot signal the agent; the embedded path is the only one where SIGSTOP/SIGCONT works. Cross-container signaling via `shareProcessNamespace: true` is explicitly prohibited by the blanket pod-security rule in [Section 13.1](13_security-model.md#131-pod-security), so there is no supported sidecar-mode variant of this path. This is **not supported** under gVisor or Kata ŌĆö signal-based checkpointing under sandboxed runtimes is unsupported; use the lifecycle channel instead. **Tool-call consistency limitation:** SIGSTOP checkpoints do not provide tool-call consistency guarantees. If the agent is mid-way through a tool call execution (e.g., a filesystem write, an HTTP request, or an LLM API call) when SIGSTOP fires, the tool call is frozen mid-execution. On SIGCONT, the tool call resumes with potentially stale or inconsistent state (half-written files captured in the workspace snapshot, timed-out HTTP connections). This is an inherent limitation of the SIGSTOP path ŌĆö unlike the Full-level lifecycle channel path, there is no quiescence handshake to wait for in-flight tool calls to settle. For periodic checkpoints, embedded adapter mode should prefer the lifecycle channel when available, falling back to SIGSTOP only when the runtime does not implement the lifecycle channel or for eviction checkpoints where time constraints preclude cooperative quiescence.

**Checkpoint timeout and recovery:** All checkpoint paths enforce a 60-second checkpoint timeout measured from the initial quiescence request to completion. For the **Full-level lifecycle channel path**: if the runtime sends `checkpoint_ready` but does not receive `checkpoint_complete` within 60 seconds, it MUST autonomously resume normal operation and log a `checkpoint_timeout` warning. This protects against adapter crashes or network partitions during the snapshot phase. For the **embedded adapter path**: the adapter MUST issue `SIGCONT` in a deferred cleanup handler (Go `defer` or equivalent) so that `SIGCONT` is sent on all exit paths ŌĆö including panics and crashes. Additionally, a 60-second watchdog timer starts when `SIGSTOP` is sent; if the checkpoint does not complete within that window, `SIGCONT` is sent unconditionally, the checkpoint is marked failed, and the adapter sets `checkpointStuck = true` immediately on failure (see liveness integration below).

**SIGCONT confirmation.** The embedded adapter runs in the same container as the agent process and reads `/proc/<adapter_self_view>/<pid>/stat` directly ŌĆö this does NOT require `shareProcessNamespace: true` since same-container processes already share the container's PID namespace. After sending `SIGCONT`, the adapter MUST confirm that the agent process has actually left the stopped state before considering the resume complete. The adapter polls `/proc/{pid}/stat` up to 5 times with 100 ms intervals; it reads the process state field (field 3 in `/proc/{pid}/stat`) and waits until it is no longer `T` (stopped) or `t` (tracing stop). If the process state remains `T`/`t` after all 5 retries (500 ms total), the adapter sets `checkpointStuck = true` immediately ŌĆö it does not wait for the watchdog timeout ŌĆö and the `/healthz` liveness endpoint returns HTTP 503, triggering Kubernetes pod restart. This closes the window where `SIGCONT` is successfully sent by the OS but the process remains pathologically stopped (e.g., due to kernel scheduling anomalies, cgroup freezer interference, or a secondary `SIGSTOP` from a tracing tool). If the adapter is ever deployed in a sibling container (future design), `shareProcessNamespace` MUST be explicitly allow-listed in the admission webhook (see [Section 13.1](13_security-model.md#131-pod-security) for the current blanket prohibition) with documented threat-model justification. On non-Linux hosts where `/proc/{pid}/stat` is unavailable, the adapter skips polling, logs a `sigcont_confirmation_unavailable` warning, and proceeds ŌĆö the watchdog timer remains the backstop.

**Checkpoint abort cleanup:** When a checkpoint is aborted ŌĆö whether by watchdog timeout, adapter crash, or any other failure ŌĆö the adapter MUST delete any partially uploaded MinIO objects for that checkpoint attempt. For multipart uploads, the adapter calls the S3 `AbortMultipartUpload` API; for single-part uploads that may have completed server-side, the adapter issues a `DeleteObject` call for the checkpoint's object key. The checkpoint metadata record in Postgres is never written for an aborted checkpoint (per the atomicity guarantee above), so no Postgres cleanup is needed. This cleanup is best-effort: if the delete call itself fails (e.g., MinIO is unreachable), the orphaned partial object is harmless (it cannot be mistaken for a valid checkpoint because no metadata references it) but consumes storage. The `lenny_checkpoint_orphaned_objects_total` metric (counter, labeled by `pool` and `trigger`) is incremented when cleanup fails, enabling operators to run periodic garbage collection sweeps against MinIO for unreferenced checkpoint objects. Any Go `recover()` call wrapping the checkpoint goroutine MUST either re-panic or explicitly set the liveness endpoint to unhealthy if `SIGCONT` cannot be confirmed sent ŌĆö silently recovering a checkpoint goroutine panic without signaling unhealthiness is forbidden, as it leaves the agent process permanently frozen while the liveness probe continues to pass.

**Embedded adapter liveness integration:** In embedded adapter mode, the `/healthz` liveness endpoint MUST incorporate checkpoint state. The adapter maintains a shared atomic flag (`checkpointStuck`) that is set in two cases: (a) by the watchdog timer when a `SIGSTOP` has been held for more than `watchdogTimeoutSeconds` (default 60s) without checkpoint completion, and (b) immediately when SIGCONT confirmation polling (see above) exhausts all 5 retries without observing the process leaving the stopped state. Setting `checkpointStuck` immediately on any SIGCONT confirmation failure avoids waiting for the full 60-second watchdog timeout in cases where the process is verifiably still frozen. When `checkpointStuck` is set, the `/healthz` endpoint returns HTTP 503 with `{"status": "unhealthy", "reason": "checkpoint_stuck"}`, causing the liveness probe to fail and Kubernetes to restart the pod. This closes the deadlock window where the adapter's HTTP server remains healthy while the agent process is permanently frozen ŌĆö the liveness probe now actively detects stuck checkpoints rather than relying solely on adapter process crashes. The `checkpointStuck` flag is cleared on successful checkpoint completion or after SIGCONT is confirmed sent (process state transitions out of `T`/`t`). If the embedded adapter process itself crashes entirely while the agent is SIGSTOPped (not a recovered goroutine panic but a full process exit), the agent process remains stopped; the pod's liveness probe will fail (since the adapter process is gone), Kubernetes will restart the pod, and the session resumes from the last successful checkpoint per [Section 7.2](07_session-lifecycle.md#72-interactive-session-model).

**Checkpoint duration SLO and workspace size impact:** Checkpoint duration is dominated by workspace tar creation and MinIO upload. Target SLO: P95 checkpoint duration < 2 seconds for workspaces Ōēż 100MB. Expected scaling: ~1 second per 100MB on typical node-local SSD with gigabit-class MinIO connectivity; 500MB workspaces may reach 5-10 seconds. For **Basic/Standard-level** (best-effort, no pause): duration affects only checkpoint freshness, not agent responsiveness. For **Full-level** (cooperative handshake): the runtime is quiesced during the snapshot phase, so checkpoint duration directly impacts agent pause time ŌĆö deployers with large workspaces should monitor the `lenny_checkpoint_duration_seconds` histogram and consider workspace hygiene (`.lennyignore` excludes, smaller working sets). For **embedded adapter mode** (SIGSTOP): the agent is fully frozen for the entire checkpoint duration, making this the most latency-sensitive path. The Phase 2 startup benchmark harness ([Section 18](18_build-sequence.md)) must include a **checkpoint duration benchmark** that measures end-to-end checkpoint time across workspace sizes (10MB, 100MB, 500MB) and storage backends, and validates the < 2s SLO for Ōēż 100MB workspaces. Incremental checkpoints (diffing against the previous snapshot) are deferred but noted as the primary mitigation if the SLO cannot be met at larger workspace sizes.

**Hard workspace size limit.** To prevent unbounded agent quiescence under Full-level or SIGSTOP checkpoint paths, `/workspace/current` (and `/workspace/slots/{slotId}/current/` in concurrent-workspace mode) MUST carry a hard `emptyDir.sizeLimit` set to `workspaceSizeLimitBytes` (pool-level configuration, default: 512Mi). This limit is enforced by the kubelet OOM eviction path ŌĆö if disk usage under the emptyDir volume exceeds the limit, the pod is evicted. This provides an unconditional hard cap and prevents a runaway workspace from making Full-level checkpoint quiescence unacceptably long. The adapter additionally performs a **pre-checkpoint workspace size probe** before initiating the quiescence handshake: it stats the total size of `/workspace/current` (or the slot-specific path). If the measured size exceeds `workspaceSizeLimitBytes`, the checkpoint is aborted immediately (without quiescing the runtime), the `lenny_checkpoint_size_exceeded_total` counter is incremented (labeled by `pool` and `level`), a `WorkspaceSizeExceeded` warning is logged, and the session emits a `checkpoint.skipped` event with `reason: "workspace_size_limit"` so the client is aware. The `lenny_checkpoint_duration_seconds` histogram (labeled by `pool`, `level`, and `trigger`) MUST be exported for all checkpoint completions (success and failure). Alert rule `CheckpointDurationHigh` ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when the P95 of `lenny_checkpoint_duration_seconds` for Full-level or embedded-adapter pools exceeds 2.5 seconds over a 5-minute window (25% headroom above the 2s SLO for Ōēż100MB workspaces). Quiescence time scales linearly with workspace size ŌĆö the direct relationship is: `expected_quiescence_seconds Ōēł workspace_bytes / (100 ├Ś 1024 ├Ś 1024)` ŌĆö deployers should set `workspaceSizeLimitBytes` to align with their acceptable agent pause budget.

**Checkpoint freshness SLO:** Every active session MUST have a successful checkpoint recorded within the last `periodicCheckpointIntervalSeconds` (default: 600s / 10 minutes). The gateway enforces this by scheduling periodic checkpoints for active sessions. The `lenny_checkpoint_stale_sessions` gauge (per pool/level, reported by the gateway) counts the number of active sessions whose last checkpoint age exceeds the interval. A `CheckpointStale` warning alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when any pool has stale sessions for > 60s. This SLO bounds the maximum workspace state that can be lost if an eviction checkpoint fails ŌĆö any session that has met the SLO loses at most 10 minutes of workspace changes, not unbounded progress.

**Periodic checkpoint scheduling:** The gateway schedules periodic checkpoints for all active sessions using a configurable interval (`periodicCheckpointIntervalSeconds`, default: 600s). Periodic checkpoints are best-effort (see non-eviction path below) ŌĆö they do not block agent progress. Failed periodic checkpoints are logged and retried on the next interval; they do not affect session state. The gateway tracks `last_successful_checkpoint_at` on the session record in Postgres, updated on every successful checkpoint regardless of trigger (periodic, eviction, pre-drain). **Jitter:** To prevent thundering-herd checkpoint storms (all sessions started in the same burst aligning their periodic checkpoints at the same wall-clock second), each session's first periodic checkpoint is scheduled at `periodicCheckpointIntervalSeconds + random(0, periodicCheckpointIntervalSeconds ├Ś periodicCheckpointJitterFraction)` seconds after session start. `periodicCheckpointJitterFraction` (default: 0.2, range: 0.0ŌĆō1.0, configurable via `gateway.periodicCheckpointJitterFraction` Helm value) spreads the initial checkpoint uniformly across a 120s window (at the default 600s interval), preventing correlated burst patterns at Tier 3 scale. Subsequent checkpoints are scheduled at the fixed interval from the previous checkpoint (no additional jitter), so sessions naturally desynchronize after the first cycle.

**Checkpoint storage failure:** MinIO uploads during checkpoint are retried with exponential backoff. The retry budget differs by checkpoint trigger:

- **Non-eviction checkpoints** (periodic scheduled checkpoints, pre-scale-down checkpoints): Retried with exponential backoff (initial 200ms, factor 2x, up to ~5 seconds total). If all upload retries fail, the adapter resumes the agent immediately ŌĆö via `checkpoint_complete` on the lifecycle channel (Full-level), `SIGCONT` (embedded adapter), or no-op (Basic/Standard-level best-effort). The failed checkpoint is discarded. The adapter logs the failure and increments the `lenny_checkpoint_storage_failure_total` metric (counter, labeled by `pool`, `level`, and `trigger`). The next scheduled checkpoint retries normally.
- **Eviction checkpoints** (preStop hook): Retried with exponential backoff (initial 500ms, factor 2x, capped at 5s per attempt, up to **30 seconds total**). The eviction retry budget is deliberately longer than non-eviction checkpoints because (a) the pod is terminating regardless ŌĆö there is no agent to resume, so retry cost is just wall-clock time, (b) `terminationGracePeriodSeconds` (240s at Tier 1/2, 300s at Tier 3 ŌĆö see [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)) provides ample headroom beyond the combined MinIO and Postgres retry budgets, and (c) transient MinIO unavailability (network blip, brief leader election) typically resolves within 10-20 seconds. If all retries fail, the gateway falls back to writing a **minimal session state record** directly to Postgres (see below). A `CheckpointStorageUnavailable` critical alert fires (see [Section 16.5](16_observability.md#165-alerting-rules-and-slos)). The adapter also increments `lenny_checkpoint_eviction_fallback_total` (counter, labeled by `pool` and `had_prior_checkpoint: true|false`) to distinguish evictions that fell back to Postgres with a prior full checkpoint from those without one ŌĆö the latter represent sessions where workspace state is permanently lost. If a previous successful full checkpoint exists, the session is resumed from that checkpoint per [Section 7.2](07_session-lifecycle.md#72-interactive-session-model). If only the minimal state record exists, the session is resumed on a fresh pod with conversation context but without workspace files ŌĆö the client receives a `session.resumed` event with `resumeMode: "conversation_only"` and `workspaceLost: true` so it can take appropriate action (re-upload files, restart workspace setup, etc.).

**Eviction fallback: Postgres minimal state record.** When an eviction checkpoint cannot be persisted to MinIO, the gateway writes a minimal state record to the `session_eviction_state` table in Postgres. This record is intentionally small ŌĆö it contains only what is needed to resume a conversation on a fresh pod without the workspace:

- `session_id` ŌĆö the session identifier
- `recovery_generation` ŌĆö the session recovery generation counter (tracks pod recoveries; used to identify which generation of the session this eviction state corresponds to)
- `coordination_generation` ŌĆö the coordinator generation counter (used for coordinator fencing on resume)
- `conversation_cursor` ŌĆö the last event cursor from the EventStore (allows replay of conversation history from the event log)
- `last_message_context` ŌĆö the last gateway-persisted prompt/response pair stored as a **MinIO object key** (`/{tenant_id}/eviction/{session_id}/context`) when the context exceeds 2KB; for contexts Ōēż 2KB the value is stored inline as plain TEXT directly in the column. This two-tier approach keeps the Postgres row small (Ōēż 2KB inline, or a short object key), avoiding TOAST entirely and honoring the "never Postgres for blobs" principle ([Section 12.1](12_storage-architecture.md#121-design-principle)). On MinIO unavailability during a fallback write, the context is truncated to 2KB and stored inline with a `context_truncated: true` flag on the row so the runtime can detect partial context on resume. The original context may reach up to 64KB before truncation.
- `evicted_at` ŌĆö timestamp of the eviction event
- `workspace_lost` ŌĆö boolean, always `true` for this record

This record is written in a single Postgres transaction. It does not store workspace file contents (no blobs, no TOAST). The workspace tar is not stored in Postgres ŌĆö this fallback provides conversation continuity, not workspace recovery. On resume from minimal state, workspace materialization is skipped and the pod starts with an empty workspace; the client is notified to re-upload files if needed.

**Postgres fallback retry during failover.** The Postgres write is retried with exponential backoff (initial 500ms, factor 2x, capped at 5s per attempt) for up to **60 seconds total**. This retry budget covers the managed Postgres failover window (RDS Multi-AZ, Cloud SQL HA: typically 15ŌĆō30s) that can prevent the first write attempt from succeeding. The 60s budget fits comfortably within `terminationGracePeriodSeconds` (240s at Tier 1/2, 300s at Tier 3), leaving ample time for stream drain after the Postgres write completes or the budget is exhausted. If all Postgres retry attempts are exhausted, the gateway proceeds to the total-loss path below.

**MinIO object key logging for manual recovery.** Before entering the total-loss path, the gateway logs a `WARN`-level structured message that includes the MinIO object keys for any partial-checkpoint chunk objects (`partial-{n}.{chunk_encoding}` ŌĆö `partial-{n}.tar` when `chunk_encoding = tar`, `partial-{n}.tar.gz` when `chunk_encoding = tar.gz`, see [┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) Partial manifest on checkpoint timeout) that were successfully uploaded before MinIO became unavailable. Fields: `session_id`, `tenant_id`, `generation`, `chunk_encoding` (`tar` or `tar.gz` ŌĆö carried explicitly so the operator knows which decode pipeline to apply without having to infer it from the object-key suffix), `committed_minio_keys: [...]` (array of object key strings, may be empty if no chunks were committed), `minio_error` (error summary). This allows operators to perform manual workspace reconstruction ŌĆö concatenating the listed chunks in index order and feeding them into the encoding-appropriate pipeline (`gzip -dc | tar -x` for `tar.gz`; `tar -x` for `tar`), per the [┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) reassembly procedure ŌĆö if Postgres eventually recovers and the session record can be written by hand. The decompression step MUST be selected from the logged `chunk_encoding` field, not from the object-key suffix. The `lenny_checkpoint_eviction_partial_keys_logged_total` counter (labels: `pool`, `keys_committed: 0|1+`) tracks how often partial key sets were logged, distinguishing total-MinIO-failure (no keys committed) from partial-upload scenarios.

The `session_eviction_state` table is cleaned up when the session reaches a terminal state. If `last_message_context` was stored as a MinIO object key (context > 2KB path), the gateway deletes the corresponding MinIO object (`/{tenant_id}/eviction/{session_id}/context`) atomically as part of the same terminal-state cleanup sequence ŌĆö the object deletion is issued before the Postgres row is removed. Failures to delete the MinIO object are retried with the same backoff as other artifact deletions; any outstanding objects that escape this path are picked up by the GC sweep ([Section 12.5](12_storage-architecture.md#125-artifact-store)).

**Total-loss path: MinIO and Postgres both unavailable during eviction.** When all Postgres retry attempts for the minimal state write are exhausted (e.g., Postgres failover exceeds the 60s retry budget, or a network partition persists throughout the grace period), all recovery paths are exhausted and the session is unrecoverable. In this scenario the gateway MUST:

1. Emit a `session.lost` event on the session's event stream with `reason: "eviction_total_loss"` and fields `session_id`, `tenant_id`, `generation`, `evicted_at`, and `minio_error` / `postgres_error` (error summaries for diagnostic context). The event is delivered on a best-effort basis ŌĆö the gateway publishes it to the event stream before the preStop hook exits; if the event stream itself is unavailable the emission is skipped (logged only), as there is no safe durable target remaining.
2. Increment the `lenny_session_eviction_total_loss_total` counter (labels: `pool`, `had_prior_checkpoint: true|false`). This counter is distinct from `lenny_checkpoint_eviction_fallback_total` ŌĆö the latter fires on MinIO failure (Postgres fallback attempted); the former fires only when both stores fail and the session is irrecoverably lost. The `had_prior_checkpoint` label allows operators to distinguish sessions that had a previous durable checkpoint (resumable from an earlier snapshot) from those with no prior checkpoint (complete data loss).
3. Log a `CRITICAL`-level structured message including `session_id`, `tenant_id`, `generation`, `evicted_at`, and the errors from both the MinIO retry exhaustion and the Postgres write failure.

The `SessionEvictionTotalLoss` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires immediately when `lenny_session_eviction_total_loss_total` increments ŌĆö any value above zero is an actionable signal. Note that the pre-drain MinIO health check webhook ([Section 12.5](12_storage-architecture.md#125-artifact-store)) prevents most planned drain-triggered instances of this scenario by blocking drains when MinIO is unhealthy; the total-loss path is most likely to occur during spontaneous node failures (not planned drains) or when the forced-drain override (`lenny.dev/drain-force: "true"`) is used while both stores are degraded.

**Storage quota accounting for eviction context objects.** When the gateway successfully writes an eviction context object to MinIO (the context > 2KB path), it MUST insert an `artifact_store` row for that object in the **same Postgres transaction** that writes the `session_eviction_state` row. The `artifact_store` row records `artifact_size_bytes` equal to the confirmed MinIO object size (bytes actually written), `object_key = /{tenant_id}/eviction/{session_id}/context`, and `artifact_type = eviction_context`. After both rows are durably committed, the gateway follows the standard post-upload increment ([Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas) step 2) and increments the tenant's Redis `storage_bytes_used` counter by the confirmed object size. This ensures eviction context bytes are included in the per-tenant quota counter and in the Postgres sum used to rehydrate Redis on restart. The GC-triggered decrement ([Section 11.2](11_policy-and-controls.md#112-budgets-and-quotas) step 3) and the GC sweep's cleanup path ([Section 12.5](12_storage-architecture.md#125-artifact-store) eviction context objects bullet) both operate on the `artifact_store` row, providing a consistent accounting lifecycle identical to other stored artifacts. On MinIO unavailability during the eviction fallback (context truncated to 2KB and stored inline with `context_truncated: true`), no MinIO object is written and therefore no `artifact_store` row is inserted and no quota increment is issued.

**Tenant isolation for `session_eviction_state`.** The table stores T3 Confidential data and carries a `tenant_id` column (non-null, indexed). It is included in the RLS policy scope identical to all other tenant-scoped tables ([Section 4.2](#42-session-manager)): an RLS policy filters rows by `current_setting('app.current_tenant', false)`, every access is wrapped in a transaction with `SET LOCAL app.current_tenant`, and the `lenny_tenant_guard` trigger covers this table. The `TestRLSTenantGuardMissingSetLocal` integration test covers `session_eviction_state` as part of its tenant-scoped table inventory.

### 4.5 Artifact Store

**Role:** Durable storage for workspace files and exports.

**Stores:**

- Original uploaded workspace files (the canonical "initial workspace")
- Sealed workspace bundles
- Exported file subsets for delegation
- Runtime checkpoints
- Large logs and artifacts

**Implementation:** MinIO (S3-compatible). Local disk for development mode. **Never** Postgres for blob storage ŌĆö the TOAST overhead and vacuum pressure degrade transactional workload performance. See [Section 12.5](12_storage-architecture.md#125-artifact-store) for retention policy.

**Tenant isolation:** All object paths in MinIO are prefixed with `/{tenant_id}/`. The `ArtifactStore` interface enforces this: every method that reads, writes, lists, or deletes objects validates that the supplied `tenant_id` matches the path prefix before issuing the S3 call. Paths that fail prefix validation are rejected without reaching MinIO. This guarantees tenant-scoped access at the interface level, prevents cross-tenant reads or writes regardless of caller bugs, and ensures that `DeleteByTenant(tenant_id)` ([Section 12.8](12_storage-architecture.md#128-compliance-interfaces)) maps to a deterministic prefix-scoped bulk delete (`/{tenant_id}/*`).

**Workspace lineage:** Each workspace snapshot is immutable and identified by a content-addressed hash (SHA-256 of the tar archive). The session record tracks lineage via a `parent_workspace_ref` field that records the session ID and artifact path of the workspace snapshot that seeded this session (from uploads or a derived session per [Section 7.1](07_session-lifecycle.md#71-normal-flow)). This field is a **metadata lineage pointer only** ŌĆö it does not create a shared MinIO object reference. When a session is derived ([Section 7.1](07_session-lifecycle.md#71-normal-flow)), the gateway copies the parent's workspace snapshot bytes into a new MinIO object scoped to the derived session's own path (`/{tenant_id}/{object_type}/{new_session_id}/...`). The derived session owns its artifact independently; GC on the parent's artifacts has no effect on the derived session's workspace. This enables lineage queries such as "which sessions were derived from this workspace?" and "what was the workspace history for this session?" Full workspace versioning (tracking incremental changes within a session) is not supported ŌĆö snapshots are full workspace captures at checkpoint or seal time.

### 4.6 Pod Lifecycle Controllers

Pod lifecycle management is split into two controllers with distinct responsibilities.

#### 4.6.1 Warm Pool Controller (Pod Lifecycle)

**Role:** Manages individual pod lifecycle, state transitions, and health.

**Responsibilities:**

- Maintain warm pods per pool (between `minWarm` and `maxWarm`)
- Manage pod state transitions via CRD status subresource
- Surface readiness status to gateway
- Claim/release/drain lifecycle
- Garbage-collect orphaned pods via owner references
- Handle node drain gracefully (checkpoint active sessions before eviction)
- Track certificate expiry on idle pods; proactively replace (drain and recreate) any idle pod whose certificate will expire within 30 minutes, preventing a claimed pod from having insufficient cert validity for a full session (see [Section 10.3](10_gateway-internals.md#103-mtls-pki) for cert TTLs)

**Implementation:** Built on **`kubernetes-sigs/agent-sandbox`** (launched at KubeCon Atlanta, November 2025). The upstream project solves the same Kubernetes-native pod lifecycle, warm pool, and claim management problem. Building on it means Lenny stops maintaining a Kubernetes controller for pod lifecycle and gets upstream community maintenance, Pod Snapshots on GKE (which directly helps with checkpointing), and a claim model designed by the Kubernetes SIG Apps community. Lenny's gateway, runtime adapter protocol, credential leasing, delegation, and MCP fabric remain entirely Lenny's own ŌĆö agent-sandbox has no opinion on any of those layers.

**Internal abstraction layer:** All Lenny components interact with pod lifecycle through two interfaces, never directly with agent-sandbox CRD types. Both embed a shared read-only `PoolReader`.

**`PoolReader`** (shared, embedded by both interfaces):

- `ListPools() ŌåÆ ([]PoolStatus, error)` ŌĆö pool health and capacity
- `GetPoolStatus(ctx, poolName) ŌåÆ (PoolStatus, error)` ŌĆö single pool status

**`PodLifecycleManager`** (gateway-facing, embeds `PoolReader`):

- `ClaimPod(ctx, poolName, sessionID, opts ClaimOpts) ŌåÆ (PodHandle, error)` ŌĆö acquire idle pod. `ClaimOpts` includes `requiresDemotion` (true when workspace plan includes `sdkWarmBlockingPaths` ŌĆö gateway signals that the claimed SDK-warm pod must be demoted before use), optional `priority`, optional `clusterID` (nullable, for future multi-cluster). `PodHandle` carries metadata: `warmMode`, `certExpiresAt`, `adapterEndpoint`.
- `ReleasePod(ctx, podHandle) ŌåÆ error` ŌĆö release pod after session ends
- `DrainPod(ctx, podHandle, checkpointFirst bool) ŌåÆ (DrainResult, error)` ŌĆö gracefully terminate. `DrainResult` includes retry state for seal-and-export hold semantics.
- `GetPodStatus(ctx, podHandle) ŌåÆ (PodStatus, error)` ŌĆö read pod state, health, cert expiry

**`PoolManager`** (controller-facing, embeds `PoolReader`):

- `ReconcilePool(ctx, poolConfig) ŌåÆ error` ŌĆö ensure pool matches desired state (minWarm/maxWarm)
- `ApplyPoolDefinition(ctx, poolDef) ŌåÆ error` ŌĆö CRUD for pool definitions (creates/updates/deletes `SandboxTemplate` and `SandboxWarmPool` CRDs)
- `ReplacePod(ctx, podHandle, reason string) ŌåÆ error` ŌĆö proactive replacement (cert expiry, health failure)
- `TransitionPodState(ctx, podHandle, from PodState, to PodState) ŌåÆ error` ŌĆö transition pod through state machine; validates legal transitions
- `GarbageCollect(ctx) ŌåÆ ([]OrphanResult, error)` ŌĆö orphan detection and cleanup. Covers orphaned `Sandbox` pods (no matching pool) and orphaned `SandboxClaim` resources (see orphan claim detection below).
- `ManageFinalizer(ctx, podHandle, action FinalizerAction) ŌåÆ error` ŌĆö add/remove finalizer; `FinalizerAction` is an enum (`Add`, `Remove`)
- `ManagePDB(ctx, poolName, config PDBConfig) ŌåÆ error` ŌĆö create/update/delete PodDisruptionBudget per pool
- `DrainPool(ctx, poolName, checkpointFirst bool) ŌåÆ error` ŌĆö drain all pods in a pool (admin API)
- `SetPoolCondition(ctx, poolName, condition PoolCondition, reason string) ŌåÆ error` ŌĆö set pool health status (e.g., Degraded when RuntimeClass missing)

Default implementations: `AgentSandboxPodLifecycleManager` and `AgentSandboxPoolManager`, which delegate to agent-sandbox CRDs.

Future extension points (reserved, not implemented now): `PodDebugger` interface for troubleshooting (exec, logs, describe); `SnapshotPod`/`RestoreFromSnapshot` for GKE Pod Snapshots integration; `clusterID` on `ClaimOpts` and `PoolStatus` for multi-cluster readiness.

This indirection means a breaking upstream change or a decision to replace agent-sandbox requires changing only the implementations behind these interfaces, not the gateway, scaling controller, or any other consumer.

**CRD mapping from agent-sandbox:**

**API group:** All four CRDs are registered under the `lenny.dev` API group (e.g., `sandboxes.lenny.dev`), not under the upstream `kubernetes-sigs/agent-sandbox` project's API group. This is a deliberate design choice: Lenny ships its own CRD manifests in `charts/lenny/crds/`, pins them to a specific schema version, and maintains full control over the API surface ŌĆö including validation rules, CEL admission policies, and webhook configurations ŌĆö independent of upstream release cadence. Re-registering under `lenny.dev` also ensures that the CRDs coexist cleanly with any standalone upstream installation and that the fallback plan (replacing agent-sandbox with custom controllers) requires no CRD migration.

| ----------------- | ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |

**ADR-007: `SandboxClaim` optimistic-locking and failover fencing verification ŌĆö Phase 1 blocking prerequisite.** Before Phase 1 implementation begins, verify that `SandboxClaim`'s optimistic-locking semantics match Lenny's claim approach and that the Kubernetes API server's `resourceVersion`-guarded update provides sufficient fencing during the 25s crash-case failover window. Document the result as ADR-007. If the semantics diverge or fencing proves insufficient, the `PodLifecycleManager.ClaimPod` implementation must compensate (e.g., by wrapping claims in a compare-and-swap loop at the gateway level). **Verification approach ŌĆö two required tests:**

1. **Concurrent-claim integration test.** Create a `SandboxClaim` resource with a `resourceVersion`-guarded update; race two concurrent claim attempts against the same `Sandbox`; confirm exactly one succeeds with HTTP 409 Conflict for the loser. If the 409 path works as expected, the ADR records "optimistic locking confirmed, no compensation needed." If it does not, the ADR must specify the compensating CAS loop design before Phase 1 proceeds.

2. **Chaos test (leader-kill acceptance test).** Run a chaos scenario that kills the WarmPoolController leader mid-claim at high concurrency (Ōēź 50 concurrent claim goroutines against a pool of 10 pods). Verify zero double-claims in the resulting `SandboxClaim` set. This test validates that any in-flight `resourceVersion`-guarded write from the killed leader is correctly rejected by the API server with HTTP 409 after the new leader begins claiming ŌĆö confirming that Kubernetes optimistic locking acts as a fencing mechanism even across the 25s failover window. ADR-007 must record the test results and explicitly accept or reject the "optimistic locking is sufficient fencing" hypothesis before Phase 1 proceeds.

**Why optimistic locking provides sufficient fencing:** The old leader's in-flight `PATCH`/`PUT` on a `SandboxClaim` carries the `resourceVersion` observed before it died. If the new leader (or any other gateway replica) has since written to the same resource, its `resourceVersion` has changed and the API server rejects the old leader's write with HTTP 409 ŌĆö the same conflict path as a concurrent gateway-vs-gateway race. The old leader's write cannot silently succeed; it either completes before the new leader touches the resource (in which case it is not a double-claim) or it arrives after and is rejected. If ADR-007's chaos test confirms this behavior holds under load, no additional fencing token or generation field is required.

**`SandboxClaim` admission webhook (double-claim prevention):** As a defense-in-depth backstop against any residual double-claim risk ŌĆö including the narrow failover window during which a new leader has not yet touched a `SandboxClaim` and an old leader's in-flight write could theoretically succeed ŌĆö a `ValidatingAdmissionWebhook` (`lenny-sandboxclaim-guard`) is deployed alongside the warm-pool controller. The webhook intercepts every `CREATE`, `PATCH`, and `PUT` operation on `SandboxClaim` resources in agent namespaces. The authoritative pod state machine lives on the referenced `Sandbox` CRD's `.status.phase` ([Section 6.2](06_warm-pod-model.md#62-pod-state-machine)), not on `SandboxClaim.status.phase` ŌĆö every phase check in this webhook reads `Sandbox.status.phase` via the claim's `.spec.sandboxRef`. For `PATCH`/`PUT`, the webhook queries the API server for the referenced `Sandbox` (via `.spec.sandboxRef`) and rejects the request if the `Sandbox.status.phase` is not `claimed` (i.e., the target pod no longer holds an active claim to this `SandboxClaim` ŌĆö e.g., it has been released back to `idle`, or has progressed into a terminal state such as `terminated`/`failed`). The rejection returns `403 Forbidden` with message `"SandboxClaim stale: referenced Sandbox <name> is in phase <phase>, not claimed; concurrent write rejected"`. This ensures that a stale in-flight write from a failed-over gateway or controller cannot mutate a `SandboxClaim` whose underlying pod is no longer bound to it. (Note: the webhook does not examine `SandboxClaim.status.phase` itself ŌĆö see the `SandboxClaim` status enumeration in [Section 4.6.3](#463-crd-field-ownership-and-write-boundaries), which is owned by the gateway and tracks session-binding state rather than pod lifecycle state.) For `CREATE`, the webhook queries the API server for any existing `SandboxClaim` whose `.spec.sandboxRef` matches the target `Sandbox`; if a non-terminal claim already exists, the creation is rejected with `403 Forbidden` and message `"SandboxClaim already exists for Sandbox <name>; concurrent claim rejected"`. This `CREATE`-time check is essential for the Postgres fallback claim path (see below), which creates a new `SandboxClaim` CRD after claiming a pod record in Postgres ŌĆö without it, the fallback path could race with the normal CRD-based claim path and produce duplicate claims for the same `Sandbox`. The webhook is deployed with `failurePolicy: Fail` (fail-closed), ensuring that a webhook outage blocks new claim operations rather than silently allowing them through. Because the webhook evaluates `Sandbox.status.phase` as persisted in etcd at admission time, it makes double-claim impossible regardless of timing: even if two writers race with the same `resourceVersion`, the second writer's request is rejected at admission before it reaches the API server's optimistic-lock check. The `lenny_sandboxclaim_guard_rejections_total` counter (labeled by `service_instance_id`) tracks admission rejections; a non-zero rate during normal operation indicates a claim-path bug. The webhook is deployed with `replicas: 2` and a `PodDisruptionBudget` (`minAvailable: 1`) matching the admission controller HA requirements in [Section 17.2](17_deployment-topology.md#172-namespace-layout). Alert `SandboxClaimGuardUnavailable` (Critical) fires when the webhook has been unreachable for more than 30 seconds ŌĆö see [Section 16.5](16_observability.md#165-alerting-rules-and-slos). The webhook is deployed as part of the Helm chart under `templates/admission-policies/` and is covered by the admission policy integration test suite (`tests/integration/admission_policy_test.go`). It is included as a check in the `lenny-preflight` Job ([Section 17.6](17_deployment-topology.md#176-packaging-and-installation)).

**Pod claim mechanism:** Gateway replicas claim pods via `SandboxClaim` resources with optimistic locking ŌĆö exactly one gateway wins; all others receive a conflict and retry with a different idle pod from the pool. This keeps the controller off the claim hot path entirely ŌĆö pod-to-session binding is resolved at the API-server level with no single-writer bottleneck. `ClaimPod` implementations must use a `resourceVersion`-guarded compare-and-swap loop: on HTTP 409, re-read the current `Sandbox` state and retry with the updated `resourceVersion` (or select a different idle pod if the target pod is no longer available). This loop is the primary defense against both concurrent gateway races and stale in-flight writes from a failed controller replica.

**Leader election:** The WarmPoolController runs as a Deployment with 2+ replicas using Kubernetes Lease-based leader election with lease name `lenny-warm-pool-controller` (lease duration: 15s, renew deadline: 10s, retry period: 2s). The PoolScalingController uses a separate lease (`lenny-pool-scaling-controller`); see [Section 4.6.2](#462-poolscalingcontroller-pool-configuration). During failover, existing sessions continue unaffected; only new pod creation and pool scaling pause. **Failover duration depends on the trigger:** on clean shutdown (rolling update, voluntary termination), the outgoing leader releases its lease via `LeaderCallbacks.OnStoppedLeading`, reducing the gap to near-zero. On crash (process kill, node failure), the worst-case failover window is `leaseDuration + renewDeadline = 25s` ŌĆö the new leader cannot acquire the lease until the old lease expires after its full duration plus the renew deadline elapses without renewal. All `minWarm` formulas and capacity planning use the 25s crash-case as the sizing basis (see below and [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)).

**Controller crash during active scale-down drain ŌĆö no additional fencing required.** When the WarmPoolController crashes while a scale-down drain is in progress (i.e., some pods have already been transitioned to `draining` phase), the new leader does not require a generation-stamped drain operation record or any other fencing token. The reason is twofold:

1. **Idle pods being scale-down drained have no checkpoint.** Warm (idle) pods being retired due to a `minWarm` reduction carry no active session and no in-flight workspace checkpoint. The new leader's reconciler, on startup, re-lists all `Sandbox` resources and finds pods already in `draining` phase. It processes them idempotently ŌĆö advancing each toward deletion ŌĆö rather than re-issuing a fresh drain signal. Re-processing a pod already in `draining` is safe and produces the same outcome: pod termination. No double-drain hazard exists.

2. **Active-session checkpointing is fenced by `coordination_generation`.** For admin-initiated pool drains (`DrainPool` with `checkpointFirst: true`) where some pods carry active sessions, the checkpoint protocol is coordinated by the gateway, not the controller. Coordinator handoff ŌĆö including the fencing that prevents a stale coordinator from re-signaling an in-progress checkpoint ŌĆö is already handled by the `coordination_generation` CAS mechanism specified in [Section 10.1](10_gateway-internals.md#101-horizontal-scaling). The controller only removes the finalizer after the gateway has written the session to a terminal state; it does not directly drive checkpointing and therefore does not require its own drain fencing record.

The K8s Lease-based leader election guarantee (single active writer at any time) means the new leader reads the current authoritative `Sandbox.status.phase` from etcd before taking any action. Pods in `draining` are not re-targeted as `idle`; pods not yet in `draining` are not spuriously advanced. No additional `SandboxWarmPool.status` drain operation record is needed.

**Scaling:** Pools support `minWarm`, `maxWarm`, and an optional `scalePolicy` with time-of-day schedules or demand-based rules. Pools referencing a `preConnect`-capable runtime warm all pods to SDK-warm state; demotion to pod-warm happens on demand at claim time (see [Section 6.1](06_warm-pod-model.md#61-what-a-pre-warmed-pod-looks-like)). Low-traffic pools can scale to zero warm pods with documented cold-start latency as fallback.

**Idle cost visibility and scale-to-zero:** The metric `lenny_warmpool_idle_pod_minutes` (counter, labeled by pool and resource class) tracks cumulative idle pod-minutes, letting deployers estimate warm pool cost from their monitoring stack. Pools support `minWarm: 0` for off-hours via time-of-day rules in `scalePolicy`: `scaleToZero: { schedule: "0 22 * * *", resumeAt: "0 6 * * *" }` sets `minWarm: 0` during the window; sessions arriving in zero-warm periods incur cold-start latency. `scaleToZero` disabled by default for `type: mcp` pools (deployer opt-in required). **Cron timezone:** Both `schedule` and `resumeAt` are interpreted as **UTC by default**. An optional `timezone` field (IANA timezone string, e.g., `"America/New_York"`) overrides the default: `scaleToZero: { schedule: "0 22 * * *", resumeAt: "0 6 * * *", timezone: "Europe/Berlin" }`. The `timezone` field is parsed by the PoolScalingController's embedded cron scheduler (Go cron library); any IANA timezone string is accepted on all Kubernetes versions. A `WarmPoolIdleCostHigh` warning alert fires when idle pod-minutes exceed a deployer-configured threshold over a 24 h window (see [Section 16.5](16_observability.md#165-alerting-rules-and-slos)).

**CRD validation:** All CRDs include OpenAPI schema validation with CEL rules to catch common misconfigurations at admission time. Key rules: `minWarm <= maxWarm`, `maxWarm > 0`, valid RuntimeClass reference format, resource class values within the allowed set, `maxSessionAge > 0`, and `taskPolicy.maxTasksPerPod > 0` when `executionMode: task`. Malformed specs are rejected by the API server before reaching the controller, preventing reconciliation loops on invalid input. The controller also validates at reconciliation time as defense-in-depth.

**Controller failover and warm pool sizing:** During a leader-election failover (near-zero on clean shutdown; up to 25s on crash ŌĆö see leader election paragraph above), the controller cannot create new pods or reconcile pool scaling. If the warm pool is undersized, this pause can exhaust available pods. To prevent this, `minWarm` should account for both the failover window and burst absorption: `minWarm >= peak_claims_per_second * (failover_seconds + pod_warmup_seconds) + burst_p99_claims * pod_warmup_seconds`. **Use `failover_seconds = 25` (the crash-case worst case) for all sizing calculations.** The burst term ensures the pool is not exhausted by demand spikes that arrive faster than replacement pods can become ready (see [Section 4.6.2](#462-poolscalingcontroller-pool-configuration) for full formula and variable definitions). For example, at 2 claims/sec with a 25s failover pause, 10s pod warmup (pod-warm baseline: container pull + runtime startup, no SDK init), and a p99 burst rate of 4 claims/sec: `minWarm >= 2 * (25 + 10) + 4 * 10 = 110`. For SDK-warm pools, `pod_warmup_seconds` additionally includes SDK initialization (typically 30ŌĆō90s); substitute the observed value in both terms. See [Section 4.6.2](#462-poolscalingcontroller-pool-configuration) for precise definitions and the `pod_startup_seconds` terminology note. Per-tier claim rate estimates and recommended minWarm values are in [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults). During the failover window, the gateway queues incoming pod claim requests for up to `podClaimQueueTimeout` (default: **60s**). If no pod becomes available before the timeout, the session creation fails with a retryable error so the client can back off and retry. Note: `podClaimQueueTimeout` (60s) provides a 35-second margin above the 25s worst-case failover window, accommodating API server slowness (elevated latency under load can extend effective failover by 5ŌĆō15s) and pod readiness probe delays. The previous default of 30s provided only a 5s margin, which was insufficient under API server pressure. As an early-warning mechanism, the `WarmPoolLow` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when available pods drop below 25% of `minWarm`, giving operators time to investigate before exhaustion occurs.

**Fallback claim path via Postgres:** If `ClaimPod` cannot find an available `Sandbox` in `idle` state via the Kubernetes API within `podClaimQueueTimeout`, the gateway attempts a **Postgres-backed fallback claim** before returning a retryable error. The fallback path queries the `agent_pod_state` table (a Postgres-side mirror of `Sandbox` CRD status, updated by the WarmPoolController on every state transition) using `SELECT ... FOR UPDATE SKIP LOCKED` to atomically claim an idle pod record, then creates the corresponding `SandboxClaim` CRD. This path is slower than the normal API-server path (one extra Postgres query + CRD create) but decouples claim availability from the API server's Endpoints/watch stream. The fallback is activated automatically when the API-server-based claim attempt exhausts `podClaimQueueTimeout`; no configuration is required. If the Postgres fallback also finds no idle pods (the warm pool is genuinely exhausted), session creation fails with `WARM_POOL_EXHAUSTED`. The `lenny_pod_claim_fallback_total` counter tracks fallback activations; a sustained non-zero rate indicates the primary claim path is degraded and operators should investigate API server health or warm pool sizing.

**Fallback preconditions (mirror freshness and admission reachability).** Because `agent_pod_state` is a read-optimized mirror and not the authoritative store in v1, the fallback path MUST gate activation on two preconditions. If either fails, the gateway skips the fallback and returns `WARM_POOL_EXHAUSTED` immediately rather than risking a stale-mirror double-claim:
1. **Mirror freshness.** The fallback is activated only when `lenny_agent_pod_state_mirror_lag_seconds` for the target pool is at or below `podClaimFallbackMaxMirrorLagSeconds` (default: **10s**; configurable via gateway flag). Above this threshold the mirror may reflect pods already claimed in etcd but not yet mirrored, so `SELECT ... FOR UPDATE SKIP LOCKED` could acquire a row-lock on a pod whose authoritative `Sandbox.status.phase` is already `claimed`. The lag is read from the same gauge described in [Section 10.1](10_gateway-internals.md#101-horizontal-scaling) "Mirror table staleness detection".
2. **Admission webhook reachability.** The `lenny-sandboxclaim-guard` CREATE-time check (deployed with `failurePolicy: Fail`) is the only defense against the fallback path racing with a concurrent CRD-based claim for the same `Sandbox`. Because the webhook callback traverses the Kubernetes API server, the fallback requires the API server to be reachable for admission calls even when it is unreachable (or degraded) for the gateway's own watch/list operations. The gateway probes API server reachability before initiating the fallback (a lightweight `GET /readyz` or equivalent); if the probe fails, the fallback is skipped. In practice the two API-server failure modes ŌĆö watch-stream degradation versus full unavailability ŌĆö are distinct, and the fallback is useful only for the former. The `lenny_pod_claim_fallback_skipped_total{reason}` counter tracks skip events with `reason` labels `mirror_stale` and `apiserver_unreachable`; a sustained non-zero rate on either reason indicates the degraded mode the fallback was designed to address has itself exceeded its safe operating envelope.

**WarmPoolController mirror reconciliation on recovery.** On startup and on every leader-election acquisition, the WarmPoolController re-lists all `Sandbox` resources in agent namespaces and re-derives the corresponding `agent_pod_state` rows (pod state, session binding, resource version, node name) with an updated `updated_at` timestamp. This is performed as a single bulk UPSERT keyed on `pod_id`, and rows whose `pod_id` no longer corresponds to any live `Sandbox` are deleted. The reconciliation establishes a steady-state invariant: after any WPC outage, the mirror converges to the authoritative etcd state before the controller resumes incremental state-transition writes. This invariant is distinct from the in-flight drain reconciliation covered in the "Controller crash during active scale-down drain" paragraph above (which addresses the `draining` phase only); mirror reconciliation covers the full `agent_pod_state` row set for all phases. Until the reconciliation completes, the `lenny_agent_pod_state_mirror_lag_seconds` gauge continues to reflect the pre-recovery lag, which ŌĆö combined with the fallback preconditions above ŌĆö keeps the Postgres fallback disabled for any pool whose lag has exceeded the threshold.

**Simultaneous controller failover:** The WarmPoolController and PoolScalingController use separate leases and may failover independently. If both crash simultaneously (e.g., both leaders co-located on a failed node): (1) pool replenishment pauses (WPC down) ŌĆö no new pods are created, (2) scaling intelligence pauses (PSC down) ŌĆö `minWarm`/`maxWarm` values are not updated from Postgres. The blast radius is additive but not multiplicatively worse: since the PSC only adjusts scaling parameters (which the WPC reads from CRDs), losing both controllers for 25s is equivalent to losing the WPC alone ŌĆö the stale CRD values from before the PSC crash remain valid and the WPC will reconcile against them once it re-elects. The `podClaimQueueTimeout` (60s) comfortably covers this scenario and provides 35s of additional margin for API server slowness. To prevent co-located leader failures, operators should use pod anti-affinity on controller deployments (see [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)) so WPC and PSC leaders run on different nodes.

**Controller leader election monitoring:** A `ControllerLeaderElectionFailed` critical alert fires when any controller's Lease has not been renewed within `leaseDuration` (15s) ŌĆö detectable via the `lenny_controller_leader_lease_renewal_age_seconds` gauge (one per controller). This gives operators early warning that failover is imminent or in progress before the lease fully expires. The alert auto-resolves when a new leader acquires the lease and begins renewing.

**API server rate limiting:** The controller uses two separate client-side rate limiter buckets to prevent pod creation starvation during scale-up:

- **Pod creation bucket:** token bucket at 20 QPS, burst 50. Dedicated to `Create` calls for new `Sandbox` pods, ensuring scale-up is never starved by status update traffic.
- **Status update bucket:** token bucket at 30 QPS, burst 100. Dedicated to `UpdateStatus` calls on `Sandbox` and `SandboxWarmPool` resources.

Both buckets are configurable via controller flags (`--create-qps`, `--create-burst`, `--status-qps`, `--status-burst`). All other API server requests (reads, deletions, finalizer updates) share the controller-runtime default rate limiter (10 QPS, burst 100).

During steady-state pool replenishment, pod creation is processed through the work queue with controlled concurrency. The work queue max depth is configurable (default: 500); if the queue exceeds this depth, new reconciliation events are dropped and a `lenny_controller_queue_overflow_total` metric is incremented. The number of concurrent reconciliation workers is configurable via `--max-concurrent-reconciles` (default: 1). For cold-start scenarios (initial pool fill from zero, cluster restart recovery), operators should increase concurrent workers to match the tier's parallelism needs ŌĆö see [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for per-tier recommendations. The rate limiter remains the throughput ceiling regardless of worker count, preventing API server or etcd overload even with multiple concurrent workers.

**Cold-start pool fill:** When a pool is created or recovers from zero (e.g., after a cluster restart or scale-from-zero resume), filling to `minWarm` takes time proportional to `minWarm / effective_create_throughput`. The `WarmPoolExhausted` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) is suppressed during an initial fill grace period (`--initial-fill-grace-period`, default: 120s from pool creation or controller startup) to avoid false-positive alerts during expected cold-start fill time. The metric `lenny_warmpool_fill_duration_seconds` (histogram, labeled by pool) tracks the time from pool creation to reaching `minWarm` ready pods, enabling operators to set SLO targets for cold-start recovery. See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for controller tuning recommendations.

**Re-activation grace period:** The fill grace period also applies whenever a pool's `minWarm` transitions from `0` to a positive value ŌĆö not just at initial pool creation. This covers experiment variant pool re-activations (`paused ŌåÆ active`, see [Section 10.7](10_gateway-internals.md#107-experiment-primitives)), scale-from-zero resume via `scaleToZero` schedule, and any operator-initiated `minWarm` increase from zero. When the WarmPoolController detects a `minWarm 0 ŌåÆ positive` transition, it resets the pool's `fill_grace_period_until` timestamp to `now() + --initial-fill-grace-period`. The `WarmPoolLow` alert is suppressed until the grace period expires, preventing alert storms on expected re-fill operations. Alert suppression does not affect pool fill metrics ŌĆö `lenny_warmpool_fill_duration_seconds` still records the actual fill duration.

**etcd write pressure at scale:** At high concurrent session counts, CRD status updates drive significant etcd write traffic. Estimated write rates by tier:

| Tier | Concurrent sessions | Estimated etcd writes/s (sustained) | Notes |
| ---- | ------------------- | ------------------------------------ | ----- |
| Tier 1 | 100 | ~8 | Well within shared etcd capacity |
| Tier 2 | 1,000 | ~80 | Shared etcd with `statusUpdateDeduplicationWindow: 500ms` |
| Tier 3 | 10,000 | ~800 (ceiling: ~120 QPS after dedup at 250ms) | Approaches shared-etcd practical limits; dedicated etcd cluster recommended ([Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)) |

The estimate assumes ~1 status write per pod per 2-minute lifetime (claim ŌåÆ active ŌåÆ released) plus warm-pool reconciliation writes. The status update rate limiter ([Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) controller tuning table: 120 QPS / burst 400 at Tier 3) and `statusUpdateDeduplicationWindow: 250ms` combine to reduce effective write rate; the ~120 QPS ceiling reflects the dedup-bounded steady-state, not the raw pre-dedup rate. At Tier 3, etcd p99 write latency (`etcd_disk_wal_fsync_duration_seconds`) should be monitored continuously ŌĆö if p99 exceeds 25ms, operators must either scale etcd resources, reduce `maxSessionsPerReplica`, or migrate to a dedicated etcd cluster. The `EtcdWriteLatencyHigh` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when p99 write latency exceeds this threshold. Mitigations: (1) The state machine uses 3 coarse label values instead of 10+ fine-grained labels, reducing label mutation frequency ([Section 6.2](06_warm-pod-model.md#62-pod-state-machine)). (2) Status updates are batched by the controller's work queue ŌĆö not every state transition is immediately written. (3) The dedicated status update rate limiter bounds the update rate while the separate pod creation bucket prevents creation starvation. (4) The `lenny_controller_workqueue_depth` metric (gauge labeled by `controller`, `queue` ŌĆö see [Section 16.1](16_observability.md#161-metrics)) tracks the instantaneous depth of the controller work queues ŌĆö a sustained high depth indicates the controller cannot process reconciliation events as fast as they arrive, which amplifies write bursts when the backlog finally drains. The `ControllerWorkQueueDepthHigh` alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when queue depth exceeds 50% of the configured `Work queue max depth` ([Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference)) for more than 2 minutes.

**etcd operational tuning:** Deployers must configure the following to maintain etcd health under sustained CRD churn:

- **Compaction:** Enable automatic compaction with `--auto-compaction-mode=periodic` and `--auto-compaction-retention=5m`. This prevents unbounded revision history growth from frequent status updates. Note: compaction mode and retention are etcd startup flags, not runtime-adjustable. Changing them requires an etcd member restart. On managed Kubernetes (EKS, GKE, AKS), the cloud provider controls etcd flags ŌĆö operators cannot change compaction settings directly. Self-managed clusters must perform rolling etcd member restarts (one member at a time, waiting for cluster health between restarts) to apply new values without downtime. See the etcd operations runbook ([Section 17.7](17_deployment-topology.md#177-operational-runbooks)) for the procedure.
- **Defragmentation:** Schedule periodic defragmentation during low-traffic windows (e.g., via a CronJob running `etcdctl defrag` on each member). Defrag reclaims free pages after compaction and should run at least once daily for clusters with high CRD write rates. **Operational caveat:** `etcdctl defrag` on a given member blocks all reads and writes to that member while defrag runs (typically 1ŌĆō30 seconds depending on database size). To avoid disrupting Kubernetes API availability, defrag one member at a time, starting with followers and finishing with the leader. Never run `etcdctl defrag --cluster` in production ŌĆö it defrags all members sequentially but provides no control over timing. On managed Kubernetes, the cloud provider handles defragmentation automatically; operators should not schedule their own defrag CronJobs. For Tier 3 deployments where "off-peak" is difficult to define (e.g., 10,000 concurrent sessions with global traffic), defrag should target the window with the lowest observed `etcd_server_proposals_committed_total` rate, or operators should use a dedicated etcd cluster (as recommended in [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults)) where defrag does not affect the main API server.
- **Quota monitoring:** Set `--quota-backend-bytes` (recommended: 8 GB) and alert when usage exceeds 80% (`etcd_server_quota_backend_bytes` minus `etcd_debugging_mvcc_db_total_size_in_bytes`). An `EtcdQuotaNearLimit` alert should fire at 80% to give operators time to defragment or increase quota before etcd enters alarm state. **Quota exhaustion recovery:** If etcd reaches its quota limit, it enters alarm state and refuses all write operations ŌĆö the Kubernetes API server becomes read-only for CRD operations, blocking pod claims, state transitions, and pool reconciliation (existing sessions continue as described in "etcd unavailability" below). Recovery requires: (1) defragment all members to reclaim space (`etcdctl defrag` on each member individually), (2) verify database size dropped below quota (`etcdctl endpoint status`), (3) disarm the alarm (`etcdctl alarm disarm`), (4) verify writes resume by checking API server health. On managed Kubernetes, quota exhaustion is a cloud-provider support escalation ŌĆö the operator cannot access etcd directly. See the etcd operations runbook ([Section 17.7](17_deployment-topology.md#177-operational-runbooks)) for the full procedure.
- **Snapshot frequency:** Ensure `--snapshot-count` is tuned for the write rate (default 100,000 is reasonable for most deployments; reduce to 10,000 for very high write rates to limit recovery time).

**etcd tuning topology matrix.** The degree to which Lenny can influence etcd tuning depends on whether the Kubernetes cluster is managed or self-managed. The table below clarifies responsibilities by topology:

| Tuning area | Managed K8s (EKS, GKE, AKS) | Self-managed (kubeadm, Talos, etc.) |
|---|---|---|

> **`statusUpdateDeduplicationWindow` semantics.** The `--status-update-dedup-window` controller flag (type: `time.Duration`, default: `500ms`) sets the **minimum interval between consecutive `UpdateStatus` writes for the same `Sandbox` resource**. When a reconciliation loop produces a status change within this window of the previous write for the same resource, the write is deferred until the window expires ŌĆö coalescing multiple rapid status transitions (e.g., `Pending ŌåÆ Creating ŌåÆ Running` within a few hundred milliseconds) into a single API server write. The deduplication is trailing: the last observed status in the window is always written (no status changes are lost, only intermediate writes are suppressed). This is the primary mechanism for reducing etcd write pressure on managed Kubernetes where compaction and quota-backend-bytes are not operator-controllable. Per-tier recommended values are in [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) controller tuning table.

**Summary:** On managed Kubernetes, the only etcd levers available to Lenny operators are Lenny-side write rate controls (row 8 above). All low-level etcd configuration is the cloud provider's responsibility. On self-managed clusters, operators are responsible for the full set of tuning parameters. The etcd operations runbook ([Section 17.7](17_deployment-topology.md#177-operational-runbooks)) covers only the self-managed procedures; managed Kubernetes operators should use the provider escalation path.

See [Section 17.8](17_deployment-topology.md#178-capacity-planning-and-defaults) for controller tuning recommendations.

**etcd unavailability (degraded mode):** etcd is a critical dependency for the Kubernetes API server. When etcd is unavailable, the API server cannot process CRD writes, which means pod claims (`SandboxClaim` creation), pod state transitions (`Sandbox` status updates), and pool reconciliation all freeze. The platform enters a degraded mode with the following behavior:

- **Existing sessions continue unaffected.** Active session pods are already running and communicate with the gateway via gRPC ŌĆö they do not depend on etcd for ongoing operation. Session state is persisted to Postgres, not etcd.
- **New session creation is rejected.** The gateway detects API server unavailability when `ClaimPod` fails and returns a retryable error to the client (same mechanism as the `podClaimQueueTimeout` path described above). Clients back off and retry.
- **Pool replenishment is frozen.** The warm pool controller cannot create new pods or update CRD status. Once etcd recovers, the controller's work queue replays pending reconciliations and the pool self-heals.
- **Alerting:** An `EtcdUnavailable` critical alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when the API server reports etcd connection failures (`etcd_request_duration_seconds` errors or `apiserver_storage_errors_total` sustained > 0) for more than 15 seconds. Operators should treat this as a cluster-wide incident.

**Disruption protection for agent pods:** The primary protection against voluntary disruption (node drains, cluster upgrades) is a **preStop hook** on every agent pod that triggers a checkpoint via the runtime adapter's `Checkpoint` RPC before allowing termination. The pod's `terminationGracePeriodSeconds` is set high enough (default: 120s) to give the checkpoint time to complete and be persisted to object storage. For active pods whose sessions have been checkpointed (or that have no session at all), the session retry mechanism described in [Section 7.2](07_session-lifecycle.md#72-interactive-session-model) handles resumption on a replacement pod.

The warm pool controller can optionally create a PDB **per `SandboxTemplate`** for warm (unclaimed) pods. The PDB MUST use `maxUnavailable: 1` rather than `minAvailable: minWarm`. Using `minAvailable: minWarm` causes a deadlock at steady state: when exactly `minWarm` idle pods exist (the normal operating condition), the PDB allows zero evictions, permanently stalling node drains. `maxUnavailable: 1` allows voluntary disruptions one pod at a time while limiting simultaneous impact. When a warm pod is evicted, the WarmPoolController proactively creates a replacement pod immediately to restore `minWarm`. The PDB targets only unclaimed (warm) pods via a label selector (`lenny.dev/state: idle`), so it does not interfere with the preStop-based protection on active session pods.

**Sandbox finalizers:** Every `Sandbox` resource carries a finalizer (`lenny.dev/session-cleanup`) to prevent Kubernetes from deleting the pod ŌĆö and its local workspace ŌĆö while a session is still active. When a `Sandbox` enters the `Terminating` state, the warm pool controller checks whether any active `SandboxClaim` still references the pod. It removes the finalizer only after confirming one of two conditions: (a) no session references the pod, or (b) the session has been successfully checkpointed and the gateway has been notified so it can resume the session on a replacement pod. If the finalizer is not removed within 5 minutes (pod stuck in `Terminating`), the controller fires a `FinalizerStuck` alert. Operators should follow the stuck finalizer runbook (`docs/runbooks/stuck-finalizer.md`; see [Section 17.7](17_deployment-topology.md#177-operational-runbooks)) to safely verify session state, confirm checkpoint and artifact integrity, and remove the finalizer without orphaning resources. This ensures that node drains, scale-downs, and accidental deletions never silently orphan an active session.

**Orphaned `SandboxClaim` detection:** Because `SandboxClaim` resources deliberately omit an `ownerReference` (to survive pod deletion and allow reassignment), Kubernetes will not garbage-collect them automatically. The WarmPoolController's `GarbageCollect` reconciliation loop detects orphaned claims as follows: every 60 seconds, **the leader replica** lists all `SandboxClaim` resources whose `metadata.creationTimestamp` is older than `claimOrphanTimeout` (default: 5 minutes, configurable via `--claim-orphan-timeout`). Because the WarmPoolController runs with Lease-based leader election (see leader election paragraph above), only the current leader executes the `GarbageCollect` loop ŌĆö non-leader replicas skip it. This is the deliberate reason orphan detection is owned by the WarmPoolController rather than the gateway: placing the GC loop in the gateway would cause all gateway replicas to run API server list operations against the agent namespace simultaneously at each 60-second tick, multiplying API load proportionally with the gateway replica count. By running GC exclusively in the single elected leader of the WarmPoolController, the API list load is constant regardless of how many gateway or controller replicas are deployed. For each candidate orphaned claim, the controller queries Postgres to check whether an active session references it. If no active session exists, the claim is deleted and the underlying `Sandbox` pod is transitioned back to `idle`, returning it to the warm pool. This handles the crash scenario where a gateway creates a `SandboxClaim` but crashes before persisting the session to Postgres ŌĆö on recovery, the gateway has no record of the claim, so the controller's orphan loop reclaims it. The metric `lenny_orphaned_claims_total` (counter, labeled by pool) is incremented for each orphaned claim deleted. A `SandboxClaimOrphanRateHigh` warning alert fires when `lenny_orphaned_claims_total` exceeds 10 in a 15-minute window, indicating potential gateway instability (see [Section 16.5](16_observability.md#165-alerting-rules-and-slos)).

**Host-node schedulability labeling (`lenny.dev/host-schedulable`):** The WarmPoolController is the sole evaluator of per-pod host-node schedulability and surfaces the result as a pod label the gateway can read without any direct Node API access. On every pod reconciliation, the leader reads the target Node (addressed by the pod's `spec.nodeName`) from its Node informer cache and sets the pod label `lenny.dev/host-schedulable` to `"true"` when the Node's `.spec.unschedulable` is `false` and no `node.kubernetes.io/unschedulable` taint is present, or `"false"` otherwise. This label is WPC-owned (field manager `lenny-warm-pool-controller`) and is **not** in the immutable set enforced by the `lenny-label-immutability` webhook ([Section 17.2](17_deployment-topology.md#172-namespace-layout) item 5) ŌĆö the webhook's immutability list covers `lenny.dev/managed`, `lenny.dev/delivery-mode`, `lenny.dev/egress-profile`, and `lenny.dev/tenant-id` only; `lenny.dev/host-schedulable` is explicitly omitted so that WPC can flip the value on every cordon/uncordon event. Node state changes (cordon, uncordon, autoscaler drain) surface on the Node informer and trigger re-reconciliation of every pod whose `spec.nodeName` matches the Node; the controller re-labels each affected pod within a single reconcile cycle (typically < 1 s per batch at Tier 3 concurrency). The gateway consumes this label via its existing `get` access on the `Pods` resource in agent namespaces (the same access it uses to read pod metadata during claim and release) ŌĆö no additional Pod verbs, no Node verbs on the Gateway ServiceAccount, and no Node informer in the gateway process are required. The normative consumer is [┬¦6.2](06_warm-pod-model.md#62-pod-state-machine) "Host-node schedulability precondition". The Node verbs required for the evaluation itself ŌĆö `get`/`list`/`watch` on `Nodes` ŌĆö are granted to the WarmPoolController ServiceAccount (see RBAC enforcement paragraph in [┬¦4.6.3](#463-crd-field-ownership-and-write-boundaries), which extends the existing CIDR-drift-detection grants with `watch` for informer support); no cluster-scoped Node grants are added to the Gateway ServiceAccount. Pods newly created by WPC carry `lenny.dev/host-schedulable: "true"` by default (WPC applies the label at creation time after the scheduler binds the pod to a node); pods whose `spec.nodeName` is not yet set ŌĆö i.e., still `Pending` at the scheduler ŌĆö are not eligible for `task_cleanup ŌåÆ sdk_connecting` transitions, so the absence of the label on an unscheduled pod is never encountered by the gateway's precondition check.

**Dependency pinning and upgrade policy:** Lenny pins `kubernetes-sigs/agent-sandbox` to a specific tagged release (initially the latest stable tag at implementation start). Because agent-sandbox is a v0.x project, minor-version releases may contain breaking API changes. The one-release-delay cadence (upgrade to release N only after release N+1 is published) is insufficient on its own for a v0.x dependency: it confirms API surface stability across two releases but does not gate against rapid successive breaking changes. Therefore the upgrade policy is augmented as follows: (1) **CI gate** ŌĆö all CRD schema changes require the integration test suite to pass before merge; (2) **breaking-change hold** ŌĆö if upstream introduces a breaking CRD API change in release N, Lenny does not upgrade until the next release (N+1) demonstrates that the N API surface has stabilized (i.e., no further structural changes to the same fields); (3) **API stability monitoring** ŌĆö the engineering team subscribes to upstream release notes and the `kubernetes-sigs/agent-sandbox` issue tracker; any field removal, rename, or semantic change to `SandboxClaim`, `Sandbox`, `SandboxTemplate`, or `SandboxWarmPool` triggers an immediate impact assessment before the next Lenny release cycle.

**Go/no-go criteria for the agent-sandbox dependency:** The Phase 1 exit gate includes an explicit dependency health assessment. Before Phase 2 begins, the team evaluates agent-sandbox against all three of the following criteria:

1. **API stability.** The CRD API (`SandboxClaim`, `Sandbox`, `SandboxTemplate`, `SandboxWarmPool`) must not have had a structural breaking change (field removal, semantic rename, or schema incompatibility) in the two most recent upstream releases. If it has, the fallback path (see below) is activated.
2. **Community support SLO.** The upstream project must demonstrate active maintenance: at least one tagged release in the prior 6 months, critical bug reports acknowledged within 30 days, and no open blocker-severity issues affecting `SandboxClaim` optimistic-locking or warm-pool reconciliation that are unaddressed for more than 60 days.
3. **Integration test pass rate.** Lenny's agent-sandbox integration test suite (covering concurrent claims, orphan detection, pool reconciliation, and the ADR-007 chaos test) must pass at 100% against the pinned upstream release.

If any criterion is not met at Phase 1 exit, the team activates the fallback implementation (see below) before Phase 2 begins. The decision and its rationale are recorded as an ADR.

**Fallback plan if agent-sandbox is abandoned or diverges:** Because all consumers use `PodLifecycleManager` and `PoolManager` interfaces, Lenny can replace the agent-sandbox backend with custom kubebuilder-based controllers implementing the same interfaces. Estimated effort: 2-3 engineering-weeks for a minimal replacement covering pod claiming, release, drain, and basic warm pool reconciliation (the interface definitions serve as functional requirements). Advanced features (Pod Snapshots on GKE, upstream community PDB patterns) would require additional effort. This fallback is viable but undesirable ŌĆö the preferred path is continued upstream contribution. **Trigger:** fallback activation is initiated when the go/no-go assessment above (at Phase 1 exit, or at any subsequent quarterly dependency review) finds one or more criteria unmet and the upstream team cannot commit to resolution within 30 days.

#### 4.6.2 PoolScalingController (Pool Configuration)

**Role:** Manages desired pool configuration, scaling intelligence, and experiment variant pool sizing. Separate from the WarmPoolController which manages individual pod lifecycle.

**Responsibilities:**

- Reconcile pool configuration from Postgres (admin API source of truth) into `SandboxTemplate` and `SandboxWarmPool` CRDs
- Manage scaling decisions: time-of-day schedules, demand-based rules, experiment variant sizing
- Integrate with experiment definitions to automatically size variant pools
- Maintain the SDK-warm circuit breaker: trip/untrip `spec.sdkWarmDisabled` based on the rolling 5-minute demotion rate and persist circuit-breaker state (`status.sdkWarmCircuitBreaker.openedAt` / `openedReason` / `minOpenUntil`) so the breaker's open/closed decision survives PSC leader failover ŌĆö see [Section 6.1](06_warm-pod-model.md#61-what-a-pre-warmed-pod-looks-like) "Circuit-breaker state persistence across PSC leader failover"

**Leader election:** The PoolScalingController runs its own Lease-based leader election using a separate lease name (`lenny-pool-scaling-controller`) from the WarmPoolController (`lenny-warm-pool-controller`). Lease parameters: duration 15s, renew deadline 10s, retry period 2s (matching WarmPoolController). The two controllers elect leaders independently ŌĆö they may run on different replicas. Failover timing follows the same model as the WarmPoolController: near-zero on clean shutdown (voluntary lease release), up to 25s on crash (see [Section 4.6.1](#461-warm-pool-controller-pod-lifecycle)).

**Pluggable `PoolScalingStrategy` interface.** Fully replaceable by deployers.

**Default formula (non-experiment pools):**

```
target_minWarm = ceil(base_demand_p95 ├Ś safety_factor ├Ś (failover_seconds + pod_warmup_seconds)
                      + burst_p99_claims ├Ś pod_warmup_seconds)
```

The first term covers steady-state demand during a failover or leader-election gap: `base_demand_p95` is the p95 claim arrival rate (claims/second); multiplying by `(failover_seconds + pod_warmup_seconds)` converts it to a pod count ŌĆö the number of claims that will arrive while the pool cannot create new ready pods **plus** the additional time a freshly-created replacement pod takes to reach ready state after failover recovery. `failover_seconds` is the worst-case controller failover time (default: 25s ŌĆö `leaseDuration + renewDeadline = 15s + 10s`). `pod_warmup_seconds` is the time from pod creation to **ready** state ŌĆö the state at which a pod can serve a claim. For pod-warm pools (no SDK pre-initialization) this equals the container-pull-plus-runtime-startup time (~10s baseline, see the `pod_startup_seconds` note below); for SDK-warm pools it additionally includes SDK initialization time (30ŌĆō90s observed, depending on the adapter). Both the base-demand term and the burst term use the same `pod_warmup_seconds` value because both measure "how many claims will arrive before a replacement pod is ready to serve them" ŌĆö they differ only in the arrival rate (p95 steady vs. p99 burst), not in the pod-readiness target. `safety_factor` scales the steady-state term to provide a buffer above the raw demand estimate.

> **Terminology note ŌĆö `pod_startup_seconds` vs. `pod_warmup_seconds`.** The spec uses `pod_startup_seconds` informally as a shorthand for the pod-warm baseline (~10s: container pull + runtime startup, no SDK init), i.e., the value of `pod_warmup_seconds` for pools that do not pre-initialize the SDK. The normative formula input is always `pod_warmup_seconds`; references elsewhere in the spec that read "startup" in this context refer to the pod-warm baseline value of `pod_warmup_seconds`. The `pod_warmup_seconds` value used in formulas corresponds to the pool's `scalingPolicy.podWarmupSecondsBaseline` field (default: 30s for SDK-warm pools, set to ~10s for pod-warm pools) ŌĆö operators must tune this field to observed pod creation-to-ready time for accurate sizing.

The second term (`burst_p99_claims ├Ś pod_warmup_seconds`) absorbs request bursts that arrive faster than the pool can refill, even outside a failover window. `burst_p99_claims` is the p99 claim arrival rate (claims/second) observed over a short window (default: 60s). During a burst, each second of warmup latency means one fewer pod available to serve the next claim ŌĆö the burst term reserves enough headroom so the pool is not exhausted before replacement pods become ready.

`safety_factor` defaults to 1.5 for agent-type pools and 2.0 for mcp-type pools **at Tier 1 and Tier 2 only**. Tier 3 deployments override these defaults to 1.2 (agent-type) and 1.5 (mcp-type) ŌĆö see [Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) warm pool sizing table, which is the normative per-tier source of truth for `safety_factor`. The Tier 3 override is applied because per-pool demand is larger, allowing a thinner fractional buffer to achieve the same absolute headroom. This formula assumes session mode (one session per pod). For task and concurrent modes, a `mode_factor` adjustment reduces pod demand based on reuse and slot multiplexing ŌĆö see [Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes) "Execution Mode Scaling Implications" for per-mode formula variants.

**Variant pool formula (experiment pools only).** For A/B experiment variant pools, `variant_weight` scales demand proportionally to the traffic fraction routed to the variant:

```
target_minWarm = ceil(base_demand_p95 ├Ś variant_weight ├Ś safety_factor ├Ś (failover_seconds + pod_warmup_seconds)
                      + burst_p99_claims ├Ś variant_weight ├Ś pod_warmup_seconds)
```

`variant_weight` is defined on each variant entry in the experiment definition (see [Section 10.7](10_gateway-internals.md#107-experiment-primitives)). This formula is not used for standard (non-experiment) pools; `variant_weight` has no meaning outside experiment context. This formula assumes session mode (`mode_factor = 1.0`). For task-mode or concurrent-mode variant pools, apply `/ mode_factor` to the steady-state term and `/ burst_mode_factor` to the burst term, consistent with the mode-adjusted formula in [Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes).

**Variant pool sizing and base pool adjustment.** When one or more experiments create variant pools on the same base pool, the PoolScalingController must recompute the **base pool's** `minWarm` to reflect the total fraction of traffic being diverted to all active variants. Without this adjustment, total warm pods across the base and all variant pools equal `(1 + ╬Ż variant_weights) ├Ś original_base_minWarm` ŌĆö a significant over-provisioning.

The adjusted base pool formula is:

```
base_minWarm = ceil(base_demand_p95 ├Ś (1 - ╬Ż variant_weights) ├Ś safety_factor ├Ś (failover_seconds + pod_warmup_seconds)
                    + burst_p99_claims ├Ś (1 - ╬Ż variant_weights) ├Ś pod_warmup_seconds)
```

This formula assumes session mode. For task-mode or concurrent-mode base pools, apply `/ mode_factor` to the steady-state term and `/ burst_mode_factor` to the burst term, consistent with the mode-adjusted formula in [Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes).

Where `╬Ż variant_weights` is the sum of `variant_weight` values across all **active** variants of **all active experiments** targeting the same base pool. When multiple experiments concurrently divert traffic from the same base pool, the controller aggregates their variant weights in a single pass ŌĆö not per-experiment ŌĆö to avoid last-write-wins overwriting. The controller recalculates this aggregate sum whenever any experiment on the pool becomes active, any variant weight changes, or any experiment is paused or concluded (see [Section 10.7](10_gateway-internals.md#107-experiment-primitives)). `╬Ż variant_weights` is clamped to `[0, 1)` ŌĆö the base pool always retains at least `1 - max_total_variant_weight > 0` of demand. If `╬Ż variant_weights >= 1`, the controller rejects the experiment configuration at admission time with `INVALID_VARIANT_WEIGHTS`.

The PoolScalingController applies this recomputation in a single reconciliation cycle: it updates both the base pool's `SandboxWarmPool` CRD and the new variant pool's `SandboxWarmPool` CRD in the same reconcile pass. Because Kubernetes does not support multi-resource atomic transactions, there is a brief window (between the two sequential CRD updates within the cycle) where total warm pod count may be inconsistent. The WarmPoolController self-corrects on its next reconciliation, so the total warm pod count never exceeds the target for more than one cycle.

**Cold-start limitation:** The formula inputs (`base_demand_p95`, `burst_p99_claims`) are derived from historical traffic metrics and are not available at first deployment. For new pools, the operator must set `minWarm` manually ŌĆö the PoolScalingController cannot auto-configure this value without observed demand. New pools automatically enter **bootstrap mode** (`status.scalingMode: bootstrap`) using a static `bootstrapMinWarm` override until convergence criteria are met. See [Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) "First deployment sizing" for the full cold-start bootstrap procedure: fallback `minWarm` starting values, the five convergence criteria for exiting bootstrap mode, the operator-facing override API (`PUT /v1/admin/pools/{name}` with `bootstrapMinWarm` and `DELETE /v1/admin/pools/{name}/bootstrap-override`), and the `lenny_pool_bootstrap_mode` gauge / `PoolBootstrapMode` alert.

**Pool phases:** pre-warm, ramp, steady state, wind-down ŌĆö all automatic. For experiment-lifecycle-driven pool teardown (what happens to variant pools when an experiment is paused or concluded), see the "PoolScalingController behavior on experiment status transitions" paragraph in [Section 10.7](10_gateway-internals.md#107-experiment-primitives).

**CRDs become derived state** reconciled from Postgres by PoolScalingController. The admin API is the source of truth for pool configuration; the controller translates that into Kubernetes CRDs.

**Reconciliation drift detection.** Because CRDs are derived state, any period where the PoolScalingController is down (crash, leader election, network partition to the API server) causes CRD specs to diverge from Postgres desired state. The WarmPoolController continues operating against stale CRD values indefinitely ŌĆö it has no mechanism to detect that the values it reads are outdated. To close this observability gap:

1. **Reconciliation lag metric.** The PoolScalingController exposes a `lenny_pool_config_reconciliation_lag_seconds` gauge (labeled by pool name) that records the time since the last successful CRD reconciliation for each pool. On every successful reconciliation cycle, the controller resets the gauge to zero. If the controller is down, the gauge value grows monotonically (scraped from the last-active replica's `/metrics` endpoint until the process exits; after exit, the metric disappears and the `PoolConfigDrift` alert fires on the absence condition below).
2. **`PoolConfigDrift` warning alert.** Fires when any pool's Postgres `pool_config_generation` (incremented on every admin API write) differs from its CRD's `metadata.annotations["lenny.dev/config-generation"]` for more than 60 seconds. Detection runs in the gateway (which reads both Postgres and CRD state) rather than in the PoolScalingController itself, so the alert fires even when the controller is completely down. The alert auto-resolves when the controller catches up and reconciles the CRD. Added to the alert inventory in [Section 16.5](16_observability.md#165-alerting-rules-and-slos).
3. **Admin API sync status.** The `PUT /v1/admin/pools/{name}` response includes a `syncStatus` field: `"pending"` immediately after a write (Postgres updated, CRD reconciliation not yet confirmed), or `"synced"` once the CRD generation matches. This prevents operators from believing changes are instantly effective. The `GET /v1/admin/pools/{name}` response also includes `syncStatus` for consistency.
4. **Sync-status endpoint.** `GET /v1/admin/pools/{name}/sync-status` returns detailed reconciliation state: `postgresGeneration`, `crdGeneration`, `lastReconciledAt`, `lagSeconds`, and `inSync` (boolean). This gives operators a single endpoint to diagnose whether the PoolScalingController is keeping up.

#### 4.6.3 CRD Field Ownership and Write Boundaries

Both the WarmPoolController and PoolScalingController interact with the same CRD types (`SandboxTemplate`, `SandboxWarmPool`, `Sandbox`). To prevent conflicting writes, each controller owns a disjoint set of fields:

| CRD               | Field / Subresource                                | Owner                      | Notes                                           |
| ----------------- | -------------------------------------------------- | -------------------------- | ----------------------------------------------- |
| `SandboxTemplate` | `spec.*`                                           | PoolScalingController      | Reconciled from Postgres pool definitions       |
| `SandboxTemplate` | `status.*`                                         | WarmPoolController         | Pool health conditions, observed generation     |
| `SandboxWarmPool` | `spec.minWarm`, `spec.maxWarm`, `spec.scalePolicy`, `spec.sdkWarmDisabled` | PoolScalingController      | Scaling parameters from Postgres; `spec.sdkWarmDisabled` is the circuit-breaker flag set by PSC |
| `SandboxWarmPool` | `status.sdkWarmCircuitBreaker.*` (carve-out) | PoolScalingController | `openedAt`, `openedReason`, `minOpenUntil` ŌĆö persisted circuit-breaker state for PSC leader-handoff continuity (see [Section 6.1](06_warm-pod-model.md#61-what-a-pre-warmed-pod-looks-like) "Circuit-breaker state persistence across PSC leader failover") |
| `SandboxWarmPool` | `status.*` (excluding `status.sdkWarmCircuitBreaker.*`) | WarmPoolController         | Current warm count, ready count, conditions     |
| `Sandbox`         | `spec.*`                                           | WarmPoolController         | Pod creation and configuration                  |
| `Sandbox`         | `status.*`                                         | WarmPoolController         | State machine transitions, health               |
| `SandboxClaim`    | `spec.*`, `status.*`                               | Gateway (not a controller) | Created/deleted by gateway during claim/release |

**Server-Side Apply (SSA) enforcement:** Field ownership is enforced via Kubernetes Server-Side Apply with named field managers. Each controller applies only its owned fields using a distinct field manager name (`lenny-warm-pool-controller` and `lenny-pool-scaling-controller`). If a controller attempts to update a field owned by another manager, the API server returns a conflict error (HTTP 409) without applying the update. This is the primary enforcement mechanism ŌĆö it is enforced by the API server itself, without any additional controller logic.

**SSA conflict retry policy (crash recovery).** When a controller crashes and restarts, its in-memory view of CRD state is stale. If the restarted controller immediately applies its owned fields using a cached resource version, the API server may return an HTTP 409 conflict (the resource has been updated by the other controller or by a prior apply that was not yet reflected). The required retry policy is:

1. **Always re-read before re-applying.** On any SSA HTTP 409 conflict error, the controller **must** discard its cached copy of the resource and issue a fresh `GET` to the API server to obtain the current `resourceVersion` and the current managed fields map. The controller then re-computes its apply patch against the freshly read state.
2. **Never force-conflict.** Controllers **must not** use `--force-conflicts` (SSA `Force: true`) except as a last-resort operator-initiated recovery action. Force-conflicts unconditionally steals field ownership from the other manager ŌĆö it bypasses the isolation invariant that prevents the two controllers from overwriting each other's fields. Using `Force: true` in reconciliation code is explicitly prohibited and would defeat the ownership boundary enforced by the SSA mechanism.
3. **Bounded retry with backoff.** After re-reading and re-computing the patch, the controller retries the apply. If a second 409 occurs (concurrent apply from the other controller), the controller backs off with jitter (initial 100ms, max 2s) and re-reads again. After 5 consecutive 409s without progress, the controller emits a `crd_ssa_conflict_stuck` structured log event (labeled by `controller`, `resource`, `name`) and increments the `lenny_crd_ssa_conflict_total` counter, then continues with exponential backoff. A `CRDSSAConflictStuck` warning alert fires when this counter exceeds 10 in a 5-minute window on any single resource, indicating an abnormal ownership dispute that requires operator investigation.
4. **No force on normal startup.** After a crash and leader-election re-acquisition, the restarted controller executes a full list-and-sync reconciliation of all resources it owns before accepting new watch events. Each reconciliation apply follows the re-read policy above. This ensures the controller's first post-crash apply is always based on the current API server state, not stale pre-crash cache.

**RBAC enforcement (coarse-grained backstop):** Each controller's ServiceAccount has RBAC rules granting access at resource and subresource granularity. The WarmPoolController ServiceAccount has `create`/`update`/`delete` on `Sandbox` and `get`/`patch` on `status` subresources of `Sandbox`, `SandboxTemplate`, and `SandboxWarmPool` (SSA status updates require `get` to read the current resource and `patch` to apply the partial object ŌĆö `update` is insufficient for SSA Apply requests), `list`/`delete` on `SandboxClaim` (required for orphan claim garbage collection ŌĆö see `GarbageCollect` above), `create`/`update`/`delete` on `PodDisruptionBudget` (required for `ManagePDB`), `get`/`create`/`update` on `Leases` in `lenny-system` (required for leader election), `get`/`list`/`watch` on `Nodes` (required for CIDR drift detection ŌĆö the continuous drift goroutine reads node `spec.podCIDR` to compute actual cluster CIDRs; see [Section 13.2](13_security-model.md#132-network-isolation) ŌĆö and for the Node informer that drives `lenny.dev/host-schedulable` pod labeling; see "Host-node schedulability labeling" above), `get`/`list` on `NetworkPolicies` in agent namespaces (required for CIDR drift detection ŌĆö the goroutine reads installed `except` blocks from `internet` egress NetworkPolicies to compare against actual CIDRs), `get`/`list`/`watch` on `Pods` in agent namespaces (required for reading pod phase/conditions and certificate annotations for cert-expiry tracking ŌĆö the WPC reads cert-manager annotations on the underlying Pod to determine remaining certificate validity for idle pods; see [Section 10.3](10_gateway-internals.md#103-mtls-pki)), and `get`/`list`/`watch` on the same CRD types listed above. The PoolScalingController ServiceAccount has `create`/`update`/`delete` on `SandboxTemplate` and `SandboxWarmPool`, `get`/`patch` on the `status` subresource of `SandboxWarmPool` (required for SSA writes to the PSC-owned `status.sdkWarmCircuitBreaker` carve-out per the ownership table above), `get`/`create`/`update` on `Leases` in `lenny-system` (required for leader election), and `get`/`list`/`watch` on remaining status subresources, with no access to `Sandbox` resources. Note: Kubernetes RBAC operates at resource and subresource granularity, not individual field granularity ŌĆö it cannot distinguish which fields within `spec` a controller may write. Field-level boundaries are enforced by SSA (above), not RBAC.

**Gateway ServiceAccount RBAC grants:** The gateway's ServiceAccount (`system:serviceaccount:lenny-system:lenny-gateway`) requires RBAC grants separate from the controller ServiceAccounts. The gateway performs pod label mutations (setting `lenny.dev/tenant-id` on first assignment and toggling `lenny.dev/state` between `idle` and `active`) and manages `SandboxClaim` resources. Required grants: `get`/`patch` on `Pods` in agent namespaces (`lenny-agents`, `lenny-agents-kata`) for tenant-id and state label mutations, `create`/`get`/`delete` on `SandboxClaim` resources for the claim/release lifecycle, and `get`/`list` on `Sandbox` resources for pod selection during claim. The `lenny-tenant-label-immutability` webhook ([Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes)) must include the gateway ServiceAccount in its allowlist for the `unset ŌåÆ {tenant_id}` initial label write ŌĆö the webhook allows the gateway SA to set `lenny.dev/tenant-id` on initial assignment, and allows the WarmPoolController SA to reset `{tenant_id} ŌåÆ unassigned` when a pod is returned to the pool.

**`SandboxClaim.status.phase` enumeration (gateway-owned binding state):** `SandboxClaim.status.phase` is distinct from the pod lifecycle state machine on `Sandbox.status.phase` ([Section 6.2](06_warm-pod-model.md#62-pod-state-machine)) ŌĆö it tracks the **session-to-pod binding** rather than the pod's warm-pool lifecycle. The gateway is the sole writer and the legal phase values are:
- `bound` ŌĆö the gateway has created the `SandboxClaim` and successfully bound it to a claimed `Sandbox`; the referenced `Sandbox.status.phase` is `claimed` (or has progressed past `claimed` into session-serving states such as `receiving_uploads`, `attached`, `running`).
- `active` ŌĆö the session has reached `attached` and is actively running work on the claimed pod.
- `released` ŌĆö the gateway has released the pod back to the pool; the referenced `Sandbox.status.phase` has transitioned to `idle` (for a recyclable pod) or `draining`/`terminated` (for a pod being replaced). The `SandboxClaim` resource is deleted shortly after entering `released`; it is a terminal phase.
- `failed` ŌĆö the claim cannot be fulfilled or the session terminated with an unrecoverable error; the `SandboxClaim` is deleted after the gateway persists session-terminal state to Postgres. Terminal.

These phase values carry no pod-lifecycle semantics and must not be confused with `Sandbox.status.phase`. The `lenny-sandboxclaim-guard` admission webhook ([Section 4.6.1](#461-warm-pool-controller-pod-lifecycle)) reads the referenced `Sandbox.status.phase` (via `.spec.sandboxRef`), never `SandboxClaim.status.phase`, when enforcing double-claim prevention and stale-write rejection. CRD OpenAPI validation restricts `SandboxClaim.status.phase` to exactly this enumeration; any write containing an undefined phase value is rejected by the API server at validation time.

**Validating webhook for Postgres-authoritative state (defense-in-depth):** The `lenny-pool-config-validator` `ValidatingAdmissionWebhook` rejects writes to `SandboxTemplate.spec` and `SandboxWarmPool.spec` based on two orthogonal rule sets, both of which must pass for a write to be admitted:

1. **Semantic/budget rules (apply to ALL writes, including PoolScalingController SSA applies):** the tiered-checkpoint-cap vs. `terminationGracePeriodSeconds` budget rule and the `checkpointBarrierAckTimeoutSeconds` floor rule ŌĆö see [┬¦10.1](10_gateway-internals.md#101-horizontal-scaling) for the exact inequalities and rejection error codes. These rules are NOT subject to the `userInfo`-based bypass in rule set 2; the PoolScalingController is rejected if it attempts to write a configuration that violates the preStop budget, because any such configuration would guarantee SIGKILL during gateway drain regardless of the writer's identity.
2. **Authorization-based denial (applies additionally):** manual `kubectl edit` or `kubectl apply` updates to these fields are rejected UNLESS the request's `userInfo` maps to the PoolScalingController ServiceAccount (`system:serviceaccount:lenny-system:lenny-pool-scaling-controller`). This uses the same `userInfo`-based authorization as the `lenny-label-immutability` webhook ŌĆö the API server populates `userInfo` from the authenticated principal, which cannot be spoofed by the caller (unlike annotations, which any principal with `patch` access could set). This prevents operators from manually editing CRD specs that would be silently overwritten on the next reconciliation cycle. The webhook returns an informative denial message directing operators to use the admin API instead. PoolScalingController writes bypass only this second rule, not the first.

The webhook runs in `Fail` mode with a 5s timeout; if the webhook is unavailable, updates are denied (fail-closed) to protect Postgres-authoritative state ŌĆö this blocks BOTH manual edits AND PoolScalingController reconciliation writes, stalling pool configuration updates until the webhook recovers. This webhook is a defense-in-depth backstop for the authorization path (rule set 2) and the sole admission gate for the semantic budget invariants (rule set 1) ŌĆö SSA field manager conflicts provide the primary enforcement layer for field ownership, but they do not enforce semantic budget invariants. Alert `PoolConfigValidatorUnavailable` (Warning) fires when the webhook has been unreachable for more than 30 seconds ŌĆö see [Section 16.5](16_observability.md#165-alerting-rules-and-slos).

**PoolScalingController admission-denial handling policy.** The SSA conflict retry policy above covers HTTP 409 responses only. When the `lenny-pool-config-validator` webhook rejects a PSC apply under rule set 1 (semantic/budget invariants ŌĆö e.g., `422 INVALID_POOL_CONFIGURATION`) or returns any other 4xx admission denial (400, 403, 422), the PSC MUST NOT treat the response as a transient SSA conflict and MUST NOT tight-loop on the offending spec. Such denials are deterministic functions of the Postgres pool configuration the PSC just read: retrying immediately against the same Postgres state (e.g., during a PgBouncer failover, an in-progress schema migration, a stale Redis-cached pool row, or an admin-left transient invalid tenant config) reproduces the same rejected spec and drives both the webhook and the API server into a hot loop. The required policy is:

1. **Log and count.** On any 4xx admission response from the pool-config validator, the PSC logs a structured event (`controller: lenny-pool-scaling-controller`, `pool`, `crd`, `http_status`, webhook-returned `reason` code, and the full rejected spec diff against the last-admitted generation) and increments `lenny_pool_scaling_admission_denied_total` (counter, labeled by `pool`, `crd` Ōłł {`SandboxTemplate`, `SandboxWarmPool`}, and `reason` ŌĆö the webhook-supplied failure code such as `tiered_cap_exceeds_grace_budget` or `barrier_ack_below_floor`). The HTTP status class is captured on the log event but not on the metric ŌĆö all 4xx denials share a single `admission_denied` retry path (see below), so the actionable distinction for operators is the `reason`, not the status code.
2. **Per-pool exponential backoff.** The PSC tracks denial state per `(pool, crd)` tuple. On the first denial, it pauses reconciliation for that tuple for 1 second. On each consecutive denial without an intervening successful apply, the backoff doubles up to a 60-second ceiling. A successful apply (HTTP 200/201) or a successful non-denial admission response resets the backoff to zero for that tuple. Backoff is maintained in-memory by the leader; a leader handoff resets the counter (the new leader re-reads Postgres and may succeed if the underlying stale read has cleared, so starting from zero is the correct behavior on failover). Backoff is per-tuple, not global ŌĆö denials on one pool never block reconciliation of other pools.
3. **Stuck-pool abort.** After `admissionDeniedRetryCeiling` (default: 10) consecutive denials on the same `(pool, crd)` without an intervening success, the PSC stops retrying that tuple until one of three conditions clears the stuck state: (a) the Postgres pool configuration for that pool changes (detected by a non-matching `pool_config_generation`), (b) the leader restarts or a new leader is elected, or (c) an operator invokes `POST /v1/admin/pools/{name}/resume-reconciliation` (which resets the in-memory denial counter for that pool without requiring a configuration change). While stuck, the `PoolConfigDrift` alert will continue to fire for the pool (the CRD generation never catches up with Postgres), and the new `PoolScalingAdmissionStuck` Warning alert fires in parallel to distinguish "controller down / missing" from "controller running, apply rejected". See [Section 16.5](16_observability.md#165-alerting-rules-and-slos).
4. **Transient vs. permanent denials.** Rule-set-1 denials are semantically deterministic, so there is no reliable a-priori classifier for "transient" vs. "permanent" from the HTTP response alone. The backoff-plus-stuck-abort policy handles both uniformly: truly transient denials (e.g., a PgBouncer failover that recovers within seconds) clear when the next Postgres read returns consistent config and the backoff resets; persistent denials (e.g., an admin-authored invalid tenant config) trigger the stuck-pool abort after the retry ceiling, forcing operator intervention rather than silently wedging the queue.

**Checkpoint / Interrupt mutual exclusion:** The adapter maintains a per-session operation lock that serializes `Checkpoint` and `Interrupt` RPCs. Only one of these operations may execute at a time; if a second arrives while the first is in progress, it is queued and executed after the first completes. Ordering semantics:

- **Interrupt during checkpoint:** The interrupt is queued until the checkpoint completes (or times out per [Section 4.4](#44-event--checkpoint-store)). After the checkpoint finishes, the queued interrupt is delivered normally. Rationale: a checkpoint in progress has already paused or quiesced the runtime (via SIGSTOP or lifecycle channel `checkpoint_request`); delivering an interrupt in that state is undeliverable (signals cannot reach a SIGSTOPped process) or would violate the quiescence guarantee.
- **Checkpoint during interrupt:** The checkpoint is queued until the runtime acknowledges the interrupt (via `interrupt_acknowledged` on the lifecycle channel for Full-level, or until the adapter observes the runtime resume output for lower levels). This prevents snapshotting mid-interrupt state.
- **Queue depth:** At most one operation may be queued. If a second operation of the same type arrives while one is already queued, the second is coalesced (checkpoint) or dropped with a `BUSY` status (interrupt). The gateway retries dropped interrupts with backoff.

**Adapter ŌåÆ Gateway events (sent over the gRPC control channel):**

#### Adapter Ōåö Runtime Protocol (Intra-Pod)

The adapter communicates with the runtime binary via two mechanisms:

**Part A ŌĆö Multiple focused local MCP servers** (intra-pod, abstract Unix socket):

- **Platform MCP server** ŌĆö Lenny-specific tools: `lenny/delegate_task`, `lenny/await_children`, `lenny/cancel_child`, `lenny/discover_agents`, `lenny/output`, `lenny/request_elicitation`, `lenny/memory_write`, `lenny/memory_query`, `lenny/request_input`, `lenny/send_message`, `lenny/get_task_tree`, `lenny/set_tracing_context`. Note: lease extension is handled via the adapterŌåögateway gRPC lifecycle, not as an MCP tool (see [Section 8.6](08_recursive-delegation.md#86-lease-extension)).
- **One MCP server per authorized connector** ŌĆö each connector in the session's delegation policy gets its own independent MCP server. No aggregated connector proxy ŌĆö aggregation is not lossless per MCP spec (capability negotiation is per-server, sampling breaks, tool name collisions, resource URI collisions).

**No workspace MCP server.** Workspace is materialized to `/workspace/current` before the runtime starts. The runtime accesses it via the filesystem directly.

**Part B ŌĆö Lifecycle channel** ŌĆö bidirectional JSON Lines stream over an abstract Unix socket (`@lenny-lifecycle`) for operational signals. The socket path is advertised in the adapter manifest (`lifecycleChannel.socket`). The runtime connects as a client; the adapter listens. Each message is a single JSON object terminated by `\n`:

```
Adapter ŌåÆ Runtime:  lifecycle_capabilities, checkpoint_request,
                    checkpoint_complete, interrupt_request,
                    credentials_rotated, terminate,
                    task_complete, task_ready
Runtime ŌåÆ Adapter:  lifecycle_support, checkpoint_ready,
                    interrupt_acknowledged, credentials_acknowledged,
                    llm_request_started, llm_request_completed,
                    task_complete_acknowledged
```

Optional. Runtimes that don't open it operate in fallback-only mode. Versioned by capability negotiation at the top. Unknown messages silently ignored on both sides.

#### Runtime Integration Levels (agent-type only)

- **Basic** ŌĆö stdin/stdout binary protocol only. Reads `{type: "message"}` from stdin, writes `{type: "response"}` and `{type: "tool_call"}` to stdout. This is the floor. Zero Lenny knowledge required.
- **Standard** ŌĆö basic plus connects to adapter's platform MCP server and connector servers via the adapter manifest. Uses platform capabilities (delegation, discovery, output parts, elicitation). Standard MCP ŌĆö no Lenny-specific code.
- **Full** ŌĆö standard plus opens the lifecycle channel. True session continuity, clean interrupt points, mid-session credential rotation via `credentials_rotated` lifecycle message.

**Credential rotation behavior by level:**

| Level    | Rotation method                                                                                                                                                                              |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Full     | Gateway calls `RotateCredentials` RPC; adapter sends `credentials_rotated` on lifecycle channel; runtime rebinds provider in-place. No session interruption.                                 |
| Standard | Gateway triggers `Checkpoint` ŌåÆ terminates pod ŌåÆ schedules replacement pod ŌåÆ `AssignCredentials` (new lease) ŌåÆ `Resume`. Session experiences a brief pause; the client sees a reconnect.     |
| Basic    | Same as Standard: checkpoint + restart. Basic-level runtimes that do not support checkpoint lose in-flight context; the gateway restarts the session from the last gateway-persisted state. |

The gateway selects the rotation strategy automatically based on the level reported in the adapter's `lifecycle_support` handshake (Full) or the absence of a lifecycle channel (Standard/Basic).

**Full-level credential rotation protocol:**

The Full-level rotation via the lifecycle channel follows a strict protocol with timeout and in-flight request handling:

1. **In-flight request completion gate.** Before sending `credentials_rotated` on the lifecycle channel, the adapter MUST wait for any in-flight LLM request to complete (or receive an error from the provider). The adapter tracks active outbound LLM requests via the lease's proxy stream count (proxy mode) or the `llm_request_started`/`llm_request_completed` lifecycle messages sent by the runtime (direct mode ŌĆö the adapter maintains a per-provider in-flight counter incremented on `llm_request_started` and decremented on `llm_request_completed`; the gate clears when the counter reaches zero). New outgoing LLM requests from the runtime are queued by the adapter during the rotation window ŌĆö they are held until the runtime sends `credentials_acknowledged`, at which point they are released using the new credential. This prevents the runtime from starting a new request with the old credential after the rotation signal has been decided but before it has been delivered. The adapter records the gate wait duration in the `lenny_credential_rotation_inflight_wait_seconds` histogram (labeled by `pool`, `provider`). If the wait exceeds 60 seconds, the adapter emits a `credential_rotation_inflight_wait_long` warning event (fields: `session_id`, `lease_id`, `provider`, `elapsed_seconds`) for operator visibility. The wait is intentionally unbounded for scheduled rotations: the old credential remains valid, the in-flight request will complete naturally, and imposing a timeout would risk an auth failure on an otherwise successful request. The queued-request impact is bounded because the rotation trigger (`RATE_LIMITED`) means the runtime was already unable to start new LLM calls.

   **Revocation-triggered rotation ceiling.** When `rotationTrigger ŌēĀ proactive_renewal` (i.e., any fault-driven, operator-initiated, or user-initiated rotation ŌĆö see [┬¦4.9](#49-credential-leasing-service) "`rotationTrigger` enum" for the full set: `fault_rate_limited`, `fault_auth_expired`, `fault_provider_unavailable`, `emergency_revocation`, `user_credential_rotated`, `user_credential_revoked`), the in-flight gate is capped at 300 seconds (5 minutes). On ceiling hit, the adapter sends `credentials_rotated` regardless of the in-flight counter and the session falls through to the standard fault-rotation path. The ceiling does NOT apply to `rotationTrigger: proactive_renewal`, which retains the unbounded wait because the old credential is still valid and imposing a timeout would risk false auth failures. This closes the attack surface in direct mode where a compromised or buggy runtime could indefinitely block `credentials_rotated` by never emitting `llm_request_completed` during a fault or revocation ŌĆö defeating the "no window where the compromised or failing key continues to reach the provider" guarantee. The threat model is identical across `RATE_LIMITED`, `AUTH_EXPIRED`, and `PROVIDER_UNAVAILABLE` fault paths and across operator/user-initiated revocations, so all non-renewal triggers are capped uniformly. A `lenny_credential_rotation_inflight_ceiling_hit_total` counter (labels `pool`, `trigger`) increments on ceiling hit, and an `OutstandingInflightAtRotationCeiling` warning alert fires at [Section 16.5](16_observability.md#165-alerting-rules-and-slos). At the same code point the adapter emits a durable `credential.rotation_ceiling_hit` audit event to the `EventStore` ([┬¦4.9.2](#492-credential-audit-events)) ŌĆö Prometheus counters reset on replica restart and alerts are delivery-best-effort, so the audit event is the forensic system of record for the identity of every ceiling-hit rotation (tenant, session, lease, pool, credential, trigger, outstanding in-flight count, elapsed seconds). This event is SIEM-streamable as a tier-1 compromise-indicator signal ([┬¦11.7](11_policy-and-controls.md#117-audit-logging)).

2. **`credentials_acknowledged` timeout.** The adapter enforces a **60-second timeout** on the `credentials_acknowledged` response, matching the checkpoint timeout ([Section 4.4](#44-event--checkpoint-store)). The timeout starts when `credentials_rotated` is sent on the lifecycle channel. If the runtime does not reply within 60 seconds, the adapter:
   - Emits a `credential_rotation_timeout` warning event (fields: `session_id`, `lease_id`, `provider`, `elapsed_seconds`).
   - Falls back to the Standard-level rotation path: triggers `Checkpoint` ŌåÆ terminates pod ŌåÆ schedules replacement pod ŌåÆ `AssignCredentials` (new lease) ŌåÆ `Resume`.
   - Increments the `lenny_credential_rotation_timeout_total` counter (labeled by `pool`, `provider`, `runtime`).

3. **Old credential grace period.** The old credential is NOT invalidated at the provider (removed from the pool's active set or returned to the pool) until one of: (a) `credentials_acknowledged` is received from the runtime, confirming the rebind is complete; or (b) the 60-second timeout elapses and the fallback path takes over. This ensures that if an in-flight LLM request is still completing with the old credential at the provider, it is not rejected mid-stream due to premature credential invalidation. After the grace period, the old lease is marked `released` and the credential is returned to the pool (or to cooldown, depending on the rotation reason). The metric `lenny_credential_rotation_grace_period_seconds` (histogram, labeled by `pool`, `provider`) records the actual duration between `credentials_rotated` and old credential release.

#### Startup Sequence for `type: agent` Runtimes

1. Pod created by Kubernetes
2. Adapter opens gRPC connection to gateway (mTLS)
3. Adapter writes placeholder manifest (connector servers unknown yet)
4. Adapter signals READY to gateway ŌĆö pod enters warm pool
5. Gateway assigns session: `PrepareWorkspace` ŌåÆ `FinalizeWorkspace` ŌåÆ `RunSetup` ŌåÆ `AssignCredentials(leases)` ŌåÆ `StartSession`
6. Adapter writes **final manifest** (connector servers now known from leases; all provider credentials available)
7. Adapter spawns runtime binary
8. Runtime reads manifest, connects to MCP servers (Standard/Full), opens lifecycle channel (Full)
9. Adapter sends `lifecycle_capabilities` (Full); receives `lifecycle_support`
10. Adapter delivers first `{type: "message"}` on stdin

#### Deployment Model

- **Default: Sidecar container** communicating with the agent binary over **abstract Unix sockets** (Linux `\0` namespace ŌĆö no filesystem path). `shareProcessNamespace: false`. The adapter writes the manifest to a dedicated `emptyDir` volume (`/run/lenny/`) mounted **read-only into the agent container** and read-write into the adapter container. Workspace is a separate `emptyDir` volume (`/workspace/`). No single shared volume carries both communication and data ŌĆö sockets are abstract (kernel-only), the manifest is read-only to the agent, and workspace is isolated. This minimizes what third-party binary authors need to implement ŌĆö just a binary that reads/writes on a well-defined socket protocol.
- **Alternative: Embedded** ŌĆö first-party binaries can embed the adapter directly and expose the same gRPC contract to the gateway.
- Same external contract either way.

**Sidecar vs embedded trade-offs:**

| Aspect                 | Sidecar (default)                                        | Embedded                                   |
| ---------------------- | -------------------------------------------------------- | ------------------------------------------ |
| Complexity for authors | Low ŌĆö implement stdin/stdout JSON protocol               | High ŌĆö implement full gRPC contract        |
| Resource overhead      | ~50 MB memory for adapter sidecar                        | None (shared process)                      |
| Language support       | Any language (stdin/stdout)                              | Go only (or language with gRPC support)    |
| Isolation              | Process isolation; abstract sockets + read-only manifest | Single process, shared memory              |
| Recommended for        | Third-party runtimes, community adapters                 | First-party runtimes where latency matters |

> **Note:** Third-party authors should always use the sidecar model. The embedded model is for first-party runtimes where the adapter and agent binary are developed together.

**Health check:** gRPC Health Checking Protocol. The warm pool controller marks a pod as `idle` only after the health check passes.

#### Adapter-Agent Security Boundary

The boundary between the adapter and the agent binary is **untrusted**. A compromised or misbehaving agent binary must not be able to escalate privileges, extract credentials, or manipulate the adapter. The following controls enforce this:

1. **Separate UIDs and connection authentication:** The adapter runs as a different UID than the agent binary (e.g., adapter as UID 1000, agent as UID 1001). Abstract Unix sockets use `SO_PEERCRED` for peer UID verification ŌĆö the adapter accepts connections only from the expected agent UID. The expected UID is the UID within the pod's user namespace (matching the `runAsUser` in the pod spec), not a host UID ŌĆö both sidecar containers share the same user namespace, so `SO_PEERCRED` reports consistent UIDs. As a defense-in-depth measure, the adapter also performs a manifest-nonce handshake on connection: the adapter writes a random nonce to the manifest file (read-only to the agent container) before spawning the agent, and the agent must present this nonce as the first message on the socket. This ensures the connecting process has legitimate access to the manifest, independent of UID verification. gVisor `SO_PEERCRED` semantics must be validated in Phase 3.5 security integration tests; if behavior diverges from the host kernel, the nonce handshake serves as the primary authentication control. The manifest volume is mounted read-only into the agent container, preventing tampering. Filesystem-level isolation prevents the agent from reading adapter config or memory.

   **Mandatory `SO_PEERCRED` startup self-test (adapter prerequisite).** Before the adapter marks itself READY to the gateway (step 4 of the Startup Sequence above), it MUST execute a loopback self-test that verifies `SO_PEERCRED` is functional in the current pod environment:
   1. The adapter opens a temporary abstract Unix socket (`@lenny-sopeercred-selftest`).
   2. A goroutine connects to the socket from the same process (loopback connection).
   3. The adapter calls `getsockopt(SO_PEERCRED)` on the accepted connection and asserts that the returned `uid` matches `os.Getuid()` (the adapter's own UID).
   4. If the call fails (error) or the returned UID does not match, the adapter logs `"FATAL: SO_PEERCRED self-test failed ŌĆö UID mismatch or syscall error; adapter security boundary cannot be enforced"`, emits the `lenny_adapter_sopeercred_selftest_failed_total` counter increment, and exits non-zero. The pod enters `CrashLoopBackOff`, preventing session assignment.
   5. If the self-test passes, the adapter closes the temporary socket and proceeds to the READY signal.

   This self-test ensures that environments where `SO_PEERCRED` is silently non-functional (e.g., some container runtimes with non-standard user namespace configurations) are caught before any agent process is spawned. The self-test runs on every pod start, not only in CI. If gVisor's `SO_PEERCRED` semantics diverge (Phase 3.5 validation), the self-test result determines whether UID verification is available; the nonce handshake remains active as the primary control in that case, and the self-test failure path is suppressed (configurable via `adapter.requireSoPeercred`, default `true`; set to `false` only when gVisor divergence is confirmed and nonce-only mode is explicitly accepted).

   **Nonce-only fallback: challenge-response supplement and escalation gate.** When `adapter.requireSoPeercred` is `false` (i.e., `SO_PEERCRED` is unavailable), the nonce handshake alone is the authentication boundary. Because the manifest nonce is static for the lifetime of a task execution, the adapter MUST supplement it with a per-connection challenge-response exchange to maintain forward security on each new connection:
   1. After the agent presents the manifest nonce, the adapter sends a 128-bit random challenge value (`adapterChallenge`) on the same socket.
   2. The agent must respond with `HMAC-SHA256(key=manifestNonce, data=adapterChallenge)` within 500 ms.
   3. The adapter validates the HMAC response. Any mismatch or timeout causes the adapter to close the socket immediately with no protocol response.
   This ensures that a process that observed the nonce (e.g., from `/proc/<pid>/fd` inspection by another pod-local process) cannot replay it across multiple connections, because each connection requires a fresh HMAC computation gated on the adapter's ephemeral challenge.

   **Escalation requirement.** Nonce-only mode (`requireSoPeercred: false`) is not a silent steady state. The adapter MUST:
   - Emit a `lenny_adapter_sopeercred_disabled_total` counter on every pod start when the setting is `false`.
   - Set a Kubernetes condition `SOPeercredDisabled=True` on the pod object, which pool controllers surface as a pool-level `SecurityDegradedMode=True` condition on the `SandboxTemplate` CRD.
   - Deployers MUST configure an alert on `lenny_adapter_sopeercred_disabled_total > 0` with a human-review SLA. The setting MUST be re-evaluated when the runtime version changes (gVisor upgrades, kernel updates) ŌĆö the Phase 3.5 `SO_PEERCRED` integration tests are re-run at that point and `requireSoPeercred` reverted to `true` if the behavior is fixed.

2. **Adapter-initiated protocol:** The adapter is the protocol initiator ŌĆö it sends messages and tool-call instructions to the agent and receives responses. The agent cannot initiate arbitrary requests to the adapter.

3. **Untrusted agent responses:** The adapter treats all data received from the agent as untrusted input. It validates, sanitizes, and size-limits all responses before forwarding them to the gateway.

4. **No credential material over socket:** LLM credentials (whether direct lease or proxy URL) are delivered to the agent process via a tmpfs-backed file (`/run/lenny/credentials.json`, mode `0440`, owned by the adapter UID and group-owned by the shared `lenny-cred-readers` supplementary group of which the agent UID is also a member) written by the adapter before spawning the runtime binary. The tmpfs volume holding this file is mounted into both the adapter and agent containers with `fsGroup: <lenny-cred-readers GID>` on the pod's `securityContext`, so the kubelet sets the group ownership and group-writable bit at mount time ŌĆö no `chown` syscall (and therefore no `CAP_CHOWN`) is required in the agent-pod containers, preserving the "all capabilities dropped" invariant in ┬¦13.1. Mode `0440` grants read to owner (adapter) and group (agent via supplementary-group membership) while denying all access to other UIDs, so no third party in the pod can read the file. Credentials are never passed via environment variables (readable via `/proc`, persist in crash dumps, often captured by framework logging), Unix sockets, or MCP servers. The adapter never sends credential material to the agent post-startup. For proxy mode, only the lease token and proxy URL are written to the file ŌĆö no real API keys enter the pod.

   **Runtime credential file contract.** The file always contains a JSON object with a top-level `providers` array ŌĆö even when only one provider is active. Each entry in the array is a credential object with the fields listed below. Runtime authors need only implement a single code path: iterate the `providers` array.
   Single-provider example:
   ```json
   {
     "providers": [
       {
         "leaseId": "lease_abc123",
         "provider": "anthropic_direct",
         "expiresAt": "2026-04-07T10:00:00Z",
         "deliveryMode": "direct",
         "materializedConfig": { "apiKey": "sk-..." }
       }
     ]
   }
   ```

   Multi-provider example:

   Note: `proxyUrl` values are always `https://` ŌĆö see the LLM Reverse Proxy section.
   ```json
   {
     "providers": [
       {
         "leaseId": "lease_abc123",
         "provider": "anthropic_direct",
         "expiresAt": "2026-04-07T10:00:00Z",
         "deliveryMode": "direct",
         "materializedConfig": { "apiKey": "sk-..." }
       },
       {
         "leaseId": "lease_def456",
         "provider": "anthropic_direct",
         "expiresAt": "2026-04-07T10:30:00Z",
         "deliveryMode": "proxy",
         "materializedConfig": { "proxyUrl": "https://proxy.lenny.internal/v1", "leaseToken": "lt-..." }
       }
     ]
   }
   ```

   Fields per entry: `leaseId` (string), `provider` (string ŌĆö the credential provider identifier), `expiresAt` (ISO 8601), `deliveryMode` (`"direct"` or `"proxy"`), `materializedConfig` (object). For `deliveryMode: proxy`, `materializedConfig` contains only `proxyUrl` (string) and `leaseToken` (string); no real API keys appear in the file. For `deliveryMode: direct`, `materializedConfig` contains the full provider credentials (see [┬¦4.9](#49-credential-leasing-service) `materializedConfig` schema). The file is rewritten by the adapter on credential rotation; after rewrite, the adapter sends `credentials_rotated` on the lifecycle channel and awaits `credentials_acknowledged` from the runtime before resuming operation.

5. **Agent crash isolation:** If the agent process crashes, the adapter detects it (socket EOF), reports the failure to the gateway, and does not restart the agent. The gateway handles retry at the session level.

6. **Credential-sensitive RPC logging exclusion:** `AssignCredentials` and `RotateCredentials` RPCs carry credential material in their payloads. These RPCs must be excluded from gRPC access log payload capture, OpenTelemetry span attributes, and any request/response logging middleware. Only the RPC name, lease ID, provider type, and success/failure status should be logged ŌĆö never the `materializedConfig` contents.

7. **MCP server security:** The local MCP servers never expose gateway credentials, mTLS certificates, other sessions, internal Lenny state, or anything about other tenants. The pod sandbox (gVisor/Kata) and network policy are the security boundary, not the protocol. The adapter never advertises the `sampling` MCP capability to the local server. The same manifest-nonce handshake used on the lifecycle channel (see item 1) is required on all intra-pod MCP connections (platform MCP server and each connector MCP server): the connecting process must present the manifest nonce as `params._lennyNonce` in the MCP `initialize` request (see [Section 15.4.3](15_external-api-surface.md#1543-runtime-integration-levels) nonce wire format) before any tools are dispatched. The adapter rejects any MCP connection that does not present a valid nonce. This prevents a compromised child process from calling privileged platform tools (e.g., `lenny/delegate_task`, `lenny/memory_write`, `lenny/request_elicitation`) by directly connecting to the MCP socket without manifest access.

### 4.8 Gateway Policy Engine

**Role:** Centralized policy evaluation on the request path.

**Physically embedded** in edge gateway replicas (not a separate service). Can be split out later if policy evaluation becomes a scaling or organizational bottleneck.

**Evaluators:**

| Module                      | Scope                                                               |
| --------------------------- | ------------------------------------------------------------------- |
| `AuthEvaluator`             | AuthN/AuthZ, user invalidation                                      |
| `QuotaEvaluator`            | Rate limits, token budgets, concurrency limits                      |
| `DelegationPolicyEvaluator` | Depth, fan-out, `DelegationPolicy` tag matching, budget inheritance |
| `RetryPolicyEvaluator`      | Retry eligibility, resume window                                    |
| `AdmissionController`       | Queue/reject/prioritize, circuit breakers                           |

**Backs onto:** SessionStore (sessions, tasks, delegation tree, lineage, retry state), QuotaStore, TokenStore, UserStateStore, RuntimeRegistry

#### `RequestInterceptor` Extension Point

Formalized interceptor interface at gateway phases: `PreAuth`, `PostAuth`, `PreRoute`, `PreDelegation`, `PreExportMaterialization`, `PreMessageDelivery`, `PostRoute`, `PreToolResult`, `PostAgentOutput`, `PreLLMRequest`, `PostLLMResponse`, `PreConnectorRequest`, `PostConnectorResponse`.

**Interceptor chain execution order:** Each phase runs its own interceptor chain independently. Within a phase, interceptors execute in ascending numeric `priority` order (lowest number runs first). An interceptor registered for multiple phases participates in each phase's chain separately. When two interceptors share the same priority within a phase, built-in interceptors run before external ones; among external interceptors with equal priority, registration order is preserved.

**Built-in interceptors** (with default priorities):

| Interceptor                   | Default Priority | Notes                                                                                                                          |
| ----------------------------- | ---------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `AuthEvaluator`               | 100              | Always active                                                                                                                  |
| `QuotaEvaluator`              | 200              | Always active                                                                                                                  |
| `DelegationPolicyEvaluator`   | 250              | Always active; fires at `PreDelegation` only (see [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease))                                                                 |
| `ExperimentRouter`            | 300              | Active when experiments are defined (see [Section 10.7](10_gateway-internals.md#107-experiment-primitives))                                                                         |
| `GuardrailsInterceptor`       | 400              | Disabled by default; deployers wire in external classifiers (AWS Bedrock Guardrails, Azure Content Safety, Lakera Guard, etc.) |
| `RetryPolicyEvaluator`        | 600              | Always active; fires at `PostRoute` only (see [Section 7.3](07_session-lifecycle.md#73-retry-and-resume))                                                                     |

> **`AdmissionController` is a pre-chain gate, not an interceptor (POL-014).** `AdmissionController` (circuit breakers, [┬¦11.6](11_policy-and-controls.md#116-circuit-breakers)) is evaluated **after `AuthEvaluator` completes at `PreAuth`** and **before the `PostAuth` and `PreDelegation` interceptor chains run**. The canonical admission chain ordering is `AuthEvaluator` (`PreAuth`, priority 100) ŌåÆ `AdmissionController` (pre-chain gate) ŌåÆ `PostAuth` / `PreDelegation` interceptor chains (priority > 100). `AdmissionController` is NOT registered in the built-in interceptor priority table above and is NOT subject to interceptor short-circuit semantics. A circuit-breaker REJECT produces a distinct audit event type `admission.circuit_breaker_rejected` (rather than `interceptor.rejected`) carrying the breaker name, open-state reason, and the request snapshot at admission time ŌĆö including the authenticated caller identity (`caller_sub`, `caller_tenant_id`) because evaluation happens after `PreAuth`. External interceptors at priorities 101ŌĆō999 run AFTER circuit-breaker evaluation ŌĆö a tripped breaker short-circuits the request before any interceptor (built-in or external) is invoked.

**Short-circuit behavior:** If any interceptor returns `REJECT`, the chain short-circuits immediately ŌĆö no subsequent interceptors are invoked. The rejection reason is logged and returned to the caller. `MODIFY` results are applied to the payload before passing it to the next interceptor in the chain; `ALLOW` passes the payload through unchanged. **Audit payload on REJECT:** When a REJECT short-circuits the chain, the audit record captures the payload as it existed at the point of rejection ŌĆö including all preceding `MODIFY` transformations applied by earlier interceptors. This is the natural behavior of the sequential pipeline and ensures audit trail accuracy.

External interceptors are invoked via gRPC (like Kubernetes admission webhooks):

> **Note for polyglot deployers:** Custom external interceptors must be implemented as gRPC services (the same protocol used by Kubernetes admission webhooks). HTTP webhook variants are planned as a post-v1 enhancement to lower the implementation barrier for deployers who prefer REST. Any language that supports gRPC code generation can implement an interceptor ŌĆö see the protobuf definition below.

```protobuf
service RequestInterceptor {
  rpc Intercept(InterceptRequest) returns (InterceptResponse);
}

message InterceptRequest {
  string phase = 1;               // e.g. "PreDelegation", "PostAgentOutput"
  string session_id = 2;
  string tenant_id = 3;
  bytes content = 4;              // phase-dependent payload (e.g. TaskSpec.input for PreDelegation)
  map<string, string> metadata = 5;
}

message InterceptResponse {
  enum Action {
    ALLOW = 0;
    REJECT = 1;
    MODIFY = 2;                   // return modified content in `modified_content`
  }
  Action action = 1;
  string reason = 2;              // human-readable; logged and returned to caller on REJECT
  bytes modified_content = 3;     // only used when action = MODIFY
}
```

**Interceptor registration** includes a numeric `priority` field and a `failPolicy`:

| Field        | Type     | Default       | Description                                                                                                                                                                                                                                                             |
| ------------ | -------- | ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `priority`   | int32    | 500           | Execution order; lower runs first. External interceptors must use priority > 100 (priorities 1ŌĆō100 are reserved for built-in security-critical interceptors). The gateway rejects registration of external interceptors with priority Ōēż 100 at configuration load time. |
| `failPolicy` | string   | `fail-closed` | Behavior on timeout or error: `fail-closed` (treat as REJECT) or `fail-open` (treat as ALLOW)                                                                                                                                                                           |
| `timeout`    | duration | 500ms         | Per-interceptor deadline                                                                                                                                                                                                                                                |
| `endpoint`   | string   | (required)    | gRPC address of the interceptor service (e.g., `my-interceptor.my-namespace.svc.cluster.local:50053` for in-cluster services, or an external host:port). For in-cluster services, the interceptor's namespace must also be declared in `gateway.interceptorNamespaces` and the interceptor Deployment's pod template must carry the `lenny.dev/component: interceptor` label so the Helm chart renders, and the rendered gateway egress NetworkPolicy's `podSelector` matches, the corresponding rule (see [Section 13.2](13_security-model.md#132-network-isolation), NET-039, NET-058). |

**Priority reservation.** Priorities 1ŌĆō100 are reserved exclusively for built-in security-critical interceptors (`AuthEvaluator` at 100). External interceptors must register at priority > 100. The gateway validates this constraint at interceptor registration time and rejects any external interceptor configuration with `priority Ōēż 100`, returning a `INVALID_INTERCEPTOR_PRIORITY` error. This prevents external interceptors from running before authentication completes, which would allow `MODIFY` actions to alter requests before auth evaluation or `fail-open` interceptors to produce audit entries from unauthenticated contexts.

**`PreAuth` phase is built-in only.** The `PreAuth` phase runs exclusively within the priority 1ŌĆō100 range and is therefore accessible only to built-in interceptors. External interceptors are **never** invoked at the `PreAuth` phase, regardless of their configured phase list. The gateway enforces this at registration time: any external interceptor registration that includes `PreAuth` in its phase set is rejected with `INVALID_INTERCEPTOR_PHASE`. This means the MODIFY semantics documented for `PreAuth` in the phase payload table apply exclusively to the `AuthEvaluator` built-in ŌĆö no external interceptor can issue a `MODIFY` at `PreAuth` to alter unauthenticated request content before authentication runs. The `ALLOW`/`REJECT` actions (e.g., IP-based blocklisting) are also not available to external interceptors at `PreAuth`; all such controls must be registered at `PostAuth` or later phases where authenticated identity is available.

**Authentication context availability.** The `InterceptRequest.metadata` map contains different context depending on the interceptor's position relative to `AuthEvaluator`:

- **Priority Ōēż 100 (built-in only):** `metadata` contains transport-level info only (source IP, TLS client cert DN). No authenticated identity is available.
- **Priority > 100 (all external interceptors):** `metadata` contains authenticated identity (`user_id`, `tenant_id`, roles/claims) populated by `AuthEvaluator`. External interceptors can always rely on authentication context being present.

**Timeout and failure mode:** External interceptors have a configurable timeout (default: 500ms). On timeout or error, the behavior is controlled by `failPolicy` on the interceptor registration: `fail-closed` (default) or `fail-open`. The default is fail-closed so that a misbehaving interceptor cannot silently bypass policy. Deployers who prefer availability over strict enforcement may override to `fail-open` on a per-interceptor basis. **Cumulative fail-open escalation:** The gateway tracks a per-interceptor rolling failure count. When a `fail-open` interceptor has failed (timeout or error) more than `interceptorFailOpenMaxConsecutive` times (default: 10) within a rolling 5-minute window, the gateway automatically escalates that interceptor to fail-closed for the remainder of the window ŌĆö subsequent requests that would invoke the interceptor are rejected with `INTERCEPTOR_TIMEOUT` as if `failPolicy: fail-closed` were set. The gateway emits an `interceptor.fail_open_escalated` audit event (containing `interceptor_ref`, `failure_count`, `window_seconds`) when escalation triggers and `interceptor.fail_open_restored` when the interceptor begins succeeding again. This prevents a content-filtering interceptor from silently failing open for extended periods without any automatic corrective action.

**Interceptor timeout error code.** When an interceptor times out, the gateway returns `INTERCEPTOR_TIMEOUT` to the caller ŌĆö a distinct error code from `LLM_REQUEST_REJECTED` (which is returned only on an explicit `REJECT` decision from the interceptor). This allows callers and operators to distinguish "policy blocked this request deliberately" from "the interceptor service is degraded or unresponsive." `INTERCEPTOR_TIMEOUT` is classified as `TRANSIENT`, HTTP 503, `retryable: true`. The audit event for a timeout includes `timeout_ms`, `interceptor_ref`, and `phase`. When `failPolicy: fail-closed`, the request is rejected with `INTERCEPTOR_TIMEOUT`. When `failPolicy: fail-open`, the request proceeds with an `interceptor_bypassed` flag in the audit event, and no `INTERCEPTOR_TIMEOUT` error is returned to the caller.

**`failPolicy` change audit event.** When an interceptor's `failPolicy` is updated from `fail-closed` to `fail-open`, the gateway emits an `interceptor.fail_policy_weakened` audit event (see [Section 11.7](11_policy-and-controls.md#117-audit-logging)) containing: `interceptor_ref` (the interceptor name), `old_fail_policy`, `new_fail_policy`, `affected_policy_count` (the number of active `DelegationPolicy` resources whose `contentPolicy.interceptorRef` references this interceptor), and `affected_policy_names` (list of those policy names). This is analogous to the `pool.isolation_warning` event emitted on pool registration ([Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease)) ŌĆö it surfaces a configuration change that silently weakens the effective enforcement posture of active policies. The event is emitted asynchronously and does **not** block the configuration update itself; however, the weakening transition additionally triggers a mandatory cooldown during which affected `delegate_task` and `lenny/send_message` calls reject with `INTERCEPTOR_WEAKENING_COOLDOWN` (see [Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease) rule 5 for the full cooldown semantics, including the `gateway.interceptorWeakeningCooldownSeconds` tunable and the companion `interceptor.weakening_cooldown_active` audit event). The reverse change (`fail-open` to `fail-closed`) emits an `interceptor.fail_policy_strengthened` audit event with the same fields, for completeness, and is **not** subject to the cooldown ŌĆö tightening posture takes effect immediately.

The `PreDelegation` phase fires before the gateway processes a `delegate_task` call, providing the full `TaskSpec.input` as the content payload. This is the hook point for the `contentPolicy.interceptorRef` field in `DelegationPolicy` ([Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease)).

The `PreExportMaterialization` phase fires **per exported file**, after `PreDelegation` has passed and before the gateway persists the file into the child's workspace. It activates only when the parent lease's effective `contentPolicy.scanExportedFiles` is `true` ([Section 8.3](08_recursive-delegation.md#83-delegation-policy-and-lease), [Section 8.7](08_recursive-delegation.md#87-file-export-model)). The interceptor invoked is always the same `contentPolicy.interceptorRef` already in force on the parent's `DelegationPolicy` ŌĆö this phase is not independently registerable. Files exceeding `contentPolicy.maxExportedFileSize` are rejected at admission with `EXPORT_FILE_SCAN_SIZE_EXCEEDED` before any interceptor call is made, bounding the per-invocation payload. Timeout/error on a per-file call follows the interceptor's existing `failPolicy`: `fail-closed` rejects the export with `EXPORT_FILE_SCAN_UNAVAILABLE`; `fail-open` admits the file and emits a `delegation.export_scan_failed_open` audit event. The ┬¦8.3 rule-5 weakening cooldown applies automatically because it is keyed on the interceptor's `failPolicy`, not on the phase.

The `PreMessageDelivery` phase fires before the gateway delivers an inter-session message (`lenny/send_message`) to the target session. The content payload is the message body. This allows deployers to apply the same content scanning (prompt injection detection, content safety) to agent-to-agent messages as they apply to delegation inputs. The `contentPolicy.maxInputSize` limit from the target session's effective `DelegationPolicy` is enforced on inter-session messages in addition to delegation inputs. The `contentPolicy.interceptorRef`, if configured, is invoked at this phase with the message content.

| `PreToolResult`      | Serialized tool result object (JSON): `{ id: string, content: OutputPart[], isError?: bool }` | The result of a tool call before it is delivered back to the agent. `id` correlates with the originating `tool_call.id`.                                                    | May modify `content` (redact sensitive data from tool output, transform structured results). May set `isError` to true to force-fail a tool call. May **not** alter `id`.                             |
| `PostAgentOutput`    | Serialized `OutputPart[]` (JSON)                                                               | The agent's response output before delivery to the client (or to the parent session in a delegation). Includes text, structured data, images, and annotations.              | May modify, redact, or truncate output content. May remove or replace individual `OutputPart` entries. May add compliance annotations. May **not** suppress delivery entirely (use `REJECT` for that). |
| `PreLLMRequest`      | Serialized upstream LLM API request body (JSON)                                                | The full LLM provider request before the gateway forwards it to the upstream endpoint. Proxy mode only ŌĆö does not fire in direct delivery mode. Includes `model`, `messages`/`prompt`, parameters. Credential headers are injected by the proxy after this phase. | May modify any request body field (model, messages, parameters). May **not** alter credential/authorization headers (those are injected by the proxy after the interceptor runs). |
| `PostLLMResponse`    | Serialized upstream LLM API response metadata (JSON)                                           | The LLM provider response before forwarding to the pod. For streaming responses, fires once on the initial response metadata ŌĆö individual stream chunks are not intercepted. Proxy mode only. | May modify response metadata fields. For streaming, modifications apply to initial metadata only ŌĆö individual stream chunks pass through unmodified. |
| `PreConnectorRequest` | Serialized MCP tool call (JSON): `{ tool_name, arguments, connector_id }` | The outgoing tool call before the gateway proxies it to the external connector MCP server. Fires on all connector calls (all are gateway-proxied). | May modify `arguments` (redact, transform). May **not** alter `tool_name` or `connector_id` (would bypass DelegationPolicy connector access control). |
| `PostConnectorResponse` | Serialized MCP tool result (JSON): `{ tool_name, connector_id, content: OutputPart[], isError? }` | The connector's response before forwarding to the pod. | May modify `content` (redact, transform). May set `isError` to true. May **not** alter `tool_name` or `connector_id`. |

**Immutable field enforcement on MODIFY.** When an external interceptor returns `MODIFY`, the gateway enforces the per-phase immutability constraints listed in the table above. Before applying `modified_content`, the gateway snapshots all immutable fields from the pre-modification payload (e.g., `user_id`, `tenant_id` at `PostAuth`/`PreRoute`; `resolved_runtime_name`, `credential_pool_id` at `PostRoute`; `id` at `PreToolResult`; `tool_name`/`connector_id` at `PreConnectorRequest`/`PostConnectorResponse`; `target`, `lease_slice`, `fileExport`, parent session ID, depth counter, fan-out counter at `PreDelegation`; `file_path`, `delegation_context` at `PreExportMaterialization` ŌĆö `file_size` must equal the post-MODIFY byte length and `sha256` is recomputed by the gateway from the returned bytes rather than compared). After deserializing `modified_content`, the gateway compares each snapshotted field against the corresponding field in the modified payload. If any immutable field differs, the gateway rejects the `MODIFY` with `INTERCEPTOR_IMMUTABLE_FIELD_VIOLATION` (category: `POLICY`, HTTP 400), logs the interceptor reference, phase, and violated field names, and short-circuits the chain ŌĆö the original pre-modification payload is preserved and no subsequent interceptors see the illegal modification. This enforcement is not bypassable by the interceptor; it runs inside the gateway process after the gRPC response is received.

**Built-in interceptor field dependencies and MODIFY interaction.** The built-in evaluators each read a specific set of fields from the intercepted payload. Interceptors at priority > 600 run after `RetryPolicyEvaluator` (600) at `PostRoute` ŌĆö this is the only phase where priority > 600 guarantees running after all built-ins. At phases where `GuardrailsInterceptor` is active (priority 400), external interceptors at priority 401ŌĆō599 run **after** guardrails; interceptors at priority 101ŌĆō399 run **before** guardrails. A `MODIFY` returned by an interceptor at any priority is passed to all subsequent interceptors, including built-ins ŌĆö `GuardrailsInterceptor` sees the modified content from upstream MODIFYs. Downstream external `MODIFY` operations (at priority > `GuardrailsInterceptor`'s priority) are not re-evaluated by guardrails. Because each phase runs its own chain, cross-phase priority comparisons do not imply execution ordering: `QuotaEvaluator` (priority 200) fires at `PostAuth` and `DelegationPolicyEvaluator` (priority 250) fires at `PreDelegation` ŌĆö an external interceptor at priority 225 registered for `PostAuth` runs after `QuotaEvaluator` at that phase but never interacts with `DelegationPolicyEvaluator` (which runs in a different phase). Similarly, `AuthEvaluator` (priority 100) fires at `PreAuth`, so an external interceptor at priority 150 registered for `PostAuth` runs before `QuotaEvaluator` within `PostAuth` ŌĆö it does not "run after `AuthEvaluator`" in any chain sense, since `PreAuth` is a separate phase that has already completed. Within a single phase, the priority ordering among co-registered interceptors holds: at `PostAuth`, an external interceptor at priority 150 runs before `QuotaEvaluator` (200); at `PreDelegation`, an external interceptor at priority 225 runs before `DelegationPolicyEvaluator` (250). When an external interceptor earlier in the chain returns `MODIFY`, the modified payload is passed to all subsequent interceptors ŌĆö including built-ins. The following table defines which fields each built-in reads at each phase, priority, and which MODIFY operations affect their behavior:

| Built-in           | Phase(s) where active               | Fields read                                                                                                                                    | Effect of upstream MODIFY on behavior                                                                                                                                               |
| ------------------ | ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `AuthEvaluator`    | `PreAuth` (priority 100)            | Raw MCP request envelope: `method`, `params`, bearer token or mTLS identity, `tenant_id` routing hint in headers                               | No upstream interceptor runs before `AuthEvaluator` (it is the first in chain at priority 100). MODIFY at `PreAuth` is issued only by `AuthEvaluator` itself (built-in only).        |
| `QuotaEvaluator`   | `PostAuth` (priority 200)           | `user_id`, `tenant_id` from authenticated metadata; `TaskSpec.input` byte length (for `maxInputSize` enforcement); session/task type from request method | An external interceptor at priority 101ŌĆō199 returning MODIFY that alters `TaskSpec.input` content will change the byte-length seen by `QuotaEvaluator` for `maxInputSize` checks. An interceptor that modifies `metadata` fields such as `user_id` or `tenant_id` will cause `QuotaEvaluator` to enforce quotas against the **modified** identity ŌĆö modifying these fields is explicitly prohibited (see PostAuth row in the phase table). |
| `DelegationPolicyEvaluator` | `PreDelegation` (priority 250) | `DelegationPolicy` tag matching fields, depth counter, fan-out counter, budget parameters from the parent session lease | An external interceptor at priority 101ŌĆō249 returning MODIFY that alters `TaskSpec.input` metadata (depth hints, delegation parameters) will affect the values `DelegationPolicyEvaluator` reads. External interceptors MUST NOT modify delegation depth, budget, or fan-out fields ŌĆö these fields are explicitly prohibited from external MODIFY (same restriction as `user_id`/`tenant_id`). `DelegationPolicyEvaluator` fires only at `PreDelegation`; it has no effect at other phases. |
| `ExperimentRouter` | `PreRoute` (priority 300)           | `TaskSpec` runtime hint fields, `user_id`, session labels used by experiment assignment                                                         | MODIFY of `TaskSpec` runtime hints by an interceptor at priority 101ŌĆō299 affects which experiment variant `ExperimentRouter` assigns. Modifying `user_id` metadata changes experiment cohort assignment. |
| `GuardrailsInterceptor` | `PreDelegation`, `PreLLMRequest`, `PostLLMResponse`, `PostAgentOutput` (priority 400; deployer-configured) | Full `content` payload for the active phase (see phase payload table above) | MODIFY by an interceptor at priority 101ŌĆō399 changes the content submitted to external classifiers wired into `GuardrailsInterceptor`. Deployers must account for this when evaluating the security value of their guardrail: if an earlier external interceptor can MODIFY delegation input, the guardrail sees the modified version. |
| `RetryPolicyEvaluator` | `PostRoute` (priority 600) | Resolved runtime name, session retry state (`retryCount`, `lastFailureCode`), `RetryPolicy` from the effective `DelegationPolicy` | `RetryPolicyEvaluator` runs after all lower-priority interceptors at `PostRoute`. An external interceptor at priority 101ŌĆō599 returning MODIFY that alters the resolved runtime or failure code fields will affect retry eligibility decisions. External interceptors MUST NOT modify `retryCount` or failure classification fields. `RetryPolicyEvaluator` fires only at `PostRoute`; it has no effect at other phases. |

**Short-circuit interaction with MODIFY:** If an interceptor at priority N returns `MODIFY`, the modified content is passed to the interceptor at priority N+1. If a built-in at priority M (M > N) reads a field that was modified, it operates on the post-modification value ŌĆö there is no mechanism to "re-evaluate" a built-in against the original content after an upstream MODIFY. Deployers who register external interceptors at priorities below a built-in within the same phase (e.g., priority 101ŌĆō199 at `PostAuth`, where `QuotaEvaluator` runs at priority 200) must be aware that any MODIFY they return will be seen by `QuotaEvaluator`. If a quota-relevant field (e.g., input byte length) is enlarged by a MODIFY, `QuotaEvaluator` will enforce quota against the enlarged value ŌĆö this is the intended behavior and does not constitute a bypass.

**LLM interceptor phases.** The `PreLLMRequest` and `PostLLMResponse` phases fire exclusively in the LLM reverse proxy path ([Section 4.9](#49-credential-leasing-service), proxy mode). They have no effect when credentials are delivered directly to the pod (`deliveryMode: direct`), because in direct mode the gateway has no visibility into LLM API traffic. The default timeout for LLM interceptor phases is **100ms** (shorter than the 500ms default for other phases) because they are on the LLM call hot path ŌĆö every millisecond of interceptor latency adds directly to LLM request latency. Deployers may override this via the `timeout` field on the interceptor registration, but should be aware of the latency impact. For `PostLLMResponse` with streaming responses: the interceptor receives the initial response headers/metadata and the first data chunk. It does **not** buffer the full stream ŌĆö buffering would destroy streaming latency and is architecturally unacceptable. Deployers who need full-response content scanning should implement it as an async post-processing step (e.g., a `PostAgentOutput` interceptor or an external audit pipeline), not in the synchronous `PostLLMResponse` interceptor. On `REJECT`, `PreLLMRequest` returns `LLM_REQUEST_REJECTED` to the pod; `PostLLMResponse` returns `LLM_RESPONSE_REJECTED` (see error catalog, [Section 15.1](15_external-api-surface.md#151-rest-api)). Typical use cases: model governance (allowlist/denylist models via `PreLLMRequest`), prompt safety scanning, PII redaction from prompts, cost controls (reject expensive models), compliance logging, and response content filtering.

**Connector interceptor phases.** The `PreConnectorRequest` and `PostConnectorResponse` phases fire on all gateway-proxied connector calls. Unlike LLM interceptors which only apply in proxy mode, connector interceptors always apply ŌĆö pods cannot reach external MCP connector servers directly (NetworkPolicy enforced), so all connector traffic flows through the gateway. The default timeout for connector interceptor phases is **200ms** (connectors are on the tool-call path which is latency-sensitive but less extreme than streaming LLM calls). The `DelegationPolicy`'s `connectorSelector` controls which connectors a session may access; these interceptors govern the **content** of calls to already-authorized connectors. On `REJECT`, `PreConnectorRequest` returns `CONNECTOR_REQUEST_REJECTED` to the pod; `PostConnectorResponse` returns `CONNECTOR_RESPONSE_REJECTED` (see error catalog, [Section 15.1](15_external-api-surface.md#151-rest-api)). Typical use cases: PII redaction before data reaches external systems, audit logging of sensitive tool calls, blocking destructive operations (e.g., reject `delete` operations on a GitHub connector), compliance controls on data leaving the platform, and redacting secrets or sensitive data from connector responses.

### 4.9 Credential Leasing Service

This section covers: Credential Providers (pluggable per-LLM-provider interfaces), Credential Pools (admin-managed credential sets), Credential Leases (session-scoped assignments), Credential Policy (tenant-level configuration for source preference and delivery mode), the Fallback Flow (rotation on provider failure), Proactive Lease Renewal, the LLM Reverse Proxy (credential-injecting proxy mode), Semantic Caching, the Credential Router (cost/latency-aware routing), Governance Boundaries, Emergency Revocation, Security Boundaries, KMS Key Rotation, and Credential Audit Events.

**Role:** Supplies runtime pods with the credentials they need to access their backing LLM provider (Anthropic API, AWS Bedrock, Vertex AI, etc.). This is distinct from the connector credential flow in [Section 4.3](#43-token-service) ŌĆö this is about the runtime's own LLM access, not downstream tool OAuth.

**Design principle:** Runtimes receive **short-lived credential leases**, never long-lived API keys or root credentials. The Token Service owns all durable credential material.

#### Credential Provider

A pluggable interface per LLM provider type. Each provider knows how to mint usable runtime credentials from its source material.

| Provider           | Source Material        | Runtime Receives                                             |
| ------------------ | ---------------------- | ------------------------------------------------------------ |
| `anthropic_direct` | API key                | Static API key (long-lived; lease TTL governs pod access duration, not key lifetime) |
| `aws_bedrock`      | IAM role / access keys | Short-lived STS session credentials + region/endpoint config |
| `vertex_ai`        | GCP service account    | Short-lived access token + project/region config             |
| `azure_openai`     | Azure AD / API key     | Short-lived token + endpoint config                          |
| `github`           | GitHub App credentials | Short-lived installation access token + app/installation IDs |
| `vault_transit`    | Vault token            | Short-lived Vault token + vault address + transit path/key   |
| Custom             | Provider-specific      | Provider-specific config bundle                              |

New providers are added by implementing the `CredentialProvider` interface ŌĆö no gateway changes required.

#### Credential Pool

An admin-managed, **tenant-scoped** set of credentials for a given provider. Each `CredentialPool` belongs to exactly one tenant; the `CredentialPoolStore` tables carry a `tenant_id` column and are covered by the same RLS policies as all other tenant-scoped tables (see [Section 4.2](#42-session-manager)). A `tenant-admin` can create, update, and delete credential pools for their own tenant via the admin API. A `platform-admin` can manage credential pools across all tenants. Cross-tenant credential pool access is impossible by construction ŌĆö RLS enforces this at the database level, and the gateway validates `tenant_id` on every pool operation.

Deployers register pools via configuration:

```yaml
credentialPools:
  - name: claude-direct-prod
    provider: anthropic_direct
    credentials:
      - id: key-1
        secretRef: lenny-system/anthropic-key-1 # K8s Secret reference
      - id: key-2
        secretRef: lenny-system/anthropic-key-2
    assignmentStrategy: least-loaded # least-loaded | round-robin | sticky-until-failure
    maxConcurrentSessions: 10 # per credential
    cooldownOnRateLimit: 60s
    leaseTTLSeconds: 3600  # optional override; defaults by provider type (see table below)

  - name: bedrock-us-east-prod
    provider: aws_bedrock
    credentials:
      - id: role-1
        roleArn: arn:aws:iam::123456789:role/lenny-bedrock
        region: us-east-1
    assignmentStrategy: sticky-until-failure
    maxConcurrentSessions: 50
    # leaseTTLSeconds omitted ŌĆö uses provider default of 3600s for aws_bedrock

  - name: github-prod
    provider: github
    hostPatterns: ["github.com"] # VCS-pool-only; required for gitClone URLŌåÆpool routing
    credentials:
      - id: app-1
        secretRef: lenny-system/github-app-1
    assignmentStrategy: least-loaded
    maxConcurrentSessions: 20
```

**`hostPatterns` ŌĆö VCS-pool-only field.** Required on pools whose provider issues leases for `vcs.<provider>.{read|write}` scopes (v1 built-in: `github`; custom providers may implement others such as GitLab, Bitbucket, Gitea, self-hosted). Each entry is either an exact host (`github.com`, `git.example.com`) or a `*.suffix` wildcard (`*.ghe.example.com`) matching the host component of a `gitClone.url`. `gitClone.url` is restricted to HTTPS in v1 ([┬¦14](14_workspace-plan-schema.md) `gitClone.url` restrictions); SSH URL forms are deferred post-V1 ([┬¦21.9](21_planned-post-v1.md#21-planned--post-v1)), so host extraction in v1 operates only on parsed HTTPS URLs. At session creation the gateway parses the HTTPS URL's host, compares it against `hostPatterns` across all VCS pools in the tenant, and uses the matched pool to resolve the `auth.leaseScope`'s `<provider>` segment. Exactly one VCS pool must match; zero matches yields `422 GIT_CLONE_AUTH_UNSUPPORTED_HOST` and multiple matches yield `422 GIT_CLONE_AUTH_HOST_AMBIGUOUS`. Non-VCS pools (LLM providers) ignore this field.

**`leaseTTLSeconds` ŌĆö per-pool lease TTL override.** Sets the maximum duration (in seconds) of a credential lease issued from this pool. Each lease's `expiresAt` is computed as `issuedAt + min(leaseTTLSeconds, providerMaxTTL)` where `providerMaxTTL` is the hard ceiling enforced by the provider type (e.g., AWS STS sessions cannot exceed 12 hours). When omitted, the provider-type default applies:

| Provider | Default `leaseTTLSeconds` | Provider ceiling (`providerMaxTTL`) | Notes |
|---|---|---|---|
| `anthropic_direct` | **3600** (1 hour) | None (key is long-lived; TTL is synthetic) | The API key itself does not expire; Lenny enforces the TTL synthetically by refusing to honor leases past `expiresAt`. In direct delivery mode, the adapter MUST set a local timer for each credential lease's `expiresAt`. When the timer fires without a replacement lease having been delivered via `RotateCredentials`, the adapter deletes the credential file (`/run/lenny/credentials.json`) for that provider's entry and reports `AUTH_EXPIRED` on the control channel, triggering the standard fallback flow. This ensures that a long-lived `anthropic_direct` key cannot be used by the runtime beyond the lease boundary even though the key itself remains valid. In proxy delivery mode, the gateway enforces expiry server-side by rejecting proxy requests for expired leases, so no adapter-side timer is needed. |
| `aws_bedrock` | **3600** (1 hour) | 43200 (12 hours, STS maximum) | STS `AssumeRole` call uses `DurationSeconds = min(leaseTTLSeconds, 43200)`. |
| `vertex_ai` | **3600** (1 hour) | 3600 (GCP access token maximum) | GCP access tokens cannot exceed 1 hour; `leaseTTLSeconds > 3600` is silently capped. |
| `azure_openai` | **3600** (1 hour) | 86400 (24 hours, Azure AD token maximum) | Azure AD tokens may be up to 24 hours; API-key-backed pools use the same synthetic TTL as `anthropic_direct`. |
| `github` | **3600** (1 hour) | 3600 (GitHub installation token maximum) | GitHub installation access tokens cannot exceed 1 hour. |
| `vault_transit` | **3600** (1 hour) | Vault policyŌĆōcontrolled | The Vault token TTL must be at least `leaseTTLSeconds`; a shorter Vault TTL takes precedence. When materializing a `vault_transit` lease, the Token Service sets `expiresAt = min(issuedAt + leaseTTLSeconds, vaultTokenExpiryTime)` so that `renewBefore` is computed from the actual Vault token expiry, not the pool's `leaseTTLSeconds`. This prevents silent credential failure when Vault policy TTL is shorter than `leaseTTLSeconds`. |
| Custom | **3600** (1 hour) | Provider-specific | Custom providers may override the default via `CredentialProvider.DefaultLeaseTTL()`. |

The `renewBefore` field in the issued `CredentialLease` is set to `expiresAt - renewBeforeBuffer`, where `renewBeforeBuffer` defaults to 300 seconds (5 minutes) and is configurable per pool via `renewBeforeBufferSeconds`. See "Proactive Lease Renewal" below for the renewal mechanism triggered by `renewBefore`.

Pool sizing scales with tier ŌĆö at Tier 3, deployers may need hundreds to over a thousand credentials per pool depending on `maxConcurrentSessions` per credential. See [Section 17.8.2](17_deployment-topology.md#1782-capacity-tier-reference) ("Credential pool sizing") for the sizing formula and per-tier starting values.

#### Credential Storage Topology

API keys referenced by `secretRef` are stored as Kubernetes Secrets. The topology and KMS key hierarchy are defined here.

**Secret-per-credential topology.** Each credential entry in a `CredentialPool` uses a **separate Kubernetes Secret**. For example, a pool with 3 credentials has 3 Secrets: `lenny-system/anthropic-key-1`, `lenny-system/anthropic-key-2`, `lenny-system/anthropic-key-3`. This topology is required (not optional) for the following reasons:

1. **Revocation granularity.** Emergency revocation of a single compromised credential (`POST /v1/admin/credential-pools/{name}/credentials/{credId}/revoke`) invalidates only the specific `secretRef` Secret. Shared-Secret topologies would require re-keying all credentials in the pool on a single credential compromise.
2. **RBAC granularity.** The Token Service's ServiceAccount has `get` permission scoped to specific Secret names (via `ResourceNames`), not `get` on all Secrets in `lenny-system`. This prevents a compromised Token Service from reading unrelated Secrets.
3. **Secret rotation isolation.** Rotating one credential does not touch the Secrets for others in the pool, enabling zero-downtime rotation (new Secret created, pool entry updated, old Secret deleted after lease drain).

**Naming convention.** Credential Secrets must follow the pattern `lenny-{pool-name}-{credential-id}` in the `lenny-system` namespace. The Helm chart creates Secrets for bootstrap credentials; operationally-added credentials are created by the operator and referenced via `secretRef`. The Token Service validates on startup that all `secretRef` values referenced in the database are accessible via its RBAC grants.

**Secret shape.** Each credential Secret has one or more `data` keys whose names are the provider's canonical key fields:

| Provider | Secret key | Value |
|---|---|---|
| `anthropic_direct` | `apiKey` | Anthropic API key (`sk-ant-ŌĆ”`) |
| `aws_bedrock` (role) | `roleArn` | IAM role ARN |
| `aws_bedrock` (access keys) | `accessKeyId` and `secretAccessKey` | Two keys in one Secret |
| `vertex_ai` | `serviceAccountJson` | GCP service account JSON |
| `azure_openai` (API key) | `apiKey` | Azure OpenAI API key |
| `azure_openai` (Azure AD) | `clientId`, `clientSecret`, and `tenantId` | Azure AD service principal credentials (three keys in one Secret); Token Service uses these to request short-lived AD access tokens |

**Encryption at rest ŌĆö KMS key hierarchy.** Kubernetes Secrets are stored in etcd. Without etcd encryption, Secrets are readable in plaintext from etcd snapshots or backups. The required configuration depends on deployment topology:

| Topology | Encryption requirement | Implementation |
|---|---|---|
| Managed K8s (EKS) | **Required** | Enable EKS envelope encryption with AWS KMS: `aws eks create-cluster --encryption-config resources=secrets,provider.key_arn=<CMK_ARN>`. Verify: `aws eks describe-cluster --query 'cluster.encryptionConfig'`. |
| Managed K8s (GKE) | **Required** | Enable CMEK (Customer-Managed Encryption Keys): `gcloud container clusters update <cluster> --database-encryption-key=<KMS_KEY>`. Or use GKE's default encryption (Google-managed key) which encrypts etcd at rest without operator action. |
| Managed K8s (AKS) | **Required** | Enable Azure Disk Encryption or use Key Vault for etcd encryption. Default AKS etcd is not encrypted at rest without explicit configuration. |
| Self-managed | **Required** | Configure `EncryptionConfiguration` on the API server with `aescbc`, `aesgcm`, or `kms` provider for the `secrets` resource type. After enabling, run `kubectl get secrets --all-namespaces -o json | kubectl apply -f -` to re-encrypt existing Secrets with the new provider. |

The preflight Job emits a non-blocking warning when etcd Secret encryption cannot be verified (see [Section 17.6](17_deployment-topology.md#176-packaging-and-installation)). Operators are responsible for verifying encryption is active before storing production credentials.

**RBAC for the Token Service.** The Token Service's ServiceAccount must have the following minimum RBAC to read credential Secrets:

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: lenny-token-service-secrets
  namespace: lenny-system
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get"]
    resourceNames: []  # populated by Helm template from credentialPool secretRef list
```

The `resourceNames` list is populated at install time from all `secretRef` values declared in bootstrap configuration. Operationally-added credentials require a manual RBAC patch or a re-run of the `helm upgrade` with updated values. `lenny-ctl admin credential-pools add-credential` includes a step that emits the required RBAC patch command.

**Admin-time RBAC live-probe (required).** Because the Token Service's `resourceNames` grants are populated only at install time, any admin path that writes a new `secretRef` value can reference a Secret that is not yet in the Token Service's allowed `resourceNames` list. Without a fail-fast check, the pool row is persisted in Postgres and the missing grant surfaces only at lease-materialization time as `CREDENTIAL_POOL_EXHAUSTED` ŌĆö indistinguishable from a transient pool-exhaustion event by clients, and blocking every session the new credential was intended to unblock. To close this gap, the admin handlers for **all three paths that can introduce a new `secretRef`** ŌĆö pool creation (`POST /v1/admin/credential-pools`, iterated over every entry in `credentials[].secretRef`), credential addition (`POST /v1/admin/credential-pools/{name}/credentials`), and credential update (`PUT /v1/admin/credential-pools/{name}/credentials/{credId}` when the update changes `secretRef`) ŌĆö MUST execute a live-probe that verifies the Token Service's own ServiceAccount can `get` each referenced Secret before committing the write. The probe is **Token-Service-owned**: the gateway-hosted admin handler MUST call a Token-Service-exposed internal probe endpoint over the existing gatewayŌåöToken-Service mTLS link, and the Token Service answers by invoking its own `SelfSubjectAccessReview` (and, on `ALLOWED`, a `get` on the named Secret to confirm the object exists) under its own ServiceAccount, returning one of `{ALLOWED, DENIED, NOT_FOUND}`. Two implementation paths are **explicitly forbidden**: (a) the gateway impersonating the Token Service SA via `TokenRequest` or the `Impersonate-*` Kubernetes API headers ŌĆö the gateway's own ServiceAccount MUST NOT hold `serviceaccounts/token` or `impersonate` verbs on the Token Service SA, and the install-time RBAC bundle MUST NOT grant either (a preflight check rejects Helm values that would produce such a grant); and (b) the gateway performing a `SelfSubjectAccessReview` under its own ServiceAccount ŌĆö the gateway's RBAC differs from the Token Service's, so such a probe is semantically meaningless and MUST NOT be used as a substitute. Error mapping: a Token Service response of `DENIED` or `NOT_FOUND` maps to `422 CREDENTIAL_SECRET_RBAC_MISSING` (category: `PERMANENT`); the error body names the missing `resourceName`(s) (for pool creation, the full list of failing Secrets is returned so the operator can patch RBAC in a single round-trip), and the response `details` field includes the exact `kubectl` / Helm RBAC patch command that `lenny-ctl admin credential-pools add-credential` would emit so the operator can apply the grant and retry. A probe transport failure (Token Service unreachable, mTLS handshake failure, upstream Kubernetes API timeout, or any non-deterministic error that prevents the Token Service from returning a definitive `ALLOWED`/`DENIED`/`NOT_FOUND` verdict) maps to `503 CREDENTIAL_PROBE_UNAVAILABLE` (category: `TRANSIENT`) and the admin write is rejected ŌĆö the handler MUST NOT fail open by persisting the CR change when the probe could not be evaluated. This distinct 5xx code lets operators discriminate a denied probe (fix RBAC) from a failed probe (fix Token Service reachability) without either case silently allowing an unprobed `secretRef` to land in Postgres. The probe is skipped for bootstrap-seeded pools (`POST /v1/admin/bootstrap` and `bootstrap.credentialPools` in Helm values) because the bootstrap path renders the RBAC `resourceNames` list from the same values block atomically. This converts a latent runtime failure into an admin-time failure with a clear remediation step, and guarantees that a `tenant-admin` who lacks rights to patch the Token Service RBAC Role cannot succeed in creating a pool whose Secrets are unreadable.

**Large pool Secret management (Tier 3).** At Tier 3 with hundreds of credentials, managing individual Secrets via `helm upgrade` on every credential addition becomes impractical. For Tier 3 deployments, operators should use an external secrets manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault) with the External Secrets Operator (`external-secrets.io`) to synchronize secrets into Kubernetes Secrets automatically. The `secretRef` in `CredentialPool` always points to a Kubernetes Secret ŌĆö the External Secrets Operator handles the synchronization from the external source. This preserves the Secret-per-credential topology while eliminating manual Helm updates.

#### Pre-Claim Credential Availability Check

Before claiming a warm pod ([Section 7.1](07_session-lifecycle.md#71-normal-flow), step 4), the gateway computes the intersection of the Runtime's `supportedProviders` and the tenant's `credentialPolicy.providerPools` keys. For each provider in the intersection, it verifies that the resolved credential source (pool, user, or fallback chain) has at least one assignable credential, evaluating pool utilization (`active leases < maxConcurrentSessions` for at least one credential), cooldown status, and health scores. If **no provider** in the intersection has an available credential, the gateway rejects the request immediately with `CREDENTIAL_POOL_EXHAUSTED` (category: `POLICY`) ŌĆö no pod is claimed, preventing a pod from being wasted on a session that would fail at credential assignment. The session can proceed if at least one provider has an available credential.

Because the availability check and the actual lease assignment are not atomic, a race condition exists where a credential becomes unavailable between the check and the assignment. If this occurs, the gateway releases the claimed pod back to the warm pool and returns `CREDENTIAL_POOL_EXHAUSTED` to the client. The metric `lenny_gateway_credential_preclaim_mismatch_total` (counter, labeled by pool and provider) tracks how often the pre-claim check passes but the subsequent assignment fails, letting operators detect pool contention and tune pool sizing.

#### Credential Lease

A session-scoped assignment from the gateway to a runtime pod. A session may hold **multiple leases** ŌĆö one per provider in the intersection of Runtime `supportedProviders` and tenant `credentialPolicy.providerPools`:

```json
{
  "leaseId": "cl_abc123",
  "sessionId": "s_xyz789",
  "provider": "anthropic_direct",
  "poolId": "claude-direct-prod",
  "credentialId": "key-1",
  "materializedConfig": {
    "apiKey": "sk-ant-...<short-lived or scoped>",
    "baseUrl": "https://api.anthropic.com"
  },
  "expiresAt": "2026-03-23T15:30:00Z",
  "renewBefore": "2026-03-23T15:25:00Z",
  "fallbackAllowed": true,
  "source": "pool"
}
```

The `source` field indicates how the credential was obtained: `pool` (admin-managed pool), `user` (user-scoped pre-registered credential). The runtime receives the `materializedConfig` ŌĆö a provider-specific bundle with everything needed to authenticate. It never receives the pool's root secret or long-lived key. Each provider's lease rotates independently via its own fallback chain; `maxRotationsPerSession` applies to the total rotation count across all providers.

#### `materializedConfig` Schema by Provider

`materializedConfig` is a discriminated map keyed on the `provider` field of the enclosing `CredentialLease`. Each built-in provider defines a fixed set of fields. Custom providers may use any fields; the Token Service passes them through without validation.

**Encoding conventions:**

- All secret values (API keys, tokens, passwords) are delivered as UTF-8 plaintext strings inside the lease. The lease itself is transmitted over mTLS and written to a tmpfs-backed file (mode `0440`, owner = adapter UID, group = `lenny-cred-readers` via `fsGroup`; see [┬¦4.7](#47-runtime-adapter) item 4 for the full ownership scheme) in the pod ŌĆö no additional encoding layer is applied. Runtimes MUST NOT log or persist these values.
- Optional fields are omitted when not applicable (not set to `null`).
- `expiresAt` (where present) is an ISO 8601 UTC timestamp matching the enclosing lease `expiresAt`.

**In proxy delivery mode** (`deliveryMode: proxy`), the Token Service omits all per-provider secret fields and `materializedConfig` collapses to a uniform shape regardless of the upstream provider ŌĆö the runtime speaks a provider-agnostic dialect to Lenny's proxy, which translates to the upstream provider's native wire format in-process before forwarding (see "LLM Reverse Proxy" below).

#### Credential Policy

Attached to the tenant configuration, controls how credentials are selected and managed across all sessions for that tenant. The Runtime definition declares what providers it *can* use (`supportedProviders`, `credentialCapabilities` ŌĆö see [Section 5.1](05_runtime-registry-and-pool-model.md#51-runtime)); the tenant-level `credentialPolicy` declares what credentials *are* available and how they are sourced. At session creation, the gateway intersects Runtime capabilities with tenant policy (see "Intersection at session creation" below).

```yaml
# Tenant-level credential policy
credentialPolicy:
  preferredSource: pool  # pool | user | prefer-user-then-pool | prefer-pool-then-user
  providerPools:
    anthropic_direct:
      defaultPool: claude-direct-prod
      fallback:
        order: [claude-direct-prod, claude-direct-backup]
    aws_bedrock:
      defaultPool: bedrock-us-east-prod
      fallback:
        order: [bedrock-us-east-prod, bedrock-us-west-prod]
  fallback:
    cooldownOnRateLimit: 60s
    maxRotationsPerSession: 3  # total across all providers in a session
  userCredentialsEnabled: true  # whether user-scoped credentials (pre-registered via POST /v1/credentials) are used
```

**Intersection at session creation.** When a session is created, the gateway computes the set of authorized providers as the intersection of the Runtime's `supportedProviders` ([Section 5.1](05_runtime-registry-and-pool-model.md#51-runtime)) and the tenant's `credentialPolicy.providerPools` keys. Only providers present in both sets are eligible for credential assignment. If the intersection is empty, session creation fails with `CREDENTIAL_POOL_EXHAUSTED`. The gateway then assigns a `CredentialLease` for each provider in the intersection that has an available credential (see "Multi-provider lease assignment" below). A session can start if at least one provider in the intersection has a successfully assigned credential ŌĆö providers without available credentials are skipped, and the session receives credentials only for the providers that were successfully assigned.

#### Three Credential Modes

| Mode                     | How It Works                                                                                                                                                                                                                                                                                                                                             | Use Case                                              |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------- |
| **Pool (admin-managed)** | Gateway picks a credential from the per-provider pool mapping (`providerPools`) using the assignment strategy. Runtime gets a lease per provider.                                                                                                                                                                                                        | Shared team/org API keys, service accounts            |
| **User-scoped**          | User pre-registers their own credential via `POST /v1/credentials`. Token Service stores it with the same envelope encryption via KMS used for OAuth refresh tokens ([Section 4.3](#43-token-service)) ŌĆö user-supplied API keys are T4 Restricted data and receive identical encryption-at-rest, key rotation, and access-control treatment. Runtime gets a lease. | "Bring your own API key", user-specific Bedrock roles |
| **Fallback chain**       | Policy determines precedence. E.g., prefer user credential, fall back to pool on failure.                                                                                                                                                                                                                                                                | Flexible enterprise deployments                       |

> The `preferredSource` field determines which mode is used. There are effectively three configurations: pool-only (`preferredSource: pool`), user-only (`preferredSource: user`), or fallback chain (`preferredSource: prefer-user-then-pool` or `prefer-pool-then-user`). Per-provider fallback across pools is configured via each provider's `fallback.order` list within `providerPools`.

> **User-scoped credential storage:** User-supplied credentials (registered via `POST /v1/credentials`) are stored in `TokenStore` (Postgres) using the same envelope encryption via KMS as OAuth refresh tokens ([Section 4.3](#43-token-service)). Each credential is encrypted with a per-tenant data encryption key (DEK), itself wrapped by the KMS key encryption key (KEK). User-scoped credentials are subject to the same KMS key rotation procedure ([Section 4.9.1](#491-kms-key-rotation-procedure)), the same access-control boundaries (tenant-scoped RLS), and the same data classification (T4 Restricted, [Section 12.9](12_storage-architecture.md#129-data-classification)). The Token Service decrypts user-scoped credentials on demand when materializing a credential lease ŌĆö the plaintext credential is never persisted outside the Token Service's in-memory processing context.

#### Pre-Authorized Credential Flow

The pre-authorized flow allows users to register their own credentials once per provider, after which every session automatically resolves the credential based on the authenticated user's identity. This works identically across all client interfaces (MCP, REST, OpenAI-compatible, A2A) ŌĆö no interactive channel is required.

**User credential management endpoints:**

| Method   | Endpoint                                     | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| -------- | -------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `POST`   | `/v1/credentials`                            | Register a credential. Body: `provider` (string, must match a known `CredentialProvider` type), `credential` (the secret material ŌĆö API key, access token, etc.), optional `label` (human-readable name). | `GET`    | `/v1/credentials`                            | List the authenticated user's registered credentials. Returns: array of `{credential_ref, provider, label, created_at, last_used_at}`. | `PUT`    | `/v1/credentials/{credential_ref}`           | Rotate (replace) the secret material for an existing user-scoped credential. Body: `credential` (new secret material), optional `label`. Use `revoke` for emergency lease invalidation; use `DELETE` for non-disruptive cleanup. The user can re-register a new credential for the same provider after either operation. |
| `DELETE` | `/v1/credentials/{credential_ref}`           | Remove a registered credential. The Token Service deletes the credential registration record from `TokenStore` and emits a `credential.deleted` audit event (fields: `tenant_id`, `user_id`, `provider`, `credential_ref`). To immediately invalidate active leases, use `POST .../revoke` instead. |

**Scoping rules:**

- Credentials are scoped to `(tenant_id, user_id, provider)`. A user can register at most one credential per provider (re-registering for the same provider replaces the previous one).
- `tenant_id` and `user_id` are derived from the authenticated JWT ŌĆö never supplied by the client.
- The Token Service validates that `provider` matches a `CredentialProvider` type known to the platform. Unknown providers are rejected with `VALIDATION_ERROR`.

**Resolution at session creation:**

- When a session is created and the `credentialPolicy.preferredSource` includes user credentials (`prefer-user-then-pool`, `prefer-pool-then-user`, or `user` only), the gateway calls the Token Service to look up a credential for `(tenant_id, user_id, provider)` for each provider in the intersection of the Runtime's `supportedProviders` and the tenant's `credentialPolicy.providerPools`.
- If found: the Token Service decrypts it and materializes a `CredentialLease` ŌĆö same as pool-managed credentials.
- If not found: depends on fallback configuration. With fallback enabled, falls through to pool. Without fallback, session creation fails with `USER_CREDENTIAL_NOT_FOUND` (see error catalog, [Section 15.1](15_external-api-surface.md#151-rest-api)).
- `last_used_at` is updated on each successful resolution.
- **Revoked credentials** (`revoked` status) are treated identically to not-found: the gateway skips the revoked credential and falls through to pool per the fallback configuration.

**`userCredentialsEnabled` semantics:**

- `userCredentialsEnabled: true` ŌĆö the gateway resolves user-scoped credentials from the credential store (registered via `POST /v1/credentials`). If no credential is registered for the requested provider, the behavior depends on the fallback configuration: with fallback enabled (`preferredSource: prefer-user-then-pool`), falls through to pool; without fallback (`preferredSource: user`), session creation fails with `USER_CREDENTIAL_NOT_FOUND`.
- `userCredentialsEnabled: false` ŌĆö user-scoped credentials are not used regardless of what is registered.

> **Why credential elicitation via MCP is not supported.** Credential elicitation via the MCP `lenny/request_input` channel is intentionally not supported. The MCP message path is subject to logging, tracing, interceptor inspection, and transcript persistence ŌĆö credential material flowing through this path would be exposed to all of these systems. User-scoped credentials must be registered via the dedicated `POST /v1/credentials` endpoint, which has explicit logging exclusions and delivers credentials directly to the Token Service without traversing the MCP message chain.

**Storage:** User credentials are stored in the `TokenStore` (Postgres) with the same envelope encryption, RLS, and access controls described in the "User-scoped credential storage" callout above. See that callout for the full encryption lifecycle.

**Security considerations:**

- The `POST /v1/credentials` and `PUT /v1/credentials/{credential_ref}` endpoints carry secret material. They MUST be excluded from HTTP access log payload capture, OpenTelemetry span attributes, and request/response logging middleware ŌĆö the same treatment as `AssignCredentials` and `RotateCredentials` RPCs ([Section 4.7](#47-runtime-adapter) item 6). Only the `credential_ref`, `provider`, and success/failure status should be logged.
- A `credential.registered` audit event is emitted on registration (fields: `tenant_id`, `user_id`, `provider`, `credential_ref`). A `credential.deleted` audit event is emitted on deletion. A `credential.rotated` audit event is emitted on rotation. A `credential.user_revoked` audit event is emitted on user-initiated revocation.

#### Fallback Flow

Fallback operates **per-provider**. When the runtime reports a credential failure for a specific provider, only that provider's fallback chain is walked. Other providers' leases are unaffected.

```
1. Runtime reports RATE_LIMITED (or AUTH_EXPIRED, PROVIDER_UNAVAILABLE) to gateway,
   identifying the affected provider
2. Gateway marks the affected provider's lease as degraded, records cooldown timestamp
3. Gateway increments the session's rotation counter (shared across all providers).
   If the counter exceeds maxRotationsPerSession (default: 3), the fallback chain
   is exhausted:
   - The session is terminated with CREDENTIAL_FALLBACK_EXHAUSTED (category: POLICY)
   - A credential.fallback_exhausted audit event is emitted (fields: session_id,
     rotation_count, last_failure_reason, fallback_chain_attempted)
   - The metric lenny_gateway_credential_fallback_exhausted_total (counter,
     labeled by pool, provider, error_type) is incremented
   - The client receives a terminal error; no further rotation attempts are made
4. Gateway evaluates the affected provider's fallback chain (from
   credentialPolicy.providerPools.{provider}.fallback.order):
   a. Same provider, different credential from pool?
   b. User credential available as fallback?
5. Gateway issues replacement CredentialLease for the affected provider only
6. Gateway pushes the replacement credential to runtime via RotateCredentials RPC,
   targeting the specific provider's lease
7. Delivery depends on runtime level and restart requirements:
   - Full: adapter delivers via lifecycle channel; runtime rebinds the specific
     provider in-place. Other providers' credentials are untouched.
   - Standard/Basic: gateway triggers Checkpoint ŌåÆ pod restart ŌåÆ Resume with
     new lease map (see Section 4.7 level table)
```

**`rotationTrigger` enum.** Every `RotateCredentials` RPC carries a `rotationTrigger` value identifying the cause of the rotation. The value is recorded in the internal rotation context, surfaced in audit events ([┬¦4.9.2](#492-credential-audit-events)) and metric labels ([┬¦16.1](16_observability.md#161-metrics)), and governs whether the in-flight gate ceiling at [┬¦4.7](#47-runtime-adapter) applies. The full enum is:

| Value | Emitted by | Cause |
| ----- | ---------- | ----- |
| `proactive_renewal` | Proactive Lease Renewal loop (see below) | Lease is approaching `renewBefore` and is being refreshed before expiry. No fault occurred; the old credential is still valid. |
| `fault_rate_limited` | Fallback Flow, step 1 (`error_type: RATE_LIMITED`) | Runtime reported the upstream provider throttled the current credential. |
| `fault_auth_expired` | Fallback Flow, step 1 (`error_type: AUTH_EXPIRED`) | Runtime reported the lease expired or was rejected by the provider. |
| `fault_provider_unavailable` | Fallback Flow, step 1 (`error_type: PROVIDER_UNAVAILABLE`) | Runtime reported the upstream provider endpoint was unreachable. |
| `emergency_revocation` | Operator via `POST /v1/admin/credential-pools/{name}/credentials/{credId}/revoke` (or the pool-wide variant) | Operator revoked a pool credential; all active leases backed by the credential are force-rotated. |
| `user_credential_rotated` | User via `PUT /v1/credentials/{credential_ref}` | User replaced the secret material of a user-scoped credential; active leases backed by it are force-rotated. |
| `user_credential_revoked` | User via `POST /v1/credentials/{credential_ref}/revoke` | User revoked a user-scoped credential; active leases backed by it are force-rotated. |

Only `proactive_renewal` rotations carry an unbounded in-flight wait at [┬¦4.7](#47-runtime-adapter) "Revocation-triggered rotation ceiling"; every other trigger is subject to the 300-second ceiling because the old credential is no longer trustworthy (fault, revocation, or user-initiated replacement).

#### Proactive Lease Renewal

The `renewBefore` timestamp in a `CredentialLease` is not decorative ŌĆö it drives a gateway-side proactive renewal loop that prevents lease expiry from interrupting active sessions.

**Renewal loop design:**

The gateway maintains a min-heap of active leases ordered by `renewBefore`. A background goroutine (`CredentialRenewalWorker`) wakes whenever the earliest `renewBefore` in the heap is reached. On wake:

1. **Check session liveness.** If the session owning the lease has ended, remove the lease from the heap and skip renewal.
2. **Issue a replacement lease.** The renewal worker calls the same path as `AssignCredentials`, selecting a credential from the same pool and credential ID (or a replacement credential if the original is in cooldown or revoked). For user-scoped leases (`source: user`), if the `credential_ref` no longer exists (user deleted the credential between tasks or during a long-lived session), the gateway falls through to the tenant's `credentialPolicy` fallback chain ŌĆö e.g., from `prefer-user-then-pool` to the next pool in the chain. If no fallback source is available, the renewal fails and enters the retry path described under "Renewal failure handling" below. The new lease has a fresh `expiresAt = now + leaseTTLSeconds` and a corresponding `renewBefore`.
3. **Push the replacement credential to the runtime** via the standard `RotateCredentials` RPC, identical to fault-driven rotation (see Fallback Flow, steps 5ŌĆō7). The runtime rebinds in-place (hot rotation) or undergoes checkpoint-and-restart, per its `credentialCapabilities.hotRotation` declaration.
4. **Reschedule.** The new lease's `renewBefore` is inserted into the renewal heap.

**Renewal does not consume `maxRotationsPerSession`.**  The rotation counter (`rotationCount` in the session record) is incremented only for **fault-driven** rotations (those triggered by `RATE_LIMITED`, `AUTH_EXPIRED`, or `PROVIDER_UNAVAILABLE` in step 3 of the Fallback Flow). Proactive renewals triggered by `renewBefore` are identified by `rotationTrigger: proactive_renewal` in the internal rotation context and are explicitly excluded from the counter increment. This preserves the full `maxRotationsPerSession` budget for fault handling regardless of how many proactive renewals occur during a long-lived session. The `credential.renewed` audit event (`rotationTrigger: proactive_renewal`) is emitted on each proactive renewal for observability.

**Metric:** `lenny_gateway_credential_proactive_renewals_total` (counter, labeled by `pool`, `provider`, `outcome: success|failure`).

**Renewal failure handling.** If a proactive renewal fails (e.g., all credentials in the pool are exhausted or in cooldown), the gateway does NOT immediately terminate the session. It retries the renewal at half the remaining TTL interval, up to 3 times. **`expiresAt` guard:** Before each retry attempt, the `CredentialRenewalWorker` checks whether `now >= lease.expiresAt`. If the lease has already expired, the retry is skipped and the worker immediately sends `RotateCredentials` with `CREDENTIAL_RENEWAL_FAILED` ŌĆö retrying an already-expired lease is pointless and delays fault handling. On the first failed attempt, the counter `lenny_credential_proactive_renewal_first_failure_total` (counter, labeled by `pool`, `provider`) is incremented as an early-warning signal before all retries are exhausted. If all retries fail and the lease has not yet expired, the gateway sends `RotateCredentials` with the error code `CREDENTIAL_RENEWAL_FAILED` ŌĆö the session falls through to the standard Fallback Flow (including `maxRotationsPerSession` budget consumption on that last-resort attempt). **This last-resort fault rotation counts against `maxRotationsPerSession`**; deployers running long-lived sessions with short-TTL credentials should set `maxRotationsPerSession >= 1 + expected_proactive_retries` to ensure the fallback rotation slot is available. The gateway emits `lenny_credential_proactive_renewal_exhausted_total` (counter, labeled by `pool`, `provider`) and a `CredentialProactiveRenewalExhausted` warning alert when all proactive retries are exhausted, providing early visibility before the fallback fault rotation fires.

**Token Service unavailability guard.** When the Token Service circuit breaker is open ([Section 4.3](#43-token-service)), the proactive renewal worker MUST NOT trigger the standard Fallback Flow for Standard/Basic-level runtimes. The Fallback Flow for these levels includes checkpoint-and-restart, but the replacement pod also requires `AssignCredentials` (which calls the Token Service), creating a restart loop. Instead, when the Token Service circuit breaker is open and the existing lease has not yet expired (i.e., `now < lease.expiresAt`), the renewal worker extends the adapter-side lease timer by one additional `renewBeforeBuffer` interval and reschedules the renewal attempt. This keeps the session alive on its current (still-valid) credential until the Token Service recovers. The Fallback Flow is triggered only when the credential has actually expired (`now >= lease.expiresAt`) or has been rejected by the upstream provider (`AUTH_EXPIRED`, `RATE_LIMITED`), not when renewal infrastructure is transiently unavailable.

**Rotation mode resolution.** The runtime's `credentialCapabilities.hotRotation` declaration ([Section 5.1](05_runtime-registry-and-pool-model.md#51-runtime)) is the authoritative signal for whether credentials can be rotated in-place, because only the runtime knows whether it can rebind credentials without a restart. The gateway resolves the effective restart requirement at session start:

1. If the runtime declares `credentialCapabilities.hotRotation: true`, the gateway uses **hot rotation** (lifecycle channel delivery, no restart). The runtime has affirmed it can handle in-place rebinding.
2. If the runtime declares `credentialCapabilities.hotRotation: false` (or omits `credentialCapabilities` entirely), the gateway uses **checkpoint-and-restart rotation**. The runtime cannot handle hot rotation.

The resolved rotation mode is recorded in the session's metadata at creation time and is visible in the `credential.leased` audit event (`rotationMode: "hot" | "restart"`).

#### LLM Reverse Proxy

For API-key-based providers that do not support short-lived token exchange (e.g., providers where the "short-lived" key is really just the long-lived key with a TTL wrapper), the gateway runs a **credential-injecting reverse proxy** so the real API key never enters the agent pod. The proxy is a single in-process component of the gateway:

- **LLM Proxy subsystem** ŌĆö the gateway's fourth internal subsystem boundary ([┬¦4.1](#41-edge-gateway-replicas)). Speaks a stable, provider-agnostic wire format to the agent pod (OpenAI Chat Completions or Anthropic Messages ŌĆö see "Proxy dialects" below). Validates the lease token, enforces SPIFFE-binding, runs the `PreLLMRequest`/`PostLLMResponse` interceptor chain ([┬¦4.8](#48-gateway-policy-engine)), translates the request into the upstream provider's native wire format using the gateway's native Go translator (Anthropic Messages API, Bedrock Converse/InvokeModel, Vertex AI `generateContent`, Azure OpenAI), injects the real upstream credentials from the Token Service's in-memory credential cache, forwards to the upstream provider, and extracts authoritative token usage from the upstream response. Translation, credential injection, and upstream forwarding all happen inside the same gateway process ŌĆö no sidecar, no loopback hop.

**Request flow:**

1. The agent pod receives a lease token, a `proxyUrl`, and a `proxyDialect` (`openai` or `anthropic`) in its `CredentialLease` instead of a materialized upstream API key.
2. The agent pod uses a standard OpenAI or Anthropic SDK pointed at `proxyUrl` with the lease token as the API key. No Lenny-specific client code and no provider-specific SDK.
3. The LLM Proxy subsystem authenticates the request: it validates the lease token against `TokenStore`, verifies the peer SPIFFE URI (for multi-tenant deployments), and runs `PreLLMRequest` interceptors on the request body. If an interceptor returns `REJECT`, the proxy returns `LLM_REQUEST_REJECTED` to the agent pod before any upstream call is made.
4. The subsystem looks up the lease's upstream provider (e.g., `anthropic_direct`, `aws_bedrock`, `vertex_ai`, `azure_openai`), translates the runtime's OpenAI/Anthropic-shaped request into that provider's native wire format using the in-process Go translator, and injects the real upstream credentials from the Token Service's in-memory cache.
5. The subsystem forwards the translated request to the upstream provider over a TLS connection governed by the gateway pod's egress `NetworkPolicy` ([┬¦13.2](13_security-model.md#132-network-isolation) NET-046). Streaming responses stream back through the subsystem.
6. The subsystem translates the upstream response back into the dialect spoken by the agent pod, runs `PostLLMResponse` interceptors (fires once on the initial response metadata; stream chunks are forwarded without per-chunk interception), and extracts token usage counts (`input_tokens`, `output_tokens`) from the upstream provider response metadata. **These extracted counts are the authoritative usage record for quota accounting (see [┬¦11.2](11_policy-and-controls.md#112-budgets-and-quotas)); `ReportUsage` calls from the pod are ignored for sessions in proxy mode** ŌĆö the gateway is the sole counter of record, eliminating the ability of a malicious runtime to underreport token consumption.
7. The response streams to the agent pod. The real upstream API key never leaves the gateway process; it never enters the agent pod's memory, environment, filesystem, or network path.

**Proxy dialects.** Every proxy-mode pool declares a `proxyDialect` matching the Runtime's declared `credentialCapabilities.proxyDialect` ([┬¦5.1](05_runtime-registry-and-pool-model.md#51-runtime)). Two dialects are supported at launch:

- `openai` ŌĆö exposes `POST {proxyUrl}/v1/chat/completions`, `POST {proxyUrl}/v1/embeddings`, and the streaming SSE variant. Accepts the standard OpenAI request body shape with a `model` field that the gateway's translator maps to the upstream provider's model ID (configured via the translator model map in `llmProxy` Helm values). Any runtime using an OpenAI-compatible SDK works against this dialect regardless of the configured upstream provider.
- `anthropic` ŌĆö exposes `POST {proxyUrl}/v1/messages` and its streaming variant. Accepts the Anthropic Messages API request body. Any runtime using the Anthropic SDK works against this dialect regardless of the configured upstream provider. The LLM Proxy subsystem accepts the `anthropic-version` header from the runtime (the Anthropic SDK sets it automatically); if the header is absent, the proxy injects the default configured via the `llmProxy.anthropicVersion` Helm value (default: the latest stable version as of each Lenny release). The injected default is advertised to runtimes via the adapter manifest's `llm.headers` field so SDKs that require an explicit version can read it at startup. Runtime-supplied versions are forwarded verbatim to the upstream provider for `anthropic_direct`, or mapped to the equivalent Bedrock/Vertex API version for cross-provider translation.

Admission control ([┬¦10](10_gateway-internals.md)) rejects any pool registration or update where the pool's `proxyDialect` does not match a dialect declared in the Runtime's `credentialCapabilities.proxyDialect` set, returning `422 INVALID_POOL_PROXY_DIALECT` with message `"pool proxyDialect <X> is not declared in runtime credentialCapabilities.proxyDialect"`. This guarantees the agent pod's SDK speaks a dialect the proxy accepts.

**Trust boundary.** The translator runs inside the gateway process and is part of the gateway's trust envelope:

- **Real credentials:** upstream API keys and STS materials live only in the Token Service's in-memory credential cache inside the gateway process. They are never written to disk, never placed on tmpfs, and never shared with any other process or container. Credential rotation refreshes the cache entry atomically; the translator reads the current entry on each outbound call, so rotation takes effect on the next request with no reload signal, file replacement, or SIGHUP required.
- **Gateway SPIFFE identity:** outbound requests to upstream providers (when mTLS is used to the provider) use the gateway pod's SPIFFE identity. There is no separate identity for the translator ŌĆö it is not a separate trust domain.

**Authoritative token counting.** The translator parses upstream-provider-native usage metadata from the response and normalizes per-provider field names (Anthropic/Bedrock Converse `usage.input_tokens`/`usage.output_tokens`, OpenAI/Azure OpenAI `usage.prompt_tokens`/`usage.completion_tokens`, Vertex `usageMetadata.promptTokenCount`/`usageMetadata.candidatesTokenCount`) into canonical `input_tokens`/`output_tokens` fields. Usage is extracted from response headers or SSE `usage`/`message_delta` events per upstream wire format. These normalized counts are the authoritative record for quota accounting; no pod-reported counts are accepted in proxy mode.

**Per-pool configuration.** Deployers choose for each credential pool whether to use direct credential leasing (simpler, lower latency, key materialized into the agent pod) or proxy mode (higher security, upstream key never leaves the gateway, in-process translation). The two modes can coexist across different pools. **Proxy mode is the recommended default for multi-tenant deployments**, since the real API key never enters the agent pod and a compromised agent cannot exfiltrate it. Direct mode is appropriate for single-tenant and development deployments where simplicity and lower latency are preferred, and where the agent pod is trusted.

The proxy enforces the same rate limits and budget constraints as direct leasing. When a lease expires or is revoked, the LLM Proxy subsystem immediately rejects requests before any upstream call is made ŌĆö there is no window of exposure where a compromised agent pod could continue using a stale key. The deny-list check is source-aware: for pool-backed leases the proxy looks up the `{source: "pool", poolId, credentialId}` entry; for user-backed leases the proxy looks up the `{source: "user", tenantId, credentialRef}` entry. See "Credential deny list" below for the tagged-union key shape, matching rules, and rebuild query.

> **Hard rejection ŌĆö `direct` + `standard` (runc) isolation in multi-tenant mode:** Combining `deliveryMode: direct` with `standard` (runc) isolation is **blocked by admission control** when `tenancy.mode: multi` is set in the platform configuration. In `standard` isolation, a container escape gives the attacker access to materialized credential material on the host node; in a multi-tenant deployment this represents cross-tenant credential exposure. The warm pool controller **rejects** any pool creation or update that sets both `deliveryMode: direct` and `isolationProfile: standard` when `tenancy.mode: multi`, returning a validation error `DirectModeStandardIsolationMultiTenantRejected`: `"deliveryMode: direct with isolationProfile: standard is not permitted in multi-tenant mode. Use deliveryMode: proxy, or set isolationProfile: sandboxed or microvm."` This is enforced at two layers: (1) the warm pool controller's pool-registration validation (rejects the admin API request before any pod is created), and (2) a `ValidatingAdmissionWebhook` (`lenny-direct-mode-isolation`) that rejects `SandboxTemplate` and `CredentialPool` resources carrying this combination in agent namespaces when the platform is in multi-tenant mode. The webhook is deployed with `failurePolicy: Fail` (fail-closed). In **single-tenant mode** (`tenancy.mode: single`) or **development mode** (`global.devMode: true`), this combination is permitted with an explicit opt-in field on the pool (`allowDirectModeStandardIsolation: true`) and emits a `DirectModeWeakIsolation` warning event and a startup log warning. The opt-in field cannot be set in multi-tenant mode ŌĆö the admission webhook rejects it regardless of the field value.

> **Monitoring direct-mode usage:** Deployers should monitor the metric `lenny_gateway_credential_leases_active{delivery_mode="direct"}` and set alerts when direct-mode lease counts exceed expected thresholds. The audit event `credential.leased` ([Section 4.9.2](#492-credential-audit-events)) includes a `deliveryMode` field, enabling compliance teams to track and review all direct-mode credential deliveries. In regulated environments, consider requiring explicit admin approval for pools configured with `deliveryMode: direct`.

**Subsystem isolation:** The LLM Proxy runs within the gateway's fourth internal subsystem boundary (see [Section 4.1](#41-edge-gateway-replicas)), with its own goroutine pool, concurrency limits, circuit breaker, and per-subsystem metrics (`lenny_gateway_llm_proxy_active_connections`, `lenny_gateway_llm_proxy_request_duration_seconds`, `lenny_gateway_subsystem_circuit_state{subsystem="llm_proxy"}`). This ensures that a surge in proxy traffic or an upstream provider outage cannot starve the Stream Proxy, Upload Handler, or MCP Fabric subsystems.

**LLM Proxy circuit breaker open-state behavior:** When the LLM Proxy circuit breaker transitions to **open** state (tracked by `lenny_gateway_subsystem_circuit_state{subsystem="llm_proxy"} = 2`), the following behavior applies:

1. **All new LLM proxy requests are immediately rejected** with a `PROVIDER_UNAVAILABLE` error. The gateway does NOT silently hang or wait for timeout.
2. **The adapter receives a `PROVIDER_UNAVAILABLE` event** on the control channel. The adapter relays this to the runtime as a tool-result error with code `PROVIDER_UNAVAILABLE`, enabling the agent to handle the failure gracefully (e.g., inform the user, retry later, or fall back to a different provider if available via the fallback chain).
3. **In-flight streaming responses that were established before the circuit opened continue until completion or upstream failure** ŌĆö the circuit breaker applies only to new outbound connections, not to already-established streams.
4. **Half-open retry:** After a configurable cooldown period (default: 30s, configurable via `llmProxy.circuitBreaker.halfOpenInterval`), the circuit transitions to half-open and allows a single probe request. If the probe succeeds, the circuit closes; if it fails, the circuit returns to open and the cooldown resets.
5. **Alert coverage:** The `GatewaySubsystemCircuitOpen` Warning alert ([Section 16.5](16_observability.md#165-alerting-rules-and-slos)) fires when the LLM Proxy circuit breaker (or any other subsystem circuit breaker) remains in open state for > 60s. No separate `LLMProxyCircuitOpen` alert is needed ŌĆö the `subsystem` label on `lenny_gateway_subsystem_circuit_state` provides per-subsystem granularity.

```yaml
credentialPools:
  - name: claude-direct-prod
    provider: anthropic_direct
    deliveryMode: proxy           # proxy | direct (default: proxy for multi-tenant, direct for single-tenant)
    proxyDialect: anthropic       # openai | anthropic; must match one of Runtime.credentialCapabilities.proxyDialect
    proxyEndpoint: https://gateway-internal:8443/llm-proxy
    # ... other pool config unchanged
```

> **Proxy endpoint transport security:** The proxy endpoint **must** use TLS (`https://`). It shares the same mTLS certificate infrastructure as the internal gRPC control plane ([Section 4.3](#43-token-service)), so lease tokens are always encrypted in transit. The controller **rejects** pool registrations where `proxyEndpoint` uses an `http://` scheme and emits a validation error (`InvalidProxyEndpointScheme`). This ensures that proxy mode's security guarantee ŌĆö real API keys never leave the gateway ŌĆö is not undermined by plaintext lease-token transmission on the cluster network.
>
> **SPIFFE-binding for proxy mode lease tokens (v1 requirement for multi-tenant deployments):** A captured lease token could theoretically be replayed from a different pod that also has a valid mTLS certificate. To eliminate this cross-pod replay risk, lease tokens in proxy mode **must** be bound to the issuing pod's SPIFFE identity for all multi-tenant deployments. The binding is implemented as follows:
> 1. At `AssignCredentials` time, the gateway records the issuing pod's SPIFFE URI (e.g., `spiffe://<trust-domain>/agent/{pool}/{pod-name}`, where `<trust-domain>` is the value of `global.spiffeTrustDomain`) alongside the lease record in `TokenStore`.
> 2. On every LLM proxy request, the gateway extracts the peer SPIFFE URI from the authenticated mTLS connection and verifies it matches the SPIFFE URI stored in the lease record. A mismatch is rejected with `LEASE_SPIFFE_MISMATCH` (category: `SECURITY`) and emits an audit event `credential.lease_spiffe_mismatch` ([Section 4.9.2](#492-credential-audit-events)).
> 3. The `leaseToken` field in `AssignCredentialsResponse` is unchanged ŌĆö SPIFFE-binding is enforced server-side; no protocol change is required for the pod.
>
> **Deployment note ŌĆö admission enforcement of SPIFFE-binding in multi-tenant mode:** SPIFFE-binding is enforced by default when `deliveryMode: proxy` is set on a pool. Combining `deliveryMode: proxy` with `spiffeBinding: disabled` is **blocked by admission control** when `tenancy.mode: multi` is set in the platform configuration, because disabling SPIFFE-binding removes the primary defense against cross-pod lease-token replay (see the threat-model paragraph below). The warm pool controller **rejects** any pool creation or update that sets `deliveryMode: proxy` together with `spiffeBinding: disabled` when `tenancy.mode: multi`, returning a validation error `ProxyModeSpiffeBindingDisabledMultiTenantRejected`: `"deliveryMode: proxy with spiffeBinding: disabled is not permitted in multi-tenant mode. Remove spiffeBinding: disabled, or set tenancy.mode: single / global.devMode: true."` This is enforced at two layers mirroring the `DirectModeStandardIsolationMultiTenantRejected` control above: (1) the warm pool controller's pool-registration validation (rejects the admin API request before any pod is created), and (2) the `lenny-direct-mode-isolation` `ValidatingAdmissionWebhook` ([┬¦17.2](17_deployment-topology.md#172-namespace-layout)) ŌĆö extended to also reject `SandboxTemplate` and `CredentialPool` resources carrying `deliveryMode: proxy` + `spiffeBinding: disabled` in multi-tenant mode (the webhook remains deployed with `failurePolicy: Fail`, fail-closed). In **single-tenant mode** (`tenancy.mode: single`) or **development mode** (`global.devMode: true`), SPIFFE-binding may be disabled via an explicit opt-in field on the pool (`allowProxyModeSpiffeBindingDisabled: true`, mirroring `allowDirectModeStandardIsolation`); disablement emits the existing `ProxyModeSpiffeBindingDisabled` warning event plus a `credential.proxy_mode_spiffe_binding_disabled` audit event ([┬¦4.9.2](#492-credential-audit-events)) and a startup log warning. The opt-in field cannot be set in multi-tenant mode ŌĆö the admission webhook rejects it regardless of the field value. The `lenny-preflight` Job additionally scans all registered `CredentialPool` resources at install and upgrade time and fails when any pool in a multi-tenant deployment has `spiffeBinding: disabled`, matching the `shareProcessNamespace` preflight pattern in [┬¦13.1](13_security-model.md#131-pod-security). Single-tenant and development deployments that legitimately disable SPIFFE-binding rely on the existing defense-in-depth controls (token visible only inside the pod on tmpfs mode `0440` with group-read restricted to the `lenny-cred-readers` supplementary group, pod sandbox isolation) as the primary mitigations against token extraction.
>
> **Threat model ŌĆö what SPIFFE-binding does and does not protect against.** SPIFFE-binding is a per-pod isolation control: it prevents a **different agent pod** (in a different pool, namespace, or tenant) from replaying a captured lease token, because the mTLS peer identity on the replay would not match the issuing pod's SPIFFE URI recorded in the lease record. The control is meaningful against an attacker who has extracted a lease token from one pod's memory (e.g., via log scraping or a memory-dump vulnerability) and attempts to use it from another pod. It does NOT protect against: (a) an attacker who has escaped into the gateway pod itself (the SPIFFE check applies at the gateway's mTLS ingress from agent pods; a gateway-pod-resident attacker can read the Token Service's in-memory credential cache directly, which is where real upstream API keys live); (b) an attacker with control over the KMS signing key (they can forge lease tokens entirely). Containment (gateway-pod hardening, seccomp, admission-verified image signatures) is the primary defense against (a); KMS HSM-backing is the primary defense against (b). SPIFFE-binding is one layer of defense against cross-pod token theft.

**Native translator.** The LLM Proxy subsystem's translator is a Go package inside the gateway binary (no sidecar, no out-of-process component, no loopback listener). It converts OpenAI Chat Completions and Anthropic Messages request/response/SSE shapes into the upstream provider's native wire format on the outbound leg and back on the return leg, and participates in the `PreLLMRequest`/`PostLLMResponse` interceptor chain on each side.

- **Supported upstream providers (v1 target set):** `anthropic_direct`, `aws_bedrock`, `vertex_ai`, `azure_openai`. Phased delivery per [┬¦18](18_build-sequence.md): Phase 5.8 ships `anthropic_direct` translation only; Phase 11 extends the native translator to the remaining three providers along with multi-provider credential rotation. Additional providers beyond v1 are added on Lenny's release cadence; each new provider requires a translator implementation and contract tests against mock upstream servers for request translation, response translation, and streaming SSE translation.
- **Credential source.** Real upstream API keys live only in the gateway process's in-memory Token Service cache ([┬¦4.3](#43-token-service)). The translator reads the current cached credential on each outbound call ŌĆö no credential file on disk, no tmpfs mount, no SIGHUP or reload dance. Credential rotation ([┬¦4.9](#49-credential-leasing-service) Fallback Flow) updates the cache in place; the next outbound call uses the new credential automatically.
- **Translation error taxonomy.** Translator failures are surfaced via `lenny_gateway_llm_translation_errors_total` with `error_type` Ōłł {`unsupported_field` ŌĆö the pod's request carries a dialect field the upstream provider does not accept; `schema_mismatch` ŌĆö either the incoming pod request or the upstream response fails translator schema validation (covers both request-side validation and response-shape drift; acts as the tripwire for unexpected input or upstream API drift); `auth_failed` ŌĆö upstream provider rejected the injected credential (invalid, expired, or rate-limited at the auth layer), triggers the Fallback Flow; `timeout` ŌĆö translator exceeded its per-request deadline waiting for the upstream to respond or complete the stream; `streaming_interrupted` ŌĆö mid-stream error, upstream early-close, or transport failure after first-byte was delivered to the pod; `upstream_4xx` ŌĆö upstream provider returned a 4xx other than auth (surfaced to the pod as `PROVIDER_REQUEST_INVALID`); `upstream_5xx` ŌĆö upstream provider returned a 5xx or pre-first-byte transport failure (surfaced to the pod as `PROVIDER_UNAVAILABLE`, feeds the circuit breaker)}.
- **Observability.** The translator emits `lenny_gateway_llm_translation_duration_seconds` (histogram, labeled by `pool`, `provider`, `proxy_dialect`, `direction: request|response`) for wall-clock translation time and `lenny_gateway_llm_translation_errors_total` (counter, same labels plus `error_type`). End-to-end proxy hop latency is measured by `lenny_gateway_llm_proxy_request_duration_seconds` ([┬¦16.1](16_observability.md#161-metrics)); the translator histogram is a subcomponent of the hop. There is no translation-overhead SLO ŌĆö in-process Go translation is covered by the LLM Proxy subsystem's existing resource envelope.
- **Upstream egress.** The gateway pod's egress `NetworkPolicy` (`allow-gateway-egress-llm-upstream`, [┬¦13.2](13_security-model.md#132-network-isolation)) enumerates every upstream LLM provider CIDR/hostname that any `CredentialPool` in the deployment references. All other egress ŌĆö including cluster-internal CIDRs and cloud instance metadata endpoints ŌĆö is blocked. The policy applies to the gateway process directly (no sidecar to cover).
- **Supply chain.** The translator is Go code in the Lenny monorepo; it ships inside the gateway image and inherits the gateway image's signing, SBOM, and CVE-scanning pipeline. There is no separate wrapper image, no cosign policy to add, no independent maintenance cadence.

**Credential health scoring:** For pooled credentials, the gateway tracks per-credential:

- Recent rate-limit events and cooldown expiry
- Auth failure count
- Concurrent session count
- Spend tracking (if provider reports it)

Assignment strategies use this health data to avoid assigning degraded credentials. All credential health scoring data (cooldown timers, rate-limit event history, auth failure counts, concurrent session counts) is persisted in Postgres as part of the `CredentialPoolStore` ([Section 12.1](12_storage-architecture.md#121-design-principle)), which is tenant-scoped with `tenant_id` column and RLS. No credential health data is stored in Redis; the Redis key prefix table in [Section 12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes) does not include credential health entries because the data is exclusively Postgres-backed.

#### Semantic Caching

Optional `CachePolicy` on `CredentialPool` backed by pluggable `SemanticCache` interface:

```yaml
cachePolicy:
  strategy: semantic
  ttl: 300
  similarityThreshold: 0.92
  backend: redis
```

Default Redis-backed implementation. Fully replaceable by deployers. Disabled by default, opt-in per pool. **Erasure contract test:** A `ValidateSemanticCacheErasure(t *testing.T, cache SemanticCache)` contract validation helper is provided for pluggable implementations. The test verifies: (a) `DeleteByUser` removes all cached entries for the specified user; (b) a subsequent `Query` for the deleted user returns zero results; (c) entries for other users in the same tenant are unaffected. Pluggable `SemanticCache` implementations must pass this test. The erasure job treats a `DeleteByUser` call that returns an error as a hard failure ŌĆö the erasure job halts and does not proceed to subsequent stores ŌĆö preventing silent no-op implementations from producing an erasure receipt that falsely records the cache as erased.

**Tenant and user isolation:** Semantic cache implementations MUST scope all cache lookups and writes to the `tenant_id` of the current session. Cross-tenant cache hits are forbidden regardless of semantic similarity. The effective cache key space is `(tenant_id, query_embedding, model, provider)` at minimum. When the credential source is `user`-scoped (i.e., the user provides their own API key), the cache key MUST additionally include `user_id`, since responses may reflect user-specific context or permissions. **For pool-scoped credentials, the default cache scope is also `per-user`: the cache key MUST include `user_id` by default.** Deployers may opt into cross-user sharing within a tenant by setting `cacheScope: tenant` on the pool's semantic cache configuration ŌĆö this explicitly acknowledges that responses may be shared across different users in the same tenant. The allowed `cacheScope` values are: `per-user` (default ŌĆö cache key includes `user_id`; no cross-user sharing), `per-session` (cache key includes `session_id`; most restrictive), `tenant` (cache key omits `user_id`; allows cross-user sharing within the tenant, deployer opt-in only). Setting `cacheScope: tenant` on a pool with a regulated `complianceProfile` (`hipaa`, `fedramp`) is rejected at pool registration time with `400 COMPLIANCE_CROSS_USER_CACHE_PROHIBITED` ŌĆö this rejection also mitigates a timing side-channel: with `cacheScope: tenant`, a user can infer whether semantically similar prompts have been issued by other users in the same tenant by measuring response latency differences between cache-hit and cache-miss paths. **Timing side-channel trade-off (`cacheScope: tenant`):** Deployers opting into `cacheScope: tenant` accept that response latency is an observable signal of prior cache population by other users within the tenant. This is analogous to the residual state vectors documented for task-mode scrub ([Section 5.2](05_runtime-registry-and-pool-model.md#52-pool-configuration-and-execution-modes)) ŌĆö a documented trade-off, not a defect. For `per-user` and `per-session` scopes, cross-user timing side-channels do not arise because cache lookups are scoped to the requesting identity. The default Redis-backed implementation inherits tenant key isolation from the Redis wrapper layer ([Section 12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes)), but pluggable implementations must enforce equivalent scoping independently. An integration test (`TestSemanticCacheTenantIsolation`) must verify that a cached response written for tenant A is not returned for a semantically identical query from tenant B.

#### `CredentialRouter` Interface

The CredentialRouter is a pluggable interface invoked at two points: (1) session creation ([Section 7.1](07_session-lifecycle.md#71-normal-flow), step 6) when the gateway assigns initial credential leases, and (2) rotation time (Fallback Flow, step 4) when the gateway selects a replacement credential for a failed provider.

**Input:**

| Field | Type | Description |
| --- | --- | --- |
| `tenantId` | string | Tenant making the request |
| `userId` | string | Authenticated user |
| `provider` | string | The provider being assigned (e.g., `anthropic_direct`) |
| `allowedPools` | list of pool descriptors | Pools eligible for this provider, each with current utilization stats (active leases, max capacity, health score) |
| `userCredentialAvailable` | boolean | Whether a user-scoped credential exists for this provider |
| `preferredSource` | string | From the tenant's `credentialPolicy.preferredSource` |
| `rotationContext` | object (nullable) | Present only during rotation: includes `previousPoolId`, `previousCredentialId`, `failureReason`, `rotationCount` |
| `hints` | map of string to string | Deployer-extensible key-value map. Populated from `runtimeOptions` at session creation or from static hints configured on pools. Examples: `model`, `cost_tier`, `region` |

**Output:**

| Field | Type | Description |
| --- | --- | --- |
| `source` | string | `pool` or `user` |
| `poolId` | string (nullable) | Selected pool ID (required when `source` is `pool`) |
| `strategyOverride` | string (nullable) | If set, overrides the pool's `assignmentStrategy` for this assignment |

**Built-in assignment strategies.** The default CredentialRouter uses the pool's `assignmentStrategy` field to select a credential within the chosen pool:

| Strategy | Behavior |
| --- | --- |
| `least-loaded` | Selects the credential with the fewest active leases. Default. |
| `round-robin` | Even distribution across credentials in the pool |
| `sticky-until-failure` | Same credential for all sessions until a failure event triggers rotation |

Deployers who need cost-aware, latency-based, or intent-based routing implement a custom CredentialRouter. The `hints` map is the primary extension point ŌĆö deployers pass deployment-specific context (e.g., `model`, `cost_tier`, `region`) without modifying the core interface.

**External LLM routing proxy integration.** Lenny's native translator covers the v1 provider set (`anthropic_direct`, `aws_bedrock`, `vertex_ai`, `azure_openai`). Deployers who need a broader provider catalog, custom routing intelligence, shared-team spend accounting, or per-request model rewriting that Lenny's interceptors cannot express can deploy an external LLM routing proxy (e.g., LiteLLM, Portkey, or a custom gateway) as a cluster service ŌĆö on the same Kubernetes cluster, on a separate cluster, or as a cloud-managed service ŌĆö and point Lenny at it. This is a first-class, supported integration pattern. Two configurations are documented:

- **Pattern A (recommended ŌĆö external proxy as the upstream provider).** Implement a custom `CredentialProvider` that mints `materializedConfig` entries with `baseUrl` pointing at the external proxy's endpoint. The pool uses `deliveryMode: direct` (the "credential" is effectively the external proxy's API key plus its base URL). Lenny's gateway hands the runtime the proxy's URL + key; the runtime's SDK talks to the external proxy directly. Routing intelligence, model catalog, and provider switching all live at the external proxy. Lenny retains its credential-governance role (who can access which external-proxy key) but **does not** intercept the LLM request content ŌĆö `PreLLMRequest` and `PostLLMResponse` interceptors do NOT fire on traffic that never passes through Lenny's LLM Proxy subsystem. Deployers must enforce policy at the external proxy's inbound side and scrape the external proxy's own metrics for observability.
- **Pattern B (external proxy behind Lenny's LLM Proxy).** Configure an existing built-in provider (e.g., `anthropic_direct`) with a `baseUrl` override on the pool pointing at the external proxy. The pool uses `deliveryMode: proxy`. Lenny's native translator converts between dialects as usual, treats the external proxy as the upstream, and forwards requests there; the external proxy accepts the Anthropic-dialect (or OpenAI-dialect) input and re-translates downstream. `PreLLMRequest` and `PostLLMResponse` interceptors fire normally on the Lenny-to-external-proxy leg. Cross-provider translation happens at the external proxy, outside Lenny's observability scope.

Both patterns preserve Lenny's credential lease governance and audit trail for who-accessed-what. See [`docs/operator-guide/external-llm-proxy.md`](../docs/operator-guide/external-llm-proxy.md) for full deployment topologies, manifest sketches, governance implications, and security considerations. For model governance specifically ŌĆö without an external proxy ŌĆö `PreLLMRequest` interceptors ([Section 4.8](#48-gateway-policy-engine)) remain the lighter-weight option: they can reject disallowed models, rewrite model fields, or redirect to approved models before the request reaches the provider. Interceptors and external proxies are composable: Pattern B with `PreLLMRequest` interceptors gives policy enforcement on the Lenny side and routing intelligence on the external-proxy side.

#### Credential Governance Boundaries

This section clarifies what Lenny governs versus what is delegated to the runtime, and how the two delivery modes affect governance scope.

**What Lenny governs:**

- **Which providers a runtime can access** ŌĆö the intersection of the Runtime's `supportedProviders` and the tenant's `credentialPolicy.providerPools` keys determines available providers.
- **Which credentials are used** ŌĆö pool assignment via the CredentialRouter, user credentials via the pre-authorized flow, and fallback chain ordering are all gateway-controlled.
- **Connector call content** ŌĆö all connector traffic is gateway-proxied, so connector call arguments and responses are interceptable via `PreConnectorRequest` and `PostConnectorResponse` interceptors ([Section 4.8](#48-gateway-policy-engine)). There is no direct-mode bypass for connectors.

**What Lenny delegates to the runtime:**

- **Which model to call for a given task** ŌĆö the runtime selects the model based on its own logic (e.g., agent configuration, task complexity). Lenny does not govern model selection by default. However, in proxy mode, deployers can enforce model policies via `PreLLMRequest` interceptors ([Section 4.8](#48-gateway-policy-engine)) ŌĆö e.g., allowlisting/denylisting specific models, or redirecting requests to approved models. Alternatively, deployers can enforce model policies via an external LLM routing proxy (e.g., LiteLLM, Portkey).
- **Cost optimization across providers/models** ŌĆö the runtime (or an external routing proxy) handles cost-aware routing. Lenny supports this via CredentialRouter hints and multi-provider credential maps.

**Proxy mode vs. direct mode:**

| Aspect | Proxy mode | Direct mode |
| --- | --- | --- |
| Credential injection | Gateway injects; key never enters pod | Pod receives materialized key |
| Request visibility | Gateway can inspect/audit LLM requests | Gateway has no visibility |
| LLM request/response interception | `PreLLMRequest` and `PostLLMResponse` interceptors fire on every proxied LLM call ([Section 4.8](#48-gateway-policy-engine)) | No interception ŌĆö gateway has no visibility into LLM traffic |
| Recommended for | Multi-tenant, regulated environments | Single-tenant, development, low-latency |

#### Emergency Credential Revocation

When a single credential within a pool is suspected compromised (e.g., `key-2` in `claude-direct-prod` has been exfiltrated), operators must be able to pull that specific credential immediately ŌĆö without deleting the entire pool or waiting for natural lease TTL expiry. During the TTL window, the compromised key would continue to be injected into upstream LLM requests for every session currently holding a lease against it.

**Revocation endpoint:**

```
POST /v1/admin/credential-pools/{name}/credentials/{credId}/revoke
```

Request body (optional):

```json
{
  "reason": "suspected_exfiltration",
  "note": "key-2 appeared in logs ŌĆö rotating immediately"
}
```

The operation is synchronous and MUST:

1. Mark the credential as `revoked` in `CredentialPoolStore` (status: `revoked`, `revokedAt`, `revokedBy`, `revocationReason`).
2. Look up all active leases backed by `credId` (via `CredentialPoolStore` lease index).
3. For each active lease: terminate it immediately via in-process signal or Redis pub/sub (matching the certificate deny-list pattern from [Section 10.3](10_gateway-internals.md#103-mtls-pki)). The credential ID is added to an **in-memory credential deny list** on every gateway replica, propagated via Redis pub/sub with Postgres `LISTEN/NOTIFY` as fallback. Entries in the deny list expire when the credential's natural lease TTL lapses ŌĆö keeping the list small.
4. For proxy mode: the LLM proxy checks the deny list on every upstream request using the source-aware matching rules defined under "Credential deny list" below ŌĆö `{source: "pool", poolId, credentialId}` for pool-backed leases (this path) and `{source: "user", tenantId, credentialRef}` for user-backed leases (populated by `POST /v1/credentials/{credential_ref}/revoke` above). Any request presenting a lease backed by a denied credential is immediately rejected with `CREDENTIAL_REVOKED` (category: `SECURITY`) ŌĆö there is no window where the compromised key continues to reach the provider.
5. For direct-delivery mode: the gateway sends a `RotateCredentials` RPC to all pods holding leases against `credId`, triggering the standard rotation fallback flow ([Section 4.9](#49-credential-leasing-service) Fallback Flow). The pod's lease is invalidated; it must acquire a new lease from a different credential in the pool (or the fallback chain) to continue. **Residual risk (direct mode only):** Lenny's revocation stops the pod from using the key *through Lenny*, but the underlying API key continues to exist at the provider. Any party that already extracted the materialized key from the pod (e.g., via a container escape or memory read) can continue using it directly until it is rotated or deleted at the provider. **Mitigating action required:** After calling the revocation endpoint for a direct-mode credential, operators MUST also rotate or delete the key at the provider (e.g., via the Anthropic Console, AWS IAM, or GCP IAM) to achieve complete revocation. See the emergency revocation runbook below for the required sequencing. This residual risk is inherent to direct-mode delivery and is one reason proxy mode is the recommended default for multi-tenant and security-sensitive deployments.
6. Emit a `credential.revoked` audit event (fields: `pool_id`, `credential_id`, `revoked_by`, `reason`, `active_leases_terminated`).
7. Return `200 OK` with a summary: `{ "revokedCredential": "key-2", "leasesTerminated": 3, "propagatedAt": "2026-04-05T..." }`.

**Force-rotate all credentials in a pool:**

```
POST /v1/admin/credential-pools/{name}/revoke
```

Revokes all credentials in the pool simultaneously. All active sessions using this pool are rotated to their fallback chain, or terminated with `CREDENTIAL_POOL_EXHAUSTED` if no fallback is available. Use when the entire pool is suspected compromised.

**Credential deny list:** Mirrors the certificate deny-list mechanism from [Section 10.3](10_gateway-internals.md#103-mtls-pki). The deny list is propagated across all gateway replicas via Redis pub/sub. Each entry is a **tagged discriminated union** covering both credential sources ([┬¦4.9](#49-credential-leasing-service) Three Credential Modes):

- `{source: "pool", poolId, credentialId}` ŌĆö populated by `POST /v1/admin/credential-pools/{name}/credentials/{credId}/revoke` (and the pool-wide `.../revoke` variant). Entries expire when the last active lease against `(poolId, credentialId)` reaches its natural TTL expiry.
- `{source: "user", tenantId, credentialRef}` ŌĆö populated by `POST /v1/credentials/{credential_ref}/revoke` (see user credential management endpoints above). Entries expire when the last active lease against `(tenantId, credentialRef)` reaches its natural TTL expiry.

**LLM Proxy matching rules.** On every inbound proxy request the LLM Proxy subsystem dispatches on the lease record's credential source: pool-backed leases are matched against `{source: "pool", poolId, credentialId}` entries using the lease's `(poolId, credentialId)` fields; user-backed leases are matched against `{source: "user", tenantId, credentialRef}` entries using the lease's `(tenantId, credentialRef)` fields. A hit on either shape rejects the request with `CREDENTIAL_REVOKED` (category: `SECURITY`) before any upstream call is made. The two keyspaces never overlap ŌĆö `source` is the primary discriminator.

**Startup rebuild query.** Newly started gateway replicas rebuild their deny list by executing a union across both stores:

- From `CredentialPoolStore`: all `(poolId, credentialId)` tuples WHERE `status = 'revoked'` AND EXISTS an active-or-recent lease ŌĆö emitted as `{source: "pool", poolId, credentialId}` entries.
- From `TokenStore`: all `(tenantId, credentialRef)` tuples WHERE `status = 'revoked'` AND EXISTS an active lease backed by that `credential_ref` ŌĆö emitted as `{source: "user", tenantId, credentialRef}` entries.

This guarantees that a gateway replica restart immediately after either a pool-credential or user-credential revocation rebuilds a complete deny list ŌĆö no revoked credential can silently become accepted on a replica that missed the original pub/sub notification.

**Integration test.** `TestUserCredentialRevocationDenyListProxy` asserts that an in-flight proxy request presenting a user-backed lease whose `credential_ref` was just revoked via `POST /v1/credentials/{credential_ref}/revoke` is rejected with `CREDENTIAL_REVOKED` before any upstream call is made ŌĆö exercising both the pub/sub propagation path (on the replica that received the revoke) and the startup rebuild path (on a replica restarted immediately after the revoke).

**Emergency revocation runbook:** See [Section 17.7](17_deployment-topology.md#177-operational-runbooks) (operational runbooks). Steps: (1) identify the compromised credential ID from audit logs (`credential.leased` events showing `credentialId`); (2) call the revocation endpoint; (3) verify `leasesTerminated` count matches expected active sessions; (4) confirm the `CredentialCompromised` alert clears within 60s (indicating revocation propagation succeeded); (5) add a replacement credential to the pool via `PUT /v1/admin/credential-pools/{name}`; (6) rotate the underlying secret (Kubernetes Secret or external secrets manager) before re-enabling the credential ID. **Direct mode additional step:** For pools using `deliveryMode: direct`, step 6 is not optional ŌĆö it is the only action that prevents continued use of the materialized key by an attacker who already extracted it from a pod. Rotate or delete the key at the provider before re-enabling it; updating only the Kubernetes Secret is insufficient until the provider-side key is also invalidated.

#### Security Boundaries

- Long-lived credentials (API keys, IAM role ARNs, service account keys) live **only** in the Token Service and Kubernetes Secrets
- Pods receive **materialized short-lived credentials** (scoped tokens, STS sessions) via the `AssignCredentials` RPC, delivered to the agent as a tmpfs-backed file (mode `0440`, owner-only-write with group-read restricted to the `lenny-cred-readers` supplementary group; see [┬¦4.7](#47-runtime-adapter) item 4) ŌĆö never via environment variables
- Leases are **revocable** ŌĆö on user invalidation, or on emergency credential revocation via `POST /v1/admin/credential-pools/{name}/credentials/{credId}/revoke`, the gateway immediately terminates affected leases and the runtime loses access. **Caveat (direct mode):** Revocation stops the pod from using the key through Lenny, but the underlying API key continues to exist at the provider until rotated there. For proxy mode, revocation is complete ŌĆö the key never left the gateway. For direct mode, provider-side key rotation is a mandatory part of the emergency revocation procedure (see "Emergency Credential Revocation" above).
- Credential material is **never logged** in audit events, transcripts, or agent output ŌĆö only lease IDs and provider/pool names are logged
- The `env` blocklist ([Section 14](14_workspace-plan-schema.md), `env` field) rejects keys matching sensitive patterns (e.g., `ANTHROPIC_API_KEY`, `AWS_SECRET_ACCESS_KEY`, `*_SECRET_*`) regardless of whether credential leasing is configured. When leasing is enabled, credentials flow through the leasing system rather than environment variables; when leasing is not configured, the blocklist still prevents accidental credential exposure via env vars

#### Credential Lifecycle and Secret Management

Credential pools span two planes: the **pool definition** (admin-API-managed, stored in Postgres) and the **credential material** (Kubernetes Secrets, referenced by `secretRef`). This is by design ŌĆö secret material belongs in Kubernetes Secrets (where it benefits from encryption-at-rest, RBAC, and external-secrets-operator integration), while pool configuration (assignment strategy, concurrency limits, policies) belongs in the admin API. Operators must understand the lifecycle for both halves.

> **Production requirement ŌĆö etcd encryption at rest for Secrets.** Kubernetes does **not** encrypt Secret data in etcd by default. Anyone with etcd access (backup tapes, compromised etcd node, etcd snapshot) sees all Secret values ŌĆö including credential pool API keys ŌĆö in plaintext. Production deployments **must** enable the Kubernetes `EncryptionConfiguration` resource with an `aescbc`, `aesgcm`, or `kms` provider for the `secrets` resource type. Cloud-managed Kubernetes services (EKS, GKE, AKS) offer envelope encryption via their respective KMS integrations (AWS KMS, GCP Cloud KMS, Azure Key Vault) and should be configured to use them. Self-managed clusters must configure `EncryptionConfiguration` in the API server manifest and verify encryption by running `etcdctl get /registry/secrets/lenny-system/<secret-name> | hexdump` ŌĆö the output should be ciphertext, not readable JSON. The `lenny-preflight` Job emits a non-blocking warning for this check (see [Section 17.6](17_deployment-topology.md#176-packaging-and-installation)), since programmatic verification requires etcd access that the preflight Job may not have.

**Initial setup workflow:**

1. **Create Kubernetes Secrets** containing credential material (API keys, service account keys, etc.) in the `lenny-system` namespace.
2. **Register the credential pool** via the admin API (`POST /v1/admin/credential-pools`) or via the bootstrap seed (`bootstrap.credentialPools` in Helm values). The pool definition references the Secrets by `secretRef`.
3. The `lenny-bootstrap` Job ([Section 17.6](17_deployment-topology.md#176-packaging-and-installation)) seeds pool definitions but does **not** create Kubernetes Secrets ŌĆö Secrets must exist before the bootstrap Job runs, or the Token Service will log errors for unresolvable `secretRef` entries and mark those credentials as `unavailable` in the pool.

**Token Service secret resolution.** The Token Service uses a Kubernetes **informer** (watch) on Secrets in the `lenny-system` namespace. When a Secret referenced by a credential pool is created, updated, or deleted, the Token Service detects the change within its informer resync interval (default: 30 seconds, configurable via `tokenService.secretResyncInterval` Helm value). No explicit reload or admin API call is required after updating a Secret ŌĆö the Token Service picks up the new value automatically. The metric `lenny_token_service_secret_reloads_total` (counter, labeled by secret name and outcome: `success`, `not_found`, `parse_error`) tracks secret resolution events.

**Credential rotation workflow:**

1. **Update the Kubernetes Secret** with the new credential material (e.g., `kubectl create secret generic anthropic-key-1 --from-literal=api-key=sk-ant-new... --dry-run=client -o yaml | kubectl apply -f -`).
2. The Token Service detects the Secret change via its informer within 30 seconds.
3. **No pool definition update is required** ŌĆö the `secretRef` has not changed, only the Secret's content.
4. New credential leases issued after the Secret update use the new material. Existing active leases continue using the previously materialized credentials until they expire naturally (governed by lease TTL). If immediate rotation of active sessions is required, use the emergency revocation endpoint (`POST /v1/admin/credential-pools/{name}/credentials/{credId}/revoke`) to force all active leases to re-acquire credentials with the updated material.

**GitOps pattern.** For teams using GitOps (ArgoCD/Flux) with external secret management:

- **Kubernetes Secrets** are managed by `external-secrets-operator` (ESO), `sealed-secrets`, or a similar controller that syncs secret material from an external vault (AWS Secrets Manager, HashiCorp Vault, GCP Secret Manager, etc.). These are standard Kubernetes resources managed through the GitOps pipeline.
- **Credential pool definitions** are managed via the `bootstrap.credentialPools` section in Helm values, committed to Git alongside other Helm values. The `lenny-bootstrap` Job applies them idempotently on every `helm upgrade` / ArgoCD sync.
- **Ordering:** ESO / sealed-secrets controllers must populate the Secrets **before** the Token Service starts resolving them. In ArgoCD, use sync-wave ordering (e.g., `ExternalSecret` at sync-wave `-3`, Lenny chart at sync-wave `0`). The Token Service tolerates temporarily missing Secrets (credentials are marked `unavailable` and re-resolved on the next informer event), so brief ordering races during initial deployment are self-healing.

#### 4.9.1 KMS Key Rotation Procedure

The Token Service uses envelope encryption (AES-256-GCM) for OAuth refresh tokens and user-supplied API keys stored in Postgres. The KMS-backed envelope key must be rotated periodically. This procedure applies to the Token Service's application-layer encryption key (distinct from etcd Secret encryption described in "Encryption at rest ŌĆö KMS key hierarchy" above).

1. **Rotation steps:**
   - Generate a new envelope key (DEK) in KMS; the old key remains active for decryption.
   - Deploy the Token Service with the new key ID as the default encryption key.
   - A background migration job re-encrypts all stored tokens: reads each token with the old DEK, re-encrypts with the new DEK, and updates the row atomically. The job is **idempotent** ŌĆö rows already at `current_version` are skipped, so the job can be safely re-run after a partial failure or restart with no risk of data corruption.
   - The `key_version` column on each encrypted token row tracks which DEK was used ŌĆö the Token Service can decrypt with any known key version.
   - **Verification before disabling old key:** Once all rows are re-encrypted, verify via `SELECT COUNT(*) FROM tokens WHERE key_version < current_version` returning 0 before proceeding.
   - Only after the verification query confirms zero remaining rows should the old key be disabled in KMS.
   - **Old key retention policy:** Disabled keys are retained in KMS for a minimum of 90 days after disabling (one full rotation cycle) before being scheduled for deletion, to allow disaster-recovery re-encryption if needed.
   - **Redis cache:** The encryption key used to protect cached access tokens in Redis ([Section 12.4](12_storage-architecture.md#124-redis-ha-and-failure-modes)) is derived from the envelope key. After rotating the envelope key, all Redis-cached tokens are invalidated ŌĆö the Token Service will re-derive tokens from Postgres using the new key on next access. No explicit Redis flush is required; cached entries simply fail decryption and are treated as cache misses.
2. **Rollback procedure:** If the re-encryption job fails and cannot be completed, rotation can be safely aborted by reverting the Token Service's default key ID to the old version (re-deploy or update the KMS key alias). Because the Token Service decrypts using the `key_version` stored per row, all rows (some re-encrypted with the new key, others still on the old key) remain readable after rollback. The partial migration is thus fully reversible. After rollback, investigate the failure cause before retrying the rotation procedure from the beginning.
3. **Frequency:** Recommended every 90 days, or immediately on suspected compromise.
4. **Monitoring:** Alert if re-encryption job hasn't completed within 24h of key rotation.
